## 1. 계산 모델 (개발팀 구현용 — 수치 확정 명세)

### 1-0. 입력 항목 정의 (디자인·개발 공통 확정)

| 입력 항목 | 타입 | 필수 | 기본값 | 비고 |
|---|---|---|---|---|
| 입사일(시작일) | date | 필수 | — | 재직을 시작한 첫날(첫 근무일) |
| 종료일(퇴사일 또는 기준일) | date | 선택 | **오늘** | 미입력 시 자동으로 "오늘" 날짜 사용(§1-1). 라벨 권장: "퇴사일(미입력 시 오늘)" |

- **입력 검증(확정)**: `종료일 ≥ 입사일`. 종료일 < 입사일이면 계산하지 않고 안내("종료일이 입사일보다 빠릅니다"). 미래 종료일(예: 퇴사 예정일)은 허용한다(예정 근속기간 조회 수요 존재). 저리스크 계산기이므로 하드 에러보다 안내 수준으로 충분.
- **입력 2개로 고정**: 단위변환에서 planning/design 목록 불일치 핑퐁이 있었던 전례에 따라, 입력은 위 2개(입사일·종료일)만이며 그 외 옵션 필드(연차 계산용 회계연도 기준 등)는 두지 않는다. 이유는 §2·§7-2.

### 1-1. 출력 항목 정의 (디자인·개발 공통 확정 — 결과 카드에 이 3개 표시)

| 출력 | 형식 | 예시(§1-3 앵커C 기준) |
|---|---|---|
| (a) 총 재직일수 | `N일` (천단위 콤마) | `1,993일` |
| (b) 근속기간 | `N년 M개월 D일` | `5년 5개월 14일` |
| (c) 근속연수(소수) | `N.N년` (소수 첫째 자리 반올림) | `약 5.5년` |

- (c)는 "대략 몇 년" 게이지 목적의 보조 표기다. 대표값(히어로)은 (b) `N년 M개월 D일` 권장(가장 직관적), (a)·(c)는 보조. 카드 레이아웃 최종 확정은 디자인팀 사안(§7-3).

### 1-2. 재직일수 계산 기준 — **양 끝 포함(+1) 채택** (핵심 확정)

**채택 기준: 재직일수는 입사일과 종료일을 모두 포함(양 끝 포함)해 센다.**

```
재직일수 = (종료일 − 입사일)의 일수차 + 1
```

- **근거**: 이 계산기 이름 자체가 "근무일수"다. 입사한 날도 하루 근무, 종료일 당일도 재직한 날이므로 양 끝을 포함하는 것이 사용자 직관에 맞다. 극단 검증: 같은 날 입·퇴사(입사일 = 종료일)면 **1일**이 나와야 정상인데, +1을 하지 않으면 0일이 되어 "하루도 근무 안 했다"는 오답이 된다(§1-3 앵커E). 만나이·D-Day 등 기존 date 계산기의 "직관적 일수" 톤과도 일치.
- **구현 권장(기존 date-diff 재사용, 두 출력의 일관성 확보)**: 내부적으로 `종료일⁺ = 종료일 + 1일`(= "재직 마지막 날의 다음 날")로 변환한 뒤, 만나이/D-Day에서 이미 쓰는 날짜차 로직을 그대로 `입사일 → 종료일⁺`에 적용한다. 그러면 재직일수와 근속기간(N년M개월D일)이 **하나의 내부 기준(입사일 → 종료일⁺)** 에서 파생되어 서로 어긋나지 않는다.

```
입력: 입사일 S, 종료일 E (E 미입력 시 E = 오늘)
검증: E ≥ S
종료일⁺ E⁺ = E + 1일

(a) 재직일수      = daysBetween(S, E⁺)          // = (E − S).days + 1, 양 끝 포함
(b) 근속기간 Y-M-D (S → E⁺, 만나이식 캘린더 분해):
      y = E⁺.year  − S.year
      m = E⁺.month − S.month
      d = E⁺.day   − S.day
      if (d < 0) { d += (E⁺ 직전 달의 일수); m -= 1; }   // 예: E⁺가 7월이면 6월의 일수(30)를 빌림
      if (m < 0) { m += 12; y -= 1; }
(c) 소수 근속연수 = round1( 재직일수 / 365 )      // 소수 첫째 자리 반올림
```

- **N년 M개월 D일 분해 로직 명시**: 위 알고리즘은 (1) 일(day) 자리가 음수면 **종료일⁺의 직전 달 실제 일수**를 빌려오고(그 달의 28/29/30/31일을 그대로 사용), (2) 월(month) 자리가 음수면 12를 더하고 연을 1 감소시킨다. **윤년(2월 29일)은 별도 분기 없이 자동 반영**된다 — 직전 달이 3월이면 빌려오는 값이 그 해 2월의 실제 일수(윤년 29, 평년 28)이고, 재직일수(a)도 실제 캘린더 일수차라 2월 29일을 포함한 구간은 자연히 1일 더 세진다.
- **소수 근속연수 divisor = 365 확정**: 퇴직금 실무(근로기준법 시행령상 평균임금 × 30일 × 재직일수 ÷ 365)와 정합을 맞추기 위해 **365로 나눈다**(365.25 아님). 소수 첫째 자리 반올림. 최종 확인은 §7-4.

### 1-2-1. 다른 기준을 쓰는 곳이 있을 수 있다 (주의문구 — 결과해석·FAQ에 반영)

- **퇴직금 실무(고용노동부) 차이**: 퇴직금 계산에서는 "마지막 근무일의 **다음 날**"을 퇴직일로 보아 `계속근로기간 = 퇴직일 − 입사일`로 계산하는 관행이 있다. 이 경우 사용자가 "마지막 근무일"을 종료일로 입력하면, 우리 재직일수(양 끝 포함)와 고용노동부식 계속근로기간이 **하루 정도 차이**날 수 있다. 이 점을 결과해석 §2 3문단·FAQ 3번에 명시했다.
- **따라서 이 계산기의 재직일수는 "일반적인 근무 일수 참고값"**이며, 실제 퇴직금 산정에 쓰이는 재직일수는 회사 규정·고용노동부 퇴직금 계산기(로드맵 #8) 기준을 따르도록 안내한다.

### 1-3. 검증용 기준값 (개발팀이 엔진 출력과 대조할 앵커) — 기획팀 직접 손계산

과거 CS 이슈 재발 방지를 위해, 결과해석·FAQ에 등장하는 예시를 아래 표로 고정한다. **앵커C(1,993일 / 5년 5개월 14일 / 5.5년)를 결과해석 대표 예시로 사용**한다. 개발팀은 엔진 출력이 이 표와 일치하는지 대조할 것.

| # | 입사일 | 종료일 | 재직일수(양 끝 포함) | N년 M개월 D일 | 소수 근속연수 | 확인 포인트 |
|---|---|---|---|---|---|---|
| A | 2020-01-01 | 2023-01-01 | **1,097일** | **3년 0개월 1일** | **3.0년** | 종료일이 입사 N주년 당일 → 하루 초과(양 끝 포함) |
| B | 2021-03-15 | 2024-03-14 | **1,096일** | **3년 0개월 0일** | **3.0년** | 종료일이 N주년 하루 전 → 딱 3년 완성, 윤년(2024-02-29) 포함 |
| C | 2020-03-02 | 2025-08-15 | **1,993일** | **5년 5개월 14일** | **5.5년** | 대표 예시(월·일 borrow + 윤년 2회) |
| D | 2022-08-20 | 2026-07-14(오늘) | **1,425일** | **3년 10개월 25일** | **3.9년** | 종료일 미입력=오늘 기준 케이스 |
| E | 2023-06-01 | 2023-06-01 | **1일** | **0년 0개월 1일** | **0.0년** | 같은 날 입·퇴사 → 1일(+1 필수 근거) |

**손계산 근거(개발 대조용):**

- **앵커 A** `2020-01-01 → 2023-01-01`: 종료일⁺ = 2023-01-02. 일수차 = (2020→2021 366) + (2021→2022 365) + (2022→2023 365) = 1,096, +1 = **1,097일**. Y-M-D: 2023-01-02 − 2020-01-01 = y3/m0/d1 → **3년 0개월 1일**. 1,097 ÷ 365 = 3.005 → **3.0년**. (해석: 3년 = 1,096일 + 1일 = 1,097일, 양 끝 포함의 정직한 결과.)
- **앵커 B** `2021-03-15 → 2024-03-14`: 종료일⁺ = 2024-03-15. 일수차 = (2021→2022 365)+(2022→2023 365)+(2023→2024 366, 2024-02-29 포함) = **1,096일**. Y-M-D: 2024-03-15 − 2021-03-15 = **3년 0개월 0일**. 1,096 ÷ 365 = 3.003 → **3.0년**. (해석: N주년 하루 전날을 마지막 근무일로 하면 딱 N년.)
- **앵커 C(대표)** `2020-03-02 → 2025-08-15`: 종료일⁺ = 2025-08-16. 일수차 = (2020-03-02→2025-03-02, 윤년 2020·2024 중 구간 내 2024-02-29 포함 = 1,826일) + (2025-03-02→2025-08-16 = 167일) = **1,993일**. 교차검증(ordinal): 2020-03-02(2020 윤년, 통산 62일)에서 2025-08-15(2025 평년, 통산 227일)까지 경과 1,992일 + 1 = 1,993일 ✓. Y-M-D: 2025-08-16 − 2020-03-02 = y5/m5/d14 → **5년 5개월 14일**(검산: 2020-03-02 +5년=2025-03-02, +5개월=2025-08-02, +14일=2025-08-16 = 종료일⁺ ✓). 1,993 ÷ 365 = 5.46 → **5.5년**.
- **앵커 D(오늘 기준)** `2022-08-20 → 2026-07-14`: 종료일⁺ = 2026-07-15. 일수차 = (2022-08-20→2026-08-20 = 1,461일) − (2026-07-15→2026-08-20 = 36일) = **1,425일**. Y-M-D: 2026-07-15 − 2022-08-20 → d = 15−20 = −5 < 0, 직전 달 6월(30일) 빌림 → d=25, m = 7−8−1 = −2 < 0, +12 → m=10, y = 4−1 = 3 → **3년 10개월 25일**(검산: 2022-08-20 +3년=2025-08-20, +10개월=2026-06-20, +25일=2026-07-15 ✓). 1,425 ÷ 365 = 3.90 → **3.9년**.
- **앵커 E(엣지)** `2023-06-01 → 2023-06-01`: 종료일⁺ = 2023-06-02. 일수차 = 1 → **1일**. Y-M-D: **0년 0개월 1일**. 1 ÷ 365 → **0.0년**. (해석: 하루라도 재직하면 1일. +1을 빼먹으면 0일이 되는 대표 반례.)

> 앵커 D의 "오늘"은 2026-07-14(문서 작성일) 기준 손계산이다. 실제 계산기의 "오늘"은 접속일마다 바뀌므로, 개발팀은 D를 "종료일 미입력 시 오늘 대입" 동작 확인용으로만 쓰고 날짜 자체는 고정 입력(A·B·C·E)으로 회귀 테스트할 것.

## 2. interpretation (결과 해석 텍스트, 5단 구조)

`lib/calculators.ts`의 `interpretation` 필드 초안. 1문단(결과 요약)은 §1-3 앵커C(입사 2020-03-02 / 퇴사 2025-08-15) 기준이며, 실제로는 **사용자 입력값에 따라 동적 생성**되어야 한다(기존 라이브 계산기와 동일 처리, §7 개발 전달).

```
입사일 2020년 3월 2일부터 퇴사일 2025년 8월 15일까지 근무한 경우, 총 재직일수는 1,993일이며 근속기간은 5년 5개월 14일, 소수로 환산하면 약 5.5년입니다.

이 근속연수 계산기는 입사일과 종료일(퇴사일 또는 오늘)을 입력하면 두 날짜 사이의 재직일수(근무일수)를 계산합니다. 재직일수는 입사일과 종료일을 모두 포함(양 끝 포함)해 세므로, 같은 날 입사·퇴사한 경우도 1일로 계산됩니다. 근속기간 'N년 M개월 D일'은 달력을 기준으로 분해하며 윤년(2월 29일)도 자동으로 반영되고, 소수점 근속연수는 재직일수를 365로 나눈 값입니다.

재직일수와 근속연수는 퇴직금·연차 산정의 기초가 되는 값입니다. 다만 회사나 기관마다 재직일수를 세는 방식이 조금씩 다를 수 있습니다. 특히 퇴직금 실무에서는 마지막 근무일의 다음 날을 '퇴직일'로 보아 계속근로기간을 계산하는 경우가 많아, 이 계산기의 재직일수와 하루 정도 차이가 날 수 있습니다. 정확한 퇴직금 산정 기준일은 회사 규정과 고용노동부 기준을 확인하시기 바랍니다.

근로기준법상 연차휴가는 1년 미만 근무 시 1개월 개근당 1일, 1년 이상 근무 시 15일이 부여되고, 3년 이상부터는 2년마다 1일씩 늘어 최대 25일까지 부여됩니다. 다만 실제 부여 일수는 회사의 회계연도 기준·입사일 기준 운영 방식과 근로계약에 따라 달라질 수 있어 이 규칙은 참고용입니다.

```

- 5단 구조 매핑(라인업 §3-1): ①결과요약 예시(1,993일 / 5년 5개월 14일 / 5.5년, 동적 생성) → ②계산기준(양 끝 포함, 윤년 자동 반영, /365) → ③실생활 팁·주의(퇴직금·연차 기초값 + 고용노동부식과 하루 차이) → ④관련 제도(연차 규칙 참고 + 회사별 상이 면책) → ⑤개인정보 처리(브라우저 내) + 면책.

## 4. 관련 계산기 링크 추천 (자동 로직과 정합 — 별도 override 불필요)

라인업 §3-3 원칙(같은 카테고리 2 + 다른 카테고리 1, 총 3개)을 **이번엔 완전 충족**한다.

- `getRelatedCalculators()`는 `status === "live"` 필터 + "같은 카테고리 우선 후 나머지, 배열 순서" 로직이다(코드 재확인 완료: `[...sameCategory, ...rest].slice(0,3)`).
- 이 계산기가 live 전환되면 같은 `date`의 **만 나이 계산기 → D-Day 계산기**가 2슬롯을 채우고, 나머지 1슬롯은 배열 순서상 다른 카테고리 첫 번째 = **연봉 실수령액 계산기**(`salary-net-calculator`, 배열에서 loan/bmi/unit보다 앞)로 자동 채워진다.
- 즉 자동 결과 = **만 나이 + D-Day + 연봉 실수령액**으로, 과제가 의도한 "date 2개 + salary 1개(퇴직금 동선)"와 **정확히 일치**한다. 지난 연봉·단위변환 기획안과 달리 이번엔 순서 강제·커스터마이즈가 **필요 없다**.
- **앵커 텍스트(문장형, 라인업 §3-3 원칙):**
  - 만 나이 계산기: "정확한 나이가 필요하다면 만 나이 계산기도 함께 이용해보세요."
  - D-Day 계산기: "특정 날짜까지 남은 기간이 궁금하다면 D-Day 계산기를 확인해보세요."
  - 연봉 실수령액 계산기: "근속연수를 확인했다면, 연봉 실수령액 계산기로 급여도 함께 점검해보세요."
- **주의(로드맵 #8 라이브 시 재배치 검토)**: 퇴직금 계산기(`salary` 카테고리)가 라이브되면, 이 계산기의 가장 자연스러운 연결은 퇴직금 계산기다. 다만 자동 로직은 "같은 카테고리(date) 우선"이라 퇴직금(salary)이 자동 3슬롯에 들어오려면 date live 계산기가 만나이·D-Day 2개뿐일 때만 salary 첫 항목이 3번째로 온다. 퇴직금이 연봉보다 배열 앞에 등록되면 3번째 슬롯이 퇴직금으로 바뀌므로 등록 순서만 확인하면 되고, 별도 기획 변경은 불필요하다(퇴직금 기획안 착수 시 재확인 항목으로만 남김).

## 7. 마스터 확인 필요 / 타팀 이관 (기획팀 단독 결정 범위 밖)

### 7-1. [마스터] 재직일수 기준 = "양 끝 포함(+1)" 채택 최종 승인
- **기획팀 채택안**: 재직일수 = 양 끝 포함(+1)(§1-2). 근거는 계산기 이름("근무일수")·사용자 직관·같은 날 입퇴사=1일 반례. 손계산 앵커 5개(§1-3)로 개발 검증 가능.
- **확인 요청**: 이 기준을 확정 승인해 주십시오. 만약 마스터가 "고용노동부 퇴직금식(계속근로기간, +1 없이 종료일 그대로 = 마지막 근무일 다음날 개념)"을 기본으로 원하면 앵커·해설 수치가 하루씩 바뀌므로 착수 전 회신 필요. (기획팀 권고: 이 계산기는 일반 유틸리티이므로 양 끝 포함 유지 + 퇴직금 차이는 해설/FAQ 고지로 처리가 최선.)

### 7-2. [마스터] 연차 정보 = "계산 출력 아님, 규칙 해설만" 채택 최종 승인
- **기획팀 채택안**: 사용자 날짜 기반 "예상 연차 = N일" 같은 **개인화 출력을 만들지 않는다**. 근로기준법 규칙(1년미만 월1일 / 1년이상 15일 / 3년이상 2년마다 +1, 최대 25일)만 결과해석 §2 ④·FAQ 4번에 **참고용 면책과 함께 텍스트로** 제공한다.
- **근거(YMYL 보수적 판단)**: 실제 연차는 회사가 회계연도 기준으로 운영하는지 입사일 기준인지, 근로계약·취업규칙에 따라 크게 달라진다. 개인화 연차 숫자를 계산기 출력으로 노출하면 상당수 사용자에게 오정보가 되어 CS·신뢰도 리스크가 크다. 반면 "연차" 키워드 검색 수요는 규칙 해설 텍스트만으로도 충분히 흡수 가능하다. → **출력=X, 해설=O**가 리스크 대비 효익 최적.
- **확인 요청**: 이 방침(연차는 해설·FAQ 텍스트 참고용, 계산 출력 아님)을 승인해 주십시오. 만약 마스터가 "예상 연차 일수도 출력하자"로 판단하면, 최소한 "입사일 기준·초년도 비례 미반영" 등 가정을 명시한 별도 면책 UI가 필요하므로 디자인·개발 추가 협의가 발생한다(권고: v1은 출력 제외).

### 7-3. [디자인] 결과 카드 레이아웃 — 대표값·보조값 위계
- 출력 3종(§1-1) 중 기획팀은 (b) `N년 M개월 D일`을 히어로 대표값, (a) 재직일수·(c) 소수년을 보조로 제안한다. 무스크롤 원칙 하에 3값 배치·강조 위계는 디자인팀 확정 사안. 종료일 미입력 시 "오늘 기준"임을 결과에 명시하는 표기(예: "2026년 7월 14일 오늘 기준")도 디자인 반영 요청.

### 7-4. [개발] 소수 근속연수 divisor = 365 확정
- §1-2대로 `재직일수 ÷ 365`(365.25 아님), 소수 첫째 자리 반올림. 퇴직금(#8) 산식과 정합 목적. 앵커 C(1,993÷365=5.46→5.5) 등과 대조.

### 7-5. [개발] 구현 주의 3건
1. **재직일수 양 끝 포함(+1)**: `종료일⁺ = 종료일 + 1일` 후 date-diff 재사용(§1-2). +1 누락 시 앵커 E가 1일이 아닌 0일로 나와 오답.
2. **N년M개월D일 borrow**: day 음수 시 **종료일⁺ 직전 달의 실제 일수**(윤년이면 2월 29)를 빌림. 30/31/28/29 고정값 하드코딩 금지, 실제 달 일수 사용.
3. **종료일 미입력=오늘 / 검증 `종료일 ≥ 입사일`**: 미래 종료일은 허용(예정 근속 조회), 종료일<입사일만 안내 처리.

### 7-6. [개발] 결과 요약 문구 동적 생성
- §2 1문단(앵커C 예시)은 정적 텍스트가 아니라 사용자 입력값(입사일·종료일·재직일수·N년M개월D일·소수년) 기반 동적 생성이어야 함(기존 라이브 계산기와 동일 처리).
