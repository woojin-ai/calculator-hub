## 2. 기존 BMI 2편과의 중복 차단 (전문 Read 완료)

### #9 `bmi-guide-2026` 이 이미 다룬 것
공식(kg ÷ m²) / 대한비만학회 4구간 + 비만 3단계 / WHO 기준 비교 / 근육량·마른비만 한계 / **경계값 반올림 함정(rawBmi 판정)** / 피트·파운드 단위 변환 안내 / 청소년·임산부 / 남녀 동일 기준.

### #15 `bmi-normal-weight-by-height-2026` 이 이미 다룬 것
정상 체중의 **역산**(키²×18.5 ~ 키²×23.0) / 키별 정상 체중 범위표 9행(150~190cm) / 정상은 "범위"라는 개념 / 상한 초과 = 과체중 / `키−100` 옛 표준체중 공식 비교 / 파운드·피트 단위 안내.

### 31편이 가져가는 **세 번째 각도** — "변화량(민감도)"
| 논점 | #9 | #15 | **31편** |
|---|---|---|---|
| BMI 값 자체 | ○ | — | — |
| 정상 몸무게 범위 | — | ○ | — |
| **1kg당 BMI 변화량** | — | — | **○ (신규)** |
| **BMI 1을 움직이는 데 필요한 kg** | — | — | **○ (신규)** |
| **키 입력 1cm 오차의 영향** | — | — | **○ (신규)** |
| **키가 클수록 둔감해지는 이유(1/h² 구조)** | — | — | **○ (신규)** |
| 경계값 반올림 | ○(판정 로직) | ○(표 약값 주의) | △(1문단, **재설명 금지·상호링크만**) |

**억지 여부 판단: 억지가 아니다.** 근거는 수학적으로 독립된 양을 다루기 때문이다. #9·#15는 모두 `BMI = W/h²`라는 **값(level)** 을 다루고, 31편은 그 **차분(Δ)** 을 다룬다. 특히 "1kg당 BMI 변화량은 몸무게 값과 무관하게 키만으로 정해진다(= 1/h²)"는 성질은 기존 두 편 어디에도 없고, 독자가 스스로 유추하기 어려우며, 실제 질문("60kg에서 1kg 빼는 거랑 90kg에서 1kg 빼는 거랑 BMI 차이가 다르지 않나?")에 정면으로 답한다.

### 중복 방지 강제 규칙 (개발/QA 공통)
1. **구간 경계표(18.5/23.0/25.0 4구간 나열) 재게재 금지.** 정상 구간 경계는 필요한 최소 1회만 언급하고, "구간 기준 전체는 #9 참고" 톤으로 처리.
2. **키별 정상 체중 범위(kg) 표 재게재 금지.** #15와 정면 중복. 31편 표는 **kg이 아니라 ΔBMI 단위**로만 구성한다.
3. **"몇 kg 빼면 정상 구간" 조견표 금지.** #15의 상한 표 + 산수면 도출되는 파생물이라 중복이고, 다이어트 처방 톤으로 미끄러진다.
4. `키−100` 표준체중, WHO 비교, 청소년·임산부 단락 **재작성 금지**(#9·#15 소유).

## 3. 프레임 정의 — **모든 숫자에 라벨 강제** (29편 실패 재발 방지)

이 글에는 서로 다른 3개의 계산 프레임이 공존한다. 값이 맞아도 프레임이 섞이면 **QA FAIL**.

| 프레임 ID | 이름 | 정의 | 의존 변수 | 단위 | 본문 표기 규칙 |
|---|---|---|---|---|---|
| **A** | 몸무게 프레임 | 몸무게 +1kg일 때 BMI 증가량 | **키만** (몸무게 무관) | BMI 포인트 | 문장에 반드시 `몸무게 1kg당` 포함. 몸무게 조건 **쓰지 말 것**(쓰면 오해 유발) |
| **A′** | 몸무게 역프레임 | BMI를 1.0 움직이는 데 필요한 몸무게 | **키만** | kg | 문장에 반드시 `BMI 1을 움직이려면` 포함 |
| **B** | 키 프레임 | 키 입력 ±1cm일 때 BMI 변화량 | **키 + 몸무게 둘 다** | BMI 포인트 | 문장에 반드시 `키 ○○cm·몸무게 ○○kg 기준` 명시. **몸무게 조건 없는 B 숫자는 금지** |

**섞임 금지 사례 (QA가 잡아야 할 것):**
- ✗ "170cm에서 1kg은 BMI 0.35, 1cm는 0.28" → B가 몸무게 조건 없이 나왔다 → **FAIL**
- ✓ "[A] 170cm라면 몸무게 1kg당 BMI 약 0.35. [B] 같은 170cm라도 키를 1cm 잘못 넣으면, 몸무게 70kg 기준으로 BMI가 약 0.28 움직인다."
- ✗ A 표와 A′ 값을 한 리스트 안에 섞어 배치 → **FAIL**. A 표(§5 list)와 A′(§5 paragraph)는 반드시 다른 섹션 블록으로 분리.

## 4. 본문에 들어갈 숫자와 산출 방법 (손계산 금지)

### 4-0. 공통 규칙
- **엔진:** `C:\claude code webside\lib\bmi.ts` → `calculateBmi(heightCm: number, weightKg: number): BmiResult | null`
- 사용 필드는 **`rawBmi`**(반올림 전). `bmi` 필드(소수 둘째 반올림)는 **중간 계산에 쓰지 말 것**.
- **중간 반올림 전면 금지.** 차분은 `rawBmi − rawBmi`로 구하고, **최종 표기 직전에 한 번만** 반올림한다.
- 표기 자리수: ΔBMI는 **소수 둘째 자리**, kg은 **소수 둘째 자리**.
- 파운드 상수는 `C:\claude code webside\lib\units.ts`의 `WEIGHT_UNITS` 중 `{ id: "lb", factor: 453.59237 }` (g 기준) → 1lb = 0.45359237kg. **다른 값(453.59 등) 사용 금지.**

### 4-1. 프레임 A — 키별 "몸무게 1kg당 BMI 변화량"

산출식(개발팀이 그대로 실행):
```
deltaA(h) = calculateBmi(h, 71).rawBmi - calculateBmi(h, 70).rawBmi
표기값 = Math.round(deltaA(h) * 100) / 100
```
> 기준 몸무게 70kg은 임의값이며, 이 차분은 몸무게에 의존하지 않는다(수학적으로 `1/(h/100)²`). QA는 70/71 대신 **55/56, 90/91로도 동일한 표기값이 나오는지** 교차 확인할 것(§7-C).

| 키 | 표기값(ΔBMI, 예상) |
|---|---|
| 150cm | 0.44 |
| 155cm | 0.42 |
| 160cm | 0.39 |
| 165cm | 0.37 |
| 170cm | 0.35 |
| 175cm | 0.33 |
| 180cm | 0.31 |
| 185cm | 0.29 |
| 190cm | 0.28 |

> 위 표기값은 기획팀이 공식으로 예측한 **참고값**이며, **개발팀은 반드시 `calculateBmi`를 실제로 실행해 재생성**하고, QA는 §7-A로 전수 대조한다. 불일치 시 엔진 출력이 정답.

### 4-2. 프레임 A′ — "BMI 1을 움직이는 데 필요한 몸무게"

산출식:
```
kgPerBmi1(h) = 1 / deltaA(h)          // deltaA는 반올림 전 원값 사용
표기값 = Math.round(kgPerBmi1(h) * 100) / 100
```
본문에 쓸 것은 **3개 키만**(표 아님, paragraph): 150cm **2.25kg** / 170cm **2.89kg** / 190cm **3.61kg** (예상값, 재생성 필수).
→ 카피 포인트: "이 값이 키(m)의 제곱과 똑같다"는 점은 우연이 아니라 공식 구조 때문 — 한 문장으로만 짚고 수식 유도는 하지 않는다.

### 4-3. 프레임 B — 키 입력 ±1cm의 영향 (몸무게 70kg 기준)

산출식:
```
base   = calculateBmi(170, 70).rawBmi
plus1  = calculateBmi(171, 70).rawBmi
minus1 = calculateBmi(169, 70).rawBmi
표기: Math.round(base*100)/100, Math.round((base-plus1)*100)/100, Math.round((minus1-base)*100)/100
환산kg = (base - plus1) / deltaA(170)   // 반올림 전 원값끼리, 마지막에만 반올림
```
예상값(재생성 필수):
- `[B] 170cm·70kg`: BMI 표시 **24.22**
- `[B] 171cm·70kg`로 1cm 크게 넣으면 BMI가 약 **0.28** 내려간다
- `[B] 169cm·70kg`로 1cm 작게 넣으면 BMI가 약 **0.29** 올라간다 → **위아래가 대칭이 아니다**(이 비대칭 자체가 이 문단의 콘텐츠)
- 환산: 키 1cm 오차 ≈ 몸무게 약 **0.82kg** 오차와 맞먹음 **(170cm·70kg 기준)**

### 4-4. 파운드 문단 (unit-converter 연결)

산출식:
```
deltaLb(170) = calculateBmi(170, 70 + 453.59237/1000).rawBmi - calculateBmi(170, 70).rawBmi
```
예상 표기값: **0.16** — `[A-파운드] 170cm 기준, 몸무게 1파운드(약 0.45kg)당 BMI 약 0.16`.
(프레임 A의 파생이므로 라벨은 A 계열로 유지하고, "1lb = 0.45359237kg" 상수 출처를 `lib/units.ts`로 못 박는다.)

### 4-5. 하루 체중 변동 예시
`[A] 170cm 기준 몸무게 1kg당 약 0.35`를 그대로 재사용해, "체중이 하루 사이 1kg 오르내리면 BMI는 약 0.35 움직인다"로만 쓴다. **새 숫자를 만들지 않는다.** "하루 1~2kg 변동은 흔하다" 같은 일반화·의학적 단정은 금지, "식사·수분 섭취에 따라 오르내릴 수 있는데, 그 폭이 1kg이라면" 조건문으로 서술.

## 8. 마스터 팀장 확인·타 팀 인계 사항

1. **[정정 보고]** dsr-calculator는 in-degree 6으로 최약체가 아님. 이후 회차의 축2 판단은 이 문서 §0-2 표를 기준으로 갱신 요망.
2. **[방향 확인 요청]** BMI 클러스터는 31편으로 3편이 된다. 계산기 1개에 글 3편이면 `getBlogPostsByCalculator`의 3편 캡을 정확히 채우므로 **네 번째 BMI 글은 노출 이득이 사실상 0**이고 thin content 리스크만 남는다. **BMI는 31편을 끝으로 종료**할 것을 제안하며, 마스터 승인 요청.
3. **[32편 방향 선제안]** 31편 반영 후 분포는 salary 8 · loan 8 · **date 7** · life 8 → 축1 최소는 date 단독. 축2에서 date 계열 최약체는 **age-calculator(4)**. 32편은 **date + age-calculator** 조합을 권고(단 기존 date 글 7편 중 age 관련 #4·#27과의 중복 검사 선행 필요). 대안으로 축2 최약체 **weekly-holiday-allowance-calculator(2, salary)**가 남아 있으나 salary가 최다 카테고리라 축1과 충돌 → 마스터가 축 우선순위를 정해주면 좋겠음.
4. **[개발팀]** §4의 모든 숫자는 **기획팀 예상값이며 확정값이 아니다.** `lib/bmi.ts`의 `calculateBmi`를 실제로 실행해 재생성한 값을 본문에 넣고, 불일치가 나오면 기획팀에 회신 바람. 특히 §7-B의 "2.86 함정"(중간 반올림) 주의.
5. **[디자인팀]** §5-5의 9행 list는 `170cm: 몸무게 1kg당 BMI 약 0.35` 형태의 긴 문자열이라 모바일 375px에서 줄바꿈이 발생한다. 기존 #15의 9행 표와 동일 렌더 패턴이므로 별도 컴포넌트는 불필요하나, 줄바꿈 시 가독성 확인 요망.
6. **[QA팀]** §7의 A~I를 그대로 집행. A·B·D·E는 `calculateBmi` 실행 결과와의 기계적 대조이므로 육안 검수 대신 실제 실행 권장.
