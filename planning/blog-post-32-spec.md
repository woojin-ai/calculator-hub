### 0-4. 계산 엔진 실측 — `lib/age.ts` `calculateManAge(birthDateISO, referenceDate)`
반환 필드 3개: `manAge`(만 나이), `yearAge`(연 나이 = 기준연도−출생연도), **`daysToNextBirthday`(다음 생일까지 남은 일수, 오늘이 생일이면 0)**.
핵심 로직(본문 숫자의 근거):
```
manAge = ref.getFullYear() - birth.getFullYear();
hasHadBirthdayThisYear =
  ref.getMonth() > birth.getMonth()
  || (ref.getMonth() === birth.getMonth() && ref.getDate() >= birth.getDate());
if (!hasHadBirthdayThisYear) manAge -= 1;   // ← 생일 '당일'은 >= 이므로 '지난' 쪽에 포함
```
→ **엔진 확인 결론:** 만 나이는 **생일 당일 0시부터 이미 +1** 된다(그 다음 날이 아니다). `daysToNextBirthday`는 오늘이 생일이면 0. 이 두 사실이 32편의 콘텐츠 축이다. **§8에 엔진 이슈 1건(윤일 2/29) 별도 보고.**

## 2. 중복 차단 (기존 2 age 글 + 형제 date 글 전문 대조)

### 2-1. 32편이 가져가는 **세 번째 각도** — "경계/타이밍(Δ)"

| 논점 | #4(개념) | #27(값) | **32편** |
|---|---|---|---|
| 만/연/세는나이가 뭔지(정의) | ○ | △(규칙만) | — (정의 재설명 금지, #4로 회유) |
| 만 나이 통일법 배경·시행일 | ○ | △ | — (금지) |
| 몇 년생=몇 살 조견표 | — | ○ | — (재게재 금지) |
| **만 나이가 '생일 당일'에 오르나 다음 날 오르나** | — | — | **○ (신규)** |
| **생일 하루 전/당일/다음 날 만 나이 갈림** | — | — | **○ (신규)** |
| **다음 생일까지 남은 일수(daysToNextBirthday)** | 계산기 티저 1줄 | 계산기 티저 1줄 | **○ (신규 본문화)** |
| **월말·윤일(2/29) 출생 경계** | 2/29 처리 1문단 | — | **○ (재구성·§2-3 주의)** |

**억지 아님 근거:** #4·#27은 만 나이라는 **값(level)** 을 정의하거나 나열한다. 32편은 그 값이 **바뀌는 시점(when)** 과 **거기까지의 잔여 일수(Δ)** 를 다룬다 — 수학적으로 level과 그 변화 시점은 독립 정보다. 실제 질문("오늘 생일인데 계산기 만 나이가 왜 벌써 올랐지?", "생일까지 며칠 남았지?")에 정면으로 답한다.

### 2-2. 중복 방지 강제 규칙 (개발/QA 공통)
1. **출생연도별 조견표(연/만/세는 3열 나열) 재게재 금지.** #27 정면 중복. 32편 예시는 **특정 날짜(2000-07-20/21/22 등) 경계 케이스**이지 출생연도 룩업표가 아니다.
2. **"세는나이 = 연 나이 + 1" 나열 금지.** #27 소유. 32편은 세는나이를 "1월 1일에 다 함께 오른다"는 **대비용 1회 언급**만 하고 값 계산은 하지 않는다.
3. **만 나이 통일법 배경·시행일·법 내용 재작성 금지.** #4 소유 → 개념·법은 #4 크로스링크로만.
4. **만/연/세는나이 '정의' 문단 재작성 금지.** 32편은 정의 없이 라벨로 **사용**만 하고, "뜻은 #4 참고"로 넘긴다.
5. **#4의 2/29 문장("평년에는 자동으로 2월 28일을 생일이 지난 기준으로 처리")을 복붙·재주장 금지.** §2-3 참조 — 32편은 2/29를 '제도·서비스마다 다룰 수 있는 사안'으로 서술하고 특정 동작을 단정하지 않는다.
6. **D-Day 세는 법(D-1/D+N/오늘 포함 여부) 재설명 금지.** #24 소유 → "세는 법 자체는 #24" 크로스링크. 32편은 남은 일수를 **결과값으로만** 사용.
7. #4·#27·#24 문장을 30자 이상 연속 복사 금지(warning callout 포함). 단 공공기관 고유명사(행정안전부·국민연금공단 등)는 carve-out(§7-G, 31편 선례).

### 2-3. ⚠️ 2/29(윤일) 취급 — 정확성 경고 (dev/QA 필독)
`lib/age.ts` 실측 결과, **윤일 출생자의 평년 처리에서 엔진 두 출력이 서로 어긋난다**(§8-1 핑퐁 이슈로 정식 보고):
- `daysToNextBirthday`는 `safeDateForYear`로 평년 생일을 **2월 28일로 클램프** → 평년 2월 28일에 **0** 반환.
- `manAge`는 `ref.getDate() >= birth.getDate()`(28 >= 29 = false) 때문에 2월 28일에는 **아직 +1 안 됨**, **3월 1일에 +1**.
- 즉 평년 2월 28일에 계산기는 "다음 생일까지 0일"(= 오늘이 생일)이라면서도 만 나이는 아직 안 오른 모순 상태가 될 수 있다.

**따라서 32편 본문은 2/29에 대해 특정 날짜(28일이다/3월 1일이다)를 단정하지 않는다.** "평년엔 2월 29일이 없어 제도·서비스마다 2월 28일 또는 3월 1일로 달리 볼 수 있다 → 정확한 적용은 행정안전부 등 관계기관·해당 서비스 확인"으로 **YMYL 면책 톤**으로만 처리한다. 엔진이 수정·확정되기 전까지 2/29 숫자를 본문에 넣지 않는다.

## 3. 프레임 정의 — **모든 숫자에 라벨 강제** (29편 실패 재발 방지)

이 글에는 성격이 다른 4개 값이 공존한다. 값이 맞아도 라벨이 없거나 섞이면 **QA FAIL**.

| 프레임 ID | 이름 | 정의 | 성질 | 본문 표기 규칙 |
|---|---|---|---|---|
| **만** | 만 나이 | `manAge` | 생일 당일에 +1, 사람마다 다른 날 | 숫자 앞뒤에 반드시 `만 ○○세`. 그냥 "○○세" 금지 |
| **연** | 연 나이 | `yearAge = 2026 − 출생연도` | 한 해 내내 고정, 생일 무관 | 반드시 `연 나이 ○○세`. 만 나이와 같은 값이어도 라벨 분리 |
| **세는** | 세는나이 | (대비용) 1월 1일 일제 상승 | 값 계산 안 함 | 값 나열 금지. "1월 1일에 함께 오른다"는 성질만, 정의는 #4 |
| **D** | 남은 일수 | `daysToNextBirthday` | 보는 날 기준, 하루 지나면 −1 | 반드시 `다음 생일까지 ○○일`. 나이 숫자 옆에 라벨 없이 두지 말 것 |

**섞임 금지 사례 (QA가 잡아야 할 것):**
- ✗ "2000년생은 26세, 364일" → 어느 나이인지(만/연) 불명 + 364 라벨 없음 → **FAIL**
- ✓ "[2000-07-20생] **만 26세** · **연 나이 26세** · **다음 생일까지 364일**"
- ✗ 만 나이와 연 나이가 같은 값(둘 다 26)이라고 "26세"로 합쳐 쓰기 → **FAIL** (경계에서 갈리는 게 콘텐츠이므로 항상 분리)
- ✗ 남은 일수 숫자를 D-Day 표기(D-364 등)로 바꿔 쓰기 → #24 세는 법과 혼입 → **FAIL** (본문은 "○○일"만)

## 4. 본문 숫자와 산출식 (손계산 금지 · dev가 엔진 실행해 재생성)

### 4-0. 공통 규칙
- **엔진:** `C:\claude code webside\lib\age.ts` → `calculateManAge(birthDateISO: string, referenceDate: Date)`.
- **기준일 고정:** `referenceDate = new Date(2026, 6, 21)` (2026-07-21, 로컬 자정). 본문의 모든 예시는 이 기준일 산출값.
- 본문에 쓰는 값은 **`manAge`·`yearAge`·`daysToNextBirthday` 세 필드뿐**. 그 밖의 파생 계산 금지.
- 2026은 **평년**(윤년 아님). 2/29 예시는 §2-3에 따라 **숫자 미기재**.

### 4-1. 핵심 경계 트리오 (§5-8 list) — 2000년생, 생일 7/20·7/21·7/22
```
calculateManAge("2000-07-20", new Date(2026,6,21))  // 어제가 생일
calculateManAge("2000-07-21", new Date(2026,6,21))  // 오늘이 생일
calculateManAge("2000-07-22", new Date(2026,6,21))  // 내일이 생일
```
| 생년월일 | 상황 | manAge(만) | yearAge(연) | daysToNextBirthday(D) |
|---|---|---|---|---|
| 2000-07-20 | 어제 생일 | 만 26세 | 연 26세 | 364일 |
| 2000-07-21 | 오늘 생일 | 만 26세 | 연 26세 | 0일(오늘) |
| 2000-07-22 | 내일 생일 | 만 25세 | 연 26세 | 1일 |

**콘텐츠 포인트:** 출생연도가 셋 다 2000년으로 같아 **연 나이는 26세로 고정**인데, 하루 차이로 **만 나이만 25↔26으로 갈린다.** 오늘이 생일이면 남은 일수 0이고 **그 순간 이미 만 26세**(다음 날 아님). → 프레임 라벨 교훈을 그대로 보여주는 표.
> 위 값은 기획팀 예측. **dev는 반드시 엔진 실행해 재생성**, QA는 §7-A 전수 대조. 특히 2000-07-20의 364(365 아님)와 2000-07-21의 0을 정확히.

### 4-2. 연말 생일 예시 (§5-14) — 연 나이와 만 나이 간극
```
calculateManAge("1990-12-25", new Date(2026,6,21))
```
예상: **만 35세 · 연 나이 36세 · 다음 생일까지 157일**.
**콘텐츠 포인트:** 생일이 늦은 사람은 한 해의 대부분을 "연 나이보다 만 나이가 한 살 어린" 상태로 보낸다. (재생성 필수)

### 4-3. 넣지 않는 것
- 출생연도별 조견표(#27), 세는나이 값 나열(#27), 2/29 특정 숫자(§2-3), 만 나이 통일법 시행일·조문(#4), D-Day 표기 변환(#24).

## 7. 검증 체크리스트 (QA 집행용)

### A. 숫자 정합성 — 경계 트리오 (필수, 3행 전수)
```
calculateManAge("2000-07-20", new Date(2026,6,21)) → { manAge:26, yearAge:26, daysToNextBirthday:364 }
calculateManAge("2000-07-21", new Date(2026,6,21)) → { manAge:26, yearAge:26, daysToNextBirthday:0 }
calculateManAge("2000-07-22", new Date(2026,6,21)) → { manAge:25, yearAge:26, daysToNextBirthday:1 }
```
본문 §5-8 list와 완전 일치. 특히 **07-20의 364(≠365)** 와 **07-21의 0** 이 핵심 — 어긋나면 FAIL.

### B. 숫자 정합성 — 연말 생일 예시
```
calculateManAge("1990-12-25", new Date(2026,6,21)) → { manAge:35, yearAge:36, daysToNextBirthday:157 }
```
본문 §5-14와 일치. manAge(35)와 yearAge(36)가 **다른 값**인지 확인(같으면 §5-14 논지 붕괴 → FAIL).

### C. 프레임 라벨 강제 (29편 실패 재발 방지)
1. 나이 숫자가 나오는 **모든 문장**에 `만` 또는 `연 나이` 라벨이 붙어 있는가? 라벨 없는 "○○세" 검출 시 FAIL.
2. 만 나이와 연 나이가 같은 값(26/26)인 곳에서 "26세"로 합쳐 쓰지 않았는가? 합치면 FAIL.
3. 남은 일수 숫자에 `다음 생일까지 ○○일` 라벨이 붙어 있는가? D-Day 표기(D-364 등)로 바뀌어 있으면 FAIL.
4. §5-8 list 3행이 각각 만·연·D 세 값을 라벨과 함께 담고 있는가?

### D. 중복 차단 (기존 2 age + date 형제)
1. 출생연도별 조견표(연/만/세는 3열 나열)가 있는가? → 있으면 #27 중복 FAIL.
2. "세는나이 = 연 나이 + 1" 값 나열이 있는가? → 있으면 #27 중복 FAIL.
3. 만/연/세는나이 **정의 문단**, 만 나이 통일법 시행일·배경이 재작성돼 있는가? → 있으면 #4 중복 FAIL.
4. D-Day 세는 법(D-1/D+N/오늘 포함 여부) 재설명이 있는가? → 있으면 #24 중복 FAIL.
5. #4·#27·#24 문장 30자 이상 연속 일치? → FAIL. **[carve-out]** 공공기관 고유명사(행정안전부·국민연금공단 등)는 §7-G 규칙대로 제외, 앞뒤 산문이 재작성됐는지로만 판정.
6. #4의 2/29 문장("평년에는 자동으로 2월 28일을 생일이 지난 기준으로 처리")이 복붙·재주장돼 있는가? → 있으면 FAIL(§2-3).

### E. 2/29(윤일) 비단정 검사
1. 본문에 윤일 출생의 **특정 날짜 단정**("2월 28일에 오른다"/"3월 1일에 오른다"/구체 만 나이 숫자)이 있는가? → 있으면 FAIL(§2-3).

### F. YMYL·톤
1. 연금 수급개시일·병역·정년 **구체 도달일/나이 숫자**가 본문에 있는가? → 있으면 高YMYL 상승, FAIL.

## 8. 마스터 팀장 확인·타 팀 인계 (핑퐁)

1. **[개발팀·핑퐁 이슈 — 엔진 불일치]** `lib/age.ts`에서 **윤일(2/29) 출생자의 평년 처리**가 두 출력 간 어긋난다(§2-3):
   - `daysToNextBirthday`는 평년 2월 28일에 **0**(생일로 클램프)인데, `manAge`는 같은 2월 28일에 **아직 안 오르고 3월 1일에 +1** 된다.
   - 결과: 평년 2월 28일에 "다음 생일까지 0일"인데 만 나이는 그대로인 모순 표시 가능.
   - 또한 #4 본문("평년에는 자동으로 2월 28일을 생일이 지난 기준으로 처리")은 **manAge 실제 동작(3월 1일 상승)과 어긋난다** → #4 문장이 부정확할 수 있음.
   - **요청:** dev가 (a) 두 출력을 한 기준(28일 또는 3월 1일)으로 일치시킬지, (b) UI에 윤일 예외 안내를 둘지 결정. 결정 전까지 32편은 2/29 숫자를 넣지 않고 면책 톤으로만 처리. #4 문장 정정 여부도 함께 판단 요청.
