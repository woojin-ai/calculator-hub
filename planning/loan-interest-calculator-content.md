## 1. 상환방식 계산 공식 (개발팀 구현용 — 정확한 스펙)

두 상환방식 모두 **고정금리, 매월 1회 상환**을 전제로 한다(변동금리·거치기간·2주 상환 등은 v1 범위 밖 — 6-2 참고).

공통 변수:
- `P` = 대출원금(원)
- `r` = 월 이자율 = 연이자율(%) ÷ 12 ÷ 100 (예: 연 5.0% → r = 0.05/12 ≈ 0.00416667)
- `n` = 총 상환개월수

### 1-1. 원리금균등상환 (매월 상환액 고정)

| 항목 | 공식 |
|---|---|
| 매월 상환액 M | `M = P × r × (1+r)^n ÷ {(1+r)^n − 1}` |
| 회차 k의 이자분 | `직전 잔액 × r` |
| 회차 k의 원금분 | `M − 이자분` |
| 잔액 갱신 | `직전 잔액 − 원금분` |
| 총 이자 | `M × n − P` |

- **r = 0(무이자 대출) 예외 처리 필수**: 위 공식은 분모가 0이 되어 계산 불가. 이 경우 `M = P / n`으로 별도 분기 처리.
- 매월 상환액이 고정되므로 사용자 입장에서 "매달 얼마씩 나가는지"가 가장 직관적인 방식.

### 1-2. 원금균등상환 (매월 원금 고정 + 이자 감소)

| 항목 | 공식 |
|---|---|
| 매월 원금분 | `P ÷ n` (전 회차 고정) |
| 회차 k의 잔액(상환 전) | `P × (n − k + 1) ÷ n` |
| 회차 k의 이자분 | `잔액(상환 전) × r` |
| 회차 k의 상환액 | `원금분 + 이자분` (1회차 최대, n회차로 갈수록 감소) |
| 총 이자 (닫힌 형태) | `r × P × (n+1) ÷ 2` |

- **원 단위 반올림/절사 시 마지막 회차 보정 필요**: `P ÷ n`이 정수로 나누어떨어지지 않는 경우(예: 3,000만원 ÷ 36개월 = 833,333.33...원), 1~(n−1)회차는 절사/반올림한 고정값을, 마지막 회차는 "잔여 원금 전액"으로 처리해 합계가 정확히 `P`와 일치하도록 구현할 것을 권장. 그렇지 않으면 총 상환 원금이 대출원금과 몇 원 단위로 어긋날 수 있음.
- 초반 상환 부담이 크고 후반으로 갈수록 상환액이 줄어드는 구조.

### 1-3. 계산 예시 (인터프리테이션·FAQ 작성에 사용한 검증값, 개발팀 QA 참고용)

조건: 대출원금 3,000만원, 연이자율 5.0%, 대출기간 36개월(r ≈ 0.00416667, n=36, (1+r)^36 ≈ 1.161472)

| 구분 | 원리금균등상환 | 원금균등상환 |
|---|---|---|
| 1회차 상환액 | 약 899,130원 (매회 동일) | 약 958,333원 (원금 833,333원 + 이자 125,000원) |
| 마지막 회차 상환액 | 약 899,130원 (매회 동일) | 약 836,806원 (원금 833,345원 + 이자 약 3,472원, 마지막 회차 원금 보정 포함) |
| 총 이자 | 약 2,368,790원 | 약 2,312,500원 |
| 총 상환액 | 약 32,368,790원 | 약 32,312,500원 |

- 위 수치는 수기 검산 값이며 **"약"으로 표기한 근사치**다. 실제 구현 시 반올림 정책(원 단위 절사/반올림, 마지막 회차 보정 방식)에 따라 1~수십 원 단위로 달라질 수 있으므로, 개발팀은 이 표를 "공식이 맞는지 확인하는 대략적 검증용"으로만 쓰고 자체 계산 결과를 최종 기준으로 삼을 것.
- 이 예시가 보여주듯 **동일 조건에서 원금균등상환의 총 이자가 원리금균등상환보다 적다**(원금을 더 빨리 갚아 이자 붙는 잔액이 빨리 줄기 때문). 이 사실은 아래 2번 해석 텍스트·3번 FAQ에서 안내한다.

## 2. interpretation (결과 해석 텍스트, 5단 구조)

`lib/calculators.ts`의 `interpretation` 필드에 사용할 초안. 1문단(결과 요약)은 위 1-3 예시(3,000만원/연 5.0%/36개월/원리금균등상환 선택 시) 기준이며, 실제로는 사용자 입력값과 **선택한 상환방식에 따라 동적으로 생성**되어야 한다(age-calculator/dday-calculator/bmi-calculator와 동일한 처리 방식).

```
대출원금 3,000만원, 연이자율 5.0%, 대출기간 36개월, 원리금균등상환 기준으로 매월 상환액은 약 899,130원이며, 총 이자는 약 2,368,790원, 총 상환액은 약 32,368,790원입니다.

원리금균등상환은 매달 갚는 총 금액(원금+이자)이 대출 기간 내내 동일하도록 설계된 방식으로, 월 상환액은 대출원금 × 월이자율 × (1+월이자율)의 상환개월수 제곱 ÷ {(1+월이자율)의 상환개월수 제곱 − 1} 공식으로 계산됩니다. 원금균등상환은 매달 갚는 원금을 대출원금 ÷ 상환개월수로 고정하고, 이자는 매달 줄어드는 잔여원금에 월이자율을 곱해 산정하므로 초반 상환액이 크고 후반으로 갈수록 상환액이 줄어듭니다.

같은 조건이라면 원금균등상환의 총 이자가 원리금균등상환보다 대체로 더 적은데, 원금을 더 빨리 갚아 이자가 붙는 잔여원금이 더 빠르게 줄어들기 때문입니다. 다만 원금균등상환은 초기 상환 부담이 커서, 매달 소득이 일정한 경우라면 상환액이 균일한 원리금균등상환을, 초기 여유자금이 있고 총이자 절감을 우선한다면 원금균등상환을 선택하는 경우가 많습니다. 이 계산기는 대출 기간 내내 고정금리가 유지된다고 가정하므로, 변동금리 대출이라면 시장금리 변동에 따라 실제 상환액이 달라질 수 있습니다.

정확한 대출 금리·한도, 중도상환수수료 등 상품별 세부 조건은 금융감독원 금융상품 통합비교공시(파인, fine.fss.or.kr)나 각 금융기관의 대출 상품 안내에서 확인하실 수 있습니다.

본 계산 결과는 대출 상담이나 자문이 아닌 단순 참고용 수치이며 법적·재무적 효력이 없습니다. 실제 대출 조건과 상환 계획은 반드시 금융기관 또는 전문가와 상담 후 결정하시기 바랍니다.
```

- 글자 수: 약 830자 내외(공백 포함, 500~1000자 범위 충족).
- 5단 구조 매핑: ①결과요약(동적 생성 예시) → ②계산기준/공식 설명(두 상환방식 공식 요약) → ③실생활 팁/주의사항(방식 선택 기준, 고정금리 가정 한계) → ④관련 제도/출처(금융감독원 파인) → ⑤면책 문구(대출 상담/자문 아님 명시).
- **"지금 대출 신청", "OO은행 특가 금리" 등 오인 유발 문구는 사용하지 않았다.** 결과 요약도 담담한 사실 나열(수치)로만 구성했다.

## 4. UI 입력 항목 제안 (참고용 — 구조 확정은 6번 참고)

| 항목 | 타입 | 필수 여부 | 비고 |
|---|---|---|---|
| 대출원금 | number (원, 천단위 콤마) | 필수 | placeholder 예: `30,000,000` |
| 연이자율 | number (%, 소수 1~2자리 허용) | 필수 | placeholder 예: `5.0` |
| 대출기간 | number (개월) | 필수 | 개월 단위 직접 입력 권장(년 단위 UI는 디자인팀 판단). 예: `36` |
| 상환방식 | 라디오 버튼/토글 2종 | 필수 | "원리금균등상환" / "원금균등상환" — 구조는 6-1 참고 |

- 거치기간(원금 상환 유예기간) 입력은 v1 범위에서 **제외 제안**(6-2 참고, 계산 복잡도·오정보 리스크 대비 실익 낮음).
- 대출기간 상한/하한 등 입력 검증 범위는 개발팀 판단 영역이나, 참고로 일반적인 신용대출(최대 60~120개월)·주택담보대출(최대 360~480개월)을 감안하면 최소 1개월 ~ 최대 480개월 정도의 넉넉한 범위를 권장.

## 6. 마스터 팀장 확인 필요 사항 (기획팀 단독 결정 범위 밖)

### 6-1. 상환방식(원리금균등/원금균등) UI 구조 — 구조적 결정

**쟁점**: 두 상환방식을 (a) 하나의 계산기 페이지 안에서 라디오 버튼/토글로 전환하는 방식으로 할지, (b) 완전히 별도의 계산기(별도 slug·URL·카드)로 분리할지.

**기획팀 추천(확정 아님)**: (a) 단일 계산기 내 토글 방식.
- 근거 1: `lib/calculators.ts`에 이미 `loan-interest-calculator` slug 1개만 등록되어 있고, `calculator-lineup.md` 46~50행도 "대출이자 계산기 (원리금균등/원금균등상환)"을 하나의 항목으로 기획했다 — 별도 계산기로 분리하면 이미 확정된 slug/로드맵 구조를 바꾸는 셈이 된다.
- 근거 2: 두 방식은 입력값(원금/금리/기간)이 완전히 동일하고 결과만 다르므로, 사용자가 같은 입력을 유지한 채 방식만 바꿔가며 즉시 비교하는 것이 UX상 자연스럽다(재입력 없이 비교 가능).
- 근거 3: 위 2번 interpretation·3번 FAQ 3번 문항이 "두 방식을 비교"하는 내용을 전제로 작성되어 있어, 토글 구조와 콘텐츠가 자연스럽게 맞물린다.

**그럼에도 마스터 확인이 필요한 이유**: 이는 사용자가 안내에서 명시적으로 예시로 든 "구조적 결정" 사안이며, 실제 UI 컴포넌트 설계(디자인팀), 계산 결과 카드 레이아웃, 페이지 SEO 타이틀(두 방식을 한 페이지에서 다루면 타이틀/메타 설명에 두 키워드를 모두 넣을지 여부)에도 영향을 준다. **디자인팀 UI 스펙 작성 전에 마스터 팀장이 (a)/(b) 중 하나로 확정해 디자인팀에 전달해줄 것을 요청한다.**

### 6-2. 두 방식 결과 동시 비교 표시 여부

6-1에서 (a)를 채택한다고 가정할 때, 결과 카드가 (i) 선택한 한 가지 방식의 결과만 보여줄지, (ii) "다른 방식으로 계산하면 총이자가 약 X원 더/덜 듭니다" 같은 간단 비교 문구를 함께 보여줄지도 UI 공간·콘텐츠 설계에 영향을 준다. 기획팀은 (ii)가 3번 FAQ 3번 문항 및 2번 interpretation 3문단(방식 비교 설명)과 자연스럽게 이어진다고 보아 (ii)를 제안하지만, 이는 결과 카드 레이아웃(디자인팀)과 계산 로직 복잡도(개발팀 — 두 방식을 동시에 계산해 함께 표시)에 걸친 사안이라 임의 확정하지 않는다.

### 6-3. 상환 스케줄(월별 상환 내역 표) 노출 여부

월별 원금/이자/잔액 내역을 표로 전부 보여주면 콘텐츠 두께·체류시간에는 유리하지만, (1) 최대 480개월(6-2 참고) 분량의 표를 렌더링하면 페이지가 무거워지고 모바일 UX가 나빠질 수 있고, (2) YMYL 등급이 "높음"인 대출 계산기에 지나치게 상세한 상환 스케줄을 제시하면 사용자가 이를 "실제 확정된 상환 계획"으로 오인할 리스크가 있다(면책 문구로 완화 가능하나 완전 제거는 어려움). 기획팀은 **v1에서는 스케줄 표를 생략하고 요약 결과(월 상환액/총 이자/총 상환액)만 제공**할 것을 제안하나, 이는 "체류시간 확대 vs 오인 리스크"의 트레이드오프 판단이라 마스터 팀장 확인을 요청한다.

### 6-4. 거치기간(원금 상환 유예) 입력 옵션

4번에서 v1 범위 제외를 제안했다. 포함 시 계산 로직이 복잡해지고(거치기간 동안 이자만 납부 vs 무이자 유예 등 방식이 다양함) 입력 항목도 늘어나 무스크롤 목표(디자인팀 원칙)에 부담을 줄 수 있다. 이 역시 임의로 "제외"를 확정하지 않고 제안으로만 남긴다.

## 7. 다른 팀에 전달 필요 사항 (마스터 팀장 경유)

1. **[마스터 팀장, 우선 확인] 위 6-1~6-4 구조적 결정**: 특히 6-1(토글 vs 별도 계산기)은 디자인팀이 UI 스펙을 작성하기 전에 먼저 정리되어야 착수 지연이 없다.
2. **[개발팀] r = 0(무이자 대출) 예외 처리**: 1-1절 공식은 연이자율 0% 입력 시 분모가 0이 되어 계산 불가하므로 별도 분기(`M = P / n`) 필요.
3. **[개발팀] 원금균등상환 마지막 회차 반올림 보정**: 1-2절에 명시한 대로 `P ÷ n`이 정수로 나누어떨어지지 않을 때 마지막 회차에서 잔여 원금 전액을 상환하도록 처리해 합계 오차를 방지할 것.
4. **[개발팀] 결과 요약 문구의 동적 생성 필요**: 2번 interpretation 1문단(3,000만원/연 5.0%/36개월 예시)은 정적 텍스트가 아니라 사용자 입력값과 **선택한 상환방식**에 따라 동적 생성되어야 한다. age-calculator/dday-calculator/bmi-calculator 구현 시 사용한 방식과 동일하게 처리 요청.
5. **[디자인팀] 오인 유발 UI 지양 재확인**: `calculator-lineup.md` 47~50행 및 이 문서 YMYL 리스크 표기에 따라, "지금 대출 신청", "최저금리 받기" 등 실제 대출 신청으로 오인될 수 있는 CTA 버튼·문구·배너 형태는 사용하지 않도록 요청한다. 계산 버튼은 "계산하기"류의 중립적 문구만 사용.
6. 위 2~5는 기획팀 단독으로 결정할 범위가 아니거나 다른 팀 작업에 직접 영향을 주는 사안이라고 판단해 마스터 팀장 경유로 전달 요청한다. 카테고리 구조·slug(`loan-interest-calculator`)·YMYL 등급 판단 등 이미 확정된 사항은 변경하지 않았다.
