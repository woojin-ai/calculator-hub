# BMI 계산기 (`life/bmi-calculator`) 콘텐츠 기획안

- 작성일: 2026-07-10
- 작성: 기획팀
- 배경: `planning/calculator-lineup.md`(2026-07-08) 전체 로드맵 3순위 "BMI 계산기"를 오늘 라이브 전환하기 위한 콘텐츠 초안. `lib/calculators.ts`에 이미 등록된 `bmi-calculator`(slug, category: `life`, status: `coming-soon`) 스텁의 `interpretation`/`faq` 필드를 채우는 작업이며, `age-calculator`·`dday-calculator`(현재 live 2건)와 동일한 톤·구조를 따랐다.
- YMYL 등급: **중간(건강 관련)** — `calculator-lineup.md` 1절 BMI 항목 판단 그대로 승계. 의료적 진단이 아니라는 면책, 체성분 미반영 한계, 과체중/저체중 시 의료진 상담 권장 문구를 반드시 포함했고, 다이어트·시술 관련 과장 문구는 전혀 사용하지 않았다.
- 기준 선택: BMI 구간 판정은 **대한비만학회(KSSO) 2018 성인 기준**을 1차 기준으로 채택했다(한국 이용자 대상 사이트 특성상 국내 임상 현장에서 통용되는 기준과 사이트 표시값이 일치하는 것이 사용자 혼란을 줄임). 세계보건기구(WHO) 국제 기준(비만 30 이상)은 본문에 참고 비교로만 언급하고, 실제 구간 판정 로직에는 사용하지 않는다. 이 기준 선택이 큰 방향과 다르다고 판단되면 마스터 팀장 확인 요망.

## 1. 체중 구간 라벨 및 임계값 (개발팀 구간 판정 로직용 — 정확한 수치)

대한비만학회(KSSO) 2018 성인 기준, BMI = 체중(kg) ÷ 신장(m)²

| 라벨 | BMI 범위 (경계값 포함 방식) |
|---|---|
| 저체중 (Underweight) | BMI < 18.5 |
| 정상체중 (Normal) | 18.5 ≤ BMI < 23.0 |
| 과체중 (Overweight) | 23.0 ≤ BMI < 25.0 |
| 비만 1단계 (Obese I) | 25.0 ≤ BMI < 30.0 |
| 비만 2단계 (Obese II) | 30.0 ≤ BMI < 35.0 |
| 고도비만 / 비만 3단계 (Obese III) | BMI ≥ 35.0 |

- UI/결과 텍스트에 큰 구간만 노출할 경우 "저체중 / 정상체중 / 과체중 / 비만"(비만은 1~3단계 통합 표시)로 4구간 단순화 가능. 단계별 세분화 표시 여부는 디자인팀·개발팀 판단에 맡긴다.
- 경계값은 모두 **이상(≥) 포함, 미만(<) 제외** 방식으로 통일했다. 예: BMI 정확히 23.0은 "과체중"으로 판정(정상체중 아님).
- 소수점 처리: BMI 값 자체는 소수 둘째 자리 반올림(예: 22.49 → 22.5) 표시를 권장하되, **구간 판정은 반올림 전의 실제 계산값 기준**으로 해야 한다(반올림 후 판정 시 예: 22.996 → 23.0으로 반올림되어 실제로는 정상체중인데 과체중으로 오판정되는 경계 오류 발생 가능 — 개발팀 구현 시 유의 요망).

## 2. interpretation (결과 해석 텍스트, 5단 구조)

`lib/calculators.ts`의 `interpretation` 필드에 그대로 사용 가능한 최종 초안. 결과 요약 1문장은 예시(키 170cm, 몸무게 65kg → BMI 22.5, 정상체중)이며 실제로는 계산 로직에서 동적으로 생성되는 문구여야 한다(`dday-calculator`와 동일한 처리 방식 권장).

```
입력하신 키 170cm, 몸무게 65kg 기준으로 BMI(체질량지수)는 22.5이며, '정상체중' 구간에 해당합니다.

BMI는 체중(kg)을 신장(m)의 제곱으로 나눈 값(체중 ÷ 신장²)으로, 대한비만학회 성인 기준 저체중 18.5 미만, 정상체중 18.5 이상 23.0 미만, 과체중 23.0 이상 25.0 미만, 비만 25.0 이상(1단계 25~29.9, 2단계 30~34.9, 3단계 고도비만 35 이상)으로 구간을 나눕니다. 세계보건기구(WHO)의 국제 기준은 이보다 완화되어 있어(비만 30 이상) 대한비만학회 기준과 다소 차이가 있습니다.

BMI는 키와 몸무게만으로 산출되어 근육량, 체지방률, 골격 구조 등은 반영하지 않습니다. 운동선수처럼 근육량이 많은 경우 BMI가 높게 나와도 체지방은 정상 범위일 수 있고, 반대로 근육량이 적은 경우 BMI가 정상이어도 체지방률이 높을 수 있습니다. 따라서 BMI는 체중 관리를 위한 여러 참고 지표 중 하나로만 활용하는 것이 바람직합니다.

보다 정확한 체성분·비만도 평가는 국민건강보험공단 일반건강검진의 체질량지수 항목이나 질병관리청 국가건강정보포털, 의료기관의 체성분 분석(인바디 등) 검사를 함께 참고하시기 바랍니다.

본 계산 결과는 의료적 진단이 아니며 참고용 지표일 뿐입니다. 과체중·저체중·비만 구간에 해당하는 경우 정확한 건강 상태 평가를 위해 의료진과 상담하시기 바랍니다.
```

- 글자 수: 약 660자 내외(공백 포함, 500~1000자 범위 충족).
- 5단 구조 매핑: ①결과요약 예시(BMI 22.5, 정상체중) → ②계산기준/공식 설명(대한비만학회 구간 전체 + WHO 비교) → ③실생활 팁/주의사항(근육량·체지방 미반영 한계) → ④관련 제도/출처(국민건강보험공단, 질병관리청, 체성분 검사) → ⑤면책 문구(의료적 진단 아님 + 의료진 상담 권장).
- **다이어트/시술 관련 문구 사용하지 않음** — "살 빼는 법", "다이어트 보조제", "시술 추천" 등 애드센스 건강 콘텐츠 정책 리스크가 있는 표현은 의도적으로 배제했다.

## 4. 관련 계산기 링크 추천

라인업 문서 3-3 원칙(같은 카테고리 2개 + 다른 카테고리 1개, 총 3개)을 이번에도 그대로 충족할 수 없다. `life` 카테고리에는 `bmi-calculator` 외 live 계산기가 없다(전기요금·단위 변환은 아직 미착수).

- **추천 1, 2 (타 카테고리, live)**: 만 나이 계산기(`age-calculator`), D-Day 계산기(`dday-calculator`) — 둘 다 `date` 카테고리이며 현재 유일하게 live 상태인 계산기들이다.
- **추천 3 (같은 카테고리)**: 채울 수 없음. `electricity-bill`, `unit-converter` 등은 아직 `lib/calculators.ts`에 등록조차 되어 있지 않다.

`planning/dday-calculator-content.md` 5번에서 이미 개발팀에 전달한 이슈("`getRelatedCalculators()`가 `status`를 필터링하지 않아 coming-soon 스텁이 관련 계산기로 노출될 수 있음")가 BMI 계산기에도 동일하게 적용된다. 이번 라이브 전환 이후 `life` 카테고리 관련 계산기 슬롯이 3개 중 0개만 채워지는 상태가 되므로, 아래 5번에 재확인 요청을 정리했다.

## 5. 다른 팀에 전달 필요 사항 (마스터 팀장 경유)

1. **[개발팀, 재확인] `getRelatedCalculators()` status 필터링 미적용 이슈**: `planning/dday-calculator-content.md`(2026-07-09)에서 이미 전달한 이슈가 BMI 계산기 라이브 전환으로 다시 발생한다. 현재 상태(`age-calculator`, `dday-calculator`, `bmi-calculator` 3개만 live)에서는 BMI 페이지의 "관련 계산기" 슬롯 3개 중 2개(만 나이, D-Day)만 자연스럽게 채워지고 1개는 비거나 coming-soon 스텁으로 채워질 수 있다. 아직 조치되지 않았다면 `status === "live"` 필터를 우선 추가해줄 것을 재요청.
2. **[개발팀] BMI 구간 판정 시 반올림 순서 유의**: 위 1절에 명시한 대로, 화면 표시용 반올림(소수 둘째 자리)과 구간 판정은 분리해서 처리해야 한다. 반올림된 값으로 구간을 판정하면 예: BMI 22.996(정상체중)이 23.0으로 반올림되어 "과체중"으로 오판정되는 경계 오류가 발생할 수 있다.
3. **[개발팀] 결과 요약 문구의 동적 생성 필요**: 위 2절 interpretation의 1번째 문단(키 170cm, 몸무게 65kg 예시)은 정적 텍스트가 아니라 사용자 입력값 기준으로 동적 생성되어야 한다. `age-calculator`/`dday-calculator` 구현 시 사용한 방식과 동일하게 처리 요청.
4. 위 1~3은 기획팀 단독 결정 범위가 아니라고 판단해 마스터 팀장 경유로 개발팀에 전달 요청한다. 카테고리 구조·slug·YMYL 등급 판단 등 이미 확정된 사항은 변경하지 않았다.
