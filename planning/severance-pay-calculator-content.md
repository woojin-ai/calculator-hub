- **핵심 확정 사항**: (a) 재직일수 = #7과 동일 "양 끝 포함(+1)". (b) 3개월 임금은 **합계 직접 입력(Option B)**. (c) 상여·연차수당은 **선택 입력 + 3/12 자동 반영**. (d) 통상임금은 **입력 제외**(면책 고지). (e) 1년 미만은 **금액 미산출 + 안내**. 손계산 검증 앵커 5개(§1-4)를 기획팀이 직접 계산해 제시했다.

## 1. 계산 모델 (개발팀 구현용 — 수치 확정 명세)

### 1-0. 입력 항목 정의 (디자인·개발 공통 확정)

| 입력 항목 | 타입 | 필수 | 기본값 | 비고 |
|---|---|---|---|---|
| 입사일 | date | 필수 | — | 재직 시작 첫날. #7 근속연수 계산기와 동일 의미 |
| 퇴사일 | date | 선택 | **오늘** | **마지막 근무일**. 미입력 시 오늘(예상 퇴직금 조회). 라벨 권장: "퇴사일(마지막 근무일, 미입력 시 오늘)" |
| 최근 3개월 급여 총액 | number(원) | 필수 | — | **세전** 기준. 퇴사일 직전 3개월간 지급된 급여의 **합계**(월 급여 3개월치 총합). placeholder 예: `9,000,000` |
| 연간 상여금 총액 | number(원) | 선택 | **0** | 퇴사일 이전 **1년간** 지급받은 상여금 총액. 내부에서 ×3/12 반영 |
| 연차수당 | number(원) | 선택 | **0** | **전년도 발생분** 미사용 연차수당. 내부에서 ×3/12 반영 |

- **입력 검증(확정)**: `퇴사일 ≥ 입사일`(위반 시 안내). 급여·상여·연차는 음수 불가(0 이상). 미래 퇴사일(퇴사 예정일)은 허용(예상 퇴직금 수요).
- **입력 5개로 고정**: 단위변환·연봉 계산기에서 planning/design 목록 불일치 핑퐁이 있었던 전례에 따라, 입력은 위 5개로 확정하고 §7 디자인 요약과 1:1 일치시킨다. 통상임금·미산입기간·수당 세부항목 필드는 v1에 두지 않는다(근거 §1-3·§8).

#### 1-0-1. 왜 이 입력 구성인가 (단순함 vs 정확도 트레이드오프 — 근거 기록)

과제가 판단을 위임한 3개 결정에 대한 기획팀 채택안과 근거다(최종 승인은 §8 마스터 확인).

1. **3개월 임금 = "합계 직접 입력"(Option B) 채택** — "월 평균임금 직접 입력"(Option A)이 아니라 3개월 급여 **합계**를 받는다. 이유: 법정 1일 평균임금의 분모는 3개월의 **달력 일수**(89~92일, 월마다 다름)인데, 합계를 받아 이 달력 일수로 나눠야 정확하다. 사용자가 "월 평균"을 넣으면 내부에서 ×3을 해야 하고 그 값을 다시 89~92로 나누면 근사 오차가 커진다. 합계 입력은 급여명세서 3장을 더하면 되는 수준이라 부담이 크지 않다.
2. **상여·연차수당 = "선택 입력 + 3/12 자동 반영" 채택**(제외 아님) — 대다수 직장인은 상여금이 있어 이를 빼면 퇴직금을 **체계적으로 과소 추정**한다. 과소 추정은 "회사가 덜 줘도 사용자가 모른다"는 방향의 리스크라 과대 추정보다 나쁘다. 연봉 계산기가 비과세·부양가족·자녀를 선택 입력으로 둔 사이트 패턴과도 일치한다. 다만 "어떤 상여/연차를 넣어야 하는지" 오해 리스크가 있으므로 라벨·FAQ·면책으로 방어한다(연간 상여금 총액 / 전년도 연차수당임을 명시).
3. **통상임금 = "입력 제외"(면책 고지) 채택** — 통상임금은 정기·일률·고정성 판단이 필요해 사용자가 정확히 입력하기 어렵고, 틀리게 넣으면 오히려 오정보가 된다. #7이 "연차 개인화 출력을 만들지 않고 규칙 해설만" 남긴 보수적 판단과 같은 논리로, 통상임금 비교는 **계산에 넣지 않고** "평균임금이 통상임금보다 낮으면 통상임금이 적용되어 실제 퇴직금이 더 많을 수 있다"는 안내만 해설·FAQ에 둔다.

### 1-1. 계산 파이프라인 (단계별 중간값 = 결과 카드에 표시)

```
입력: 입사일 S, 퇴사일 E(미입력 시 오늘), 3개월급여합계 P, 연간상여금 Bonus, 연차수당 Annual

STEP 1  재직일수(계속근로기간) D_work = daysBetween(S, E) + 1     // 양 끝 포함(+1), #7과 동일
STEP 2  1년 미만 게이트: D_work < 365 → 금액 미산출, 안내 출력하고 종료
STEP 3  퇴직일 D_ret = E + 1일                                    // = #7의 종료일⁺. 평균임금 산정기간의 끝점
STEP 4  평균임금 산정기간 총일수 N_avg = daysBetween(D_ret − 3개월, D_ret)   // 달력 일수(89~92)
STEP 5  임금총액 W = P + Bonus × (3/12) + Annual × (3/12)
STEP 6  1일 평균임금 A = W ÷ N_avg                                // 반올림하지 않고 전 정밀도 유지
STEP 7  예상 퇴직금 = round( A × 30 × (D_work ÷ 365) )            // 최종만 원단위 반올림
```

- **파이프라인 의존성**: STEP 4의 `D_ret − 3개월`은 D_ret의 3개월 전 같은 날짜(달을 3 빼고, 존재하지 않는 날이면 말일로 clamp)다. `N_avg`는 그 구간의 순수 달력 일수차(양 끝 미포함 방식 = 3개월의 실제 일수 합).
- **결과 카드에는** 최소 (예상 퇴직금) + (1일 평균임금 A) + (평균임금 산정기간 총일수 N_avg) + (임금총액 W) + (재직일수 D_work·근속 N년 M개월)을 표시. 중간값 노출 정도는 디자인 협의(§7-3).

### 1-2. 재직일수 기준 — **#7과 동일 "양 끝 포함(+1)" 채택** (핵심 확정)

**채택: 재직일수(계속근로기간) = `daysBetween(입사일, 퇴사일) + 1` (양 끝 포함).** #7 근속연수·근무일수 계산기와 **완전히 동일**하다.

- **왜 세트 일관성이 성립하나(중요 발견)**: 고용노동부/사람인 퇴직금 계산기는 "퇴사일 = 마지막 근무일의 **다음 날**(자격상실일)"로 입력받아 `계속근로기간 = 퇴사일 − 입사일`로 센다. 우리는 "퇴사일 = **마지막 근무일**"로 입력받아 `+1`을 한다. **두 방식은 수치가 정확히 같다**:
  - 고용노동부: `daysBetween(입사일, 마지막근무일+1)`
  - 우리: `daysBetween(입사일, 마지막근무일) + 1` = 위와 동일
  - 예) 입사 2020-01-01, 마지막근무일 2022-12-31 → 둘 다 **1,096일**.
- 따라서 사용자가 근속연수 계산기(#7)와 퇴직금 계산기에 **같은 입사일·퇴사일(마지막 근무일)을 넣으면 재직일수가 정확히 일치**한다. 세트로서 값이 어긋나지 않는다.
- **소수 근속연수도 divisor 365 동일**: #7이 `재직일수 ÷ 365`(365.25 아님)를 쓴 것은 이 퇴직금 산식(`… × 재직일수 ÷ 365`)과 정합을 맞추기 위함이었다(#7 §1-2). 그대로 승계한다.
- **외부 계산기 교차검증 시 주의(마스터 대조용)**: 마스터가 고용노동부/사람인 퇴직금 계산기와 대조할 때, 그 계산기들의 "퇴사일" 칸에는 **우리 퇴사일 + 1일**(= 마지막 근무일 다음 날)을 넣어야 재직일수가 일치한다. 같은 날짜를 넣으면 하루 차이가 나는데, 이는 오류가 아니라 "퇴사일" 정의(마지막 근무일 vs 상실일) 차이다. 이 점을 FAQ·§8에 명시했다.

### 1-3. 단순화한 부분 (근사·한계 — 결과해석·FAQ·면책에 반영)

v1에서 실무 대비 단순화한 항목. 모두 YMYL-중간에 맞춰 고지한다.

1. **통상임금 비교 미반영**: 평균임금 < 통상임금이면 통상임금을 평균임금으로 봐야 하나, 통상임금을 입력받지 않는다. → 이 경우 **실제 퇴직금이 계산값보다 많을 수 있음**을 고지(해석 ③·FAQ 2).
2. **부분월·미산입기간 미반영**: 퇴사일이 월 중간이거나 산정기간에 무급휴직·결근 등 미산입기간이 있으면, "3개월 급여 합계"와 달력 일수(N_avg)가 실제와 어긋난다. v1은 "산정기간 3개월을 온전히 근무하고 급여를 받았다"고 가정한다. → 부분월·휴직 이력이 있으면 결과가 근사임을 고지(FAQ 4).
3. **상여·연차수당 3/12 반영은 사용자 입력에 의존**: 어떤 상여/연차가 대상인지는 사용자가 판단해 넣는다. 라벨로 "연간 상여금 총액 / 전년도 연차수당"을 안내하되, 정확한 대상 여부는 회사 규정·전문가 확인으로 넘긴다(FAQ 2·면책).

### 1-4. 검증용 기준값 (QA·마스터가 독립 재현할 앵커) — 기획팀 직접 손계산

과거 CS 이슈(해설 예시 수치 ≠ 엔진 실측) 재발 방지를 위해, 결과해석·FAQ의 모든 예시를 아래 표로 고정한다. **앵커 A를 결과해석 대표 예시로 사용**한다. 개발팀은 엔진 출력이 이 표와 일치하는지, QA·마스터는 아래 손계산 근거로 독립 재현할 것.

**손계산 근거(독립 재현용):**

- **앵커 A(대표)** `2019-01-01 → 2024-12-31`, P=900만, 상여=600만:
  - D_work: daysBetween(2019-01-01, 2024-12-31)= 2019(365)+2020(366)+2021(365)+2022(365)+2023(365)=1,826, +2024-01-01→2024-12-31(365)=2,191. +1 = **2,192일**(= 입사 후 만 6년, 퇴직일 2025-01-01, 윤년 2020·2024 2회 포함).
  - D_ret = 2025-01-01. D_ret−3개월 = 2024-10-01. N_avg = daysBetween(2024-10-01, 2025-01-01)= 10월31+11월30+12월31 = **92**.
  - W = 9,000,000 + 6,000,000×3/12 = 9,000,000 + 1,500,000 = **10,500,000**.
  - A = 10,500,000 ÷ 92 = **114,130.4348**.
  - 퇴직금 = 114,130.4348 × 30 × 2,192 ÷ 365 = 3,423,913.043 × 2,192 ÷ 365 = 7,505,217,391 ÷ 365 = 20,562,239.4 → **20,562,239원**.
- **앵커 B** `2020-03-02 → 2025-08-31`, P=1,050만, 상여=400만, 연차=120만:
  - D_work: daysBetween(2020-03-02, 2025-08-31)=2,008(교차검증: #7 앵커C 2020-03-02→2025-08-15가 1,993일, +16일=2,009 즉 daysBetween 2,008), +1 = **2,009일**.
  - D_ret = 2025-09-01. D_ret−3개월 = 2025-06-01. N_avg = daysBetween(2025-06-01, 2025-09-01)= 6월30+7월31+8월31 = **92**.
  - W = 10,500,000 + 4,000,000×3/12 + 1,200,000×3/12 = 10,500,000 + 1,000,000 + 300,000 = **11,800,000**.
  - A = 11,800,000 ÷ 92 = **128,260.8696**.
  - 퇴직금 = 128,260.8696 × 30 × 2,009 ÷ 365 = 3,847,826.087 × 2,009 ÷ 365 = 7,730,282,609 ÷ 365 = 21,178,856.5 → **21,178,856원**.
- **앵커 C** `2021-05-01 → 2025-04-30`, P=960만:
  - D_work: daysBetween(2021-05-01, 2025-05-01)=1,461(윤년 2024 포함) − 1(4-30은 5-01 하루 전) = 1,460. +1 = **1,461일**(만 4년).
  - D_ret = 2025-05-01. D_ret−3개월 = 2025-02-01. N_avg = daysBetween(2025-02-01, 2025-05-01)= 2월28(2025 평년)+3월31+4월30 = **89**.
  - W = **9,600,000**. A = 9,600,000 ÷ 89 = **107,865.1685**.
  - 퇴직금 = 107,865.1685 × 30 × 1,461 ÷ 365 = 3,235,955.056 × 1,461 ÷ 365 = 4,727,730,337 ÷ 365 = 12,952,685.9 → **12,952,686원**.
- **앵커 D** `2022-01-01 → 2024-04-30`, P=900만:
  - D_work: daysBetween(2022-01-01, 2024-01-01)=730, +daysBetween(2024-01-01, 2024-04-30)=120(1월31+2월29+3월31+4월29, 2024 윤년) = 850. +1 = **851일**.
  - D_ret = 2024-05-01. D_ret−3개월 = 2024-02-01. N_avg = daysBetween(2024-02-01, 2024-05-01)= 2월29(2024 윤년)+3월31+4월30 = **90**.
  - W = **9,000,000**. A = 9,000,000 ÷ 90 = **100,000.00**(엣지: 정확히 나누어떨어짐).
  - 퇴직금 = 100,000 × 30 × 851 ÷ 365 = 3,000,000 × 851 ÷ 365 = 2,553,000,000 ÷ 365 = 6,994,520.5 → **6,994,521원**.
- **앵커 E(엣지)** `2025-01-01 → 2025-08-31`:
  - D_work: daysBetween(2025-01-01, 2025-08-31)= 1월31+2월28+3월31+4월30+5월31+6월30+7월31 = 212, +8월(1→31)=30 → 242. +1 = **243일**.
  - 243 < 365 → **법정 퇴직금 지급 대상 아님**(계속근로 1년 미만). 금액 미산출, 안내 출력.

> 반올림 기준(검증 전제): 1일 평균임금 A는 **반올림하지 않고 전 정밀도 유지**, 최종 퇴직금만 **원단위 반올림(round)**. 고용노동부/사람인은 평균임금을 원단위로 먼저 정리하는 경우가 있어 수십~수백 원 차이가 날 수 있다(§8 라운딩 확정 요청). 개발팀은 이 표를 "산식이 맞는지 확인하는 검증용"으로 쓰고, 최종 라운딩 정책 확정 후 회귀 테스트 값으로 고정할 것.

## 2. interpretation (결과 해석 텍스트, 5단 구조)

`lib/calculators.ts`의 `interpretation` 필드 초안. 1문단(결과 요약)은 §1-4 앵커A 기준이며, 실제로는 **사용자 입력값에 따라 동적 생성**되어야 한다(기존 라이브 계산기와 동일 처리, §7 개발 전달).

```
입사일 2019년 1월 1일부터 퇴사일 2024년 12월 31일까지 근무하고 최근 3개월 급여 합계 900만원, 연간 상여금 600만원을 입력한 경우, 1일 평균임금은 약 114,130원, 재직일수는 2,192일(약 6년)이며 예상 퇴직금은 약 2,056만원입니다.

이 퇴직금 계산기는 근로자퇴직급여 보장법상 법정 퇴직금 공식인 '1일 평균임금 × 30일 × (재직일수 ÷ 365)'로 계산합니다. 1일 평균임금은 퇴사일 직전 3개월간 지급된 임금 총액을, 그 기간의 달력 일수(월에 따라 89~92일)로 나눈 값입니다. 연간 상여금과 전년도 연차수당은 각각 3/12(3개월분)만 임금 총액에 더해 반영합니다. 재직일수(계속근로기간)는 입사일과 퇴사일을 모두 포함해 계산하며, 근속연수·근무일수 계산기의 재직일수와 동일한 기준입니다.

법정 퇴직금은 계속근로기간이 1년 이상이고 4주 평균 주 15시간 이상 근무한 경우에 지급 대상이 됩니다. 재직일수가 1년(365일) 미만이면 지급 대상이 아니므로 금액을 산출하지 않습니다. 또한 평균임금이 통상임금보다 적은 경우에는 통상임금을 평균임금으로 보는데, 이 계산기는 통상임금을 입력받지 않으므로 이 경우 실제 퇴직금이 계산값보다 많을 수 있습니다.

입력하신 값은 브라우저 내에서만 계산에 사용되며 서버로 전송·저장되지 않습니다. 본 계산 결과는 참고용이며, 정확한 퇴직금은 회사 규정·급여명세서와 고용노동부 퇴직금 계산기, 노무 전문가를 통해 확인하시기 바랍니다.
```

- 글자 수: 약 700자(공백 포함, 500~1000자 충족).
- 5단 구조 매핑: ①결과요약(동적, 앵커A) → ②계산기준/법정 공식·평균임금 정의·상여 3/12·재직일수(#7 동일) → ③지급요건·한계(1년 이상+주15시간, 통상임금 미반영 시 과소 가능) → ④범위 고지(세전, 퇴직소득세 별도) → ⑤개인정보(브라우저 내)+면책(회사 규정·고용노동부·전문가).
- **SEO 키워드 자연 포함**: "퇴직금 계산기"·"법정 퇴직금"·"1일 평균임금"·"재직일수"·"근속연수" 자연 배치.
- **등장 수치(114,130원 / 2,192일 / 약 2,056만원 / 900만·600만)는 §1-4 앵커A와 일치**(임의 수치 금지 원칙).

## 4. 관련 계산기 링크 추천 (세트 연결 — override 제안)

- `getRelatedCalculators()`는 `status === "live"` 필터 + "같은 카테고리 우선 후 나머지, 배열 순서" 로직(`[...sameCategory, ...rest].slice(0,3)`)이다.
- 이 계산기가 live 전환되면 `salary` 카테고리 live는 `salary-net-calculator`와 이 계산기 2개다. self 제외하면 sameCategory=[연봉 실수령액] 1개뿐이라, 나머지 2슬롯은 배열 순서(age → dday → service-period …)로 채워져 **자동 결과 = 연봉 실수령액 + 만 나이 + D-Day**가 된다.
- **문제**: 세트 파트너인 **근속연수·근무일수 계산기(#7)가 자동 3슬롯에 안 들어온다**(배열상 dday 다음). 근속연수 계산기는 이 퇴직금 계산기를 가리키는데, 되돌아오는 링크가 없으면 세트 동선이 한 방향만 된다.
- **기획팀 제안(override)**: 이 계산기의 관련링크를 **연봉 실수령액 + 근속연수·근무일수 + 만 나이** 순으로 지정. 근속연수 계산기(#7)는 재직일수를 먼저 확인하는 자연스러운 사전 단계라 세트 상호링크로 반드시 포함해야 한다. 구현은 §8에서 개발팀에 override 여부 확인 요청.
- **앵커 텍스트(문장형):**
  - 근속연수·근무일수 계산기: "퇴직금의 기초가 되는 재직일수부터 확인하려면 근속연수·근무일수 계산기를 함께 이용해보세요."
  - 연봉 실수령액 계산기: "매달 실수령액이 궁금하다면 연봉 실수령액 계산기도 확인해보세요."
  - 만 나이 계산기: "정확한 만 나이가 필요하다면 만 나이 계산기를 이용해보세요."

## 6. 확정한 모델 요약 (개발팀 즉시 참조용)

| 항목 | 확정값 |
|---|---|
| 입력 | 입사일(필수) / 퇴사일(선택, 미입력=오늘, 마지막 근무일) / 최근 3개월 급여 합계(필수, 세전) / 연간 상여금(선택, 0) / 연차수당(선택, 0) |
| 재직일수 D_work | **양 끝 포함(+1)** = `daysBetween(입사일, 퇴사일)+1`. #7과 동일, `lib/service-period.ts` 재사용 가능 |
| 1년 미만 게이트 | `D_work < 365` → 금액 미산출, 안내 출력 |
| 산정기간 총일수 N_avg | `daysBetween(퇴직일 D_ret − 3개월, D_ret)`, D_ret = 퇴사일+1. 달력 일수 89~92 |
| 임금총액 W | `3개월합계 + 상여×3/12 + 연차×3/12` |
| 1일 평균임금 A | `W ÷ N_avg` (반올림 없이 전 정밀도) |
| 예상 퇴직금 | `round(A × 30 × D_work ÷ 365)` (최종만 원단위 반올림) |
| 통상임금 | 입력 제외(면책 고지, 실제 퇴직금 ≥ 계산값 가능) |

| YMYL | 중간. 통상임금 한계·세전·부분월 3중 면책 |

## 7. 디자인팀에 넘길 확정 요약 (화면 설계용 — 목록 불일치 핑퐁 방지)

단위변환 때 planning/design 목록 불일치가 재발하지 않도록, 아래 입력·출력 목록을 **그대로** 화면 설계에 사용한다. §1-0·§6과 1:1 일치.

### 7-1. 입력 필드 (5개, 순서 고정)

1. **입사일** — date, 필수
2. **퇴사일** — date, 선택(미입력 시 오늘). 라벨 "퇴사일(마지막 근무일, 미입력 시 오늘)"
3. **최근 3개월 급여 총액** — number(원, 천단위 콤마), 필수. 보조문구 "세전 기준, 최근 3개월 급여 합계"
4. **연간 상여금 총액** — number(원), 선택, 기본 0. 보조문구 "최근 1년간 받은 상여금 (없으면 0)"
5. **연차수당** — number(원), 선택, 기본 0. 보조문구 "전년도 발생 미사용 연차수당 (없으면 0)"

### 7-2. 출력 값 (결과 카드)

- **대표값(히어로)**: 예상 퇴직금 `N원` (천단위 콤마) — 세전임을 배지/각주로 명시
- **보조값**: (a) 1일 평균임금 `N원` / (b) 평균임금 산정기간 총일수 `N일`(89~92) / (c) 임금총액(3개월) `N원` / (d) 재직일수 `N일` + 근속 `N년 M개월`
- **1년 미만 상태**: 재직일수 < 365이면 대표값 자리에 "법정 퇴직금 지급 대상이 아닙니다(계속근로 1년 미만)" 안내 + 재직일수만 표시, 금액 미표시
- **퇴사일 미입력 시**: "오늘(2026년 X월 X일) 기준 예상 퇴직금" 표기

### 7-3. 디자인 협의 필요(참고)

- 상단 근사·범위 고지 배너(YMYL-중간, "안내" 톤): 연봉 계산기 배너 선례처럼 amber `role="note"`, 항상 노출. 카피 예: "예상 퇴직금은 세전 금액이며, 통상임금·부분월 등에 따라 실제 지급액과 다를 수 있습니다. 참고용으로만 활용해 주세요." (최종 카피는 디자인 스펙 회신 후 기획팀 확정)
- 보조값(a~d) 접이식(`<details>`) 여부: 연봉 계산기처럼 기본 접힘 검토 가능. 확정은 디자인 스펙 수령 후.
- 퇴직·이직 부추김 문구, 오인 CTA 지양(YMYL-중간).

## 8. 마스터 확인 필요 결정사항 (기획팀 단독 결정 범위 밖)

### 8-1. [마스터] 3개월 임금 입력 방식 — Option A(월 평균) vs **Option B(3개월 합계)** 승인
- **기획팀 채택안**: **Option B(3개월 급여 합계 직접 입력)**. 근거(§1-0-1): 법정 평균임금 분모가 달력 일수(89~92)라 합계÷달력일수가 정확. 월 평균 입력은 ×3 근사 오차 발생.
- **확인 요청**: Option B 확정 승인 요망. 만약 마스터가 사용성(월급만 아는 사용자)을 우선해 Option A(월 평균 입력)를 원하면 앵커·해설 수치가 바뀌므로 착수 전 회신 필요.

### 8-2. [마스터] 상여금·연차수당 — **선택 입력 + 3/12 자동 반영** vs 제외(단순화) 승인
- **기획팀 채택안**: **포함(선택 입력, 기본 0, ×3/12 자동)**. 근거: 제외 시 대다수 직장인 퇴직금을 체계적으로 과소 추정(과소 추정이 더 나쁜 리스크). 연봉 계산기의 선택 입력 패턴과 일치.
- **확인 요청**: 포함 확정 승인 요망. 마스터가 "v1은 3개월 급여만으로 최대 단순화" 방침이면 상여·연차 2필드를 제거하고 해설에 "상여·연차 미반영, 실제 퇴직금이 더 많을 수 있음" 면책만 남기는 대안으로 전환 가능(앵커 A·B 재계산 필요).

### 8-3. [마스터] 통상임금 입력 제외 승인
- **기획팀 채택안**: **통상임금 입력 제외**, "평균임금 < 통상임금이면 통상임금 적용 → 실제 퇴직금이 더 많을 수 있음"을 해석·FAQ 면책으로만 처리(§1-3, #7의 "연차 출력 안 만들고 해설만" 보수 판단과 동일 논리).
- **확인 요청**: 제외 확정 승인 요망. 통상임금 비교까지 넣으려면 통상임금 입력 UI + 정의 안내가 필요해 별도 태스크(개발·디자인 추가)로 분리해야 한다.

### 8-4. [마스터] 1년 미만 = 금액 미산출 + 안내 승인
- **기획팀 채택안**: 재직일수 < 365면 퇴직금 **금액을 표시하지 않고** "지급 대상 아님" 안내만. 근거: 대상 아닌 금액을 노출하면 오정보(YMYL). 회사 규정상 별도 지급은 FAQ 3에서 안내.
- **확인 요청**: 승인 요망.

### 8-5. [개발] 원단위 라운딩 정책 확정
- §1-4 검증값은 "1일 평균임금 전 정밀도 유지 + 최종 퇴직금만 원단위 반올림" 기준. 고용노동부/사람인 대조 시 평균임금 선반올림 여부로 수십~수백 원 차이 가능. 최종 라운딩 정책 확정 후 앵커를 회귀 테스트 값으로 고정 요청.

### 8-6. [개발] 재직일수 로직 재사용 + 외부 대조 주의
- D_work = 양 끝 포함(+1)로 `lib/service-period.ts` 재사용(§1-2). 외부(고용노동부/사람인) 대조 시 그쪽 "퇴사일" 칸에는 **우리 퇴사일+1일**을 넣어야 재직일수 일치(정의 차이, 오류 아님).

### 8-7. [개발] 관련 계산기 override
- §4대로 자동 결과에 세트 파트너 근속연수 계산기(#7)가 빠진다. 관련링크를 **연봉 실수령액 + 근속연수·근무일수 + 만 나이**로 override할지 확인 요망(기획팀 권고: override).

### 8-8. [개발] 결과 요약 문구 동적 생성
- §2 1문단(앵커A 예시)은 정적 텍스트가 아니라 사용자 입력값 기반 동적 생성(기존 라이브 계산기와 동일).

### 8-9. [디자인] §7-3 배너 카피·접이식·1년미만 안내 UI
- 상단 고지 배너 최종 카피, 보조값 접이식 여부, 1년 미만/퇴사일 미입력 상태 표기는 디자인 스펙 수령 후 기획팀 확정 회신.

## 참고 (기준 근거)
- 법정 퇴직금 산식·평균임금 정의(근로자퇴직급여 보장법, 근로기준법 제2조): 퇴직금 = 1일 평균임금 × 30일 × (재직일수 ÷ 365); 1일 평균임금 = 사유발생일 이전 3개월 임금총액 ÷ 그 기간 총 일수. 상여금은 퇴직 전 1년치의 3/12, 연차수당은 전년도분의 3/12 가산. 평균임금 < 통상임금이면 통상임금 적용. 찾기쉬운 생활법령정보/고용노동부.
- 지급 요건: 계속근로 1년 이상 + 4주 평균 주 15시간 이상.
- 재직일수 "양 끝 포함(+1)" = 고용노동부식 계속근로기간(퇴직일−입사일)과 수치 동일(§1-2 검증). #7 `service-period-calculator`와 완전 정합.
- 대조 검증용 외부 계산기: 고용노동부 퇴직금 계산기(www.moel.go.kr / labor.moel.go.kr), 사람인 퇴직금 계산기.
