(1) 결과가 대출 규제 정책(DSR 규제비율·스트레스 DSR)에 직접 연동되고, (2) 정책이 수시로(지역·시점별) 바뀌며, (3) 은행별·차주별 실제 심사 기준(장래소득 인정·스트레스 금리 가산·대출종류별 원리금 산정방식)이 달라 이 계산기 값과 실제 심사 DSR이 다르게 나올 수 있다. 따라서 **"이 결과로 대출 승인 여부를 판단하지 말라"는 강한 면책을 결과해석·FAQ·상단배너·면책 4중으로 배치**하고, "지금 대출 신청/한도 조회" 류 오인 유발 문구는 전혀 쓰지 않는다(대출이자·중도상환수수료 원칙 승계).
- **정확성 리서치 핵심 발견(중요)**:
  1. **DSR 정의는 명확·안정적**: `DSR = (모든 대출의 연간 원리금 상환액 합계) ÷ 연소득 × 100(%)`. 이 공식 자체는 정책이 바뀌어도 변하지 않는다. **변하는 것은 "규제 통과 기준선"(40%/50% 등)과 스트레스 금리 가산분**이라, 계산식은 하드코딩하되 규제 기준선은 상수 분리 + "참고용·변동" 면책으로 처리한다(전기요금 부담금·중도상환 요율 전례와 동일).
  2. **규제비율(2026-07 시점)**: 일반적으로 **차주단위 DSR 은행권 40%, 제2금융권 50%** 수준. **스트레스 DSR 3단계**가 단계적 시행 중이며(수도권·규제지역 등 선적용, 지방 주담대는 유예 이력 있음), 스트레스 금리 가산으로 **실제 한도 산정 DSR은 이 계산기의 단순 DSR보다 높게 잡힐 수 있다**. 지역·시점·대출종류별로 적용이 갈려 **단정하지 말고 "일반적으로 ~수준"으로 완곡히** 서술한다(§1-2·§2·§3).
  3. **원리금 산정 단순화**: 실제 규정은 대출종류별로 원금·이자 반영 방식이 세분(주담대는 전액 분할상환 원리금, 일부 신용대출은 만기 균등분할 가정 등)돼 매우 복잡하다. v1은 **신규 대출은 원리금균등 월상환액×12, 기존 대출은 사용자가 아는 "연간 원리금 상환액"을 직접 입력·합산**하는 실용 근사로 두고, "실제 심사 산정방식과 다를 수 있음"을 면책 처리한다(§1-3·§8-2).

## 1. 계산 모델 (개발팀 구현용 — 수치 확정 명세)

### 1-0. 입력 항목 정의 (디자인·개발 공통 확정)

**입력 방식 결정(A 간편형 vs B 상세형) — 채택: (B) 상세형 기반 + (A) 간편형 자동 포섭.** 과제가 제시한 (A)(연간 총 원리금 직접입력)와 (B)(신규대출 원리금 자동계산 + 기존대출 합산) 중, **(B)를 채택하되 신규 대출 입력을 비우면 자동으로 (A)와 동일해지도록** 설계한다(별도 모드 토글 없이 하나의 화면으로 둘 다 커버). 근거는 §1-0-1.

| # | 입력 항목 | 타입 | 필수 | 기본값 | 비고 |
|---|---|---|---|---|---|
| 1 | 연소득(세전) | number(원, 천단위 콤마) | **필수** | — | DSR 분모. placeholder 예: `60,000,000`. 보조문구 "세전 연소득(원)" |
| 2 | 신규 대출금액 | number(원, 천단위 콤마) | 선택 | 0(빈칸=0) | 이번에 새로 받을(또는 계산할) 대출 원금. 비우면 기존대출만으로 DSR 산출(간편 경로). placeholder 예: `300,000,000` |
| 3 | 신규 대출 연이자율 | number(%, 소수 2자리) | 신규금액>0이면 필수 | — | placeholder 예: `4.5` |
| 4 | 신규 대출기간 | number(개월) | 신규금액>0이면 필수 | — | placeholder 예: `360`(30년). 보조문구 "개월 단위" |
| 5 | 신규 대출 상환방식 | select 2종 | 신규금액>0이면 필수 | 원리금균등 | "원리금균등"/"원금균등". 기본 원리금균등(§1-0-1-4·§8-3) |
| 6 | 기존 대출 연간 원리금 상환액 | number(원, 천단위 콤마) | 선택 | 0(빈칸=0) | 이미 받은 다른 대출들의 1년치 원리금 합계. placeholder 예: `5,000,000`. 보조문구 "이미 있는 대출의 1년간 원금+이자 상환액 합계" |

- **입력 검증(확정)**:
  - 연소득: `> 0` 필수(0/음수/공란/NaN → 계산 불가, §9 엣지). 상식 상한(예: 100억) 초과 시 안내 권장.
  - 신규 대출금액·기존 연상환액: `≥ 0`(공란은 0으로 취급).
  - 신규 대출 연이자율: `≥ 0`(0% = 무이자 → §1-1 STEP r=0 분기). 상식 범위(예: 0~20%) 밖 안내 권장.
  - 신규 대출기간: `≥ 1개월`(신규금액>0일 때). 상한 넉넉히(예: 600개월).
- **입력 6개로 고정**: 과거 단위변환·연봉·전기요금에서 planning/design 목록 불일치 핑퐁이 있었던 전례에 따라, 입력은 위 **6개로 확정**하고 §7 디자인 요약과 1:1 일치시킨다.

#### 1-0-1. 왜 이 입력 구성인가 (근거 기록 — §8에서 최종 승인)

1. **(B) 상세형 채택 + (A) 자동 포섭** — (A) 간편형만 두면 "연간 총 원리금상환액"을 사용자가 스스로 계산해 넣어야 해서 정작 계산기의 존재 이유가 반감된다(대다수 사용자는 신규 주담대의 연 원리금을 모른다). (B)는 신규 대출의 연 원리금을 **대출이자 계산기(`loan-interest-calculator`)의 원리금균등 로직을 재사용**해 자동 산출하므로 사용자 부담이 낮다. 동시에 **신규 대출금액을 비우면 신규 연원리금=0이 되어 기존 연상환액만으로 DSR이 나오므로, (A) 간편형 사용자도 별도 UI 없이 그대로 수용**된다. 개발 부담이 (B)로 커지지만 대출이자 로직 공유로 상쇄한다(§개발팀 전달 이슈).
2. **분모=연소득 단일** — DSR 정의 분모는 연소득이다. 월소득만 아는 사용자를 위해 "월소득×12" 보조안내는 디자인 협의(§7-3). 세전 기준 명시(세후 입력 시 DSR 과대계산 오류 방지).
3. **기존 대출은 "연간 원리금상환액 직접입력"으로 근사** — 기존 대출은 종류·잔여기간·상환방식이 제각각이라 원금·기간·금리를 다 받으면 입력이 폭증한다. 사용자가 대출 앱/약정서에서 비교적 쉽게 확인 가능한 **"1년치 원리금 합계"**를 직접 받는다(#10 중도상환이 "잔존기간 직접입력" 대신 명확한 값을 받은 원칙과 동일 계열). 여러 건이면 합산해 입력하도록 안내(§7-3, v1은 단일 필드; 다건 분리입력은 §8-4 대안).
4. **신규 상환방식 기본 원리금균등** — 주담대·대부분 분할상환 대출이 원리금균등이라 기본값으로 두되, 대출이자 계산기가 이미 원금균등을 지원하므로 동일 로직 재사용으로 옵션 제공(§8-3에서 상환방식 UI 노출 여부 마스터 확정). 원금균등 선택 시 DSR 산정용 "연 원리금"은 §1-1 STEP 3-b로 처리.

### 1-1. 계산 파이프라인 (단계별 중간값 = 결과 카드에 표시)

```
입력:
  Y      = 연소득(원)                 [> 0, 필수]
  P      = 신규 대출금액(원)          [≥ 0, 기본 0]
  i      = 신규 대출 연이자율(%)      [≥ 0, 신규금액>0이면 필수]
  n      = 신규 대출기간(개월)        [≥ 1, 신규금액>0이면 필수]
  method = 신규 상환방식              ["원리금균등"(기본) | "원금균등"]
  E_year = 기존 대출 연간 원리금상환액(원)   [≥ 0, 기본 0]

STEP 0  유효성 검사
          if (Y ≤ 0 || Y is NaN)  →  계산 불가: 결과 대신 안내 메시지("연소득을 올바르게 입력하세요") → 종료
          if (P ≤ 0)              →  신규 연원리금 New_year = 0 으로 두고 STEP 3 건너뜀(간편 경로)

STEP 1  월이자율
          r = i / 12 / 100

STEP 2  (신규금액 P>0 인 경우만) 신규 대출 월 상환액 — 대출이자 계산기 로직 재사용
          [원리금균등]  if (r == 0)  M = P / n            // 무이자 예외 분기(대출이자 §1-1 승계)
                        else         M = P × r × (1+r)^n / ((1+r)^n − 1)
                        New_year = M × 12
          [원금균등]    연 원리금은 회차별로 달라지므로 "1년차(초기 12개월) 원리금 합계"를 산정(가장 보수적·큰 값)
                        → §1-1-b 참조

STEP 3  신규 연 원리금 확정
          New_year = (P ≤ 0) ? 0 : STEP 2 결과

STEP 4  총 연간 원리금 상환액
          Total_year = New_year + E_year

STEP 5  DSR
          DSR(%) = Total_year / Y × 100        // 소수 1자리 반올림 표시 권장

STEP 6  규제선 비교(해석용, 판정 아님)
          은행권 기준선 40%, 제2금융권 50%와 대소만 안내(§1-2). "심사 통과/부결"로 단정 금지.
```

#### 1-1-b. 원금균등 선택 시 "연 원리금" 산정(보수적 근사)

원금균등은 매월 원금 = `P/n` 고정, 이자는 잔액에 비례해 감소하므로 **연 원리금이 해마다 다르다**. DSR은 통상 상환 부담이 가장 큰 초기를 보므로, v1은 **1년차(1~12회차) 원리금 합계**를 신규 연원리금으로 사용한다.
- 1년차 원금 합계 = `(P/n) × 12`
- 1년차 이자 합계 = `r × [ 12·P − (P/n)·(0+1+2+…+11) ] = r × [ 12·P − (P/n)·66 ]`  (회차 k 상환 전 잔액 = P − (P/n)·(k−1))
- `New_year(원금균등) = (P/n)×12 + r×(12P − 66P/n)`
- **주의**: 이 근사는 "가장 부담 큰 첫해" 기준이라 실제 심사 산정방식과 다를 수 있다(면책, §1-3·§8-3). 원금균등 UI를 v1에서 뺀다면 이 분기는 미구현(§8-3 마스터 확정).

- **결과 카드 표시 최소 항목**: (a) DSR `%`(대표값) / (b) 총 연간 원리금상환액 Total_year / (c) 신규 연원리금 New_year(있을 때) + 월상환액 M / (d) 기존 연원리금 E_year / (e) 연소득 Y / (f) 규제 기준선(40%·50%) 대비 위치 해석. 노출 정도는 디자인 협의(§7-2).
- **부동소수 주의**: `(1+r)^n` 지수·나눗셈은 대출이자 계산기와 동일 함수를 쓰면 라운딩 일관성이 유지된다(공유 리팩터 권장, §개발팀 전달 이슈). DSR%는 소수 1자리 반올림 표시.

### 1-2. 규제 상수 (기준선·스트레스 DSR) — 리서치값 + "참고·변동" 강제 표기

**아래는 WebSearch로 조사한 2026-07 시점 값이다. 대출 규제는 지역·시점·대출종류별로 갈리고 수시 개정되므로, 규제 기준선·스트레스 금리는 계산 로직에 넣지 않고(계산은 순수 DSR 공식만) "해석·안내용 참고값"으로만 쓰며 상수 모듈로 분리한다(§8-5).**

| 상수 | 값(2026-07 기준) | 성격 | 확인 상태 |
|---|---|---|---|
| 은행권 차주단위 DSR 기준선 | **일반적으로 40%** | 해석 안내용(참고) | 교차확인·단, "일반적으로" 완곡 표기 |
| 제2금융권 DSR 기준선 | **일반적으로 50%** | 해석 안내용(참고) | 교차확인·완곡 표기 |
| 스트레스 DSR | **3단계 단계적 시행 중**(수도권·규제지역 선적용, 지방 주담대 유예 이력) | 안내 문구용 | 지역·시점별 상이 — 단정 금지 |
| DSR 공식 | `연 원리금상환액 합계 ÷ 연소득 × 100` | **계산 로직(고정)** | 정의 안정적, 하드코딩 |

- **정확성 주의(개발·마스터 필독)**:
  1. **계산기가 산출하는 것은 "단순 DSR"이다.** 실제 은행 심사 DSR은 **스트레스 금리 가산분**이 붙어 더 높게 잡히고, **장래소득 인정·소득 산정방식**에 따라 분모(연소득)도 달라진다. 따라서 이 계산기 값이 40% 미만이어도 실제 심사에서는 초과할 수 있다 → §2·§3·면책에서 강조.
  2. **규제 기준선(40%/50%)은 "심사 통과 보증"이 아니다.** 결과 카드는 "은행권 기준선 40% 대비 여유/근접/초과" 수준의 **위치 안내**만 하고, "대출 가능/불가"로 단정하지 않는다(§7-2).
  3. **스트레스 DSR 수치·적용지역은 변동성이 커서 계산기 본문에 특정 스트레스 금리(%)를 하드코딩하지 않는다.** "스트레스 DSR이 단계적으로 시행되어 실제 한도는 더 보수적으로 산정될 수 있다"는 정성적 안내만 둔다(§2 3문단·§3 3번).

### 1-3. 단순화한 부분 (근사·한계 — 결과해석·FAQ·면책에 반영)

v1에서 실제 심사 대비 단순화한 항목. 모두 YMYL-높음에 맞춰 고지한다.

1. **스트레스 금리 미가산**: 계산기는 입력 금리 그대로 원리금을 산정 → 실제 한도 심사보다 DSR이 낮게 나올 수 있음(FAQ 4·면책).
2. **기존 대출 원리금 = 사용자 입력 근사**: 대출종류별 규정 산정방식(만기 균등분할 가정 등) 미반영 → 실제와 차이(FAQ 4·면책).
3. **소득 산정 단순화**: 세전 연소득 단일 입력. 장래소득 인정·소득 종류별 인정비율·증빙 미반영(면책).
4. **규제 기준선·스트레스 DSR 시점·지역 의존**: 40%/50%는 일반값, 스트레스 DSR 적용은 지역·시점별 상이 → "참고용·변동" 반복 고지(§2·§3).

### 1-4. 검증용 기준값 (QA·마스터가 독립 재현할 앵커) — 엔진 실측 정합

과거 CS 이슈(해설 예시 수치 ≠ 엔진 실측) 재발 방지를 위해, 결과해석·FAQ의 모든 예시를 아래 표로 고정한다. **앵커 B를 결과해석 대표 예시로 사용**(연소득 6천·신규 주담대 3억 = 이 계산기 주 검색의도). 신규 대출 월상환액·연원리금은 **엔진(`lib/loan.ts`의 `calculateLoan`, 대출이자 계산기와 동일 로직·Math.round 라운딩)의 실측값**으로 고정하며 표기상 "약(근사)"을 붙인다. **DSR%가 대표값**이다. DSR은 소수 1자리 반올림.

| # | 연소득 Y | 신규 P / i / n / 방식 | 신규 월상환 M | 신규 연원리금 New_year | 기존 E_year | 총 Total_year | **DSR%** | 확인 포인트 |
|---|---|---|---|---|---|---|---|---|
| A(간편) | 50,000,000 | 없음(0) | — | 0 | 20,000,000 | 20,000,000 | **40.0%** | 신규 비움=간편경로, 기준선 40% 정확히 도달 |
| **B(대표)** | 60,000,000 | 3억 / 4.5% / 360 / 원리금균등 | 약 1,520,056 | 약 18,240,672 | 0 | 약 18,240,672 | **약 30.4%** | 주담대 30년, 기준선 40% 이하(여유가 크지는 않음) |
| C(합산·초과) | 50,000,000 | 2억 / 4.5% / 360 / 원리금균등 | 약 1,013,369 | 약 12,160,424 | 10,000,000 | 약 22,160,424 | **약 44.3%** | 신규+기존 합산, 은행권 40% 초과·제2금융 50% 이하 |
| D(무효) | 0 | — | — | — | — | — | **계산 불가** | 연소득 0 → STEP 0 종료, 안내 메시지 |

**손계산 근거(독립 재현용):**

- **앵커 A** `연소득 5천·신규 없음·기존 연 2천`:
  - 신규 P=0 → STEP 0에서 New_year=0. Total_year = 0 + 20,000,000 = **20,000,000**.
  - DSR = 20,000,000 / 50,000,000 × 100 = **40.0%**. (간편 경로 = 과제 (A) 방식 그대로 수용됨을 검증)
- **앵커 B(대표)** `연소득 6천·신규 3억/4.5%/360개월/원리금균등·기존 0`:
  - r = 4.5/12/100 = 0.00375. (1+r)^360 = **3.8476980**(엔진 실측 기준. 초기 손계산의 3.847721 근사는 오차가 있어 폐기).
  - M = 300,000,000 × 0.00375 × 3.8476980 / (3.8476980 − 1) → **엔진(`calculateLoan`, Math.round) 실측 = 1,520,056원**.
  - New_year = 1,520,056 × 12 = **18,240,672원**. Total_year = 18,240,672 + 0 = 18,240,672.
  - DSR = 18,240,672 / 60,000,000 × 100 = 30.401…% → **약 30.4%**. (금액은 엔진 실측값, DSR 대표값 30.4%는 손계산·엔진 동일)
- **앵커 C** `연소득 5천·신규 2억/4.5%/360개월/원리금균등·기존 연 1천`:
  - 원금 2억은 3억의 2/3 → M ≈ 1,520,056 × (2/3) ≈ **1,013,369원**(엔진 실측 반올림). New_year = 1,013,369 × 12 ≈ **12,160,424원**(엔진 Math.round 기준).
  - Total_year = 12,160,424 + 10,000,000 ≈ **22,160,424원**.
  - DSR = 22,160,424 / 50,000,000 × 100 = 44.32…% → **약 44.3%**. (은행권 40% 초과 → "실제 심사 시 은행권 한도를 넘을 수 있음"을 안내하되 승인 여부 단정 금지)
- **앵커 D** `연소득 0`: STEP 0에서 `Y ≤ 0` → 계산 종료, DSR 미산출. 결과 카드 대신 "연소득을 올바르게 입력하세요" 안내(§7-2·§9).

> **라운딩 전제(검증)**: 신규 월상환액 M·New_year는 **대출이자 계산기와 동일한 엔진(`lib/loan.ts` `calculateLoan`, Math.round)**을 공유하는 것을 전제로 엔진 실측값을 고정하고 "약" 표기했다(원 단위 처리 방식이 대출이자 계산기와 어긋나면 CS·회귀테스트 불일치 발생 → §개발팀 전달 이슈). DSR%는 소수 1자리 반올림. 개발팀은 이 표를 "산식 검증용 회귀 테스트 값"으로 그대로 고정할 것. **모든 앵커는 원리금균등·세전 연소득·스트레스 금리 미가산 기준 — 마스터가 산정방식을 바꾸면 값 재계산 필요.** (2026-07-15 QA 독립검증에서 앵커B 금액을 엔진 실측 1,520,056 / 18,240,672로 정정 완료.)

## 2. interpretation (결과 해석 텍스트, 5단 구조)

`lib/calculators.ts`의 `interpretation` 필드 초안. 1문단(결과 요약)은 §1-4 앵커B(연소득 6천·신규 주담대 3억·원리금균등) 기준이며, 실제로는 **사용자 입력값에 따라 동적 생성**되어야 한다(기존 라이브 계산기와 동일, §8-6 개발 전달). 무효입력(앵커D)·간편경로(앵커A)·초과(앵커C) 케이스는 문안 분기 필요(§8-6).

```
연소득 6,000만원, 신규 주택담보대출 3억원(연 4.5%, 360개월, 원리금균등상환) 기준으로 이 대출의 월 상환액은 약 1,520,056원, 연 원리금상환액은 약 18,240,672원이며, 다른 기존 대출이 없다면 DSR은 약 30.4%입니다.

DSR(총부채원리금상환비율)은 '모든 대출의 연간 원리금 상환액 합계 ÷ 연소득 × 100'으로 계산합니다. 이 계산기는 신규 대출의 연 원리금을 원리금균등상환 방식으로 자동 계산하고, 여기에 이미 있는 대출의 연간 원리금 상환액을 더해 연소득으로 나눕니다. 일반적으로 은행권은 DSR 40%, 제2금융권은 50%를 기준선으로 두는 경우가 많은데, 위 예시의 30.4%는 은행권 기준선 40%를 넘지는 않지만 여유가 크지는 않은 편입니다.

다만 이 값은 단순 DSR이며 실제 대출 심사 결과가 아닙니다. 은행은 심사 시 '스트레스 DSR'에 따라 향후 금리 상승분을 더한 스트레스 금리로 원리금을 다시 계산하므로 실제 산정 DSR은 이보다 높게 잡힐 수 있고, 소득 인정 방식·대출 종류별 원리금 산정 기준·은행별 내부 기준도 제각각입니다. DSR 규제 비율과 스트레스 DSR의 적용 범위는 지역과 시점에 따라 달라지고 정책도 수시로 바뀌므로, 이 계산기 결과만으로 대출 가능 여부를 판단하지 마세요.

정확한 DSR 규제 기준과 스트레스 DSR 적용 여부는 금융위원회·금융감독원(금융소비자포털 파인, fine.fss.or.kr)과 실제 대출을 취급하는 금융기관에서 확인하시기 바랍니다.

입력하신 값은 브라우저 내에서만 계산에 사용되며 서버로 전송·저장되지 않습니다. 본 계산 결과는 대출 상담이나 심사 결과가 아닌 단순 참고용 수치이며, 실제 대출 승인·한도는 은행별·차주별 심사 기준에 따라 달라지므로 정확한 사항은 반드시 해당 금융기관에서 확인하시기 바랍니다.
```

- 글자 수: 약 690자(공백 포함, 500~1000자 충족).
- 5단 구조 매핑: ①결과요약(동적, 앵커B) → ②계산기준/공식·규제 기준선 안내 → ③핵심 주의(단순 DSR≠심사 DSR, 스트레스 DSR, 정책 변동·판단 금지) → ④공식 출처(금융위·금감원 파인) → ⑤개인정보(브라우저 내)+면책(심사결과 아님, 은행별 상이).
- **SEO 키워드 자연 포함**: "DSR"·"총부채원리금상환비율"·"DSR 계산"·"DSR 40%"·"스트레스 DSR"·"주택담보대출" 자연 배치.
- **등장 수치(6,000만 / 3억 / 4.5% / 360 / 1,520,056 / 18,240,672 / 30.4%)는 §1-4 앵커B(엔진 실측)와 일치**(임의 수치 금지 원칙).
- **2문단 톤(QA 관찰 반영)**: 30.4%는 코드 구간상 "주의 구간(30~40%)"이고 결과카드 동적요약이 "기준선 40%에 가깝습니다"로 표시되므로, 정적 해설도 "안심"으로 기울지 않게 "40%를 넘지는 않지만 여유가 크지는 않은 편"으로 완곡·중립 서술(단정·안심 과잉 금지, 동적 배지와 톤 일치). §7-2 위치 안내 문구·§8-6 동적요약과 정합.
- YMYL-높음 4중 고지: ③(단순 DSR≠심사·스트레스·판단 금지) + ④(공식 출처) + ⑤(참고용·심사결과 아님·은행별 차이) + 상단 배너(§7-3). "지금 대출/한도 조회" 류 상업적 부추김 표현 없음.

## 4. 관련 계산기 링크 추천 (loan 카테고리 §3-3 원칙 최초 정상 충족)

- DSR 계산기가 live 전환되면 `loan` live는 `loan-interest-calculator`·`loan-prepayment-fee`·`dsr-calculator` **3개**. 라인업 §3-3의 "같은 카테고리 2개 + 타 카테고리 1개"를 loan에서 처음으로 온전히 충족할 수 있다.
- **기획팀 권고 relatedSlugs(§8-7 확정)**: `["loan-interest-calculator", "loan-prepayment-fee", "salary-net-calculator"]`
  - `loan-interest-calculator`(loan, **직접 동선 파트너**): "신규 대출의 월 상환액과 총 이자가 궁금하다면 대출이자 계산기로 확인해보세요."
  - `loan-prepayment-fee`(loan, **같은 카테고리 2번째**): "대출을 미리 갚을 때 드는 비용은 중도상환수수료 계산기로 확인해보세요."
  - `salary-net-calculator`(salary, **분모 연소득 파악·타 카테고리 유입**): "DSR의 기준이 되는 소득은 연봉 실수령액 계산기로 함께 확인해보세요."
- 라인업 §3-3(같은 카테고리 우선 + 타 카테고리 유입)을 loan 2 + salary 1로 정확히 충족. 타 카테고리는 금융 인접성이 높은 salary(연봉)를 채택(만나이 등 date보다 관련성 큼).
- **자동 로직 대비**: `getRelatedCalculators()`는 self 제외 후 sameCategory(loan 2개=대출이자·중도상환) + rest 1개로 채워, override 없이도 loan 2개는 확보된다. 다만 3번째 자동 채움이 date 계산기가 될 수 있어 salary 고정을 위해 **override 권고**.
- **역방향 링크 정리(§8-7)**: 이제 loan 3계산기가 모두 live이므로, 중도상환(현재 `["loan-interest", "salary-net", "age"]`)의 3번째 `age-calculator`를 **`dsr-calculator`로 교체**해 loan 내부 순환을 강화하는 재조정을 권고(마스터 판단). 대출이자 계산기 relatedSlugs 지정 여부도 함께 검토.

## 6. 확정한 모델 요약 (개발팀 즉시 참조용)

| 항목 | 확정값 |
|---|---|
| 입력 | 연소득 Y(원, 필수>0) / 신규금액 P(원, 선택) / 신규금리 i(%) / 신규기간 n(개월) / 상환방식(기본 원리금균등) / 기존 연원리금 E_year(원, 선택) |
| 신규 연원리금 | 원리금균등: `M = P·r·(1+r)^n/((1+r)^n−1)`, `New_year = M×12` (r=0이면 M=P/n) / 원금균등: §1-1-b |
| DSR | **`DSR% = (New_year + E_year) / Y × 100`** (소수 1자리 반올림) |
| 간편경로 | P 비움 → New_year=0, 기존만으로 DSR (과제 (A)형 자동 수용) |
| 무효처리 | Y ≤ 0 / NaN → 계산 종료, 안내 메시지("연소득을 올바르게 입력하세요") |
| 규제 기준선 | 은행권 40%·제2금융 50%(참고·해석용, 상수 분리, 판정 아님) |
| 로직 공유 | 신규 월상환·라운딩은 **대출이자 계산기 엔진(`lib/loan.ts` `calculateLoan`) 재사용**(§개발팀 전달 이슈) |
| 검증 앵커(엔진 실측) | A 40.0% · B 30.4%(월 1,520,056/연 18,240,672) · C 44.3% · D 계산불가 |
| YMYL | 높음. 단순 DSR≠심사 DSR·스트레스 DSR·정책 변동·심사결과 아님 4중 면책 |

## 7. 디자인팀에 넘길 확정 요약 (화면 설계용 — 목록 불일치 핑퐁 방지)

§1-0·§6과 1:1 일치. 아래 목록을 **그대로** 화면 설계에 사용한다.

### 7-1. 입력 필드 (6개, 순서 고정)

1. **연소득(세전)** — number(원, 콤마), 필수. placeholder `60,000,000`, 보조문구 "세전 연소득(원)"
2. **신규 대출금액** — number(원, 콤마), 선택. placeholder `300,000,000`, 보조문구 "이번에 새로 받을 대출(없으면 비워두세요)"
3. **신규 대출 연이자율** — number(%, 소수 2자리), 신규금액 입력 시 필수. placeholder `4.5`
4. **신규 대출기간** — number(개월), 신규금액 입력 시 필수. placeholder `360`, 보조문구 "개월 단위(30년=360)"
5. **신규 상환방식** — select 2종("원리금균등"/"원금균등"), 기본 원리금균등. (v1 노출 여부 §8-3 마스터 확정)
6. **기존 대출 연간 원리금상환액** — number(원, 콤마), 선택. placeholder `5,000,000`, 보조문구 "이미 있는 대출의 1년치 원금+이자 합계(여러 건이면 합산)"

### 7-2. 출력 값 (결과 카드)

- **대표값(히어로)**: DSR `NN.N%` — "참고용·심사 결과 아님" 배지/각주 필수. **무효(연소득 0 등)**: 수치 대신 "연소득을 올바르게 입력하세요" 안내.
- **보조값(계산 근거 3~5행)**: (a) 총 연간 원리금상환액 / (b) 신규 연원리금(월상환액 M 포함) / (c) 기존 연원리금 / (d) 연소득 / (e) 규제 기준선(은행권 40%·제2금융 50%) 대비 **위치 안내 문구**("은행권 기준선 40%에 가깝습니다" 등 — "가능/불가" 단정 금지). 구간 톤: 30~40%는 "주의 구간(기준선에 가까움)"으로 안내하고 §2 정적 해설과 톤 일치.
- **규제선 게이지(선택)**: 0~50% 막대에 DSR 위치와 40%/50% 눈금 표시하면 직관적(체류시간 기여). 단 "통과/부결"로 오인될 색상(빨강=거절 등)·문구는 지양, 중립 톤 권장.

### 7-3. 디자인 협의 필요(참고)

- **상단 고지 배너(YMYL-높음, 항상 노출)**: 대출이자 계산기 배너 선례처럼 amber `role="note"`. 카피 예: "DSR 공식에 따른 단순 참고 수치이며 실제 대출 심사 결과가 아닙니다. 은행은 스트레스 DSR 등 별도 기준으로 심사하며, 규제 비율은 지역·시점에 따라 다릅니다." (최종 카피는 디자인 스펙 회신 후 기획팀 확정)
- 월소득만 아는 사용자를 위한 "월소득×12" 보조 입력/안내 추가 여부(디자인 판단).
- 기존 대출 다건 입력(행 추가 UI) 채택 여부 — v1은 단일 합산 필드 제안(§8-4).
- **오인 유발 UI 전면 금지(대출·중도상환 원칙 승계)**: "지금 대출 신청", "한도 조회", "최저금리 조회", "대출 가능" 등 신청·승인으로 오인될 CTA·배너·색상 금지. 버튼은 "계산하기"류 중립 문구만.

## 8. 마스터 확인 필요 결정사항 (기획팀 단독 결정 범위 밖)

### 8-1. [마스터] slug·카테고리 확정 — `dsr-calculator` / `loan`
- 라인업 개념 URL과 코드 slug가 일치(`dsr-calculator`)해 접두 이슈 없음. 등록 승인만 요망.

### 8-2. [마스터/개발, 우선] 입력 방식 확정 — (B)상세형+(A)간편 자동포섭
- **기획팀 채택안**: 입력 6개, 신규 대출은 대출이자 로직 재사용해 연원리금 자동계산, 기존 대출은 "연간 원리금상환액 직접입력" 합산. 신규 비우면 간편(A)형 자동 수용(§1-0-1).
- **확인 요청**: (a) 이 혼합 방식 승인, (b) 개발 부담이 크면 v1을 (A)간편형(연간 총 원리금 직접입력)만으로 축소할지 — 이 경우 계산기 유용성이 크게 떨어짐을 명시하며 (B) 채택 권고.

### 8-3. [마스터/개발] 신규 상환방식 UI 노출 여부
- **기획팀 채택안**: 기본 원리금균등, 원금균등은 대출이자 로직 재사용해 옵션 제공(원금균등 연원리금은 §1-1-b 1년차 기준 근사).
- **확인 요청**: (a) 상환방식 select를 v1에 노출할지, (b) v1은 원리금균등 고정(입력 5개로 축소)하고 원금균등은 후속으로 뺄지. YMYL-높음 오정보 최소화 관점에서 (b)도 무방.

### 8-4. [개발] 기존 대출 입력 형태 — 단일 합산 vs 다건
- v1은 "연간 원리금상환액" **단일 합산 필드**(사용자가 직접 합산) 제안. 다건 분리입력(대출별 원금·금리·기간 → 각 원리금 산정)은 입력 폭증·YMYL 오차 확대로 v1 제외 제안. 마스터/개발 확인.

### 8-5. [개발] 규제 상수 모듈 분리 + 갱신 거버넌스
- 규제 기준선(40%/50%)·스트레스 DSR 안내 문구를 **상수/카피 모듈로 분리**하고, 정책 개편(스트레스 DSR 단계·지역 적용 변경) 시 갱신 체크리스트를 운영 워크플로에 추가 제안(전기요금 §8-5·중도상환 §8-5와 동일 취지). **DSR은 정책 민감도가 계산기 중 가장 높아 갱신 주기 관리가 특히 중요**(로드맵 #12가 "관리 부담 큼"으로 후순위였던 이유).

### 8-6. [개발] 결과 요약 동적 생성 + 케이스 분기
- §2 1문단은 정적 텍스트가 아니라 입력값 기반 **동적 생성**(기존 라이브 계산기 동일). **분기 필요**: (i) 간편경로(신규 없음), (ii) 무효(연소득 0/NaN → 안내), (iii) 기준선 초과 케이스(규제선 대비 위치 문구 변경, 단 "불가" 단정 금지). **구간 톤 정합(QA 반영)**: 결과카드 동적요약이 30~40%를 "기준선 40%에 가깝습니다(주의 구간)"로 표시하므로, 동적 생성되는 1문단·2문단도 같은 구간 톤(안심 과잉 금지)으로 문구를 맞출 것.

### 8-7. [마스터/개발] relatedSlugs 확정 + loan 3계산기 재조정
- **기획팀 권고**: `dsr-calculator`에 `relatedSlugs = ["loan-interest-calculator", "loan-prepayment-fee", "salary-net-calculator"]`(§4).
- **확인 요청**: (a) override 승인, (b) 중도상환(`loan-prepayment-fee`) relatedSlugs 3번째를 `age-calculator`→`dsr-calculator`로 교체해 loan 내부 순환 강화할지, (c) 대출이자 계산기 relatedSlugs 지정 여부.

### 8-8. [디자인] §7-3 배너 카피·규제선 게이지·월소득 보조입력·무효 표시
- 상단 고지 배너 최종 카피, 규제선 게이지 채택·색상 톤(오인 방지), 월소득×12 보조안내, 무효입력 결과 표시 방식은 디자인 스펙 수령 후 기획팀 확정 회신.

| E5 | 신규 대출 금리 0%(무이자) | STEP 2 `M = P/n` 분기(대출이자 계산기 r=0 로직 재사용) |
| E6 | 신규 대출기간 0 또는 음수 | 검증 오류 안내(≥1개월) |
| E7 | DSR > 100% (과다 부채) | 그대로 표시(예: 120.0%) + "연소득 대비 상환액이 매우 큽니다·심사 결과 아님" 안내 |
| E8 | 매우 큰 연소득/대출금액(오버플로) | 콤마 포맷·수치 안정성 확인, 상식 상한 안내(디자인 협의) |
| E9 | 소수점·콤마 혼입 입력 | 파싱 정상화(대출이자·연봉 계산기 입력 파서 재사용 권장) |
| E10 | 원금균등 선택 + 매우 짧은 기간(n<12) | §1-1-b 합산 상한을 n회차까지로 보정(1~n회차 합, 12 초과분 방지) |

## 개발팀 전달 이슈 (마스터 팀장 경유)

1. **[개발, 우선] 대출이자 계산기 원리금 로직 공유 리팩터 필요** — DSR 계산기의 신규 대출 연원리금(월상환액 M·라운딩·r=0 분기·원금균등 회차 로직)은 대출이자 계산기(`loan-interest-calculator`)와 **동일해야** 한다(같은 대출 조건에서 두 계산기 값이 다르면 CS·신뢰도 문제). 대출이자 계산기의 계산 함수(`lib/loan.ts`의 `calculateLoan`)를 **공유 lib로 재사용**해 두 계산기가 import하는 구조를 권고한다. 인라인 중복 구현 금지. 라운딩 정책(Math.round)도 공유해 §1-4 앵커가 엔진과 일치하도록 할 것. (앵커B는 엔진 실측 1,520,056/18,240,672로 확정됨 — 회귀 테스트 기준값으로 고정.)
2. **[마스터/개발, 우선] §8-2 입력 방식((B)+간편 자동포섭) 승인**. 개발 부담 시 §8-3(상환방식 고정)로 축소 가능하나 (A)단독 축소는 유용성 저하로 비권장.
3. **[개발] §1-1 파이프라인 준수** — STEP 0 무효분기(연소득≤0) 필수, 신규 비움 시 New_year=0(간편경로), DSR 소수 1자리 반올림. 규제 기준선은 **계산이 아니라 해석·안내에만** 사용(판정 로직 금지).
4. **[개발] §8-5 규제 상수 분리·갱신 / §8-6 동적요약+케이스 분기 / §9 엣지케이스 전건 처리**.
5. **[디자인] §7 입력 6개/출력 목록 그대로 사용, §7-3 오인 유발 CTA·색상 전면 금지, §8-8 배너·게이지·무효 표시**.
6. **YMYL 리스크(높음) 재강조** — 이 계산기는 로드맵 12개 중 정책 민감도가 가장 높다. **핵심 리스크는 "사용자가 계산 결과를 실제 대출 승인/한도로 오인하는 것"**이다. 이를 (i) 상단 배너, (ii) interpretation 3·5문단, (iii) FAQ 4번, (iv) 결과 카드 "심사 결과 아님" 배지 **4중으로 차단**했고, 규제 기준선은 "일반적으로 ~수준"·스트레스 DSR은 정성 서술로 단정을 회피했다. 규제 상수 갱신 거버넌스(§8-5)를 반드시 운영 워크플로에 포함할 것. 카테고리(`loan`)·YMYL 등급(높음)·확정 로드맵(#12)·컨셉·Next.js 스택은 변경하지 않았다.

- **리서치 방법·한계**: 규제 비율·스트레스 DSR 수치는 2026-07 시점 교차확인값이나, 대출 규제는 지역·시점·대출종류별로 갈리고 수시 개정된다. 따라서 계산 로직에는 넣지 않고(순수 DSR 공식만) 해석·안내용 참고값으로만 쓰며 "일반적으로 ~수준"·"변동 가능" 완곡 표기로 처리했다.
