## 2. 조견표 12행의 heading + list 분할 방식

표 렌더러가 없으므로 마크다운 표 대신 **heading 3개 + list(ordered:false) 3개**로 쪼갠다. 분할 축은 **입사연도**다(연차별이 아니라).

| 섹션 heading | 담는 행 | 행 수 |
| --- | --- | --- |
| "2020~2021년 입사자 만근일" | 2020-01-01, 2020-03-02, 2021-04-01, 2021-07-01 | 4 |
| "2022~2023년 입사자 만근일" | 2022-01-03, 2022-09-01, 2023-02-01, 2023-06-15 | 4 |
| "2024~2025년 입사자 만근일" | 2024-01-02, 2024-02-29, 2024-03-04, 2025-01-02 | 4 |

- 연차(1/3/5/10년)를 heading 축으로 잡으면 같은 사람이 4개 섹션을 오가며 자기 줄을 4번 찾아야 하므로, **입사연도 축 + 한 줄 안에 4개 마일스톤 전부**를 선택했다. 사용자는 자기 입사연도 섹션 하나만 보면 끝난다.
- 한 list 항목의 리터럴 형식은 전 행 동일: `"{입사일} 입사 → 1년 {만근일}({재직일수}일) / 3년 ... / 5년 ... / 10년 ..."`. 읽기 규칙은 표 앞 "표를 읽기 전에" 섹션 list 3번째 항목에서 미리 설명한다.
- 12행 × 4마일스톤 = 48개 값 전부 마스터 앵커 그대로. 창작·보간 없음.

## 3. 선정 근거

- **검색 의도 전담**: "입사일 1년 채우는 날", "근속 3년 되는 날 계산", "2월 29일 입사 1년" 류는 개념 설명이 아니라 **날짜 값 lookup** 의도다. 형제글(원리 해설)로는 만족되지 않는 갭이라, service-period-calculator의 두 번째 전용 랜딩으로 이 글이 전담한다.
- **비자명 인사이트 4종**(전부 앵커 파생): (1) 만근일 = 응당일 −1일, (2) 윤일 입사자의 1년 만근일은 2025-02-28로 예외 없이 떨어짐, (3) 같은 만 1년인데 재직일수 365 vs 366 / 10년 3652 vs 3653, (4) 만근일 하루 전은 0년 11개월 27~30일로 뚝 떨어짐. 단순 나열형 조견표와 차별화되는 지점.
- **내부링크 3종**: `service-period-calculator`(주 CTA, 사용 절차 뒤) + `dday-calculator`·`age-calculator`(하단) — 지시된 relatedCalculatorSlugs와 정확히 일치, 전부 date 클러스터. severance/annual-leave 계산기는 카니벌·YMYL 회피 목적으로 의도적으로 제외했고 본문에서도 링크하지 않는다.

## 4. 카니벌 회피 근거 요약

형제글 `service-period-guide-2026`("근속기간 계산법 완전정리")이 이미 깊게 다루는 4개 영역을 이 글은 **한 문장도 재설명하지 않는다**.

| 형제글 영역 | 이 글의 처리 |
| --- | --- |
| 재직일수 / 근속기간 / 소수 근속연수 3값 구분 | 설명 없음. 도입 info callout에서 형제 글로 defer |
| +1 원리(양 끝 포함)의 상세 설명 | 원리 설명 없음. "양쪽 다 세는 값이라 365 또는 366" 한 줄의 **결과**만 언급 |
| 윤년이 왜 하루를 더하는지의 원리 | 원리 설명 없음. "사이에 2월 29일이 들어 있는지에 따라 갈린다"는 **표 안 두 줄의 비교**로만 처리 |
| "소수 근속연수 1년 ≠ 만 1년" 함정 | 언급 없음. 이 글은 소수 근속연수를 계산기 출력 항목으로 1회 나열만 하고 해석하지 않음 |

- 도입 info callout이 4개 영역을 **이름으로 열거하며 형제 글로 넘긴다** → 검색엔진·독자 양쪽에 역할 분담이 명시된다.
- 이 글의 고유 자산은 **48개 날짜 값**이며, 형제글에는 이 표가 없다. 중복 본문 0.
