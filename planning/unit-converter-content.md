- **YMYL 등급: 없음(저리스크)** — 마스터 결정 승계. 의료·금융·법률 조언 요소가 전혀 없고, 클라이언트단 순수 곱셈/오프셋 변환 로직이다. 연봉·대출·BMI와 달리 "오정보로 인한 실질 피해" 경로가 없어 면책은 필수가 아니나, 사이트 통일성을 위해 결과 해석 말미에 1줄만 권장(§2).
- **구조 결정(중요·변경 금지)**: 이번엔 "평형별/단위별 롱테일 페이지를 프로그래매틱으로 대량 생성"하지 **않는다**. **단일 인터랙티브 다중단위 변환기 페이지 1개**만 만든다. 대량 URL 생성은 사이트맵 규모·크롤 예산·"얇은 콘텐츠" 판정에 영향을 주는 사안이라 사용자(마스터) 승인이 필요하며 이번 범위에서 제외한다. 이 페이지는 기존 계산기와 동일한 단일 템플릿(계산기 UI + 결과 해석 500~1000자 + FAQ 3~5개 + 관련 계산기 링크)을 그대로 따른다.
- 지원 카테고리는 마스터 확정대로 **5종(길이·무게·넓이·부피·온도)** 고정. 넓이의 **평↔㎡ 부동산 환산을 대표(히어로) 유스케이스**로 강조한다.

## 1. 단위별 변환 계수 (개발팀·디자인팀 검증 앵커 — 정확한 수치)

### 1-0. 변환 방식 2가지 (구현 구조 결정에 영향)

- **곱셈형(길이·무게·넓이·부피)**: 각 단위를 "기준 단위(base)로의 배율(factor)"로 정의하고, `기준값 = 입력값 × from_factor`, `결과값 = 기준값 ÷ to_factor`로 변환한다. 단일 계수표 하나로 카테고리 내 모든 조합(N×N)을 처리할 수 있다.
- **오프셋형(온도)**: 온도는 0점이 단위마다 달라 **단순 배율(곱셈)로 변환 불가**하다. 반드시 오프셋을 포함한 공식으로 개별 처리해야 한다(§1-5). 개발팀은 온도만 곱셈형 계수표에서 분리해 별도 함수로 구현할 것.

### 1-1. 길이(length) — 기준 단위 = m

| 단위 | 표기 | m 기준 배율(factor) | 근거 |
|---|---|---|---|
| 밀리미터 | mm | 0.001 | 정의 |
| 센티미터 | cm | 0.01 | 정의 |
| 미터 | m | 1 | 기준 |
| 킬로미터 | km | 1000 | 정의 |
| 인치 | inch | **0.0254** | 1 inch = 2.54 cm(국제 정의값) |
| 피트 | ft | **0.3048** | 1 ft = 12 inch |
| 야드 | yard | **0.9144** | 1 yard = 3 ft |
| 마일 | mile | **1609.344** | 1 mile = 1760 yard |

### 1-2. 무게(weight) — 기준 단위 = g

| 단위 | 표기 | g 기준 배율(factor) | 근거 |
|---|---|---|---|
| 밀리그램 | mg | 0.001 | 정의 |
| 그램 | g | 1 | 기준 |
| 킬로그램 | kg | 1000 | 정의 |
| 톤 | t | 1000000 | 1 t = 1000 kg |
| 온스 | oz | **28.349523125** | 국제 상용 온스 |
| 파운드 | lb | **453.59237** | 1 lb = 16 oz |
| 근 | 근 | **600** | 한국 육류 기준 1근 = 600 g (§3 FAQ에서 채소·과일 375g 관행 고지) |

### 1-3. 넓이/면적(area) — 기준 단위 = ㎡  ★대표 유스케이스

| 단위 | 표기 | ㎡ 기준 배율(factor) | 근거 |
|---|---|---|---|
| 제곱미터 | ㎡ | 1 | 기준 |
| 평 | 평 | **400/121 = 3.305785123966942…** | 1평 = 400/121 ㎡ (즉 1㎡ = 121/400 = 0.3025평) |
| 제곱킬로미터 | ㎢ | 1000000 | 1㎢ = 100만 ㎡ |
| 헥타르 | ha | **10000** | 1 ha = 10000 ㎡ |
| 제곱피트 | ft² | **0.09290304** | 1 ft² = 0.3048² ㎡ |

- **평은 반드시 400/121 상수(≈3.3057851)를 코드에 그대로 넣을 것.** 3.3058이나 3.3으로 반올림한 상수를 하드코딩하면 33평·84㎡ 등 대표 예시에서 검증값(§1-6)과 어긋난다. 나눗셈 상수 400/121을 부동소수로 계산해 factor로 쓰거나, 평↔㎡는 `× 400 ÷ 121` / `× 121 ÷ 400`로 직접 처리 권장.

### 1-4. 부피(volume) — 기준 단위 = L

| 단위 | 표기 | L 기준 배율(factor) | 근거 |
|---|---|---|---|
| 밀리리터 | mL | 0.001 | 정의 |
| 리터 | L | 1 | 기준 |
| 세제곱미터 | ㎥ | 1000 | 1㎥ = 1000 L |
| 컵 | 컵 | **0.2** | 한국 요리 기준 1컵 = 200 mL |
| 갤런 | gallon | **3.785411784** | 미국 갤런(US liquid gallon) |

- 컵·갤런은 **국가·용도별로 값이 여러 개**다(한국 계량컵 200mL vs 미국 컵 236.6mL, 미국 갤런 3.785L vs 영국 갤런 4.546L). 본 계산기는 마스터 앵커대로 **컵 = 200mL(한국), 갤런 = 미국 갤런 3.785L**로 고정하고, 이 기준을 단위 라벨과 FAQ에 명시해 혼동을 차단한다(§3 FAQ). 라벨 표기 권장: "컵(200mL)", "갤런(미국)".

### 1-5. 온도(temperature) — 오프셋 변환(곱셈 불가) — 기준 단위 = ℃

곱셈형 계수표에 넣지 말고 아래 공식으로 개별 구현한다.

```
입력을 먼저 ℃로 환산:
  ℉ → ℃ :  ℃ = (℉ − 32) × 5/9
  K  → ℃ :  ℃ = K − 273.15
  ℃ → ℃ :  그대로

℃를 목표 단위로 환산:
  ℃ → ℉ :  ℉ = ℃ × 9/5 + 32
  ℃ → K  :  K = ℃ + 273.15
  ℃ → ℃ :  그대로
```

- **왜 곱셈이 아닌가**: 길이·무게는 0점이 공통(0cm = 0inch)이라 배율만으로 변환되지만, 온도는 0점이 단위마다 다르다(0℃ = 32℉ = 273.15K). 그래서 "×배율"이 아니라 "×기울기 + 오프셋"이 필요하다. 개발팀이 온도를 다른 카테고리처럼 factor 곱셈으로 처리하면 32℉(=0℃)가 아니라 0℉로 나오는 등 전면 오류가 난다 — **온도는 별도 함수 필수**.
- **켈빈 하한 검증(입력 유효성)**: 절대영도(0 K = −273.15℃) 미만은 물리적으로 불가능하다. K 입력 음수 방지, ℃ 입력 −273.15 미만 시 경고 처리를 권장(§7 개발 전달). 저리스크 계산기이므로 하드 에러보다는 "물리적으로 불가능한 온도입니다" 안내 수준으로 충분.

### 1-6. 검증용 기준값 (개발팀이 엔진 출력과 대조 — 소수 둘째 자리 반올림 기준)

과거 연봉 계산기에서 해설 예시 수치가 엔진값과 안 맞아 CS 이슈가 났던 전례에 대응해, 결과 해석·FAQ에 등장하는 모든 예시를 아래 표로 고정한다. **표시 반올림 = 소수 둘째 자리 반올림(round)** 기준이며, 개발팀은 엔진 출력이 이 표와 일치하는지 대조할 것.

| 카테고리 | 변환 | 정확값 | 표시값(소수 둘째 자리 반올림) |
|---|---|---|---|
| 길이 | 1 inch → cm | 2.54 | **2.54 cm** |
| 길이 | 1 ft → cm | 30.48 | **30.48 cm** |
| 길이 | 1 mile → km | 1.609344 | **1.61 km** |
| 무게 | 1 oz → g | 28.349523125 | **28.35 g** |
| 무게 | 1 lb → g | 453.59237 | **453.59 g** |
| 무게 | 3 근 → kg | 1.8 | **1.80 kg** |
| 면적 | 1 평 → ㎡ | 3.305785124 | **3.31 ㎡** |
| 면적 | 33 평 → ㎡ | 109.09090909… | **109.09 ㎡** |
| 면적 | 84 ㎡ → 평 | 25.41 (84 × 0.3025) | **25.41 평** |
| 면적 | 59 ㎡ → 평 | 17.8475 | **17.85 평** |
| 부피 | 1 갤런 → L | 3.785411784 | **3.79 L** |
| 부피 | 1 L → 컵 | 5 | **5.00 컵** |
| 온도 | 100 ℃ → ℉ | 212 | **212.00 ℉** |
| 온도 | 70 ℉ → ℃ | 21.111… | **21.11 ℃** |
| 온도 | 25 ℃ → K | 298.15 | **298.15 K** |

- **주의(표시 반올림 정책 미결)**: 위 "소수 둘째 자리 반올림"은 **결과 해석 예시 문구용** 표기 규칙이다. 실제 계산기 결과 필드의 표시 규칙은 카테고리 특성상 고정 2자리로는 부족하거나 과하다(예: 1 mm → km = 0.000001 km가 2자리 반올림 시 `0.00 km`로 소멸, 반대로 1 km → mm = 1,000,000 mm는 소수 불필요). **표시 자리수 정책은 개발·디자인 확정 사안(§7-1)**으로 넘긴다. 결과 해석 본문(§2)의 예시 수치만 위 표대로 고정하면 CS 대조에는 충분하다.

## 2. interpretation (결과 해석 텍스트, 5단 구조)

`lib/calculators.ts`의 `interpretation` 필드 초안. 1문단(결과 요약)은 대표 유스케이스인 평↔㎡(33평 예시) 기준이며, 실제로는 **사용자가 선택한 카테고리·단위·입력값에 따라 동적 생성**되어야 한다(기존 라이브 계산기와 동일 처리, §7 개발 전달).

```
예를 들어 넓이 33평을 제곱미터로 변환하면 약 109.09㎡가 됩니다(1평 = 400/121 ㎡ ≈ 3.31㎡ 기준). 이 계산기는 길이·무게·넓이·부피·온도 5개 분야의 단위를 상호 변환해 줍니다.

각 단위는 국제 표준 정의값을 그대로 사용합니다. 길이는 1인치 = 2.54cm, 1피트 = 30.48cm, 1마일 ≈ 1.61km로, 무게는 1온스 ≈ 28.35g, 1파운드 ≈ 453.59g, 1근(한국 육류 기준) = 600g으로 변환합니다. 넓이는 부동산에서 가장 많이 쓰는 평과 제곱미터를 1평 = 약 3.31㎡(1㎡ = 0.3025평)로 환산하며, 부피는 1컵 = 200mL(한국 요리 기준), 1갤런 = 약 3.79L(미국 갤런)로 계산합니다.

실생활에서는 부동산 면적(평↔㎡), 모니터·타이어·해외 직구 사이즈(인치↔cm), 해외 날씨나 오븐 레시피 온도(화씨↔섭씨)를 확인할 때 특히 유용합니다. 예를 들어 전용면적 84㎡ 아파트는 약 25.41평, 화씨 70℉는 약 21.11℃에 해당합니다.

온도는 길이·무게와 달리 단순히 배율을 곱하는 방식이 아니라, 섭씨·화씨·켈빈의 기준점(0점)이 서로 달라 오프셋을 포함한 공식(℉ = ℃ × 9/5 + 32, K = ℃ + 273.15)으로 변환합니다. 그래서 0℃가 32℉, 273.15K로 나오는 것입니다.

모든 계산은 브라우저 안에서 즉시 처리되며 입력값은 서버로 전송·저장되지 않습니다. 변환 결과는 일상 참고용이며, 정밀 측정·공식 서류에는 해당 기관의 기준값을 확인하시기 바랍니다.
```

- 글자 수: 약 640자 내외(공백 포함, 500~1000자 범위 충족).
- 5단 구조 매핑: ①결과요약 예시(33평 → 109.09㎡, 동적 생성) → ②계산기준/정의값 설명(5개 카테고리 대표 계수) → ③실생활 유스케이스 팁(부동산·직구·날씨/레시피 + 84㎡·70℉ 실측 예시) → ④온도의 오프셋 원리(왜 곱셈이 아닌지) → ⑤개인정보 처리(브라우저 내 계산) + 1줄 면책(정밀·공식용은 기관 기준 확인).
- 등장 수치(109.09㎡, 3.31㎡, 2.54cm, 30.48cm, 1.61km, 28.35g, 453.59g, 3.79L, 25.41평, 21.11℃)는 전부 §1-6 검증표와 일치시켰다 — 개발팀 엔진 대조 시 이 값들이 그대로 나와야 한다.

## 4. UI 입력·표시 제안 (참고용 — 구조 확정은 §7, 디자인팀 사안)

## 5. 관련 계산기 링크 추천

라인업 §3-3 원칙(같은 카테고리 2 + 다른 카테고리 1, 총 3개)을 이번엔 부분 충족한다. `life` 카테고리 live 계산기는 `bmi-calculator`와 이번 `unit-converter` 2개뿐이라, "같은 카테고리 2개"를 온전히 채울 다른 life 계산기가 부족하다.

- `getRelatedCalculators()`는 `status === "live"` 필터 + "같은 카테고리 우선 후 나머지, 배열 순서" 로직이다(연봉 기획안 §5에서 코드 확인 완료, status 필터 적용 확인됨). 이 페이지가 live 전환되면 같은 `life`의 **BMI 계산기**가 우선 채워지고, 나머지 2슬롯은 배열 순서상 다른 카테고리(만 나이 계산기 → D-Day 계산기 등)로 자동 채워진다.
- 기획팀 선호 배치: **BMI 계산기(같은 life) + 만 나이 계산기 + D-Day 계산기**. 앵커 문구 예시 — "생활 속 다른 계산도 필요하다면 BMI 계산기를 확인해보세요" / "날짜 계산이 필요하면 만 나이 계산기, D-Day 계산기도 함께 이용해보세요".
- 실제 링크 선정은 `getRelatedCalculators()` 자동 로직이 처리하므로 위는 참고 제안이며, 순서 강제는 요청하지 않는다(현 자동 결과도 라인업 원칙에 부합).

## 7. 마스터 확인 필요 / 타팀 이관 (기획팀 단독 결정 범위 밖)

### 7-1. [개발·디자인] 결과 표시 자리수(반올림) 정책 — 결정 필요
- **쟁점**: §1-6의 "소수 둘째 자리 반올림"은 **결과 해석 예시 문구용**이다. 실제 결과 필드는 카테고리·배율 차이가 커서 고정 2자리로는 부적절하다. 예: 1 mm → km = 0.000001(2자리 시 `0.00`으로 소멸), 반대로 큰 단위→작은 단위는 정수라 소수 불필요.
- **기획팀 제안(택1 협의)**: (A) **유효숫자 기반**(예: 유효숫자 6자리 + 불필요한 말미 0 제거) — 대소 배율에 강건, 권장. (B) 소수 최대 4~6자리 + 말미 0 제거. 어느 쪽이든 결과 해석 본문의 예시 수치(§1-6)는 소수 둘째 자리로 고정하므로 CS 대조에는 영향 없음.
- 확정 필요 주체: 개발(로직)·디자인(표시 폭) 협의 사안. 표시 정책이 확정되면 §1-6 예시와의 정합만 재확인.

### 7-2. [마스터/디자인] 진입 기본 카테고리 = "넓이(평↔㎡)"로 둘지
- 마스터가 평↔㎡를 대표 히어로 유스케이스로 지정했으므로, 페이지 진입 시 기본 선택 탭을 **넓이**로 두어 평↔㎡ 변환기가 첫 화면에 노출되도록 제안한다. 다만 첫 진입 기본값은 사용자 체감·SEO(길이 변환 검색량이 더 클 가능성)에 영향을 줄 수 있어 마스터 확인 사안으로 남긴다.

### 7-3. [디자인] 다중 단위 동시 표시(1:N 표) 채택 여부
- §4의 "1개 입력 → 카테고리 내 전 단위 동시 표시" 방식 채택 여부는 디자인 레이아웃 사안(무스크롤 원칙 대 정보량). 단일 페이지 내 표시 방식이라 URL/사이트맵과 무관.

### 7-4. [개발] 온도 = 오프셋 별도 함수 구현 필수 (구현 주의)
- §1-5대로 온도는 곱셈형 factor 테이블에서 **반드시 분리**해 오프셋 공식으로 구현. factor 곱셈으로 처리 시 0℃→0℉ 같은 전면 오류 발생. 켈빈 하한(0 K = −273.15℃) 미만 입력 방지 안내도 함께 요청.

### 7-5. [개발] 평 상수는 400/121로 정밀 처리
- §1-3대로 평 factor는 3.3058 등 반올림 상수 하드코딩 금지, 400/121(또는 `×400÷121` / `×121÷400`)로 처리. 33평·84㎡ 검증값 정합의 전제.

### 7-6. [개발] 결과 요약 문구 동적 생성
- §2 1문단(33평 예시)은 정적 텍스트가 아니라 사용자 선택 카테고리·단위·입력값 기반으로 동적 생성되어야 함(기존 라이브 계산기와 동일 처리).

### 7-7. [마스터·구조] 롱테일 대량 페이지 생성은 이번 범위 제외(재확인)
- 서두 구조 결정대로 "평형별/단위별 프로그래매틱 대량 페이지"는 이번엔 만들지 않는다(단일 인터랙티브 페이지 1개). 향후 SEO 확장 차원에서 대량 페이지 생성을 검토하려면 사이트맵 규모·크롤 예산·"얇은 콘텐츠" 리스크가 있으므로 **별도 태스크로 마스터 승인 후** 진행할 것을 제안한다. 이번 문서는 그 결정을 내리지 않는다.
