## 0. 목적 / 범위

계산기 상세 페이지에는 이미 "관련 글" 블록이 구현돼 있다(`RelatedBlogPosts` 재사용, 05:30 실행 완료).
이번 작업은 그 **카테고리 허브 레벨 미러링**이다: `/salary /loan /date /life` 카테고리 페이지
(`components/CategoryPage.tsx`) 하단, 계산기 그리드 아래에 **그 카테고리의 블로그 글** 블록을 신설한다.
목적은 카테고리 허브로 진입한 사용자를 같은 주제(silo)의 블로그 글로 순환시켜 PV·체류시간·애드센스
지면 노출을 늘리고, 카테고리 ↔ 블로그 내부링크를 양방향으로 완성하는 것이다.

- 이 작업은 **순수 내부링크 재배치**다. 신규 카피(설명글/FAQ)·신규 데이터 필드를 만들지 않는다.
- 코드는 개발팀 몫. 이 문서는 헬퍼 설계·배치·문구·빈상태·SEO·헤딩 위계·핸드오프 이슈까지만 다룬다.
- 확정 사항(계산기 허브 컨셉, Next.js 스택, 4개 카테고리 정보구조, `BlogPost` 데이터모델)은 그대로
  두고 그 위에서 확장한다. **정보구조 변경 없음** — 기존 4개 카테고리 재사용.

### 실측 현황 (2026-07-15 기준, 반드시 인지)
- 블로그 4편, 카테고리별 **정확히 1편씩**:
  - salary → `salary-net-income-guide-2026`
  - loan → `dsr-loan-limit-guide-2026`
  - life → `summer-electricity-progressive-rate-2026`
  - date → `age-korean-age-law-2026`
- 4편 모두 `publishedDate: "2026-07-15"`로 **동일** → 발행일 단독으로는 정렬이 결정되지 않는다.
- 따라서 현재는 **어느 카테고리도 빈 상태(0편)가 아니고**, 어떤 상한(3·6)에도 걸리지 않는다.
  이 스펙의 정렬·상한·빈상태 규칙은 전부 **향후 글이 늘어날 때를 위한 방어·확장 규격**이다.

## 1. 헬퍼 설계 (결정 1)

### 1.1 신규 헬퍼 — `getBlogPostsByCategory(category)`
`lib/blog.ts`에 기존 `getBlogPostsForCalculator`를 **미러링**한 순수 헬퍼 1개 추가.

```
getBlogPostsByCategory(category: BlogCategory): BlogPost[]
  = blogPosts
      .filter(p => p.category === category)
      .sort((a, b) =>
        b.publishedDate.localeCompare(a.publishedDate) ||  // 1차: 발행일 내림차순
        a.slug.localeCompare(b.slug))                       // 2차: slug 오름차순(SSG 결정성)
      .slice(0, LIMIT)
```

- 필터: `p.category === category` (계산기 역인덱싱이 아니라 **카테고리 필드 직접 일치** — 더 단순).
- 정렬: 기존 헬퍼와 **완전히 동일** (발행일 desc → tie면 slug asc). 4편이 같은 발행일이므로 지금은
  slug가 실질 정렬키. SSG 빌드 산출물이 재현 가능(결정적)해야 하므로 tie-breaker는 필수.
- **신규 데이터 필드 0.** `BlogPost.category`는 이미 존재(`lib/blog.ts` L6, L24). 신규 필드·역매핑
  테이블 불필요.
- 의존성 방향: `blog.ts` 내부에서 완결. `calculators.ts`를 import하지 않아도 됨(카테고리 리터럴만 사용).

### 1.2 개수 상한 (권장: **6**, 계산기 상세의 3과 의도적으로 다름)
- 계산기 상세는 상한 **3**(단일 도구 관점, 3열 그리드 한 줄). 카테고리 허브는 **6** 권장.
- **6 권장 근거:**
  1. **허브 성격:** 카테고리 페이지는 단일 계산기가 아니라 그 주제 영역 전체를 대표하는 허브다.
     같은 silo의 읽을거리를 더 깊게 모아 보여주는 것이 주제 응집·내부링크 밀도·크롤 깊이에 유리하다.
  2. **시각 정합:** 기존 계산기 그리드가 `lg:grid-cols-3`(3열)이고 관련 글도 동일 3열 그리드 재사용
     (§2.3). 6 = 정확히 **2행**으로 떨어져 시각 리듬이 깔끔하다.
  3. **헤드룸:** 6은 각 카테고리가 **4편 이상**이 되기 전엔 발동하지 않는다. 즉 당장의 레이아웃 변화
     0, 향후 성장 여지만 확보.
- **대안(3):** 카테고리 페이지의 1차 콘텐츠는 어디까지나 계산기 목록이고 블로그는 하단 보조 블록이다.
  블로그가 시각적으로 계산기 목록을 압도하지 않게 3으로 억제하는 선택도 정당하다. → 디자인팀이
  "블로그 블록을 얼마나 부각할지" 관점에서 최종 택1(기본안 6).
- 상한은 **상수 1개**(예: `CATEGORY_RELATED_BLOG_LIMIT`)로 두어 나중에 조정 쉽게. 정렬 후 `slice`.

## 2. 섹션 배치 / 컴포넌트 재사용 (결정 2)

### 2.1 배치 위치 — 계산기 그리드 **아래**
`CategoryPage.tsx` 렌더 순서(위 → 아래):

```
[h1 카테고리 제목 + 설명]        (기존)
```

계산기 목록이 이 페이지의 1차 콘텐츠, 블로그는 "더 읽을거리" 보조 블록이므로 **반드시 그리드 아래**.
(계산기 상세 스펙 §2의 intent-ladder 논리와 동일: 도구 먼저, 읽을거리 나중.)

### 2.2 컴포넌트 재사용 — `RelatedBlogPosts` 그대로
- 카드: 기존 `BlogPostCard`(headingLevel="h3") 재사용. 신규 카드 불필요.
- 래퍼: 기존 `RelatedBlogPosts` **그대로 재사용**. 이미 `posts.length === 0 → null`, 3열 그리드
  (`grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`), 카드 headingLevel="h3"를 갖춰 계산기 그리드와
  시각 리듬이 일치한다. 단, 섹션 헤딩 문구/헤딩레벨은 §3·§5 참조(현재는 h2 "관련 글" 하드코딩).

### 2.3 호출 규격 (개발 참고, 구현은 개발팀 판단)
`CategoryPage.tsx`에서:
```
const relatedPosts = getBlogPostsByCategory(category);
...
<RelatedBlogPosts posts={relatedPosts} />   // (headingText 파라미터화 시 §3 참조)
```
게이팅: 계산기 상세와 달리 카테고리 페이지에는 `status` 게이팅이 없다(페이지 자체가 항상 live).

## 3. 섹션 헤딩 문구 (결정 2b — 디자인팀과 정합 필요)

현재 `RelatedBlogPosts`는 헤딩 문구가 **"관련 글"로 하드코딩**돼 있다. 계산기 상세 페이지에서는
바로 위에 "관련 계산기"가 있어 "관련 글"이 대구를 이뤄 자연스럽다. 그러나 카테고리 페이지에는
"관련 계산기" 형제 섹션이 없고 페이지 전체가 계산기 목록이므로, "관련 글" 단독은 **"무엇에 관련?"**
이라는 맥락이 약해진다.

- **권장(기본안): 카테고리 맥락 전용 문구 + `headingText` prop 파라미터화.**
  - `RelatedBlogPosts`에 **선택적 `headingText` prop**을 추가하고 **기본값을 "관련 글"**로 둔다
    → 계산기 상세의 기존 호출은 **무변경**(하위호환), 카테고리 페이지만 문구를 주입.
  - 카테고리 페이지 주입 문구 권장: **"이 주제의 블로그 글"** (카테고리명 하드코딩 회피, 4개 카테고리
    공통 재사용 가능, 맥락 명확).
    - 대안 A: "이 카테고리의 블로그 글" — 명확하나 "카테고리"가 다소 기능적 단어.
    - 대안 B: 카테고리별 맞춤(예: salary→"연봉·급여 관련 글") — 클릭 유인 최고이나 문자열 4종 관리
      필요·유지보수 부담. 1차에서는 지양, 후속 A/B 후보로 마케팅팀과 별도 논의.
- **대안(최소변경): "관련 글" 그대로 재사용, 컴포넌트 무변경.**
  - 장점: 코드 변경 0, 사이트 전역 어휘 일관("관련 글"이 상세·카테고리 공통). 최속 배포.
  - 단점: 위의 맥락 약화. 다만 바로 위가 계산기 그리드이므로 "이 계산기들과 관련된 글"로 자연 해석될
    여지도 있어 치명적이진 않음.
- **판단 이관:** 헤딩 문구는 디자인팀도 병렬로 결정 중이다. `headingText` 파라미터화 여부와 최종
  문자열은 **디자인팀 결정과 반드시 정합**시킬 것. 기획팀 기본안 = 파라미터화 + "이 주제의 블로그 글".
  디자인이 어휘 일관("관련 글" 유지)을 택하면 컴포넌트 무변경으로 즉시 진행 가능.

## 4. 빈 상태 (결정 3 — 방어 확인)

- 특정 카테고리에 글이 0편이면 `getBlogPostsByCategory`가 `[]`를 반환 → `RelatedBlogPosts`가
  `posts.length === 0`에서 `null` 반환 → **섹션 전체(헤딩 포함) 미표시**. 빈 헤딩·빈 그리드가
  남지 않는다. 기존 컴포넌트 로직으로 이미 충족, 추가 처리 불필요.
- **현재는 4개 카테고리 모두 1편씩이라 빈 상태 케이스가 실제로 발생하지 않는다.** 이 규칙은 방어용.
- 계산기 그리드는 항상 ≥1개(모든 카테고리에 live 계산기 존재)라 §5의 그리드 섹션 헤딩은 항상 표시.

## 5. 헤딩 계층 (결정 5) — ★핵심 발견: 현재 구조는 그대로 두면 위계가 어긋난다

### 5.1 실측된 문제
카테고리 페이지의 현재 DOM 헤딩 구조(`CategoryPage.tsx` 실측):
- `h1` = 카테고리 제목 (L28)
- 계산기 그리드: `CalculatorCard`가 카드 제목을 **`h3`**로 렌더(`CalculatorCard.tsx` L17).
  **그리드 위에 섹션 헤딩(h2)이 없다.**

즉 현재도 이미 **`h1 → h3` 스킵(h2 건너뜀)**이 존재한다. 여기에 이번 "관련 글" 섹션을 `h2`로 붙이면:

```
h1 (카테고리 제목)
  h3 (계산기 카드) × N      ← h2 없이 h3부터 (스킵)
h2 (관련 글)                ← h3들보다 뒤에 h2 등장 (순서 역전)
  h3 (블로그 카드) × N
```

- **문제 1 (기존):** h1 다음 바로 h3 → 한 단계 건너뜀.
- **문제 2 (신규 유발):** 낮은 레벨(h3 카드)이 먼저 나오고 **더 높은 레벨 h2가 뒤에** 나와 문서
  아웃라인 순서가 뒤집힌다. 접근성(스크린리더 아웃라인)·SEO 관점에서 바람직하지 않다.

참고로 **계산기 상세 페이지**는 이 문제가 없다: 거기엔 "관련 계산기"(h2)·"관련 글"(h2) 섹션 헤딩이
있어 `h1 → h2 → h3`로 평탄하다. 카테고리 페이지만 그리드 섹션 헤딩이 없어서 생기는 고유 이슈다.

### 5.2 권장 해법 — 계산기 그리드 위에 섹션 헤딩(h2) 신설
"관련 글" 블록을 붙이기 전에, **계산기 그리드 위에도 `h2` 섹션 헤딩을 추가**해 아웃라인을 정상화한다.
목표 구조:

```
h1 (카테고리 제목 + 설명)
  h2 (계산기 목록 섹션 헤딩)   ← 신설
    h3 (계산기 카드) × N
  h2 (관련 글)                 ← 이번 블록
    h3 (블로그 카드) × N
```

`h1 → h2 → h3 → h2 → h3` 으로 스킵·역전이 모두 해소된다.

- **문구 권장:** 시각적으로는 h1이 이미 카테고리를 설명하므로 눈에 보이는 "계산기" 헤딩이 다소
  중복돼 보일 수 있다. 두 옵션을 디자인팀에 넘긴다:
  1. **시각적 숨김(sr-only) h2** — 예: sr-only "계산기 목록". 화면상 변화 0, 아웃라인만 정상화.
     **기획팀 기본 권장**(레이아웃/디자인 영향 최소, 접근성·SEO 이득만 취함).
  2. **노출 h2** — 예: "이 카테고리의 계산기". 페이지에 명시적 구획을 만들되 h1과 문구가 겹치지
     않게 조정. 디자인이 명시적 구획을 원하면 채택.
- 이는 **기존 정보구조 변경이 아니다**(카테고리 4개 그대로, 계산기 목록 그대로). 오직 마크업 위계
  보정이다. 다만 `CategoryPage.tsx`에 헤딩 요소가 하나 추가되므로 **디자인팀 사인오프 필요**(§7).

### 5.3 대안 (권장하지 않음)
- "관련 글" 헤딩을 h2 대신 낮춰서 순서 역전을 피하려는 시도는 부적절: "관련 글"은 독립 섹션이므로
  h2가 시맨틱상 맞다. 카드(h3)보다 낮출 수 없다.
- 아무것도 안 하고 h1 → h3 → h2를 방치하는 것도 비권장(문제 2가 새로 생김). 최소한 sr-only h2로
  보정할 것.

## 6. SEO / 구조화데이터 (결정 4) — 카테고리 JSON-LD에 블로그 글 **넣지 말 것**

- `buildCategoryJsonLd`(`lib/site-jsonld.ts`)는 현재 **CollectionPage + ItemList** 2노드다.
  ItemList의 각 `ListItem`은 **그 카테고리의 live 계산기 URL**(`/calculator/{slug}`)만 담는다.
  이 ItemList의 의미는 명확히 "이 카테고리가 모은 **계산기 목록**"이다.
- **권장: 이 ItemList에 블로그 글을 섞지 않는다. 신규 구조화데이터도 추가하지 않는다.** 근거:
  1. **의미 오염 방지:** ItemList에 블로그 URL을 섞으면 "계산기 컬렉션"이라는 리스트 의미가 흐려진다.
     Google이 이질적 아이템이 섞인 ItemList를 신뢰도 낮게 볼 수 있다.
  2. **역할 분리:** 블로그 글은 각 상세 페이지에서 자체 `BlogPosting`(계획된 블로그 구조화데이터)로
     표현되는 것이 정석이다. 카테고리 페이지의 CollectionPage는 계산기 컬렉션을 기술하면 충분.
  3. **유지보수:** JSON-LD와 화면 렌더가 각기 다른 목록을 담게 되면 불일치·회귀 위험이 커진다.
     화면엔 관련 글을 보여주되 구조화데이터는 계산기 컬렉션으로 고정하는 편이 안전.
- 결론: **`buildCategoryJsonLd` 무변경.** 화면상 "관련 글" 블록은 순수 UI/내부링크이고 구조화데이터
  대상이 아니다. (블로그의 구조화데이터는 별도 블로그 JSON-LD 건에서 다룬다.)

---

## 7. 리스크 / 콘텐츠(YMYL) — 낮음

- 이 작업은 **이미 게시된 글로의 내부링크 추가**뿐이다. 신규 본문·설명·FAQ를 만들지 않으므로 얇은
  콘텐츠(thin content)를 새로 생성하지 않는다. 애드센스 정책 리스크 **낮음**. 오히려 카테고리 ↔
  블로그 순환으로 내부링크·크롤 경로가 좋아져 SEO에 우호적.
- 링크 대상 글(연봉/DSR/전기요금)은 YMYL 주제지만 각 글이 이미 면책 `callout`을 포함해 게시된
  상태다. 이번 작업이 그 리스크를 추가로 늘리지 않는다(새 주장·수치 도입 없음).
- 유일한 유지보수 주의: 글의 `category`를 바꾸거나 글을 삭제하면 카테고리 관련 글 목록이 자동 반영된다
  (정상 동작, 별도 동기화 불필요).

## 8. 디자인/개발 이관 이슈 (핸드오프)

1. **[디자인+개발] 계산기 그리드 위 h2 섹션 헤딩 신설(§5.2).** 헤딩 위계(h1→h3 스킵 + h2 순서 역전)
   해소를 위해 그리드 위에 h2를 추가해야 한다. **sr-only(시각 숨김)** vs **노출 헤딩** 택1은 디자인팀
   판단. 기획 기본 권장 = sr-only "계산기 목록"(화면 변화 0). `CategoryPage.tsx` 마크업 변경 수반.
2. **[디자인+개발] `RelatedBlogPosts` 헤딩 문구 파라미터화(§3).** `headingText` 선택적 prop 추가
   (기본값 "관련 글" → 상세 페이지 무변경), 카테고리 페이지에 "이 주제의 블로그 글" 주입 권장.
   디자인팀이 "관련 글" 어휘 일관을 택하면 컴포넌트 무변경으로 진행. **디자인 결정과 정합 필수.**
3. **[개발] `getBlogPostsByCategory` 헬퍼 추가(§1)** — 정렬 발행일 desc + slug asc, `slice(0, LIMIT)`,
   LIMIT 상수화(기본 6). 신규 데이터 필드 0.
4. **[개발] `buildCategoryJsonLd` 무변경(§6)** — 관련 글은 구조화데이터에 넣지 않는다.
5. **[개발] 스모크 체크:** 4개 카테고리 각각에서 관련 글 블록에 해당 카테고리 글 1편이 노출되는지,
   (가상) 0편 카테고리에서 블록이 완전히 사라지는지, 아웃라인이 h1→h2→h3→h2→h3인지 확인.

> 큰 방향 전환(데이터모델 변경, 신규 필드 도입, 카테고리 구조/정보구조 변경 등)이 필요하다고 판단되면
> 개발/디자인팀이 임의 결정하지 말고 마스터를 통해 기획팀으로 되돌릴 것(핑퐁).
