## 0. 이자과세율 상수 단일 소스 (개발 필수 전제)

- **이자소득 과세율(마스터 웹검색 확정, 2026-07-16)**:
  - 일반과세 = **15.4%** (이자소득세 14% + 지방소득세 1.4%)
  - 세금우대 = **9.5%**
  - 비과세 = **0%**
- #13 `lib/minimum-wage.ts` 선례처럼 이 세율은 **상수 1곳에서만 관리**한다: 신규 `lib/interest-tax.ts` (예시).

```ts
// lib/interest-tax.ts — 이자소득 과세율 단일 소스 (하드코딩 금지)
export type InterestTaxType = "general" | "preferential" | "taxFree";

export const INTEREST_TAX_RATE: Record<InterestTaxType, number> = {
  general: 0.154,      // 일반과세 15.4% (이자소득세 14% + 지방소득세 1.4%)
  preferential: 0.095, // 세금우대 9.5%
  taxFree: 0,          // 비과세 0%
};

export const INTEREST_TAX_LABEL: Record<InterestTaxType, string> = {
  general: "일반과세",
  preferential: "세금우대",
  taxFree: "비과세",
};
```

- 화면 3곳이 전부 이 상수를 참조한다: ① 과세 세그먼트 버튼 라벨(`INTEREST_TAX_LABEL`) ② 세율 안내 헬퍼(`15.4% / 9.5% / 0%`) ③ 결과 Tier② "이자과세" 행의 적용세율 병기. 세율 개정 시 상수 1곳만 바꾸면 전 화면 갱신.
- 세율 수치는 **버튼 라벨에 직접 박지 말고**(375px 폭·개정 리스크) 상수에서 문자열 조합(`${(rate*100).toFixed(1)}%`)해 노출한다. `15.4%` 문자열이 화면에 3번 이상 등장하므로 하드코딩 금지.

## 1. 전체 레이아웃 구조 (섹션 배치)

계산기 페이지 표준 템플릿을 따른다(위→아래). #13과 동일 골격, 게이트 카드만 부재(§4 참조):

```
[페이지 제목 H1 + 리드 문장]          ← 페이지 셸(계산기 상세 공통)
─────────────────────────────
① 상단 amber 근사 고지 배너 (role="note", 항상 노출)
② 입력 폼 카드
    - [모드 탭] 예금 / 적금        ← 폼 최상단 (탭 성격, §3-1)
    - 그룹 "저축 정보"
        · [금액] 전체 폭 (라벨이 모드 따라 변함)
        · [기간(개월)] · [연이율(%)]  (2열)
        · [이자 계산방식] 단리 / 월복리  (컴팩트 토글, §3-2)
    - 그룹 "이자 과세"
        · [과세 유형] 일반과세 / 세금우대 / 비과세  (세그먼트 3옵션)
    - [계산하기] 버튼
    - 브라우저 내 계산 안내 문구
③ 결과 카드 (계산 후 노출) — 예금/적금 변형 1종, 게이트 없음
    - Tier① 세후 수령액  +  Tier② 보조값 행 + 동적 요약
─────────────────────────────
④ (게이트 부재 명시 — §4)
⑤ 결과 해석 텍스트(interpretation, 500~1000자)   ← calculators.ts
⑥ FAQ 4개                                        ← calculators.ts
⑦ 관련 계산기 3개 링크                            ← 페이지 셸 공통
⑧ 하단 면책 + 공식출처 병기                        ← §5 참조
```

- ②③은 이 컴포넌트가 렌더링. ①은 폼 카드 바깥 최상단. ⑤~⑧은 페이지 셸/`calculators.ts` 담당(기존 계산기와 동일).
- **광고 배치**(대출 계산기 관례 계승): 결과 카드(③)와 해석 텍스트(⑤) **사이**, 그리고 관련 계산기(⑦) **아래**에만. 입력 폼 내부·계산 버튼 주변·대표값 바로 옆 금지(클릭 유도 오인 방지). AdSense 심사 대기 중이므로 실제 유닛은 승인 후 활성화하되 **레이아웃상 광고 자리(placeholder 여백)를 결과-해석 사이에 확보**.

## 2. 입력 필드 레이아웃 상세

폼 카드: `rounded-xl border border-brand-border bg-brand-surface p-4 sm:p-6`, 내부 `<form className="flex flex-col gap-4">`. 그룹 라벨은 `text-xs font-semibold text-brand-text-secondary`(퇴직금 그룹 라벨 패턴). `fieldset/legend`로 그룹핑해 스크린리더 접근성 확보(대출 계산기 L117~120 패턴).

### 필드 순서·폭·문구 초안

| # | 필드 | 폭 | 라벨 | placeholder | 힌트 | 필수 |
|---|------|-----|------|-------------|------|------|
| 0 | 모드 | 전체 폭 | (탭, 라벨 "계산 유형") | — | (예금=목돈 일시 예치 / 적금=매월 납입) | — |
| 1 | 금액 | 전체 폭 | 예금: `예치금 (원)` / 적금: `월 납입액 (원)` | 예금 `10,000,000` / 적금 `500,000` | 모드별로 문구 스위치(아래) | 필수 |
| 2 | 기간 | 2열 좌 | `기간 (개월)` | `12` | `저축(예치) 기간을 개월 수로 입력하세요. 12개월 = 1년.` | 필수 |
| 3 | 연이율 | 2열 우 | `연이율 (%)` | `3.5` | `세전 연 이자율(약정금리)을 입력하세요.` | 필수 |
| 4 | 이자 계산방식 | 전체 폭 | (컴팩트 토글, 라벨 "이자 계산방식") | — | 단리 / 월복리 (§3-2) | — |
| 5 | 과세 유형 | 전체 폭 | (세그먼트, 라벨 "이자 과세") | — | `${세율}%` 적용 안내(§3-3) | — |

- **모드에 따른 라벨/힌트 스위치**(필드 1): 예금 = `예치금 (원)` / placeholder `10,000,000` / 힌트 `한 번에 예치하는 목돈을 입력하세요.` · 적금 = `월 납입액 (원)` / placeholder `500,000` / 힌트 `매월 납입하는 금액을 입력하세요. 총 납입원금은 월납입액 × 개월 수입니다.` **라벨·placeholder·힌트가 모드 상태(state)에 종속되므로 개발은 상수/헬퍼로 분기**.
- **입력 방식**: 금액은 `type="text" inputMode="numeric"` + 천단위 콤마(대출 `formatPrincipalInput` 재사용). 기간·연이율은 대출과 동일하게 `type="number"`(기간 `inputMode="numeric" step="1"`, 연이율 `inputMode="decimal" step="0.01"`). 클래스 `${INPUT_BASE} border-brand-border`(에러 시 `border-brand-warning`).
- **에러 처리**: 대출 계산기와 동일. 필드 하단 `text-xs text-brand-warning` + `role="alert"`, 입력에 `aria-invalid`. 트리거: 금액 미입력/0 이하, 기간 미입력/0 이하/비정수, 연이율 미입력/음수. 상한(비현실적 금액·이율·기간) 정책은 엔진 사안(§9).
- **기본값 설계(토글 부담 완화의 핵심)**: 모드·계산방식·과세를 모두 **합리적 기본값**으로 초기화해, 사용자가 **금액·기간·연이율 3개만 입력하면 즉시 결과**가 나오게 한다. 세 토글을 건드리지 않아도 완결된다 → "토글 과다" 체감 감소. 기본값: 모드 = `적금`(검색량·모집단 우위, §9 기획 확인), 계산방식 = `단리`(국내 예·적금 상품 관례), 과세 = `일반과세`(대다수 상품 기본).

## 3. 토글 3종 UX + 계산 버튼 + 결과 카드

> **핵심 설계 의도(기획 §4-4 "모드 토글이 한 페이지에 과하다" 선제 완화)**: 성격이 다른 3개 토글을 **시각 위계로 분리**해 "토글 벽"으로 보이지 않게 한다.
> - ① 모드(예금/적금) = **탭 성격**(폼 최상단, 가장 큼) — "어느 계산을 하는가"의 문맥 스위치. 옵션 토글이 아니라 계산기 종류 선택처럼 보이게.
> - ② 단리/월복리 = **컴팩트 서브 토글**(대출 년/개월 토글 크기 `h-9`) — 금액·기간·이율 저축정보 그룹에 부속.
> - ③ 과세 = **표준 세그먼트**(별도 "이자 과세" 그룹) — 금액 계산과 분리된 후처리 옵션이라 그룹을 나눔.
> → 사용자가 인지하는 건 "탭 1개 + 옵션 토글 2개"이고, 기본값이 다 잡혀 있어 실제로 만질 필요도 낮다. 3개가 같은 크기로 나란히 서는 "과다" 인상을 회피.

### 3-1. 모드 토글 (예금 / 적금) — 탭 성격, 폼 최상단

- **형태: 세그먼트(탭 성격), 폼 최상단 전체 폭.** 대출 상환방식 세그먼트(L117~146) 구조 계승하되 **약간 크고 강조**되게 하여 "탭"으로 읽히게: 컨테이너 `flex items-stretch gap-1 rounded-lg border border-brand-border bg-white p-1`, 버튼 `flex min-h-11 flex-1 ... text-sm font-semibold`, 선택 `bg-brand-primary text-white` / 비선택 `bg-transparent text-brand-text-secondary`.
- 옵션 2개: **`예금`** / **`적금`** (기본값 `적금`, §9 기획 확인). 각 버튼 아래 서브라벨은 넣지 않고(폭 절약), 헬퍼 1줄로 안내: `예금은 목돈을 한 번에 예치, 적금은 매월 일정액을 납입합니다.`(`text-xs text-brand-text-secondary`, 모드 전환과 무관하게 고정 노출).
- 그룹 라벨(위): `계산 유형` (`legend`, `text-xs font-semibold text-brand-text-secondary`).
- **동작**: 모드 전환 시 (a) 금액 필드 라벨/placeholder/힌트 스위치(§2) (b) **이전 결과 카드는 초기화**(예금 결과를 띄운 채 적금으로 바꾸면 값이 어긋나므로 재계산 유도). (c) 입력값(금액/기간/이율/토글)은 유지해 재입력 부담 최소화.

### 3-2. 이자 계산방식 토글 (단리 / 월복리) — 컴팩트 서브 토글

- **형태: 컴팩트 세그먼트.** 대출 기간단위 토글(L230~258)을 그대로 계승 — `role="radiogroup"`, `type="button" role="radio" aria-checked`, `h-9 flex-1 rounded text-xs font-semibold`, 컨테이너 `flex items-stretch gap-0.5 rounded-md border border-brand-border bg-white p-0.5`. 연이율/기간 2열 아래 전체 폭에 배치(저축정보 그룹의 마지막 요소).
- 옵션 2개: **`단리`** / **`월복리`** (기본값 `단리`). 헬퍼(`text-xs text-brand-text-secondary`): `단리는 원금에만, 복리는 이자에 다시 이자가 붙습니다.`
- 대출의 년/개월과 시각적으로 동일 크기 → 사용자가 "보조 옵션"으로 자연 인지. 모드 토글(3-1)보다 확실히 작게 하여 위계 구분.

### 3-3. 과세 유형 토글 (일반과세 / 세금우대 / 비과세) — 세그먼트 3옵션 [권장안]

- **권장: 세그먼트 3옵션**(드롭다운 대비). 근거:
  1. 사이트 내 소수 이산 옵션은 전부 세그먼트(상환방식·년/개월·단리/복리)라 **일관성** 유지.
  2. 3개는 세그먼트 한계 내(4개 이상이면 드롭다운 고려). 375px에서도 `flex-1` 3등분 + `text-xs`면 `일반과세`(4자)/`세금우대`(4자)/`비과세`(3자) 모두 한 줄에 수용.
  3. 선택지가 **항상 보여** 세율 3종을 한눈에 비교·전환(체류·상호작용↑). 드롭다운은 접혀 있어 탐색 1스텝 추가.
- 컨테이너/버튼: 대출 상환방식 세그먼트와 동일(`min-h-11 flex-1 text-sm font-semibold`), 선택 `bg-brand-primary text-white`. **버튼 라벨엔 세율 숫자 미포함**(폭·개정 리스크) — 라벨은 `INTEREST_TAX_LABEL` 텍스트만.
- **적용세율 헬퍼**(세그먼트 아래, `text-xs text-brand-text-secondary`): 선택된 유형의 세율을 동적 노출 — 예 `일반과세: 이자소득의 15.4%가 세금으로 공제됩니다 (소득세 14% + 지방소득세 1.4%).` / `세금우대: 9.5% 공제.` / `비과세: 세금이 공제되지 않습니다.` 문자열은 §0 상수에서 조합.
- 그룹 라벨(위): `이자 과세` (`legend`).
- 만약 개발 검수에서 375px 3버튼이 빠듯하면 **폴백: `SELECT_BASE` 드롭다운**(3옵션)으로 전환 가능(스코프 내 대안, §9). 1순위는 세그먼트.

### 3-4. 계산 버튼

- 문구: **`이자 계산하기`**
- 스타일: 대출과 동일 `h-12 w-full rounded-lg bg-brand-primary px-5 text-base font-semibold text-white transition-colors hover:bg-brand-primary-hover`.
- 버튼 아래 안내(`text-xs text-brand-text-secondary`): `입력하신 금액·이율 정보는 브라우저 안에서만 계산되며 서버에 저장·전송되지 않습니다.`

### 3-5. 결과 카드 (예금/적금 공통 골격, 변형 1종)

컨테이너: 대출 정상 결과와 동일 `mt-6 rounded-xl border-l-4 border-brand-accent bg-gradient-to-br from-emerald-50 to-white p-4 sm:p-6`. 계산 후 `resultRef.scrollIntoView({behavior:"smooth", block:"nearest"})`(대출 L90~92 관례).

**Tier ① 대표값 — 세후 수령액**
- 상단 라벨(`text-sm font-medium text-brand-text-secondary`): `만기 세후 수령액` (예금·적금 공통. 원금+세후이자 = 실제 손에 쥐는 금액이라 대표값으로 최적)

## 6. 모바일(375px) 뷰포트 레이아웃 고려사항

토글 3개 + 결과 다행 붕괴 여부를 375px 기준으로 명시:

- **모드 탭(3-1)**: `flex-1` 2등분 → 각 버튼 폭 충분(`예금`/`적금` 각 2자). `min-h-11` 터치 타깃 유지. 붕괴 없음.
- **단리/월복리(3-2)**: `flex-1` 2등분, `h-9` 컴팩트. `단리`/`월복리` 짧아 한 줄 수용. 붕괴 없음.
- **과세 3옵션(3-3)**: `flex-1` 3등분 + `text-xs`. `일반과세`(4자)/`세금우대`(4자)/`비과세`(3자)가 375px에서 한 줄에 들어가는지가 **최대 리스크 포인트**. 세율 숫자를 버튼에 넣지 않기로 한 이유가 이것(§0·§3-3). 개발 검수 시 375px에서 글자 말줄임(`…`) 발생하면 즉시 §3-3 드롭다운 폴백으로 전환. 버튼 세로는 `min-h-11` 유지.
- **금액 필드**: 전체 폭이라 문제 없음. 콤마 서식 8자리(`10,000,000`)도 `INPUT_BASE` 폭 내 수용.
- **2열(기간·연이율)**: 둘 다 짧은 숫자(2~4자)라 `grid grid-cols-2 gap-3 sm:gap-4` 유지 자연스러움. 라벨 `기간 (개월)`/`연이율 (%)`는 1줄 유지(퇴직금처럼 `min-h` 강제 불요, 짧음).
- **Tier① 대표값 숫자**: 예금 예치금이 크면 세후수령액이 8자리(수천만원)까지 가능 → `text-4xl`(약 36px) 8자리는 375px 폭(약 343px 내부)에 빠듯. **`flex-wrap items-baseline`으로 `원` 단위가 자연 줄바꿈**되게 처리(대출 L286 동일). 그래도 넘치면 폴백으로 `text-3xl sm:text-[2.5rem]`(모바일만 축소). 1순위는 대출과 동일한 `text-4xl` + wrap.
- **Tier② 보조값 행(5~6행)**: `flex items-center justify-between` 라벨-값 양끝 정렬(AmountRow 기본). 라벨 긴 행(`이자과세 (15.4%)`, `세후 실효수익률`)도 값과 겹치지 않음(값이 짧음). 6행이어도 세로 스크롤 자연 수용, 붕괴 없음.
- **계산 버튼**: `w-full h-12` 풀폭 유지(엄지 도달 영역).
- **광고 자리**: 결과-해석 사이 광고는 모바일 반응형(`display:block`, 폭 100%). 결과 카드와 광고 사이 `mt` 여백 확보해 대표값을 가리지 않게.

## 7. interpretation / FAQ 방향성 (디자인 권고 — 최종 카피는 기획/개발이 `calculators.ts`에 확정)

### 7-1. interpretation (결과 해석 텍스트, 500~1000자) 방향

한 문단 몰아쓰기 금지, 2~3개 소주제로 스캔 가능하게. 담을 내용:
1. **예금 vs 적금, 왜 같은 금리인데 이자가 다른가**: 예금은 전액이 전 기간 이자를 받지만, 적금은 매월 납입분이 남은 개월 수만큼만 이자가 붙어 **실효수익률이 명목금리의 약 절반**이 되는 원리(단리 기준 `n(n+1)/2` 가중). "600만원 넣었는데 이자가 왜 이것밖에" 오해에 답.
2. **단리 vs 복리·과세의 영향**: 복리는 이자에 이자가 붙어 기간이 길수록 유리, 이자소득세 15.4%가 실수령을 얼마나 깎는지(세전≠세후). 세금우대·비과세 상품의 존재.
3. **실효수익률의 의미·활용**: 광고 문구의 "연 O%"와 실제 세후 손에 쥐는 수익률의 차이. 저축 vs 대출상환 의사결정(내부링크: loan-prepayment-fee), 월 저축여력(salary-net-calculator) 연결.
- 톤: 정보전달·중립. 특정 상품·금융사 추천 금지. 세율 15.4%는 §0 흐름과 일치하게 표기.

### 7-2. FAQ 4개 방향

1. **`예금과 적금은 뭐가 다른가요? 이자가 왜 차이 나나요?`** → 일시예치 vs 매월납입, 적금 이자가 예금의 약 절반인 이유.
2. **`이자소득세 15.4%는 어떻게 계산되나요? 세금우대·비과세는요?`** → 소득세 14% + 지방소득세 1.4%, 세금우대 9.5%·비과세 0% 요건 개요(면책 톤).
3. **`단리와 복리 중 뭐가 유리한가요?`** → 기간·금리에 따른 차이, 국내 적금 상당수가 단리 표기라는 점.
4. **`실효수익률이 약정금리보다 낮은 이유는?`** → 적금 이자 가중 + 과세 효과. 계산기 결과와 연결.

- FAQ 렌더는 기존 계산기 FAQ 컴포넌트 관례 그대로 — 디자인 신규 요소 없음.

## 9. 핑퐁 / 확인요청

**기획팀 확인요청**
1. **기본 모드**: 디자인은 `적금`을 기본으로 제안(검색량·모집단 우위, 매월납입이 대중적). 예금 기본이 더 자연스럽다는 판단이면 회신 바람(라벨/힌트 초기 상태만 스위치, 구조 무변).
2. **적금 복리(월복리 누적) 노출 여부**: 기획 §14는 "적금 복리는 월복리 누적 — 개발이 엔진에서 확정"으로 유보. 디자인은 4개 조합(예금·적금 × 단리·복리)을 모두 UI에서 선택 가능하게 그렸다. 만약 적금 복리를 스코프에서 빼면 **적금 모드일 때 단리/복리 토글을 숨기거나 비활성**해야 한다(선택은 되는데 결과가 단리와 같으면 혼란). 적금 복리 지원 여부 확정 요청.
3. **실효수익률 정의**: 세후이자÷원금(예금)·÷총납입원금(적금)의 **단순 세후수익률**로 표기할지, 연환산(APY)까지 갈지. 디자인은 단순 세후수익률(소수 2자리) 기준 요약을 그렸다. 표기 방식 확정 필요(라벨 문구·소수 자릿수 영향).

**개발팀 인계 이슈**
4. **토글 3종 UX 과다 여부(기획 §4-4 선제 대응)**: 디자인은 "탭 1 + 컴팩트 토글 1 + 세그먼트 1"의 **위계 분리** + **기본값 프리셋**으로 "토글 벽" 인상을 완화했다(§3 상단 박스). 그럼에도 구현상 한 폼에 과하다는 반송이 오면, 대안으로 (a) 과세를 드롭다운(`SELECT_BASE`)으로 강등, (b) 단리/복리를 "고급 옵션" 접힘 영역으로 이동 중 택1을 재설계하겠다. 구현 전 판단 회신 바람.

6. **입력 상한/엣지 정책**: 비현실적 금액(수백억)·이율(음수/수백%)·기간(0/비정수/초장기) 클램프·에러 규칙은 엔진 스펙. 디자인은 필드 에러(§2)·이율 0% 정상표시(§4)만 규정했다. 상한 정책 정해 회신 바람.
