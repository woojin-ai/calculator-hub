# /privacy, /terms — 텍스트 페이지 UI 구성안 (v1.0, 2026-07-13)

> 대상: 신규 라우트 `/privacy`(개인정보처리방침), `/terms`(이용약관)
> 콘텐츠(실제 법률 문구)는 기획팀이 `planning/privacy-policy-content.md`,
> `planning/terms-content.md`로 별도 작업 중이며, 이 문서는 콘텐츠가 채워질 **화면 레이아웃
> 틀**만 정의한다. 두 페이지는 성격(약관/법률 문서)이 동일하므로 하나의 공통 템플릿으로 다룬다.

현재 `app/privacy`, `app/terms` 라우트는 존재하지 않는다(확인 완료) → 개발팀 착수 전 디자인팀이
먼저 구성안을 작성하는 정상 순서다. `design/design-system.md`를 그대로 따르며 코드는 작성하지
않는다.

---

## 0. 설계 전제

- 법률 문서 특성상 계산기 페이지처럼 인터랙션·강조 요소가 많을 필요가 없다. **가독성 최우선,
  화려함 최소화**가 원칙(마스터 지시사항 그대로).
- `cs/static-pages-checklist.md` 기준 `/privacy`는 8개 항목, `/terms`는 9개 항목으로 구성되며,
  각 항목이 하나의 섹션(`h2`)이 된다. 항목 수가 많으므로 목차(TOC)를 둔다(3장).
- 두 페이지 모두 "전문가 검토 필요" 표시가 붙은 항목이 있다(체크리스트 참고) — 검토·확정은
  기획팀/마스터 팀장 몫이며 이 문서는 관여하지 않는다.
- 4장(비주얼 리치니스 보강 방향, `design/visual-richness-plan.md`)의 이미지/아이콘 보강
  대상에서 **`/privacy`, `/terms`는 의도적으로 제외한다.** 법률 문서에 장식적 이미지를 넣는
  것은 신뢰감을 높이기보다 오히려 문서의 격식/진지함을 해칠 수 있다고 판단.

## 1. 공통 레이아웃 구조 (와이어프레임)

```
[H1] 개인정보처리방침 / 이용약관
[메타 정보] 시행일자 · 최종 수정일
──────────────────────────────
[목차(TOC)] — 접기/펼치기 가능한 카드
──────────────────────────────
[본문 섹션 1] 제목 + 본문
[본문 섹션 2] 제목 + 본문
...
[본문 섹션 N]
──────────────────────────────
[하단 문의 안내] "궁금한 점은 고객센터에서 문의해주세요" + 링크
[선택: 광고 1개, 7장 참고]
```

컨테이너: `mx-auto max-w-3xl px-4 py-10 sm:px-6 sm:py-14`. 그리드/카드 나열이 없는 순수
텍스트 문서이므로 홈/about의 `max-w-5xl`보다 좁게 잡아 한 줄 길이를 읽기 편하게 제한한다
(16px 본문 기준 적정 가독 폭).

---

## 2. 상단: 제목 + 메타 정보

- H1: "개인정보처리방침" / "이용약관" — `text-2xl font-bold text-brand-text sm:text-[2rem]`
  (기존 Display 패턴 그대로).
- 메타 정보(시행일자·최종 수정일): H1 바로 아래, `mt-2 text-xs text-brand-text-secondary`
  (Caption 스케일). 예: "시행일자 2026-07-13 · 최종 수정일 2026-07-13". 두 날짜가 같으면
  하나만 표기해도 무방(콘텐츠팀 판단).
- 좌측 정렬 고정(about/home처럼 모바일 중앙 정렬 패턴을 쓰지 않는다) — 법률 문서는 격식 있는
  좌측 정렬이 관례이며, 목차·섹션 번호와도 자연스럽게 이어진다.

---

## 3. 목차(TOC)

- 위치: 메타 정보 다음, 본문 섹션 시작 전. `mt-6` 간격.
- 스타일: `rounded-xl border border-brand-border bg-brand-surface p-4 sm:p-5` 카드 하나로
  감싼다(기존 FAQ 아코디언 카드와 동일한 "카드로 묶어 하나의 블록처럼 보이게" 원칙 재사용).
- 카드 제목: "목차" — `text-sm font-semibold text-brand-text mb-2`.
- 목차 항목: `ol` 또는 `ul`로 각 섹션 제목을 나열, 각 항목은 해당 섹션 `id`로 가는 앵커 링크
  (`<a href="#section-1">`, 순수 HTML 앵커, JS 불필요). 스타일: `text-sm text-brand-text-
  secondary hover:text-brand-primary`, 항목 간 `space-y-1`.
- **모바일에서는 기본 접힘 상태로 제공할 것을 권장한다** — `FaqAccordion.tsx`와 동일한
  `<details>/<summary>` 패턴 재사용 가능(질문 대신 "목차 보기"를 summary로). 데스크톱(`sm`
  이상)에서는 펼친 상태로 기본 노출해도 무방(공간 여유가 있음). 접기/펼치기는 순수 HTML
  `<details>` 기본 동작이라 별도 상태 관리(JS) 없이 구현 가능 — 기능 로직 영역이 아니다.
- 목적: 8~9개 섹션을 스크롤로만 훑어야 하는 부담을 줄이고, "이용자의 권리" "제3자 제공 여부"처럼
  특정 조항을 찾는 사용자가 빠르게 이동할 수 있게 한다(법률 문서에서 자주 발생하는 "원하는
  조항을 못 찾겠다"는 유형의 불편을 사전에 방지).

---

## 4. 본문 섹션 스타일

- 각 섹션에 `id`를 부여(TOC 앵커 대상): 예 `id="collected-info"` 등, 개발팀이 항목별로 슬러그
  형태로 정하면 됨(이 문서에서 정확한 slug까지 강제하지 않음).
- 섹션 제목(`h2`): "1. 수집하는 개인정보 항목"처럼 **번호 + 제목** 형식 — `text-lg font-bold
  text-brand-text sm:text-xl mt-8` (첫 섹션은 `mt-6`). 번호를 붙이는 이유: 목차와 1:1로
  대응되어 사용자가 "지금 몇 번째 섹션을 보고 있는지" 스스로 위치를 파악하기 쉬움.
- 본문 문단: `mt-3 text-sm leading-relaxed text-brand-text-secondary sm:text-base` — 기존
  about 페이지 본문과 동일한 컬러 컨벤션(제목만 Text Primary, 본문은 Text Secondary 톤)을
  그대로 따라 사이트 전체 일관성 유지.
- 하위 목록이 필요한 섹션(예: "수집 항목: 쿠키, 접속 로그, IP")은 `mt-2 list-inside list-disc
  space-y-1 text-sm sm:text-base` (about 페이지 목록 스타일 재사용).
- 섹션 간 간격은 제목의 `mt-8`이 자연스러운 구분자 역할을 하므로 별도 구분선(`<hr>`)은 넣지
  않는다(디자인 과잉 방지, 가독성 우선 원칙).
- 강조가 필요한 조항(예: 면책 조항, 준거법)이 있다면 별도 색상 박스보다 **본문 내 굵게
  (`font-semibold`) 처리만 허용**한다. 계산기 결과 카드 같은 컬러 강조 박스를 법률 조항에
  쓰면 "이 조항이 광고/프로모션처럼 보인다"는 오해를 줄 수 있어 지양.

## 5. 하단: 문의 연결

- 위치: 마지막 섹션 다음, `mt-10`.
- 스타일: about 페이지의 "문의 안내 블록"과 동일한 카드 패턴 재사용 — `rounded-xl border
  border-brand-border bg-brand-surface p-5 sm:p-6`.
- 문구 예: "개인정보 처리방침에 대해 궁금한 점이 있으시면 고객센터에서 문의해주세요." (개인정보
  보호책임자 연락처가 본문 섹션에 이미 명시되어 있다면 중복이니 링크만 배치해도 무방 — 콘텐츠팀
  판단)
- 링크: "고객센터로 이동" → `/support`(또는 질문하기로 바로 이동하고 싶다면 `/support#ask`,
  `design/support-page-ui-spec.md` 참고). 버튼 스타일은 about 페이지와 동일하게 **아웃라인**
  (`border border-brand-primary text-brand-primary hover:bg-blue-50`) — 계산 액션용 solid
  버튼과 위계 구분.

## 6. 반응형 요약

| 브레이크포인트 | 정렬 | 목차 | 본문 |
|---|---|---|---|
| 기본(모바일) | 좌측 정렬 고정 | 기본 접힘(`<details>`) | 전체 폭, `text-sm` |
| `sm`(640px~) | 동일 | 펼침 기본 노출 가능 | `text-base`로 확대 |
| `lg`(1024px~) | 동일 | 동일 | 컨테이너 폭(`max-w-3xl`) 그대로 유지, 사이드바 도입 없음 |

## 7. 광고 배치

**권장: `/privacy`, `/terms`에는 광고를 넣지 않는다.**

이유:
- 법률/정책 문서는 사이트의 신뢰도를 담보하는 페이지다. 애드센스 정책상 광고 자체가 금지되는
  것은 아니지만, 이 두 페이지에 광고를 넣으면 "정책 문서를 광고 지면 확보 수단으로 쓴다"는
  인상을 줄 수 있고, 법률 문서 특성상 체류시간 확보 목적의 광고 배치 실익도 낮다(about 페이지
  스펙에서도 동일하게 "저트래픽 정보성 페이지는 광고 우선순위 낮음"으로 판단한 바 있음 — 여기서는
  한 단계 더 나아가 아예 넣지 않는 쪽을 권장).
- 넣더라도 최하단(문의 카드 다음) 1개로 제한하고, `text-xs text-brand-text-secondary` "광고"
  라벨을 명시하며 위아래 24px 이상 이격한다.

**최종 채택 여부는 광고 수익 우선순위에 대한 마스터 팀장 판단이 필요할 수 있다** — 이 문서는
"넣지 않는 것을 권장"까지만 제시하고 강제하지는 않는다.

## 8. 다른 팀 전달 사항

**기획팀 확인 요청**
- `planning/privacy-policy-content.md`, `planning/terms-content.md` 작성 시 섹션 제목을
  "N. 제목" 형식(예: "1. 수집하는 개인정보 항목")으로 정리해주면 개발팀이 목차와 본문 `id`를
  1:1로 매핑하기 쉽다.
- `cs/static-pages-checklist.md`의 "전문가 검토 필요" 항목이 실제 게시 전 검토를 거쳤는지
  최종 확인 필요(디자인팀 범위 밖, 리마인드 차원).

**개발팀 전달 사항**
- `app/privacy/page.tsx`, `app/terms/page.tsx` 신규 구현, 공통 템플릿(1~5장)을 그대로
  따르되 콘텐츠(섹션 개수·문구)만 다르게 채운다. 공통 부분이 많으므로 필요하면
  `components/LegalPageLayout.tsx` 같은 공통 레이아웃 컴포넌트로 추출하는 것도 고려 가능
  (채택 여부는 개발팀 재량 — 기능적으로 동일하면 구조는 개발팀 판단에 맡김).
- 목차 앵커(`<details>` + `<a href="#...">`)는 순수 HTML이라 별도 상태 관리 라이브러리 불필요.
- `SiteFooter.tsx`에 "개인정보처리방침"(`/privacy`), "이용약관"(`/terms`) 링크 추가 필요
  (`design/about-page-ui-spec.md` 7장에서 이미 이 시점에 함께 추가하도록 예고된 사항).

구현 완료 후 실제 화면(콘텐츠 채워진 상태)을 본 스펙과 대조해 검수할 예정이다.
