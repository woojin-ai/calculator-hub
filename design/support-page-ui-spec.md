# 고객센터(/support) — UI 구성안 (v1.0, 2026-07-13)

> 대상: 신규 라우트 `/support` (오늘 마스터 팀장 결정: 헤더 끝에 "고객센터" 신설, FAQ +
> "질문하기" 포함)
> 현재 `app/support` 라우트는 존재하지 않는다(확인 완료) → 개발팀이 착수하기 전에 디자인팀이
> 먼저 구성안을 작성하는 정상 순서다.

본 문서는 `design/design-system.md`를 그대로 따르며, 코드는 작성하지 않는다. 개발팀이 본
문서를 참고해 `app/support/page.tsx`(신규)를 구현한다.

## 0. 현황 확인 (설계 전제)

- `components/SiteHeader.tsx`의 `NAV_ITEMS`는 급여/대출/날짜/생활/블로그 5개뿐이며 고객센터
  링크 없음 → 1장에서 변경안 제시.
- `components/FaqAccordion.tsx`는 `{question, answer}[]`만 받는 범용 컴포넌트로, 특정
  계산기 전용 로직이 없다 → **그대로 재사용 가능**, 신규 컴포넌트 불필요.
- FAQ 문구 자체(콘텐츠)는 `cs/faq-draft.md`(CS팀 초안, 현재 10문항, "전문가 검토 필요"
  표시 일부 있음)가 이미 존재한다. 이 문서는 화면 구성만 다루며, 문구 확정은 CS팀/기획팀
  담당이다.
- `/support`에는 서버/DB가 없다고 가정한다. "질문하기" 폼은 마스터 지시대로 **mailto 기반**
  (서버 저장/자동응답 없음) — 이 제약이 5장 UX 설계의 핵심 전제다.
- **참고(연결 제안):** `design/about-page-ui-spec.md`, `planning/about-page-content.md`에서
  "`/contact`가 아직 없어 `/about`의 문의 버튼이 갈 곳이 없다"는 이슈가 미해결로 남아 있었다.
  오늘 `/support`가 "질문하기"를 갖추게 되므로, `/about`과 `SiteFooter`의 문의 관련 링크를
  `/contact` 대신 `/support#ask`로 연결하면 이 이슈를 별도 `/contact` 페이지 없이 해소할 수
  있다. 이는 새로운 톤앤매너 변경이 아니라 기존에 비어 있던 링크를 채우는 것이므로, 채택 여부만
  마스터 팀장 확인 후 개발팀에 전달할 것을 제안한다(8장 참고).

## 1. 헤더 내비게이션 변경안

**결론: `NAV_ITEMS` 배열 끝에 일반 항목으로 "고객센터"(`/support`) 1개만 추가하면 충분하다.
구분선(divider) 등 별도 시각적 구분은 넣지 않는다.**

이유:
- 현재 5개 항목(급여/대출/날짜/생활/블로그) 중 "블로그"도 계산기 카테고리와 성격이 다른
  콘텐츠 유형이지만 동일한 스타일로 나란히 배치되어 있다. 성격이 다른 메뉴를 구분선 없이
  같은 톤으로 배치하는 것이 이미 이 사이트의 관례이므로, "고객센터"만 별도로 구분하면 오히려
  일관성이 깨진다.
- 구분선을 넣을 만큼 시각적 위계 차이를 줘야 하는 상황(예: 로그인/CTA 버튼과 일반 메뉴 구분)이
  아니다. 고객센터도 탐색 메뉴이지 액션 버튼이 아니다.

**단, 모바일 폭 이슈를 반드시 확인해야 한다 (개발팀 QA 요청 사항):**
- 현재 헤더는 텍스트 링크 5개를 `gap-1 sm:gap-2`, `px-2 sm:px-3`로 한 줄에 배치하는 구조이며
  햄버거 메뉴 등 축약 UI가 없다. "고객센터"(4글자)까지 추가되면 6개 항목이 되어, 375px급
  소형 모바일 뷰포트에서 로고("계산기 허브") + 6개 메뉴가 한 줄에 다 들어가지 않을 가능성이
  높다.
- **권장 안전장치:** `nav` 요소에 `overflow-x-auto whitespace-nowrap` 성격의 처리를 추가해,
  들어가지 않을 경우 메뉴 줄이 가로 스크롤되도록 한다(레이아웃 깨짐/줄바꿈으로 인한 헤더 높이
  변화 방지). 이는 순수 CSS 처리이며 새 컴포넌트나 상태 로직이 필요 없다.
- **구조 변경(햄버거 메뉴 등)은 이번 범위에 포함하지 않는다.** 항목이 앞으로 더 늘어날
  계획이라면 그때는 모바일 전용 축약 메뉴 도입을 별도로 마스터 팀장에게 제안해야 한다(내비게이션
  전체 구조 변경은 톤앤매너 확인 대상).
- 개발팀은 실제 iPhone SE(375px) 등 소형 뷰포트에서 6개 메뉴 + 로고가 줄바꿈 없이 보이는지,
  또는 가로 스크롤로 자연스럽게 처리되는지 반드시 확인할 것.

## 2. 페이지 레이아웃 구조 (와이어프레임)

```
[H1] 고객센터
[리드 문단] + [바로가기 링크 2개: "자주 묻는 질문 보기" / "바로 질문하기"]
──────────────────────────────
[선택: 본문 중간 광고 1개] ← 인트로와 FAQ 사이
──────────────────────────────
[FAQ 아코디언]  id="faq"        ← FaqAccordion 재사용
──────────────────────────────
[질문하기 폼]   id="ask"        ← 신규 구성 (5장)
──────────────────────────────
[선택: 하단 광고 1개] ← 페이지 최하단, 폼과 충분히 이격
```

컨테이너: `mx-auto max-w-3xl px-4 py-10 sm:px-6 sm:py-14`. 홈/about처럼 카드 그리드가 없는
단일 컬럼 텍스트+폼 페이지이므로, 기존 계산기 상세 페이지(예: D-Day 스펙의 단일 컬럼
`max-w-3xl`)와 동일한 폭 기준을 따른다.

## 3. 인트로 섹션

- H1: "고객센터" — `text-2xl font-bold text-brand-text sm:text-[2rem]` (기존 Display 패턴)
- 리드 문단(카피는 기획팀/CS팀): 자주 묻는 질문을 먼저 확인하도록 유도하는 문구를 넣을 것을
  권장한다. 예: "궁금하신 내용은 아래 자주 묻는 질문에서 먼저 확인해보세요. 원하시는 답을
  찾지 못하셨다면 바로 질문을 남겨주시면 됩니다." — `mt-2 text-sm text-brand-text-secondary
  sm:text-base leading-relaxed`
- **바로가기 링크 2개 (신규 제안, 발견성 강화용):** 리드 문단 아래 `mt-4 flex flex-wrap gap-3`
  로 앵커 링크 2개 배치.
  - "자주 묻는 질문 보기 ↓" → `#faq`로 스무스 스크롤(순수 HTML 앵커, JS 불필요)
  - "바로 질문하기 →" → `#ask`
  - 스타일: 작은 아웃라인 pill 버튼, `rounded-full border border-brand-border px-4 py-2 text-sm
    font-medium text-brand-text-secondary hover:border-brand-primary hover:text-brand-primary`
  - **의도:** FAQ가 길어질 경우(현재 CS팀 초안만 10문항) 질문하기 폼을 찾기 위해 전부
    스크롤해야 하는 상황을 막기 위한 사전 조치다. "사용자가 특정 버튼/메뉴를 못 찾는다"는
    유형의 CS 이슈가 향후 이 페이지에서 발생할 가능성을 미리 차단하려는 목적.

## 4. FAQ 섹션 (`id="faq"`)

- **`components/FaqAccordion.tsx`를 그대로 재사용한다.** 구조 변경 불필요 — `items` prop에
  사이트 공통 FAQ 배열만 넘기면 된다.
- 콘텐츠 소스: `cs/faq-draft.md` 10문항을 다듬어 사용(문구 확정은 CS팀/기획팀 담당, 이 문서
  범위 아님). 단, `FaqAccordion`은 정적 배열을 받는 구조이므로 콘텐츠는 `lib/`쪽에 별도 상수로
  정의하거나 페이지 컴포넌트 안에 직접 배열로 두는 방식 모두 가능 — 구현 방식은 개발팀 재량.
- 섹션 제목: `FaqAccordion` 내부에 이미 "자주 묻는 질문" H2가 고정되어 있으므로 별도 제목을
  중복 추가하지 않는다.
- 문항 수가 10개 안팎으로 많은 편이라 아코디언 자체 스크롤 처리는 불필요(페이지 전체 스크롤로
  충분), 다만 3장의 "바로 질문하기" 바로가기 링크가 이 부담을 상쇄한다.

## 5. 질문하기 폼 (`id="ask"`) — mailto 제약 UX 설계

**핵심 제약: 서버가 없는 mailto 방식이므로, 제출 = "사용자의 메일 앱에 초안을 채워서 여는 것"이지
"실제 전송 완료"가 아니다.** 이 차이를 사용자에게 숨기면 "질문했는데 답이 없다"는 CS 이슈로
그대로 되돌아온다. 따라서 아래 3가지를 반드시 화면에 반영한다.

### 5-1. 폼 진입 전 안내 박스 (필수)

- 위치: 폼 필드 바로 위, `id="ask"` 섹션의 최상단.
- 스타일: 경고가 아닌 정보 톤 — `rounded-lg bg-blue-50 border border-blue-100 p-4 text-sm
  text-brand-text-secondary leading-relaxed` (Warning/Danger 컬러 사용 금지 — 문제 상황이
  아니라 정상 동작 안내이므로).
- 문구(예시, 확정은 기획팀): "질문 보내기 버튼을 누르면 입력하신 내용이 담긴 이메일 작성
  화면이 열립니다. 평소 사용하시는 메일 앱(또는 웹메일)이 설정되어 있어야 하며, 열린 화면에서
  마지막으로 [보내기]를 눌러야 문의가 실제로 접수됩니다."
- 이 박스는 폼 위에서 항상 노출되며, 제출 후에도 사라지지 않는다(사라지게 하려면 JS 상태
  관리가 필요한데 이는 기능 로직이라 이 문서 범위 밖 — 정적으로 항상 보이는 편이 안전).

### 5-2. 입력 필드

| 필드 | 타입 | 필수 여부 | 스펙 |
|---|---|---|---|
| 이름 | text | 선택 | `h-12`, radius 8px, 라벨 "이름 (선택)" |
| 답장받으실 이메일 | email | **필수** | `h-12`, 라벨 "답장받으실 이메일". 안내 caption 추가: "메일 본문에 자동으로 포함되어 답변 시 참고됩니다." (mailto는 발신자를 자동으로 사용자 메일 계정으로 채우므로, 이 필드는 실제 발신 주소가 아니라 "이 주소로 답장해달라"는 참고용 정보임을 명확히 해야 오해가 없다) |
| 문의 내용 | textarea | **필수** | `min-h-[140px]`, radius 8px, 라벨 "문의 내용", placeholder 예: "사용하신 계산기 이름, 입력한 값, 문의 내용을 자세히 적어주시면 빠르게 확인할 수 있습니다." (버그 제보 시 필요한 정보를 placeholder로 유도 — `cs/static-pages-checklist.md`의 "/contact 버그 제보용 안내 문구" 요건을 이 필드 placeholder로 흡수) |

디자인 시스템 3-1(입력 필드 스펙) 그대로 준수: 포커스 시 Primary 링, 에러 시 Warning/Danger
테두리 + Caption 에러 메시지(`role="alert"`), 라벨은 필드 위 Body Small + Text Secondary.

### 5-3. 제출 버튼 및 대체 수단 (필수)

- 버튼: "질문 보내기" — solid Primary, `h-12`, 모바일 `w-full`, `sm` 이상 `min-w-[180px]`
  (about 페이지처럼 전체 폭으로 과하게 강조하지 않고 일반 폼 액션 크기 유지).
- **버튼 바로 아래, 대체 연락 수단을 반드시 텍스트로 노출한다:** 예: "메일 작성 화면이 열리지
  않는다면 [이메일 주소]로 직접 문의해주세요." — `mt-3 text-xs text-brand-text-secondary`,
  이메일 주소는 선택/복사가 쉽도록 일반 텍스트로 노출(이미지·아이콘 안으로 숨기지 않음).
  - **마스터 팀장 확인 필요:** 이 대체 이메일 주소가 아직 확정되지 않았다(`planning/about-
    page-content.md`에서도 동일 이슈가 이월된 상태). 실제 공개할 주소를 확정해줘야 개발팀이
    하드코딩할 수 있다.
- 제출 후 "전송 완료" 같은 성공 메시지는 **넣지 않는다** — mailto는 실제 전송 여부를 사이트가
  알 수 없으므로 거짓 확인 메시지가 된다. 대신 5-1의 안내 박스가 "메일 앱에서 보내기까지
  눌러야 완료"라는 사실을 이미 전달하고 있으므로 별도 후속 메시지는 불필요.

## 7. 광고 배치

design-system.md 5번 가이드를 그대로 적용하되, 이 페이지 특성(문의/신뢰 페이지)을 반영해
아래처럼 제한한다.

- 인트로와 FAQ 사이: 광고 1개 배치 **가능**(선택, 여백 24px 이상 분리).
- **FAQ와 질문하기 폼 사이, 폼 내부/주변에는 광고를 넣지 않는다.** 문의 폼은 사용자가 실제
  액션(입력·제출)을 취하는 구간이라 오클릭 유도 리스크가 가장 크고, 문의 페이지 특성상
  신뢰감이 최우선이므로 클러터를 만들지 않는다.
- 하단(폼 다음, 푸터 위)에 광고 1개는 가능하나 우선순위 낮음.

## 8. 다른 팀 전달 사항

**마스터 팀장 확인 필요**
1. 0장 "연결 제안": `/about`, `SiteFooter`의 문의 관련 링크를 `/support#ask`로 연결하는 것에
   동의하는지 확인 요청. 동의 시 개발팀에 `/about` 수정도 함께 전달 예정.
2. 5-3 "대체 연락 수단" 이메일 주소 확정 필요 — 아직 공개 이메일 주소가 정해지지 않음.
3. 1장 헤더 모바일 폭 이슈는 확인 필수는 아니나(구조 변경 아님), 개발팀 QA에서 실제 줄바꿈이
   발생하면 재보고하겠다.

**개발팀 전달 사항**
