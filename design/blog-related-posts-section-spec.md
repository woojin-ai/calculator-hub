# 블로그 상세 — 형제 블로그 글 링크 섹션 화면 구성안

- 대상 페이지: `app/blog/[slug]/page.tsx` (본문 컨테이너 `max-w-3xl`)
- 재사용 컴포넌트: `components/RelatedBlogPosts.tsx`, `components/BlogPostCard.tsx`
- 신규 컴포넌트/신규 색 토큰/신규 간격 값: **없음** (기존 클래스만 조합)
- 배경: `docs/team-log/2026-07-19.md` §5 "blog↔blog 내부링크 부재" 이슈 대응
- 이 문서 범위: 레이아웃/위계/스타일 스펙까지. **어떤 글을 형제로 고를지(선정 로직)는 기획+개발 소관이며 여기서 정의하지 않는다.**

---

## 0. 사전 사실 확인 (실제 코드 기준 — 태스크 설명 정정 1건)

태스크 브리핑에는 `RelatedBlogPosts`가 "현재 카테고리 페이지 전용"이라고 되어 있으나, **실제로는 호출부가 2곳**이다:

| 호출부 | 컨테이너 폭 | 코드 |
|---|---|---|
| `components/CategoryPage.tsx:48` | `max-w-5xl` (1024px) | `<RelatedBlogPosts posts={getBlogPostsByCategory(category)} />` |
| `app/calculator/[slug]/page.tsx:146` | **`max-w-3xl` (768px, line 95)** | `<RelatedBlogPosts posts={relatedPosts} />` |

즉 **"max-w-3xl 안의 3열 카드 그리드"는 이미 계산기 상세 페이지에서 라이브로 돌아가고 있다.** 이 사실이 §1 판단의 전제이고, 동시에 §7 이슈의 원인이다. 하위호환 검증 대상도 1곳이 아니라 2곳이다.

---

## 1. [핵심] 3열 grid를 max-w-3xl에 그대로 재사용해도 되는가 — 결론: **안 된다. 2열로 파라미터화한다.**

### 1-1. 실제 폭 계산

컨테이너 클래스: `mx-auto max-w-3xl px-4 py-8 sm:px-6 sm:py-12`
- `max-w-3xl` = 48rem = **768px**
- 좌우 패딩: `<640px`에서 `px-4`(16px×2=32px), `≥640px`에서 `sm:px-6`(24px×2=48px)
- 그리드 간격 `gap-4` = 16px

**데스크톱(뷰포트 ≥1024px, `lg:` 적용 구간) — 콘텐츠 폭 = 768 − 48 = 720px**

| 열 수 | 카드 1장 폭 | 카드 내부 폭 (`p-4` = 16px×2 차감) |
|---|---|---|
| 3열 (현행 클래스) | (720 − 16×2) ÷ 3 = **229.3px** | **197.3px** |
| 2열 (제안) | (720 − 16) ÷ 2 = **352px** | **320px** |
| 참고: 카테고리 페이지 3열 (`max-w-5xl`, 콘텐츠 976px) | (976 − 32) ÷ 3 = **314.7px** | 282.7px |

→ **같은 3열 클래스인데 카드 폭이 314.7px vs 229.3px로 27% 차이가 난다.** `lg:grid-cols-3`는 카테고리 페이지의 976px 컨테이너를 전제로 정해진 값이며, 720px 컨테이너에 그대로 옮기면 디자인 의도(카드 최소 폭 ~300px대)가 유지되지 않는다.

### 1-2. 근거 A — 실제 제목이 카드 안에서 얼마나 잘리는가 (결정적)

`BlogPostCard`의 제목은 `line-clamp-2 text-base ... sm:text-lg`, 즉 데스크톱에서 **18px 2줄**이다. 한글 글리프 폭 ≈ 폰트 크기(1em)로 잡으면 줄당 노출 글자 수는 `내부폭 ÷ 18`.

이 사이트의 실제 블로그 제목 길이(`lib/blog.ts`, 27편)는 대부분 **30~45자**다. 예:
- `겨울 전기요금이 여름보다 비싼 이유 — 계절별 누진구간 차이 (2026년 주택용 저압)` (45자)
- `주휴수당 조건 총정리 — 주 15시간·단시간 알바·40시간 초과 실전 케이스 (2026년)` (44자)
- `입사일별 근속 1·3·5·10년 만근일 조견표 — 언제 채워지나 (2026년)` (39자)

| 배치 | 내부폭 | 줄당 글자수 | 2줄 노출 글자수 | 40자 제목 노출률 |
|---|---|---|---|---|
| 데스크톱 3열 (현행) | 197px | ≈10.9자 | **≈21자** | **약 53%** |
| 데스크톱 2열 (제안) | 320px | ≈17.8자 | **≈35자** | 약 88% |
| 모바일 375px 1열 | 311px | ≈19.4자 (16px 기준) | **≈39자** | 약 98% |
| 카테고리 3열 (기존, 정상) | 283px | ≈15.7자 | ≈31자 | 약 78% |

→ 3열을 그대로 쓰면 **데스크톱(21자)이 모바일(39자)보다 제목을 절반밖에 못 보여준다.** 화면이 넓어질수록 정보가 줄어드는 역전이다. 이 사이트 제목은 앞부분이 주제, `—` 뒤가 변별 정보("계절별 누진구간 차이", "소득구간별 근로자 부담 조견표")인 패턴이 많은데, 21자 컷에서는 변별 정보가 통째로 잘린다. "다음에 뭘 읽을지" 고르게 하는 것이 이 섹션의 유일한 목적(체류시간 확보)이므로, 이건 스타일 취향이 아니라 **기능 손실**이다.

### 1-3. 근거 B — 역방향 반응형(뷰포트가 넓어질 때 카드가 좁아짐)

`max-w-3xl` 컨테이너는 뷰포트 약 816px(768 + 48)부터 폭이 768px에 고정된다. 반면 `lg:` 브레이크포인트는 **뷰포트** 1024px에서 발동한다. 따라서:

| 뷰포트 | 열 수 | 카드 폭 |
|---|---|---|
| 640px | 2열 | (592−16)÷2 = 288px |
| 1023px | 2열 | (720−16)÷2 = **352px** |
| **1024px** | **3열** | (720−32)÷3 = **229px** ← 뷰포트 1px 넓어졌는데 카드가 123px 좁아짐 |

컨테이너 폭이 고정된 영역에서 뷰포트 기준 `lg:` 브레이크포인트를 쓰는 것 자체가 부정합이다. 1023px→1024px 구간에서 카드가 급격히 쪼그라드는 것을 브라우저 리사이즈로 육안 확인 가능하다.

### 1-4. 채택안 및 기각안

| 안 | 판정 | 사유 |
|---|---|---|
| (a) 3열 그대로 재사용 | **기각** | §1-2, §1-3. 계산기 상세에 선례가 있다는 건 "이미 같은 문제가 한 곳 더 있다"는 뜻이지 정당화가 아니다. |
| (b) 이 페이지 전용 리스트형(가로 1열 리스트) 변형 | **기각** | 신규 마크업 계열이 생겨 유지보수 대상이 늘고, 사이트 전역의 카드 언어(rounded-xl 보더 카드)와 어긋난다. 폭 문제는 열 수 조정으로 충분히 해결된다. |
| **(c) 열 수 파라미터화 (`columns` prop, 기본값 3)** | **채택** | 기본값이 현행과 동일 → 기존 호출부 2곳 무영향(하위호환). 블로그 상세만 `columns={2}` 주입. 변경량 최소. |

**결론: `RelatedBlogPosts`에 선택적 `columns?: 2 | 3` prop을 추가(기본값 `3`)하고, 블로그 상세에서만 `columns={2}`를 넘긴다.**

## 2. 모바일 375px 검토

**콘텐츠 폭 = 375 − 32(`px-4`) = 343px, 카드 내부 = 343 − 32(`p-4`) = 311px**

- **1열로 떨어지는가**: 떨어진다. `grid-cols-1`이 기본이고 2열 전환은 `sm:`(640px)부터다. 375px에서는 `columns` 값과 무관하게 1열이므로, §1의 변경은 모바일 렌더에 **아무 영향이 없다**.
- **긴 제목 가로 오버플로**: 위험 없음.
  - 제목 `h3`에 `line-clamp-2`가 걸려 있고, `line-clamp`는 `overflow: hidden`을 포함하므로 세로/가로 모두 컨테이너를 넘지 않는다.
  - 한글은 임의 위치에서 줄바꿈되므로 `white-space` 관련 넘침이 없다. 실제 제목 중 최장 무공백 토큰은 `제곱미터(㎡)`·`총부채원리금상환비율` 수준(≈10자 ≈ 160px)으로 311px 안에 여유롭게 들어간다.
  - "입사일별 근속 1·3·5·10년 만근일 조견표 — 언제 채워지나 (2026년)"(39자) → 16px 기준 줄당 ≈19자, 2줄 ≈39자로 **거의 전문 노출**. 클램프도 거의 걸리지 않는다.
- **가장 빡빡한 행은 제목이 아니라 메타행**이다. `<div className="mt-1 flex items-center gap-2 ...">`에 `flex-wrap`이 없다:
  - `2026. 7. 15.`(tabular-nums, 12px, ≈80px) + gap 8 + `·`(≈5px) + gap 8 + `읽는 시간 8분`(≈73px) ≈ **174px**
  - 모바일 311px, 데스크톱 2열 320px 모두 충분(여유 137px 이상). **데스크톱 3열이었다면 197px로 여유 23px까지 줄어들어 두 자리 분수(`읽는 시간 12분`)에서 위험 구간에 들어간다** — 2열 채택의 부수적 이점.
- **조치**: 모바일 관련 **마크업 변경 없음**. 검수 시 375px에서 가로 스크롤바 미발생만 확인.

## 3. 섹션 배치 순서 — 결론: `관련 계산기` → `관련 글` (계산기 먼저)

기획팀도 같은 항목을 판단하므로, 여기서는 **시각적 위계와 스캔 흐름 근거만** 제시한다.

1. **행동 비용 오름차순이 아니라, 전환 가치 내림차순으로 쌓아야 스캔이 멈추지 않는다.** 본문을 다 읽은 사용자의 시선은 아래로 감속하며 내려온다. 가장 가치 높은 액션(계산기 = 이 사이트의 목적 페이지)을 감속 구간의 **첫 카드 블록**에 둬야 한다. 읽을거리를 먼저 두면, 사용자가 거기서 이탈(다른 글로 점프)해 계산기 블록은 아예 시야에 들어오지 않는다.
2. **본문 내 `calculatorCta` 블록과의 리듬 연속성.** 본문 중간에 이미 `calculatorCta`(브랜드 컬러 채운 버튼)가 반복 등장한다. 본문 종료 직후 `관련 계산기`가 오면 "계산기로 가는 동선"이 끊기지 않고 이어진다. 사이 끼어드는 블록이 있으면 이 학습된 리듬이 깨진다.
3. **하단 링크와의 의미 클러스터링.** 페이지 최하단은 `← 블로그 목록으로`(블로그 맥락)다. `관련 글`을 그 바로 위에 두면 "블로그성 요소"가 아래쪽에 모여 **계산기 영역 / 블로그 영역**의 2블록 구조가 자연스럽게 형성된다. 순서를 뒤집으면 `관련 글 → 관련 계산기 → 블로그 목록으로`가 되어 블로그 요소가 계산기 블록을 사이에 두고 분리된다.
4. **기존 페이지와의 일관성.** `app/calculator/[slug]/page.tsx`는 이미 `RelatedCalculators`(142) → `RelatedBlogPosts`(146) 순이다. 두 상세 페이지가 동일한 "도구 먼저, 읽을거리 다음" 규칙을 따르면 사용자가 사이트 하단 구조를 한 번만 학습한다.

**따라서 기존 `RelatedCalculators` 렌더 위치는 그대로 두고, 그 바로 아래에 신규 섹션을 삽입한다.**

## 4. 헤딩 위계 검증 — 결론: 신규 섹션 `h2`, 카드 제목 `h3`가 정확함

현행 `RelatedBlogPosts`는 이미 섹션 `<h2>` + `<BlogPostCard headingLevel="h3">`로 되어 있어 **변경 불필요**하다. 최종 아웃라인:

```
h1  {post.title}                                  ← 페이지 제목 (1개)
├─ h2  (본문 heading 섹션들, renderSection의 mt-8 text-xl font-bold)
├─ h2  관련 계산기            ← RelatedCalculators.tsx:13
│   └─ h3  {calculator.title} ← CalculatorCard.tsx:18 (하드코딩 h3, 확인 완료)
└─ h2  함께 읽으면 좋은 글    ← RelatedBlogPosts.tsx:12 (신규 배치)
    └─ h3  {post.title}       ← BlogPostCard, headingLevel="h3" (RelatedBlogPosts.tsx:15에 이미 고정)
```

- h1 → h2 → h3 순차. **레벨 건너뜀 없음.**
- h1은 페이지당 1개 유지.
- `CalculatorCard`는 `headingLevel` prop이 없고 `h3` 하드코딩 — 이 페이지 위계와 우연히 맞아떨어진다. 손댈 것 없음.
- 본문 h2와 섹션 h2가 같은 레벨인 것은 의미상 옳다(본문의 하위 절이 아니라 형제 블록이므로). 다만 **시각적으로 구분이 안 되는 문제가 있어 §5에서 처리한다.**

## 5. 두 카드 섹션의 시각적 구분

### 5-1. 문제 정의 (실제 클래스 비교)

| 요소 | 클래스 |
|---|---|
| 본문 heading (`renderSection`) | `mt-8 text-xl font-bold text-brand-text` |
| `관련 계산기` h2 | `mb-4 text-xl font-bold text-brand-text` (섹션 `mt-10`) |
| `관련 글` h2 | `mb-4 text-xl font-bold text-brand-text` (섹션 `mt-10`) |

세 종류의 h2가 **font-size/weight/color가 전부 동일**하다. 게다가 `CalculatorCard`와 `BlogPostCard`는 컨테이너 클래스가 `group flex flex-col gap-2 rounded-xl border border-brand-border bg-brand-surface p-4 transition-all hover:border-brand-primary hover:shadow-md`로 **완전히 같다**. 즉 아무 조치 없이 붙이면 하단에 "구분 안 되는 회색 카드 그리드 2덩어리"가 생기고, 본문이 어디서 끝났는지도 흐려진다. 구분 장치는 3중으로 건다.

### 5-2. 장치 ① 본문 종료 구분선 (신규)

`</article>` 직후, `RelatedCalculators` 앞에 시맨틱 `<hr>`을 넣는다.

```tsx
<hr className="mt-10 border-brand-border" />
```

- `<hr>`은 HTML 명세상 "주제 전환(thematic break)"이므로 **읽을거리 본문 종료 → 회유 동선 시작**을 마크업 수준에서도 표현한다(스크린리더 `separator` 역할, 별도 aria 불필요).
- 간격: `hr`의 `mt-10`(40px) + `RelatedCalculators`의 자체 `mt-10`(40px) = 구분선 위아래 40px씩. 본문 내 h2 간격(`mt-8`, 32px)보다 명확히 넓어 위계가 드러난다. **신규 간격 값 발명 없음**(사이트 관례인 mt-8 / mt-10 리듬 그대로).
- 색: `border-brand-border` — 기존 카드 보더와 동일 토큰. 신규 색 없음.

### 5-3. 장치 ② 헤딩 문구 차별화 (`headingText` prop 신규)

현행 `RelatedBlogPosts`의 헤딩은 `"관련 글"` 하드코딩이다. `관련 계산기` / `관련 글`은 어미가 같아 훑을 때 대비가 약하고, 블로그 상세에서는 "관련"이라는 말이 **본문과의 관계**인지 **카테고리 관계**인지도 모호하다.

- `RelatedBlogPosts`에 `headingText?: string` 추가, **기본값 `"관련 글"`**.
- 블로그 상세에서만 `headingText="함께 읽으면 좋은 글"` 주입.
- 효과: 글자 수(5자 vs 11자)와 어형이 달라져 스캔 시 두 블록이 즉시 분리되고, "지금 읽은 글에 이어서"라는 관계가 문구로 명시된다.
- 이 prop은 `planning/category-related-blog-spec.md` §3에서 이미 제안됐다가 미구현 상태인 것과 동일한 설계다(중복 발명 아님).

### 5-4. 장치 ③ 카드 자체의 대비 (기존 자산으로 이미 충족 — 추가 작업 없음)

- `CalculatorCard`: 우측 상단에 **에메랄드 배지**(`bg-emerald-50 text-emerald-600` + 점) `바로 사용 가능`. 제목이 카드 최상단.
- `BlogPostCard`: 좌측 상단에 **회색 카테고리 배지**(`bg-brand-bg`), 하단에 **날짜 + 읽는 시간** 메타행.

색상(에메랄드 vs 그레이)·배지 위치(우 vs 좌)·메타행 유무가 이미 다르므로 **카드 스타일은 손대지 않는다.** 여기에 열 수까지 3열 vs 2열로 달라지므로(§1) 그리드 리듬 자체가 구분 신호로 추가된다 — 이것이 §1 결정의 부수 효과다.

### 8-1. 필수
- [ ] 데스크톱 1280px: `함께 읽으면 좋은 글`이 **2열**, 카드 폭 ≈352px. `관련 계산기`는 **3열** 유지.

### 8-2. 판단 참고 — 관련 글 편수와 2열 정렬
2열에서 **3편**이면 마지막 카드가 2번째 줄에 홀로 좌측 반폭으로 남는다. 이는 **허용**한다. 카드를 중앙 정렬하거나 늘려 채우면 그리드 축이 어긋나 오히려 정렬 규칙이 깨지고, 좌측 정렬 잔여 카드는 웹에서 보편적으로 학습된 패턴이다. 시각적 완결성을 우선한다면 **4편**이 2×2로 가장 안정적이므로, 기획/개발이 편수를 자유롭게 정할 수 있다면 **4편을 권장**한다.

## 9. blocker / 이슈

**blocker 0** (구현 착수 가능). 단, 마스터 판단이 필요한 이슈 2건:

1. **[중] 계산기 상세 페이지에 동일한 폭 부정합이 남는다.** `app/calculator/[slug]/page.tsx`도 `max-w-3xl`(line 95)인데 `RelatedCalculators`·`RelatedBlogPosts` 둘 다 `lg:grid-cols-3`을 쓴다 — §1-2·§1-3의 문제가 그대로 있다(카드 229px, 제목 21자 컷). 이번 스펙대로 하면 **블로그 상세는 관련 글 2열, 계산기 상세는 관련 글 3열**이 되어 과도기 불일치가 생긴다. 이번 범위에서는 **의도적으로 손대지 않았다** — 라이브 페이지 16개의 하단 레이아웃을 바꾸는 변경이라 마스터 승인 없이 끼워 넣을 수 없기 때문. 후속 티켓으로 "계산기 상세의 두 섹션도 `columns={2}` 적용" 검토를 권한다. `columns` prop이 이미 들어가 있으므로 후속 작업은 호출부 2줄 수정으로 끝난다.
2. **[소] 형제 글 선정 로직 미정.** `siblingPosts` 소스(같은 카테고리 / 같은 `relatedCalculatorSlugs` 공유 / 태그 기반)는 기획+개발 결정 사항이다. 화면 스펙은 편수(3~4편, 상한 6)와 자기 자신 제외만 요구하며, 어느 방식이든 레이아웃은 동일하게 성립한다.

또한 §0에서 확인했듯 **`RelatedBlogPosts`는 카테고리 페이지 전용이 아니라 호출부가 2곳**이다. 하위호환 검증 시 계산기 상세도 함께 확인해야 한다(§8-1에 반영).
