## 0. 이 페이지가 다른 계산기와 다른 점 (설계 전제)

- 입력 필드가 4종(원금/이자율/기간/상환방식)으로 기존 라이브 계산기(1~2개) 대비 많다 →
  design-system.md 4번 원칙 1 "입력→결과 무스크롤"을 그대로 달성하기 어렵다. 4장에서 실측
  기반 판단과 절충안을 제시한다.
- 상환방식(원리금균등/원금균등)에 따라 **결과의 구조 자체가 달라진다**(고정 월 상환액 1개
  vs 회차별로 감소하는 범위) → 결과 카드에 두 가지 모드를 설계한다(3장).
- YMYL 리스크 "높음" 등급(`calculator-lineup.md` 50행) — 면책 문구·CTA 금지 원칙이 레이아웃에
  직접 반영되어야 한다(6장, 이 문서에서 가장 중요한 절).

## 1. 페이지 전체 레이아웃

`app/calculator/[slug]/page.tsx` 기존 골격을 그대로 따른다. 신규 설계는 "2. 입력 설계"와
"3. 결과 카드"이며, 추가로 이 페이지에는 기존에 없던 **상단 면책 배너**를 신규 삽입한다.

```
[Breadcrumb: 홈 / 대출 계산기]
[H1] 대출 이자 계산기
[본문 요약 1줄]
[⚠ 상단 면책 배너]  ← 신규 요소, 6장 참고. 광고 아님, 항상 노출.
──────────────────────────────
[계산기 카드]  ← 이번 스펙의 핵심 (2~3장)
  - 상환방식 선택 (세그먼트 토글, 2옵션)
  - 대출 원금 입력
  - 연 이자율 / 대출 기간(+단위 토글) — 2열
  - 계산 버튼
  - 결과 표시 영역 (모드별 분기, 3장)
──────────────────────────────
[결과 해석]         ← ResultInterpretation 재사용 (카피는 기획팀, YMYL 면책 문구 필수 — 6장)
[본문 중간 광고 여백]  ← 6장 광고 원칙 준수, "지금 대출 신청"류 오인 방지 규칙 포함
[FAQ 아코디언]        ← FaqAccordion 재사용
[관련 계산기]         ← RelatedCalculators 재사용
[하단 광고]           ← 6장 광고 원칙 준수
```

데스크톱에서도 기존 계산기 페이지(만 나이/D-Day/BMI)와 동일하게 사이드바 없는 단일 컬럼
`max-w-3xl`을 유지한다(5장 참고). 사이드바 도입은 페이지 공통 이슈이므로 이 문서 범위 밖.

## 2. 입력 설계

### 2-1. 필드 순서 및 배치 원칙

**상환방식 선택을 맨 위에 배치한다** (다른 3개 숫자 입력보다 먼저). 이유: 상환방식은 단순
입력값이 아니라 이후 결과가 "고정 월 상환액"으로 나올지 "회차별로 줄어드는 범위"로 나올지를
가르는 분기점이다. 사용자가 숫자를 입력하기 전에 먼저 어떤 방식을 비교하려는지 인지하도록
모드 선택을 최상단에 두는 것이 데이터 입력 후 결과 형태가 바뀌어 혼란스러운 것보다 낫다고
판단했다.

| 순서 | 필드 | 배치 |
|---|---|---|
| 1 | 상환 방식 선택 | 전체 폭, 세그먼트 토글 (2-3) |
| 2 | 대출 원금 | 전체 폭 (큰 금액이라 한 줄 확보 — BMI처럼 2열로 묶으면 자릿수가 잘려 보여 가독성이 떨어짐) |
| 3 | 연 이자율(%) / 대출 기간(+단위 토글) | 2열 그리드 (`grid grid-cols-2 gap-3 sm:gap-4`, BMI 패턴 재사용) |
| 4 | 계산 버튼 | 전체 폭 |

### 2-2. 필드별 스펙

design-system.md 3-1 스타일(테두리 `#E5E7EB`, radius 8px, 높이 `h-12`/48px, 포커스 시 Primary
컬러 링)을 모든 입력에 동일 적용한다.

| 필드 | id | 라벨 | type | placeholder | 비고 |
|---|---|---|---|---|---|
| 대출 원금 | `principal` | "대출 원금 (원)" | `text` + `inputMode="numeric"` 권장 (아래 참고) | `50,000,000` | 3~9자리 숫자, 콤마 서식 권장 |
| 연 이자율 | `interestRate` | "연 이자율 (%)" | `number`, `step="0.01"` | `4.5` | |
| 대출 기간 | `loanPeriod` | "대출 기간" | `number` (정수) | `30` | 아래 기간 단위 토글과 세트 |
| 기간 단위 | `loanPeriodUnit` | (라벨 없음, 대출기간 입력 바로 아래) | 토글(년/개월) | — | 기본값 "년" |
| 상환 방식 | `repaymentType` | "상환 방식" | 세그먼트 토글(2옵션) | — | 기본값 미선택 상태 없음 — 최초 진입 시 "원리금균등상환"을 기본 선택값으로 노출 (아래 참고) |
| 계산 버튼 | — | "대출 이자 계산하기" | `submit` | — | AgeCalculator/BmiCalculator와 동일하게 클릭(제출) 방식. 사이트 전체 인터랙션 패턴 통일. |

**대출 원금 숫자 서식 관련 권장(강제 아님)**: 금액이 억 단위까지 갈 수 있어(예:
`500000000`) 콤마 없이는 자릿수를 즉시 파악하기 어렵다. 입력 중 자동으로 `50,000,000`처럼
천단위 콤마가 붙는 서식을 강하게 권장하나, 이는 입력값 실시간 가공(로직) 영역이라 최종
구현 방식(타이핑 중 포맷팅 vs `type="number"` 순수 숫자 입력)은 개발팀 판단에 맡긴다. 콤마
서식을 넣지 않는 경우에도 최소한 결과 카드의 금액 표시(3장)에는 반드시 천단위 콤마를
적용해야 한다 — 이건 필수.

**상환 방식 기본값**: 원리금균등상환을 기본 선택으로 둔다(가장 널리 알려진 방식이라 초심자
혼동이 적음). BMI/D-Day처럼 "선택 안 함" 상태를 만들지 않는다 — 세그먼트 토글은 라디오
버튼과 동일한 성격이라 항상 하나가 선택되어 있어야 자연스럽다.

### 2-3. 세그먼트 토글 컴포넌트 (신규 UI 패턴 — design-system.md에 없음)

이 페이지에서 처음 필요해진 컴포넌트 유형이다. 두 가지 크기로 사용한다.

**(A) 상환 방식 토글 (풀사이즈, 2옵션)**
- 컨테이너: `rounded-lg border border-brand-border bg-white p-1 flex gap-1`, 전체 폭
- 버튼(옵션) 각각: `flex-1 h-11 rounded-md text-sm font-semibold transition-colors`
  - 선택 상태: `bg-brand-primary text-white`
  - 비선택 상태: `bg-transparent text-brand-text-secondary`
- 옵션 텍스트: "원리금균등상환" / "원금균등상환" (풀 명칭 그대로 노출 — 축약하면 초심자가
  구분 못 할 위험, 아래 5장 모바일에서 줄바꿈 허용)
- 접근성: 시각적으로는 버튼처럼 보이되, 실제 마크업은 라디오 그룹(`name="repaymentType"`,
  숨겨진 `input[type=radio]` + `label`)으로 구현할 것을 권장 — 폼 시맨틱과 키보드 접근성
  확보 목적 (구현 방식은 개발팀 재량, 시각 결과만 동일하면 됨).

**(B) 기간 단위 토글 (컴팩트, 2옵션, 대출기간 입력 하위)**
- 대출기간 숫자 입력(`h-12`) 바로 아래 `mt-1.5`, 같은 컬럼 폭 안에 배치
- 컨테이너: `rounded-md border border-brand-border bg-white p-0.5 flex gap-0.5`
- 버튼 각각: `flex-1 h-9 rounded text-xs font-semibold` (36px — 44px 터치 타겟 기준보다
  작음, 4장에서 모바일 터치 타겟 예외 사유 명시)
- 옵션: "년" / "개월", 선택 상태 `bg-brand-primary text-white`, 비선택 `text-brand-text-secondary`

> **마스터 팀장 확인 요망**: 두 토글은 design-system.md 3장(입력/결과/FAQ/카드)에 없는 신규
> 컴포넌트 유형이다. 향후 DSR 계산기·중도상환수수료 계산기(로드맵 10·12순위) 등 `/loan`
> 카테고리 다른 계산기에서도 유사한 "옵션 선택형 토글"이 필요할 가능성이 높으므로, 이번
> 정의를 `design-system.md`에 공식 컴포넌트로 편입할지 여부를 확인해달라(10장 재정리).

### 2-4. 에러 상태

AgeCalculator/BmiCalculator 패턴 재사용: 미입력 또는 0 이하 값 입력 후 계산 시 해당 입력
테두리 `border-brand-warning` + 하단 `text-xs text-brand-warning` 메시지, `role="alert"`.
예: "대출 원금을 입력해주세요.", "연 이자율은 0보다 커야 합니다." (정확한 검증 규칙·문구는
개발팀 재량, 이 문서는 시각 스타일만 규정)

## 3. 결과 카드 설계 (상환방식별 분기 — 핵심)

공통 골격은 design-system.md 3-2를 따른다: `rounded-xl border-l-4 p-4 sm:p-6`, 폼과
`mt-6`(24px) 간격, 좌측 강조선 + 연한 배경 틴트(Accent `#10B981` / `bg-emerald-50`, BMI의
위험색 전환 같은 상태별 색상 분기는 이 페이지에는 적용하지 않는다 — 대출 상환액에 "위험"
색상 의미를 부여하는 것은 오해를 부를 수 있어 항상 Accent 컬러 고정).

### 3-1. 공통 구조 (위 → 아래)

1. 결과 라벨(Body Small, Text Secondary): 선택된 상환방식을 그대로 반영
   - 예: "매월 상환액 (원리금균등상환)" / "매월 상환액 (원금균등상환)"
2. **주요 숫자 영역** — 3-2/3-3에서 모드별로 분기
3. 구분선(`border-t border-brand-border/60`, `mt-4 pt-4`)
4. **보조 항목 리스트** (두 모드 공통, 3-4)
5. 갱신 트랜지션: design-system.md 3-2 그대로(배경 fade/미세 scale 0.15~0.2s)

### 3-2. 모드 A — 원리금균등상환 (고정 월 상환액)

기존 계산기(만 나이/BMI)와 동일하게 **단일 히어로 숫자**로 표시한다.

```
매월 상환액 (원리금균등상환)
1,234,560 원          ← Result 스케일: text-4xl sm:text-[2.5rem] font-bold, Accent 컬러, tabular-nums
매월 동일한 금액을 상환합니다.   ← 보조 설명 1줄 (Body Small, Text Secondary)
```

### 3-3. 모드 B — 원금균등상환 (회차별 감소 — 범위로 표시)

**설계 판단(마스터 지시서의 "표시 방식 결정" 요구에 대한 디자인팀 결론)**: 평균값 1개로
표시하지 않는다. 원금균등상환은 매달 상환액이 다르므로 "평균 월 상환액"이라는 단일 숫자를
Result 스케일로 크게 보여주면, 이용자가 이를 "매달 고정으로 내는 금액"으로 오인할 위험이 있다
— 이는 실제 상환 계획(초반에 더 많이, 후반에 적게 냄)과 달라 YMYL 성격상 오정보에 가깝다.
따라서 **"1회차 상환액 → 마지막 회차 상환액"의 범위(구간)로 표시**한다. 이 방식이 "매달
줄어든다"는 사실 자체를 결과에서 바로 드러내 오해 소지가 가장 적다고 판단했다.

```
매월 상환액 (원금균등상환, 회차마다 감소)

┌─────────────┐      ┌─────────────┐
│ 1회차        │  →   │ 마지막 회차   │
│ 1,483,333 원 │      │   837,500 원 │
└─────────────┘      └─────────────┘
회차가 지날수록 원금이 줄어들며 매월 상환액도 함께 감소합니다.
```

- 레이아웃: `flex items-center justify-between gap-3` (데스크톱), 모바일은 4장에서 별도 규정
- 숫자 크기: **`text-3xl sm:text-4xl` (30px→36px)** — design-system.md Result 스케일
  규정("모바일에서도 28px 이하로 축소 금지")을 준수하는 선에서, 두 숫자를 나란히 배치해야
  하므로 단일 히어로 숫자(36~40px)보다 한 단계 축소했다. 두 숫자 모두 동일 크기·Accent 컬러,
  `tabular-nums` 적용.
- 두 블록 사이 화살표(`→`, 데스크톱) 또는 아래방향 화살표(모바일, 4장)는 Text Secondary
  컬러의 단순 텍스트/아이콘, 장식적 강조(색상 변경 등) 넣지 않음.
- 각 블록 상단 라벨(Body Small, Text Secondary): "1회차" / "마지막 회차"
- 하단 보조 설명 1줄(Body Small, Text Secondary): 감소 추세를 명시하는 문구(정확한 카피는
  기획팀)

> 이 "1회차/마지막회차" 방식은 design-system.md에 없던 "결과 카드 내 2개 병렬 숫자" 신규
> 패턴이다. 10장에서 마스터 확인 항목으로 재정리한다(다만 진행을 막을 사안은 아니라고 판단해
> 우선 이 안으로 확정하고 진행함).

### 3-4. 보조 항목 리스트 (두 모드 공통)

주요 숫자 아래 구분선 다음, 총 이자와 총 상환액을 나열한다. Result 스케일을 쓰지 않는
"2차 지표"로, 히어로 숫자와 시각적 위계를 명확히 구분한다.

```
총 이자           1,234,568,00 원
총 상환액         51,234,568,00 원
```

- 각 행: `flex items-center justify-between py-1.5 text-sm`
- 라벨(좌): `text-brand-text-secondary`
- 값(우): `font-semibold tabular-nums text-brand-text` (Accent 컬러 아님 — 컬러는 최상단
  히어로 숫자에만 사용해 시선 위계를 지킨다), 천단위 콤마 필수
- 2행 고정(총 이자 / 총 상환액), 추가 지표(예: 회차별 상세 스케줄표)는 이번 스펙 범위 밖 —
  필요 시 별도 "상세 상환 스케줄" 섹션은 후속 스펙에서 논의(10장 참고)

## 4. 모바일(375px) 반응형 규칙

### 4-1. 무스크롤 목표 재검토 — 이 페이지는 다른 계산기와 달리 목표 달성이 어렵다

BMI/D-Day 문서와 같은 방식으로 iPhone SE급(375×667, 실사용 높이 약 550~600px) 기준
개략 높이를 추정했다.

| 영역 | 예상 높이 |
|---|---|
| 헤더(`h-16`) | 64px |
| Breadcrumb + H1 + 요약문 | 약 160px |
| **상단 면책 배너(신규, 6장)** | 약 72px |
| 계산기 카드 전 여백(`mt-6`) | 24px |
| 카드 상단 padding | 16px |
| 상환방식 토글(라벨+`h-11`) | 약 57px |
| gap-4 | 16px |
| 대출 원금 필드(라벨+`h-12`) | 약 74px |
| gap-4 | 16px |
| 연이자율/대출기간 2열 행 (대출기간 쪽이 입력+단위토글로 더 높음) | 약 116px |
| gap-4 | 16px |
| 계산 버튼(`h-12`) | 48px |
| 카드 하단 padding | 16px |
| **입력 폼까지 소계** | **약 695px** |
| 결과 카드(라벨+주요숫자+보조항목 2행, padding 포함) | 약 260~280px |
| **전체 합계** | **약 955~975px** |

→ **입력 폼만으로도 이미 375×667 뷰포트(약 600px 실사용 높이)를 초과하며, 결과까지
포함하면 iPhone 12~14급(실사용 높이 약 700~750px)에서도 한 화면에 들어오지 않는다.**

**결론 및 절충안**: 이 페이지는 design-system.md 4번 원칙 1("입력→결과 무스크롤")을
문자 그대로 달성할 수 없다고 판단한다. 필드 수(4종)와 결과 지표 수(모드별 최대 4개 숫자:
1회차/마지막회차/총이자/총상환액)가 만 나이·D-Day·BMI 계산기보다 구조적으로 많기 때문이며,
입력 필드를 억지로 압축하면(예: 라벨 생략, 입력 높이 축소) 44px 터치 타겟 기준이나 YMYL
페이지에 요구되는 정보 명확성이 훼손될 위험이 커서 압축보다 아래 절충안을 택한다.

1. **상단 배너 광고는 이 페이지에 넣지 않는다** (모바일·데스크톱 공통). 이유는 지면 예산
   문제도 있지만, 그보다 이 페이지 최상단은 광고보다 면책 배너가 우선이어야 한다는 원칙을
   지키기 위함(6장).
2. **입력 폼 자체는 최대한 압축**해 스크롤 1회 이내로 도달 가능한 범위(~700px대)에 두는
   것을 목표로 하고, 결과는 스크롤 이후 노출되는 것을 허용한다.
3. **계산 버튼 클릭 시 결과 영역으로 자동 스크롤(smooth scroll)** 하는 UX를 개발팀에
   권장한다 — 사용자가 버튼을 누른 뒤 결과를 놓치지 않도록 하는 보완책. 이는 인터랙션
   동작에 대한 제안이며, 실제 구현 여부·방식은 개발팀 재량(기능 로직 범위이므로 강제하지
   않음).
4. 기간 단위 토글(`h-9`, 36px)만 예외적으로 44px 터치 타겟 기준에 못 미친다(2-3 (B) 참고).
   대출기간 숫자 입력(`h-12`, 48px)과 인접해 있고 오조작 시 되돌리기 쉬운 보조 옵션이라
   허용 가능한 예외로 판단했으나, 개발팀이 터치 여백(`padding`)을 추가로 확보해 실질 터치
   영역을 넓히는 것은 권장한다.

### 4-2. 모드 B(원금균등) 결과 카드의 모바일 배치

3-3의 "1회차 → 마지막 회차" 가로 배치(`flex justify-between`)는 375px 폭에서 두 블록 폭이
좁아져(각 카드 border-l-4 padding 감안 시 약 140px) 7자리 숫자(`1,483,333`)가 줄바꿈되거나
잘릴 위험이 있다. **모바일(375px, `sm` 미만)에서는 세로 스택으로 전환**한다.

```
1회차
1,483,333 원
   ↓
마지막 회차
837,500 원
```

- `sm:flex-row` (즉 기본은 `flex flex-col gap-2`, `sm:flex-row sm:items-center sm:justify-between`)
- 화살표도 `→`(가로, `sm` 이상) ↔ `↓`(세로, 모바일)로 전환
- 숫자 크기는 4-1과 동일하게 `text-3xl`(모바일) 유지 — 세로 스택이라 가로 폭 제약이
  사라지므로 축소할 필요 없음

### 4-3. 상환방식 토글의 모바일 텍스트 처리

"원리금균등상환"/"원금균등상환" 각 5~6자 한글 라벨이 375px에서 2등분된 버튼(각 약
165px 폭) 안에 들어가는지 확인 필요 — `text-sm font-semibold` 기준 실측상 한 줄에 들어갈
가능성이 높으나, 폰트 로딩(Pretendard) 전 대체 폰트 렌더링 시 폭이 달라질 수 있으므로
`whitespace-nowrap`을 강제하지 말고 **필요 시 자동 줄바꿈을 허용**한다(버튼 높이가
2줄일 때 `h-11` 대신 `min-h-11`로 유연하게). 줄바꿈되어도 두 버튼 높이가 같아지도록
`items-stretch` 유지.

## 5. 데스크톱 레이아웃

- 컨테이너: 기존과 동일 `mx-auto max-w-3xl px-4 py-8 sm:px-6 sm:py-12`.
- 2-1의 필드 순서·2열 그리드(연이자율/대출기간)는 데스크톱에서도 동일 유지.
- 원금균등상환 결과의 "1회차 → 마지막 회차"는 데스크톱에서 가로 배치(4-2 참고), 두 블록
  사이 여유 폭이 충분해 숫자 잘림 없음.
- 사이드바 2단 구성은 도입하지 않는다(만 나이/D-Day/BMI와 통일, 1장 참고).
- 데스크톱은 세로 공간 여유가 있어 4-1의 무스크롤 절충안(스크롤 허용)이 모바일보다 부담이
  적다 — `lg` 이상에서는 입력 폼+결과 카드가 대부분 한 화면 또는 최대 1.2스크롤 이내에
  들어올 것으로 예상(별도 실측표는 생략, 실 구현 후 6장 검수 시 확인 예정).

## 6. YMYL 리스크 반영 — 면책 문구 배치 및 CTA 금지 원칙 (이 페이지에서 가장 중요)

`planning/calculator-lineup.md` 50행: "대출 상담/자문이 아닌 단순 참고용 계산 도구'임을
명확히 고지, 실제 금리·상환조건은 금융기관 확인 유도 문구 필수. 광고 배치 시 '지금 대출
신청' 류 오인 유발 디자인 지양"을 레이아웃에 아래와 같이 반영한다.

### 6-1. 상단 면책 배너 (신규 요소 — 이 페이지에서 처음 도입)

기존 라이브 계산기(만 나이/D-Day/BMI) 어디에도 없던 요소다. 면책 문구는 지금까지
`ResultInterpretation` 본문 하단(결과 계산 후에만 노출)에만 있었는데, 이 페이지는 YMYL
리스크가 "높음"이라 **계산 실행 여부와 무관하게 페이지 진입 즉시 보이는 위치**에 한 번 더
배치한다.

- 위치: H1/요약문 아래, 계산기 카드 위 (1장 와이어프레임 참고)
- 스타일: `rounded-lg border border-amber-200 bg-amber-50 px-4 py-3 flex items-start gap-2`
  (design-system.md Warning 컬러 계열 재사용, 신규 색상 추가 아님)
- 아이콘: 좌측 `ⓘ` 또는 느낌표 아이콘(Warning 컬러), 장식이 아니라 "주의" 의미 전달용
- 텍스트: `text-sm text-brand-text-secondary` (또는 대비를 위해 `text-amber-900` 계열 — 최종
  색상은 개발팀 구현 시 WCAG AA 대비 확인), 정확한 카피는 기획팀 담당이며 아래 골자를 포함
  해야 한다: "이 계산기는 대출 상담·신청이 아닌 참고용 도구이며, 실제 금리·한도·상환조건은
  금융기관마다 다릅니다."
- **광고 유닛이 아니다** — "광고" 캡션을 붙이지 않으며, 항상 노출(닫기 버튼으로 숨길 수 있게
  할지는 개발팀 재량이나 기본 상태는 항상 노출 권장)

### 6-2. 결과 해석 텍스트 하단 면책 문구 (기존 패턴 재사용)

`planning/calculator-lineup.md` 3-1 5단계 구조 그대로: `ResultInterpretation` 본문 최하단에
"본 계산 결과는 참고용이며 법적/의료적/재무적 효력이 없습니다..." 공통 면책 문구 + 대출
특화 문구("정확한 금리·한도는 금융감독원 금융상품통합비교공시 등 공식 채널 또는 해당
금융기관에 확인하시기 바랍니다" 류)를 배치한다. 실제 문장은 기획팀 콘텐츠안에서 확정.

### 6-3. CTA 디자인 금지 원칙의 레이아웃 반영

- 계산기 카드 안에는 **"대출 이자 계산하기" 버튼 1개만** 존재한다. "지금 대출 신청",
  "대출 상담 받기", "최저금리 확인하기" 류의 2차 버튼을 결과 카드나 계산기 카드 주변에
  절대 추가하지 않는다.
- 결과 해석 텍스트 안에서 공식 기관(금융감독원 등)을 링크할 필요가 있다면, **일반 본문
  링크 스타일**(밑줄 + Primary 컬러, 인라인 텍스트)로만 표기하고 버튼(`bg-brand-primary`
  배경의 박스형 CTA) 스타일을 절대 사용하지 않는다 — 버튼 스타일은 계산 버튼 전용으로
  예약한다.
- "관련 계산기" 카드(design-system.md 3-4)는 계산기 내부 링크로만 구성하고, 외부 대출
  비교/신청 서비스로 연결되는 카드는 넣지 않는다.
- 6-4 광고 규칙과 별개로, **광고 소재 자체(애드센스가 게재하는 크리에이티브)는 사이트가
  직접 통제할 수 없다.** 따라서 레이아웃 차원에서 할 수 있는 최선은 "광고 영역을 콘텐츠와
  명확히 시각적으로 분리"하는 것뿐이며, 이는 6-4에서 구체화한다.

### 6-4. 광고 배치 (design-system.md 5장 + 이 페이지 전용 추가 규칙)

- 상단 배너 광고: **이 페이지는 넣지 않는다** (4-1 절충안, 6-1 면책 배너가 그 자리를 대체).
- 본문 중간 광고: 결과 해석 텍스트와 FAQ 사이(margin 24px 이상), "광고" 캡션 라벨 필수
  표기(design-system.md 5장 공통 규칙).
- 사이드바(`lg` 이상): 현재 페이지 구조가 사이드바 없는 단일 컬럼이라 해당 없음(5장).
- 하단: 관련 계산기 카드 다음, 푸터 위.
- **이 페이지 전용 추가 규칙**: 광고 컨테이너의 배경/테두리를 결과 카드(연한 에메랄드 틴트
  `bg-emerald-50` + 좌측 강조선)나 면책 배너(연한 amber 틴트)와 유사한 톤으로 만들지 않는다
  — 대출/금융 광고 소재가 우연히 "지금 신청" 같은 문구를 포함하더라도, 사이트 자체 UI(결과·
  경고 영역)와 명확히 다른 중립적 배경(White 또는 Gray 50, 얇은 회색 테두리)으로 감싸
  콘텐츠와 혼동되지 않게 한다. "광고" Caption 라벨은 이 페이지에서는 다른 어떤 계산기보다도
  누락 없이 반드시 표기할 것을 강조한다(리스크가 높은 페이지이므로).

| 관련 계산기 | `getRelatedCalculators`는 `status: "live"`인 항목만 노출. 이 문서 작성 시점 기준 `/loan` 카테고리에는 라이브 계산기가 이 페이지 자신뿐이라, 라이브 전환 직후에는 "같은 카테고리 2개+다른 카테고리 1개"(`calculator-lineup.md` 3-3) 원칙을 채울 수 없고 age/dday/bmi 등 타 카테고리 계산기만 노출된다 — BMI 문서에서도 동일하게 지적된 사안, 로드맵 진행(DSR·중도상환수수료 계산기 라이브화)에 따라 자연 해결되므로 지금 조치 불필요 |

## 9. 대안으로 검토했으나 채택하지 않은 안

- **결과를 "평균 월 상환액" 1개 숫자로 통일 표시하는 안**: 구현이 가장 간단하고 다른
  계산기와 시각적으로도 가장 통일되지만, 3-3에서 설명한 오인 위험(고정 금액으로 착각) 때문에
  기각했다. YMYL 페이지에서는 구현 단순성보다 정보 정확성을 우선했다.
- **대출기간 단위(년/개월)를 드롭다운(`<select>`)으로 처리하는 안**: 토글보다 구현이
  간단하지만, 이 페이지의 다른 요소(상환방식)가 이미 토글 패턴을 쓰므로 페이지 내 상호작용
  방식을 통일하기 위해 토글로 결정했다. 드롭다운이 개발 공수상 유리하다면 개발팀 판단으로
  대체 가능(시각적 일관성은 다소 낮아지나 기능상 문제 없음) — 이 부분은 강제하지 않는다.

## 10. 마스터 팀장 확인 필요 사항

1. **콘텐츠 기획안과의 상충 가능성**: `planning/loan-interest-calculator-content.md`가 아직
   없어 본 문서는 마스터 지시서 원문(상환방식 2종, 출력 3종)만 근거로 작성했다. 기획안이
   나온 뒤 (a) 상환방식 명칭 표기, (b) 원금균등상환 결과를 "1회차/마지막회차 범위"로 보여주는
   방식에 대해 콘텐츠팀 카피(해석 텍스트·FAQ)가 동일한 전제로 쓰였는지 반드시 대조 확인이
   필요하다. 상충 시 이 레이아웃 또는 콘텐츠 중 어느 쪽을 조정할지 마스터 판단 필요.
2. **신규 컴포넌트(세그먼트 토글) 2종의 design-system.md 공식 편입 여부**: 2-3에서 정의한
   상환방식/기간단위 토글은 향후 DSR·중도상환수수료 계산기(로드맵 10·12순위)에서도 재사용될
   가능성이 높다. `design-system.md` 3장에 정식 컴포넌트로 추가할지 확인 요망.
3. **결과 카드 "1회차 → 마지막 회차" 병렬 숫자 표시(3-3)**: design-system.md의 Result 스케일
   규정(단일 히어로 숫자 36~40px)을 벗어나는 첫 사례로, 두 숫자를 30~36px로 나란히 배치하는
   신규 패턴이다. BMI 문서에서 `brand-danger` 색상 확장을 마스터 확인 후 팔레트에 반영한
   전례처럼, 이 병렬 숫자 패턴도 공식 반영 여부 확인 후 `design-system.md`에 소급 반영할지
   결정 필요. (승인 전까지는 이 문서의 제안 단계로만 취급하며, 진행 자체를 막지는 않음)
4. **상단 면책 배너(6-1)의 페이지 공통화 여부**: 이번에는 대출이자 계산기 전용으로
   신규 도입했으나, 로드맵상 DSR·퇴직금·4대보험 등 YMYL "중간" 이상 등급 계산기에도 동일한
   상단 배너를 공통 적용할지(`calculator-lineup.md`가 요구하는 면책 문구 필수 규정과 연결)
   마스터 팀장 판단 필요 — 사이트 전체 톤앤매너에 영향을 주는 결정이라 임의로 다른
   계산기에까지 확장 적용하지 않았다.
