## 0. 이 페이지가 다른 계산기와 다른 점 (설계 전제)

지금까지의 계산기(BMI·대출·연봉)와 근본적으로 다른 성격이라, 몇 가지 패턴을 의도적으로
분기한다. 각 분기는 근거를 명시하고 §11에서 마스터에게 governance 확인을 요청한다.

1. **실시간 변환(계산 버튼 없음) — 사이트 첫 "실시간 계산기"다.** BMI·연봉은 `form onSubmit` +
   "계산하기" 버튼 후 결과 카드가 나타나는 방식이지만, 단위 변환기는 입력하는 즉시 결과가
   바뀌는 것이 도구의 존재 이유다(§3-1 근거). 이 분기는 사이트 계산기 인터랙션 패턴을 하나 더
   늘리는 것이라 §11에서 마스터 확인.
2. **양방향(from↔to) + 스왑.** 단일 방향 "입력→결과"가 아니라 두 단위 사이를 오가며 쓴다(§3-3).
3. **하나의 페이지에서 5개 카테고리를 전환**(길이/무게/면적/부피/온도)하며 단위 드롭다운 목록이
   카테고리에 따라 통째로 교체된다(§2, §3-5).
4. **온도만 곱셈이 아니라 오프셋 변환**(℉=℃×9/5+32, K=℃+273.15)이라 스왑·역변환 구현에 주의가
   필요하다(§4).
5. **YMYL 등급 = 없음(일반 생활 유틸).** 연봉(중간)·대출(높음)과 달리 면책 배너·CTA 가드가
   불필요하고, 결과에 위험/경고 색을 부여하지 않는다. 결과 강조는 순수하게 "값 확인" 목적의
   Accent(에메랄드) 하나로 고정한다.
6. **design-system.md 4장 원칙 1("모바일에서 입력→결과 무스크롤")을 이 계산기는 실제로 달성한다.**
   입력·결과가 모두 콤팩트해서(연봉처럼 공제 10줄이 없다) 모바일 1스크린 안에 카테고리 탭 + 입력 +
   결과가 전부 들어온다(§7-4). 이 사이트에서 원칙 1을 문자 그대로 만족하는 몇 안 되는 계산기다.

## 1. 페이지 전체 레이아웃 (와이어프레임)

`app/calculator/[slug]/page.tsx` 기존 골격을 그대로 따른다. 이번 스펙의 신규 설계 범위는
"카테고리 탭 → 변환기 카드 → 참조표"이며, 이 셋은 모두 `UnitConverter` 컴포넌트 내부다.

```
[Breadcrumb: 홈 / 단위 변환 계산기]
[H1] 단위 변환 계산기
[본문 요약 1줄]
──────────────────────────────
[UnitConverter 컴포넌트]  ← 이번 스펙 범위 (2~7장)
  [카테고리 탭]  길이 · 무게 · 면적 · 부피 · 온도     ← §2 (기본 선택 = 면적)
  [변환기 카드]                                       ← §3
    [입력]   [ 20        ] [ 평 ▾ ]
                  ( ⇅ 스왑 )                          ← §3-3
    [결과]   ┌─────────────────────────────┐
             │ 결과            [ 제곱미터 ▾ ]│
             │ 66.12 ㎡                     │          ← 히어로(에메랄드), 실시간
             │ 20평은 약 66.12㎡입니다.      │          ← 동적 요약 1줄
             └─────────────────────────────┘
  [자주 쓰는 환산 참조표]  (선택 카테고리에 따라 교체)   ← §6
──────────────────────────────
[결과 해석 / 설명글]  ← ResultInterpretation 재사용 (콘텐츠=기획팀, 공통 템플릿 — 범위 밖)
[본문 중간 광고]      ← design-system.md 5장 공통 규칙 (범위 밖, §10에서 원칙만 재확인)
```

(연봉·대출 문서와 동일 판단; 사이드바 도입은 페이지 공통 이슈로 범위 밖).

## 2. 카테고리 선택 UI — [탭 vs 셀렉트 결정]

### 2-1. 결정: **가로 세그먼트 탭 5등분 (셀렉트 아님, 가로 스크롤 아님)**

**근거 (모바일 375px 실측 기준):**

- 카테고리 라벨 5종이 모두 **정확히 2글자**다: `길이`·`무게`·`면적`·`부피`·`온도`. 라벨이 짧아
  5개가 한 줄에 여유 있게 들어간다.
- 375px에서 페이지 좌우 여백(약 16px×2)을 빼면 가용 폭 ≈ **343px**. 5등분 시 세그먼트당
  **약 68.6px**. 2글자(`text-sm` 14px, 약 28px) + 좌우 패딩이 68px 안에 충분히 들어간다 →
  **가로 스크롤 불필요**, 5개 전부 항상 노출.
- **셀렉트보다 탭이 우월한 이유**: (1) 카테고리 전환이 이 페이지의 **1순위 상시 인터랙션**이라
  5개 선택지를 항상 보여주는 편이 발견성·전환속도에서 유리. 셀렉트는 "탭 → 펼침 → 스크롤 →
  선택"의 3동작이 필요하고 나머지 4개 카테고리가 접혀 보이지 않는다. (2) 탭은 "지금 어떤
  카테고리인지"를 활성 상태로 즉시 보여준다. (3) 5개 2글자라는 조건에서 탭이 가로로 깔끔히
  들어가므로 셀렉트를 쓸 이유(공간 절약)가 없다.
- **폴백 규칙**: 만약 향후 카테고리가 늘거나 라벨이 길어져 한 줄에 안 들어가게 되면, 그때
  `overflow-x-auto flex-nowrap`(가로 스크롤) 또는 2행 랩으로 전환한다. **현재 5×2글자에서는
  스크롤·랩 모두 불필요**하며, 등분 탭이 정답이다.

### 2-2. 탭 스타일 스펙 (세그먼트 컨트롤)

design-system.md에 아직 정의되지 않은 신규 컴포넌트 유형이다(§11 편입 확인). 기존 대출/연봉의
세그먼트 토글 톤과 일관되게, iOS식 세그먼트 컨트롤로 정의한다.

- 트랙(컨테이너): `flex w-full rounded-lg border border-brand-border bg-brand-bg p-1`
  (`bg-brand-bg` = design-system Background Gray 50).
- 각 탭 버튼: `flex-1 rounded-md text-sm font-medium transition-colors`,
  **최소 높이 44px**(`min-h-11`, 터치 타깃 확보 — design-system 4장 원칙 3).
  - 비활성: `text-brand-text-secondary` (배경 없음, 트랙 색 그대로).
  - 활성: `bg-brand-surface(White) text-brand-primary font-semibold shadow-sm`.
- 대안(진한 버전): 활성 `bg-brand-primary text-white`. 5개 중 하나만 강조하는 세그먼트 성격상
  위 "흰 카드 + Primary 텍스트"의 은은한 버전을 1안으로 하되, 최종 명도는 §11 design-system 편입
  시 확정(개발팀이 둘 중 택1 가능).
- 접근성: `role="tablist"` + 각 탭 `role="tab"` `aria-selected`, 또는 라디오 그룹
  (`role="radiogroup"`)로 구현. 키보드 좌우 화살표 이동 지원 권장.

### 2-3. 카테고리 전환 동작

- **기본 선택 = 면적**(마스터 확정: 평↔㎡ 부동산 수요가 대표 유스케이스). 초기 로드 시 활성 탭 =
  `면적`, from = `평`, to = `제곱미터(㎡)`(§3-5 기본 단위쌍).
- 카테고리 탭을 바꾸면: (1) from/to 단위 셀렉트의 **옵션 목록이 해당 카테고리 단위로 통째 교체**
  (§3-5), (2) from/to는 그 카테고리의 **기본 단위쌍**으로 세팅, (3) `자주 쓰는 환산 참조표`(§6)도
  해당 카테고리 표로 교체.
- 입력값 유지 여부: 카테고리 전환 시 입력값은 **해당 카테고리 기본 예시값으로 리셋**을 권장한다
  (단위가 완전히 달라져 이전 숫자를 유지하면 무의미한 결과가 되기 때문). 기본 예시값은 §3-4의
  프리필과 동일 정책. (개발팀 재량으로 "숫자 유지"도 허용하나, 리셋이 혼란이 적다.)

## 3. 변환기 본체 (입력 → 스왑 → 결과)

### 3-1. 결정: **실시간 변환 (계산 버튼 없음)**

**근거 (BMI/연봉 버튼 방식과의 일관성 고려 포함):**

- **도구의 본질이 다르다.** BMI·연봉은 "여러 입력을 채우고 한 번 계산해 결과를 정독"하는
  1회성 계산이며, YMYL이라 "계산하기"라는 명시적 확정 동작이 자연스럽다. 반면 단위 변환기는
  "값을 계속 바꿔가며 여러 번 탐색"(20평? 25평? 33평?)하고 단위를 스왑하는 **연속 탐색 도구**다.
  키 입력마다 버튼을 눌러야 하면 핵심 루프에 마찰이 생긴다.
- **프로젝트 배경 문서의 명시 지침과 부합**: "계산기 페이지는 입력 → 즉시 결과가 뜨는 인터랙션이
  핵심(체류시간 확보)". 단위 변환은 이 즉시성이 가장 잘 들어맞는 유형이다.
- **YMYL 위험이 없다.** 연봉처럼 세액 근사 고지가 필요한 값이 아니라 결정론적 단일 공식이므로,
  "확정 버튼"으로 사용자를 한 박자 멈춰 세울 이유가 없다.
- **업계 관례**: 구글/스마트폰 기본 변환기 모두 실시간이라 사용자 기대와 일치한다.
- **일관성에 대한 답**: 버튼 방식을 깨는 것은 맞지만, 이는 "계산기 유형에 따라 인터랙션이 다른"
  의도된 분기다(단발 계산=버튼 / 연속 변환=실시간). §11에서 마스터에게 이 이원화를 사이트
  공통 원칙으로 승인받기를 요청한다.

**구현 함의**: `form`/`onSubmit`/`handleSubmit`을 쓰지 않는다. `onChange`로 입력·단위·카테고리
state를 갱신하고 결과는 파생 계산한다. "계산하기" 버튼과 계산 후 `scrollIntoView` 스무스 스크롤은
**넣지 않는다**(결과가 항상 화면에 있으므로 불필요).

### 3-2. 입력부 스펙 (변환할 값 + from 단위)

- 라벨: `변환할 값` — `text-sm font-medium text-brand-text-secondary`(BMI 라벨과 동일).
- 배치: `grid grid-cols-[1fr_1.2fr] gap-3`(값 입력 좌, 단위 셀렉트 우; 단위명이 길어 셀렉트에
  폭을 조금 더 준다). 폭이 부족하면 `grid-cols-2`로 폴백 — 어느 쪽이든 §7-3의 이유로 가로
  오버플로는 발생하지 않는다.
- **값 입력**:
  - `type="text"` + `inputMode="decimal"`. (BMI는 `type="number"`지만, 여기서는 **온도 음수 입력**
    과 소수점 입력 자유도를 위해 `text` 권장 — 일부 모바일 키보드에서 `number`+`decimal`이 마이너스
    키를 숨기는 문제 회피. 개발팀이 `type="number"`로 가려면 `min` 미설정 + 음수 허용 확인.)
  - 스타일: 연봉 컴포넌트의 `inputBase` 상수 그대로 재사용
    (`h-12 rounded-lg border px-4 text-base text-brand-text outline-none transition-colors
    focus:border-brand-primary focus:ring-4 focus:ring-brand-primary/15`), 테두리 `border-brand-border`.
  - **입력 필드에는 천단위 콤마 실시간 서식을 적용하지 않는다**(§5-1 근거). `inputMode="decimal"`로
    원시 숫자/소수점/마이너스를 자유롭게 입력.
- **from 단위 셀렉트**: 연봉의 `selectBase` 상수 재사용
  (`h-12 w-full appearance-none rounded-lg border border-brand-border bg-white pl-4 pr-9 ...`),
  우측 `pr-9`에 chevron `▾`를 `pointer-events-none absolute`로 겹쳐 배치. 옵션 목록 = 현재
  카테고리의 단위 목록(§3-5).

### 3-3. 스왑 버튼 (from↔to)

- 위치: 입력부와 결과부 **사이 중앙**. 세로 스택 레이아웃이므로 아이콘은 **`⇅`(상하 화살표)**
  (데스크톱·모바일 동일). 두 행 사이 간격에 살짝 겹치게(`-my-2`, `z-10`) 배치해 "두 값을 잇는
  스위치"임을 시각적으로 전달.
- 스타일: `h-11 w-11 rounded-full border border-brand-border bg-brand-surface`(44px 터치 타깃),
  hover 시 `border-brand-primary text-brand-primary`. 아이콘 `text-brand-text-secondary`.
- 접근성: `type="button"` `aria-label="입력 단위와 결과 단위 바꾸기"`.
- **동작(중요)**: 클릭 시 `from`↔`to` 단위를 교환하고, **직전 결과값을 입력값으로 이동**시킨다.
  즉 "20평 → 66.12㎡" 상태에서 스왑하면 "66.12㎡ → 20평"이 되어 방향이 자연스럽게 뒤집힌다.
  - **드리프트 방지(개발 주의)**: 입력으로 옮길 값은 **화면 표시용으로 반올림된 문자열이 아니라
    full-precision 결과값**이어야 한다. 반올림값을 되먹이면 스왑을 반복할수록 오차가 누적된다.
  - 온도는 오프셋 변환이라 이 스왑이 특히 민감하다 — §4 참조(정방향/역방향 공식을 별도로 짜지
    말고 §4의 canonical-base 방식으로 구현하면 스왑이 자동으로 정확해진다).

### 3-4. 결과부 스펙 (결과값 + to 단위) — design-system 3-2 결과 카드

design-system.md 3-2(결과 표시 = 시각적 위계 최상위)를 그대로 적용. **버튼 방식과 달리 결과
카드는 조건부가 아니라 항상 렌더**되며 실시간 갱신된다.

- 카드: `mt-4 rounded-xl border-l-4 border-brand-accent bg-gradient-to-br from-emerald-50 to-white
  p-4 sm:p-6`(연봉 결과 카드와 동일 골격, BMI처럼 상태별 색 분기 없음 — 항상 Accent 고정).
- 카드 상단 행(`flex items-center justify-between`):
  - 좌: 라벨 `결과` — `text-sm font-medium text-brand-text-secondary`.
  - 우: **to 단위 셀렉트** — `selectBase` 재사용(단, 폭은 내용에 맞게 `w-auto` 또는 최대폭
    제한). 에메랄드 카드 위 흰색 셀렉트라 "선택 가능한 컨트롤"임이 분명히 드러난다.
- 히어로 결과값: `mt-1 flex items-baseline gap-2 tabular-nums text-4xl font-bold sm:text-[2.5rem]
  text-brand-accent`(BMI/연봉 히어로와 동일). 뒤에 단위 기호(㎡, ㎏ 등)를 `text-base font-medium
  text-brand-text-secondary`로 작게 병기(design-system 3-2 단위 규칙).
- 동적 요약 1줄: `mt-2 text-sm text-brand-text-secondary`. 예: "20평은 약 66.12㎡입니다." — BMI의
  `getResultSummary(...)`처럼 컴포넌트 내부 함수로 입력값·단위·결과를 받아 조립(SEO/가독성 +
  체류시간). 정적 텍스트 금지, 실시간 동적 생성. 정식 카피 톤은 기획팀(§11).
- **초기/빈 상태**: 프리필된 기본 예시가 항상 살아있는 결과를 보여준다(도구 작동 방식을 즉시
  시연 — AdSense 리뷰어에게도 유리). 입력이 비거나 숫자로 해석 불가하면 히어로를 `—` 또는
  `0`으로 조용히 표시하고 요약 문구는 숨긴다. **BMI/연봉식 빨간 에러 배너는 쓰지 않는다**(실시간
  유틸에서 매 키 입력마다 경고를 띄우면 거슬림 — §8).
  - 기본 프리필(면적): 값 `1`, from `평`, to `㎡` → 결과 `3.31㎡`. (참조표의 10평=33.06 등과
    이어지도록 값 `10`을 쓰는 것도 가능 — 정확한 기본 예시값·단위쌍은 기획팀 확정, §11.)
- **트랜지션 주의**: design-system 3-2는 결과 갱신 시 미세 페이드/스케일을 권하지만, 실시간
  입력에서는 키 입력마다 애니메이션이 겹쳐 깜빡일 수 있다. **카드 배경에만 `transition-colors
  duration-200`을 두고, 히어로 숫자 텍스트 자체에는 매 입력 애니메이션을 걸지 않는다**(플리커
  방지). 카테고리/스왑 등 "큰 전환"에서만 은은한 트랜지션 허용.
- 접근성: 히어로 값 컨테이너에 `aria-live="polite" aria-atomic="true"` — 버튼이 없으므로
  스크린리더가 실시간 갱신 결과를 낭독하도록 보장(입력이 멈춘 뒤 조용히 안내).
- (선택 강화) "결과 복사" 버튼: 히어로 옆 작은 아이콘 버튼으로 결과값 클립보드 복사 — 유틸리티성·
  체류시간에 도움. 필수 아님, §11에서 옵션으로 제안(개발 로직).

### 3-5. 카테고리별 단위 목록 (개발팀 참고 — 최종 확정은 기획팀)

아래는 디자인팀 **제안**이다. 실제 단위 구성(특히 한국 전통 단위 평·근·관·홉·되·말)은 SEO 검색
수요와 직결되므로 **기획팀이 최종 확정**한다(§11). 내부 변환은 §5·§4의 canonical-base 방식으로
"입력→기준단위→목표단위" 2단계로 구현한다.

| 카테고리 | 기준단위(내부) | 단위 목록(제안) | 기본 from→to(제안) |
|---|---|---|---|
| 길이 | 미터(m) | 밀리미터㎜·센티미터㎝·미터m·킬로미터㎞·인치in·피트ft·야드yd·마일mi | 센티미터㎝ → 인치in |
| 무게 | 그램(g) | 밀리그램㎎·그램g·킬로그램㎏·톤t·온스oz·파운드lb·근(600g)·관(3.75㎏) | 킬로그램㎏ → 파운드lb |
| 면적 | 제곱미터(㎡) | 제곱미터㎡·제곱킬로미터㎢·헥타르ha·아르a·평·제곱피트ft²·에이커ac | **평 → 제곱미터㎡** (기본 카테고리) |
| 부피 | 리터(L) | 밀리리터㎖·리터L·세제곱미터㎥·갤런gal·액량온스floz·홉·되·말 | 리터L → 갤런gal |
| 온도 | 섭씨(℃) | 섭씨℃·화씨℉·켈빈K | 섭씨℃ → 화씨℉ |

- 셀렉트 옵션 표기: `제곱미터 (㎡)`처럼 "한글명 (기호)" 형태 권장(SEO·이해도). 닫힌 셀렉트에서
  긴 옵션명은 브라우저가 자연 클리핑하므로 **가로 오버플로를 유발하지 않는다**(§7-3).
- 면적의 평↔㎡ 계수는 `1평 = 3.305785㎡`(=400/121)로, 참조표의 10평=33.06·20평=66.12·30평=99.17·
  33평=109.09와 일치한다(§6).

## 4. 온도 변환 특수 처리 (오프셋 — 스왑/역방향 주의)

온도는 비례(곱셈) 변환이 아니라 **오프셋 포함 변환**이라 스왑·역방향에서 실수가 나기 쉽다.

- 공식: `℉ = ℃ × 9/5 + 32`, `K = ℃ + 273.15`. (역: `℃ = (℉ − 32) × 5/9`, `℃ = K − 273.15`.)
- **구현 원칙 = canonical-base(기준단위 경유)**: 다른 카테고리와 동일하게 "입력값 → 기준단위(℃) →
  목표단위" 2단계로 변환한다. 즉 어떤 단위든 우선 ℃로 바꾸고, ℃에서 목표 단위로 바꾼다. 이렇게
  하면 **정방향/역방향 공식을 개별로 하드코딩할 필요가 없고**, §3-3 스왑(from↔to 교환 + 결과값을
  입력으로 이동)이 자동으로 정확해진다.
  - **하지 말 것**: 다른 카테고리의 곱셈 계수처럼 온도를 "역방향 = 1/계수"로 처리하면 오프셋
    때문에 완전히 틀린다. 반드시 기준단위 경유 공식으로.
- **음수 입력 필수 지원**: `−40℃ = −40℉`, `−273.15℃ = 0K` 등. §3-2에서 값 입력을 `type="text"
  inputMode="decimal"`로 둔 이유가 이 마이너스 입력 보장이다.
- 반올림: 온도도 §5의 기본(소수 둘째 자리 반올림 + 후행 0 절삭) 규칙을 그대로 적용해도 자연스럽다
  (체온 36.5℃ → 97.7℉는 후행 0 절삭으로 `97.7`로 표기).
- 참조표(§6): 온도는 비례쌍이 없으므로 대표 지점들로 구성 — `0℃ = 32℉`, `36.5℃ = 97.7℉(체온)`,
  `37℃ = 98.6℉`, `100℃ = 212℉`, `−40℃ = −40℉`(교차점).

## 5. 표시 / 포맷 규칙

### 5-1. 천단위 구분(,) — 입력 아니오 / 결과 예

- **입력 필드: 천단위 콤마 서식 미적용.** 연봉 계산기는 입력에 실시간 콤마를 넣지만(정수 금액),
  단위 변환기는 소수점·마이너스 입력이 잦아 실시간 콤마 서식이 커서 점프·소수점 처리와 충돌한다.
  입력은 원시값(`inputMode="decimal"`)으로 자유롭게.
- **결과·요약·참조표: 정수부 천단위 콤마 적용**(`toLocaleString("ko-KR")` 계열). 큰 결과값
  (예 `1,000,000㎜`)의 가독성 확보. 모든 숫자는 `tabular-nums`로 자릿수 정렬(design-system 2장).

### 5-2. 소수 자리수 / 반올림 규칙

기획팀이 해설 예시를 **소수 둘째 자리 반올림**으로 적을 예정이므로, UI 기본도 이에 맞춘다.
다만 단위 변환은 아주 작은/큰 값이 흔해 단일 규칙만으로는 부족하므로 3단 규칙을 제안한다.

| 구간 | 규칙 | 예 |
|---|---|---|
| **일반** (0.01 ≤ \|값\| < 1e12) | 소수 **둘째 자리 반올림** + **후행 0 절삭** + 천단위 콤마 | `66.12㎡`, `33.06㎡`, `2m→200㎝`(200.00 아님), `97.7℉` |
| **아주 작은 값** (0 < \|값\| < 0.01) | 둘째 자리 반올림이 `0.00`이 되어 무의미 → **유효숫자 기준**(예 `toPrecision(4)`) + 후행 0 절삭 | `1㎜→0.000001㎞` (0.00 아님) |
| **아주 큰 값** (\|값\| ≥ 1e12 또는 표시 문자열 과다) | 콤마 표기가 히어로 폭을 넘길 위험 → §7-3 폰트 단계 축소 우선, 그래도 넘치면 **압축(과학) 표기** `1.2346 × 10¹²` 폴백 | `1㎢→1,000,000,000,000㎟` 등 극단값 |

- **후행 0 절삭**이 핵심: 항상 `.00`을 붙이면 `200.00㎝`처럼 어색해진다. 반올림 후 후행 0을 떼어
  `200㎝`로. 기획팀 예시(10평=33.06 등)는 둘째 자리가 0이 아니라 그대로 `33.06`으로 표기되어
  **모순 없음**.
- **일관성 확인 요청(§11)**: 기획팀 해설/참조표 수치가 위 "둘째 자리 반올림 + 후행 0 절삭"과
  어긋나지 않는지, 그리고 아주 작은/큰 값에 대한 표기(유효숫자·과학 표기)를 해설에서 언급할지
  기획팀이 확인.

## 6. 자주 쓰는 환산 참조표 — [포함 결정]

### 6-1. 결정: **포함한다.**

- **근거**: "10평 몇 제곱미터", "1마일 몇 킬로미터", "1근 몇 그램" 같은 **구체 수치 검색 수요**가
  매우 크다. 정적 참조표는 이 롱테일 검색을 흡수해 SEO·체류시간에 직접 기여하고, 사용자가 직접
  입력하기 전에도 즉시 답을 얻어 페이지 가치를 높인다. 마스터 지시서도 "있으면 SEO/체류시간에
  좋음"으로 긍정 검토를 요청했다.

### 6-2. 배치 · 스타일

- 위치: **변환기 카드 바로 아래**(컴포넌트 내부). 선택 카테고리에 종속되므로 카테고리 전환 시
  함께 교체된다.
- 블록 스타일: `mt-6 rounded-xl border border-brand-border bg-brand-surface p-4 sm:p-6`
  (변환기 카드와 구분되는 중립 카드). 제목 `자주 쓰는 [카테고리] 환산` — `text-sm font-semibold
  text-brand-text-secondary`.
- **행 구조 = `flex items-center justify-between py-1.5 text-sm`**(연봉 `AmountRow`와 동일 패턴).
  좌: "10평", 우: "33.06㎡"(`tabular-nums`). **HTML 다열 `<table>`을 쓰지 않는다** → 375px에서
  가로 오버플로가 구조적으로 발생할 수 없다(§7-3).
- 데스크톱에서는 여백이 남으므로 `sm:grid sm:grid-cols-2 sm:gap-x-8`로 참조 행을 2열로 배치해도
  좋다(각 열 내부는 여전히 `flex justify-between`). 모바일은 1열.

### 6-3. 카테고리별 참조 내용 (제안 — 정확한 항목/수치·SEO 카피는 기획팀 확정)

- 면적: `10평=33.06㎡`, `20평=66.12㎡`, `30평=99.17㎡`, `33평=109.09㎡` (+ 역방향 `1㎡=0.3025평` 등).
- 길이: `1인치=2.54㎝`, `1피트=30.48㎝`, `1야드=0.91m`, `1마일=1.61㎞`.
- 무게: `1근=600g`, `1관=3.75㎏`, `1파운드=453.59g`, `1온스=28.35g`.
- 부피: `1홉=180㎖`, `1되=1.8L`, `1말=18L`, `1갤런=3.79L`.
- 온도: `0℃=32℉`, `36.5℃=97.7℉(체온)`, `37℃=98.6℉`, `100℃=212℉`, `−40℃=−40℉`.
- 표기 반올림은 §5-2 규칙과 동일(둘째 자리 + 후행 0 절삭).

### 6-4. (선택 강화) 클릭 시 채우기

참조 행을 탭하면 그 값이 변환기 입력에 채워지도록 하면 참조표→변환기 연결로 상호작용·체류시간이
는다. 유용하지만 로직이 추가되므로 **필수 아님**. §11에서 옵션으로 제안(개발 로직 판단).

## 7. 반응형 (모바일 375 / 데스크톱 1280)

### 7-1. 카테고리 탭 — 가로 스크롤 없이 5등분 (모바일)

§2-1 실측대로 375px에서 5×2글자 탭이 한 줄에 들어간다(세그먼트당 ≈68.6px). **가로 스크롤을 쓰지
않는다.** `flex w-full` + 각 탭 `flex-1`로 등분. 데스크톱(1280, `max-w-3xl` ≈ 768px 내)에서는 더
여유로우므로 동일 등분 유지(탭이 과도하게 넓어지면 트랙 `max-w`로 제한 가능).

### 7-2. from/to 세로 스택 — 데스크톱·모바일 동일

- **양쪽 뷰포트 모두 세로 스택**(입력 위 → `⇅` 스왑 → 결과 아래)으로 통일한다. 근거: (1) `⇅`
  상하 스왑 메타포와 일치, (2) 결과를 design-system 3-2의 "큰 에메랄드 히어로"로 강조하려면 결과가
  독립 카드여야 하는데, 데스크톱에서 좌우로 나란히 두면 히어로 위계가 약해지고 높이 정렬이
  어색해진다, (3) 단일 레이아웃이 리플로우만 하면 되어 개발·QA가 단순하고 오버플로 리스크가 낮다.
- 데스크톱은 폭이 넓을 뿐 구조 동일: 입력 행 `[값][단위]`와 결과 카드가 더 넉넉한 폭을 갖는다.
- (검토했으나 미채택: 데스크톱 좌우 `입력 | ⇆ | 결과` 가로 배치 — §12.)

### 7-3. 가로 오버플로 0 규칙 (필수)

이 페이지의 최대 반응형 리스크는 **긴 단위명 셀렉트**와 **아주 큰 결과 숫자**다. 원천 차단 규칙:

- **단위 셀렉트**: `appearance-none` + 고정/그리드 폭. 옵션명이 길어도 닫힌 셀렉트에서 브라우저가
  자연 클리핑하므로 페이지 가로 폭을 늘리지 않는다. 그리드 셀·플렉스 자식에 `min-w-0`을 부여해
  셀렉트가 컨테이너를 밀어내지 못하게 한다.
- **히어로 결과 숫자**: `tabular-nums` + `whitespace-nowrap`. 자릿수가 길어질 때 **줄바꿈이 아니라
  폰트 단계 축소**로 대응(`text-4xl`(36px) → 필요 시 `text-3xl` → `text-2xl`). design-system Result
  스케일 "모바일 28px 이하 축소 금지"는 일반 구간에서 지키고, 12자리↑ 극단값에서만 그 이하로
  축소하거나 §5-2의 압축(과학) 표기로 폴백. 일반적인 변환값(최대 8~10자리)은 375px 카드 안쪽
  (약 311px)에서 `text-4xl`로 무리 없이 들어간다.
- **참조표·결과 보조행**: 모두 `flex justify-between`(좌 라벨 줄바꿈 허용, 우 값 `nowrap`
  우측 고정). 다열 테이블 미사용 → 가로 스크롤 구조적 불가.
- 카테고리 탭: `flex-1` 등분 + `min-w-0`로 텍스트 오버플로가 트랙을 넘기지 않게.

### 7-4. 무스크롤 목표 — 이 계산기는 달성

iPhone SE급(375×667, 실사용 높이 약 550~600px) 개략 높이 추정.

| 영역 | 예상 높이 |
|---|---|
| 카테고리 탭(트랙 패딩 포함) | 약 52px |
| 카드 전 여백 + 카드 상단 padding | 약 24px |
| 입력 행(라벨 + `h-12` 값/셀렉트) | 약 76px |
| 스왑 버튼 영역(겹침 고려 순증) | 약 24px |
| 결과 카드(상단행 + 히어로 + 요약 1줄, padding 포함) | 약 150~170px |
| **입력→결과 소계** | **약 326~346px** |

→ **카테고리 탭 + 입력 + 결과가 실사용 뷰포트(약 550~600px) 한 화면에 모두 들어온다.** 연봉·대출과
달리 이 계산기는 design-system 4장 원칙 1(무스크롤)을 문자 그대로 달성한다. 참조표(§6)는 그 아래로
자연스럽게 이어지며 스크롤 대상이지만, 핵심 도구(입력↔결과)는 above-the-fold를 지킨다.

### 7-5. 부수 규칙

- 입력 행 `grid-cols-[1fr_1.2fr]`은 375px에서도 값 입력 약 130px·셀렉트 약 160px로 나뉘어 둘 다
  터치·판독에 충분. 좁게 느껴지면 `grid-cols-2` 폴백 허용.
- 스왑 버튼 44px 터치 타깃 유지. 결과 카드 상단 to-단위 셀렉트도 `h-12` 유지.

## 8. 접근성 / 입력 상태 처리

- **결과 라이브 영역**: 히어로 값에 `aria-live="polite" aria-atomic="true"`(§3-4) — 버튼이 없어
  실시간 결과를 스크린리더에 전달.
- **카테고리 탭**: `role="tablist"`/`tab`+`aria-selected` 또는 라디오 그룹, 키보드 좌우 이동.
- **스왑 버튼**: `aria-label="입력 단위와 결과 단위 바꾸기"`.
- **셀렉트**: 각 셀렉트에 시각 라벨 또는 `aria-label`(from=`변환할 단위`, to=`결과 단위`) 연결.
- **입력 오류의 우아한 처리**: 실시간 유틸이므로 **BMI/연봉식 빨간 에러 배너·`role="alert"`를
  쓰지 않는다.** 빈 값/비숫자면 결과를 `—`로 조용히 표시하고 요약 문구를 숨긴다. 음수는 온도에서
  필수 허용, 그 외 카테고리(길이/무게/면적/부피)에서 물리적으로 무의미하지만 수식은 성립하므로
  차단하지 않고 그대로 계산(과도한 검증으로 사용자를 막지 않음 — 로직 세부는 개발팀 재량).

## 10. 광고 배치 (design-system.md 5장 + 이 페이지 규칙)

- **상단 배너 광고: 넣지 않는다.** 카테고리 탭 + 변환기를 최상단 above-the-fold에 두는 것이 이
  콤팩트 도구의 핵심(§7-4). 상단에 광고를 끼우면 무스크롤 이점을 스스로 깎는다.
- **인터랙티브 클러스터(카테고리 탭 ↔ 입력 ↔ 결과 ↔ 참조표) 내부/사이에는 광고를 넣지 않는다.**
  이 넷은 하나의 도구 흐름이므로 중간 삽입은 오인 클릭·흐름 방해가 된다.
- 본문 중간/하단 광고: 공통 템플릿이 처리(범위 밖). **참조표 다음(설명글/FAQ 사이)** 여백부터
  광고 배치 가능. design-system 5장 공통 규칙 준수(위아래 margin 24px+, "광고" Caption 라벨,
  결과 카드의 에메랄드 틴트/좌측 강조선과 유사하지 않은 중립 배경으로 명확히 구분).

## 11. 마스터 확인 필요 / 타팀 이관 (핑퐁)

**마스터 팀장 확인 필요**
1. **실시간(버튼 없음) 인터랙션 패턴 승인(§3-1)**: 이 계산기는 사이트 최초의 실시간 계산기로,
   BMI·연봉의 "form + 계산하기 버튼" 패턴을 의도적으로 깬다. "단발 계산=버튼 / 연속 변환=실시간"의
   이원화를 사이트 공통 원칙으로 승인해 줄 것을 요청한다(향후 환율 등 유사 변환기에도 적용
   가능). 진행은 막지 않고 본 스펙대로 설계하되, 톤앤매너/일관성 결정 사안이라 문의.
2. **design-system.md 신규 컴포넌트 편입 여부**: (a) 5등분 세그먼트 카테고리 탭(§2-2, 활성 색
   1안/2안 확정 포함), (b) 스왑 버튼(§3-3), (c) 버튼 없는 실시간 결과 카드 + `aria-live`(§3-4),
   (d) 참조표 블록(§6). 재사용 가능성이 높아 연봉 문서의 셀렉트/`<details>`처럼 정식 편입할지 결정
   요망(승인 전까지 제안 단계로 취급, 진행은 막지 않음).
3. **양방향 구현 방식(§3-3 vs §12)**: 디자인팀은 "입력 1개 + 큰 에메랄드 결과 표시 + 스왑으로
   방향 전환"을 택했다(결과 위계를 design-system 3-2로 살리기 위함). 대안은 "값 필드 2개가 모두
   편집 가능한 트윈 인풋(구글식)". 후자가 변환기 UX로는 더 표준적이나 히어로 결과 강조가 약해진다.
   현 방식으로 확정해도 되는지, 트윈 인풋을 원하는지 확인(둘 다 실시간·스왑은 동일).

**기획팀 확인 요청**
- **단위 목록·기본 단위쌍 최종 확정(§3-5)**: 특히 한국 전통 단위(평·근·관·홉·되·말)와 각 카테고리
  단위 구성은 SEO 검색 수요와 직결되므로 기획팀이 확정. 카테고리별 기본 from→to 쌍, 기본 프리필
  예시값(면적 1평/10평 등)도 확정 요망(§3-4).
- **참조표 항목·수치·SEO 카피(§6-3)**: 어떤 "자주 쓰는 환산"을 몇 개 노출할지, 각 수치·표기가
  §5-2 반올림 규칙과 일치하는지 확인.
- **반올림/표기 일관성(§5-2)**: 해설 예시가 "둘째 자리 반올림 + 후행 0 절삭"과 어긋나지 않는지,
  아주 작은/큰 값의 유효숫자·압축(과학) 표기를 해설에서 다룰지 확인.
- **동적 요약 문구(§3-4) 카피 톤**: "20평은 약 66.12㎡입니다." 류 1줄 템플릿 문안 확정.

**개발팀 전달 사항**
- `components/UnitConverter.tsx`(§9), `lib/units.ts`(canonical-base 변환, **온도는 오프셋 공식**,
  계수/공식 상수 모듈화) 신설. `CALCULATOR_COMPONENTS` 매핑 추가, `lib/calculators.ts` 콘텐츠 확보 후
  `status: "live"`.
- **온도 주의(§4)**: 역방향을 "1/계수"로 처리 금지, 반드시 기준단위(℃) 경유. 음수 입력 지원.
- **스왑 드리프트 방지(§3-3)**: 입력으로 되먹일 값은 반올림 표시값이 아니라 full-precision 결과값.
- 입력엔 천단위 콤마 미적용, 결과·참조표엔 콤마 적용(§5-1). 히어로는 폰트 단계 축소로 오버플로
  방지(§7-3).
- (옵션) 참조표 클릭-채우기(§6-4), 결과 복사 버튼(§3-4) — 채택 여부 회신.
- 구현 후 375×667 / 390×844 / 1280에서 §7-3 가로 오버플로 0, §7-4 무스크롤을 디자인팀이 재검수.

## 12. 대안으로 검토했으나 채택하지 않은 안

- **카테고리 = 셀렉트 드롭다운**: 공간은 아끼나 4개 카테고리가 접혀 발견성이 떨어지고 전환에
  3동작이 필요. 5×2글자가 탭으로 한 줄에 들어가므로 셀렉트 이점이 없음 → 세그먼트 탭 채택(§2-1).
- **계산 버튼 방식(BMI/연봉과 동일)**: 연속 탐색·스왑이 핵심인 변환기에서 매번 버튼은 마찰.
  실시간 채택, 대신 §11에서 패턴 이원화를 마스터에 승인 요청(§3-1).
- **트윈 편집 인풋(값 필드 2개 모두 편집)**: 표준적 변환기 UX지만 design-system 3-2의 "큰 에메랄드
  히어로 결과" 위계가 약해지고 두 필드 높이 정렬이 어색. 결과를 히어로로 살리는 "입력 1개 + 결과
  카드 + 스왑" 채택. 단 §11에서 마스터 선택지로 남김.
- **데스크톱 좌우 `입력 | ⇆ | 결과` 가로 배치**: 폭 활용엔 좋으나 결과 히어로 위계 약화 + 데스크톱/
  모바일 이중 레이아웃으로 QA·오버플로 리스크 증가. 단일 세로 스택으로 통일(§7-2).
- **항상 소수 둘째 자리 고정 표기**: `200.00㎝`처럼 어색 + 아주 작은 값에서 `0.00`이 됨. 후행 0
  절삭 + 3단 규칙 채택(§5-2).

---

이상으로 단위 변환 계산기(`components/UnitConverter.tsx`) 개발 착수가 가능하다. §11 확인 사항
처리 및 콘텐츠(기획 interpretation·FAQ·단위 목록·참조표 수치) 확보 후 구현을 진행하고, 완료 후
데스크톱(1280)/모바일(375·390) 실제 화면을 본 스펙과 대조해 검수할 예정이다.

   공통 원칙으로 승인. 향후 유사 변환형 계산기에도 적용 가능.
2. **design-system.md 정식 편입 = 보류.** 세그먼트 탭/스왑 버튼/실시간 결과 카드/참조표 블록은
   이번 컴포넌트 내 사용만 승인, 정식 문서 편입은 연봉 계산기 때와 동일하게 추후 일괄 재검토
   (누적된 신규 패턴이 여러 개 쌓이면 한 번에 정리하는 편이 효율적).
3. **양방향 구현 = §3-3 방식 채택(입력 1개 + 에메랄드 히어로 결과 + 스왑), 트윈 인풋 미채택.**
   결과 위계를 유지하는 디자인팀 권장안 그대로 확정.
4. **단위 목록 = 기획팀 `planning/unit-converter-content.md` §1 확정표를 그대로 채택.**
   디자인팀이 SEO 목적으로 제안한 관(무게)·아르/에이커(면적)·홉·되·말(부피)은 **이번 v1 범위에서
   제외**한다 — 기획팀이 교차검증한 정확도 앵커(§1-6)가 없는 상태로 수치를 하드코딩하면 연봉
   계산기 CS 이슈(해설-엔진 수치 불일치)와 같은 유형의 리스크가 생긴다. 대신 **부피 카테고리는
   디자인팀이 실수로 누락한 "컵(200mL)"을 반드시 포함**한다(기획팀이 앵커까지 확정한 단위이며
   interpretation 본문에도 이미 등장). 최종 단위 목록(개발팀 구현 기준, 총 28개):
   - 길이(8): mm·cm·m·km·inch(0.0254)·ft(0.3048)·yard(0.9144)·mile(1609.344) — 기준 m
   - 무게(7): mg·g·kg·t·oz(28.349523125)·lb(453.59237)·근(600) — 기준 g
   - 면적(5): ㎡·평(400/121)·㎢·ha(10000)·ft²(0.09290304) — 기준 ㎡
   - 부피(5): mL·L·㎥·컵(0.2, 200mL 한국 기준)·gallon(3.785411784, 미국) — 기준 L
   - 온도(3): ℃·℉·K — 오프셋 공식(§4)
   - 한국 전통 단위(관/아르/에이커/홉/되/말) 확장은 **다음 실행으로 이월하는 백로그**로 남긴다
     (기획팀이 정확도 앵커까지 갖춰 확정하면 재검토).
5. **기본 from→to·프리필 = 디자인팀 §3-5 표 + §1 와이어프레임 그대로 승인.** 면적 카테고리 기본
   프리필은 와이어프레임 예시대로 **20평 → 66.12㎡**로 확정(참조표 §6-3의 20평 행과도 일치).
   그 외 카테고리 기본쌍(cm→in, kg→lb, L→gal, ℃→℉)도 그대로 채택.
6. **반올림 = 디자인팀 §5-2 3단 규칙(일반: 둘째자리+후행0절삭 / 초소형: 유효숫자 / 초대형:
   폰트축소→과학표기) 그대로 승인.** 기획팀 해설 예시(§1-6)와 충돌 없음 확인됨.
7. **옵션 기능(결과 복사 버튼 §3-4, 참조표 클릭 채우기 §6-4) = 이번 v1은 스킵.** 필수 경로가 아니며
   범위를 늘리지 않기 위해 다음 확장 때 재검토(백로그).
8. **동적 요약 문구는 디자인팀 템플릿("{입력값}{from}은 약 {결과}{to}입니다.") + 기획팀 interpretation
   톤을 그대로 따르면 충분** — 별도 기획 왕복 없이 개발팀이 BMI/연봉의 `getResultSummary` 패턴대로
   구현.

이 결정에 따라 개발팀은 두 스펙 문서(본 문서 + `planning/unit-converter-content.md`)를 그대로
구현하되, 단위 목록만 위 4번 표를 최종본으로 삼는다.
