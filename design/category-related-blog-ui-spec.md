# 화면 구성안 — 카테고리 페이지 하단 "관련 글" 블록

- 문서 유형: 개발 전 화면 구성안(레이아웃 스펙)
- 대상 페이지: `components/CategoryPage.tsx` (/salary /loan /date /life 공통)
- 재사용 자산: `components/RelatedBlogPosts.tsx`, `components/BlogPostCard.tsx` (신규 컴포넌트/색 토큰 도입 없음)
- 작성: 디자인팀 / 기준일 2026-07-15
- 관련 선행 문서: `design/related-blog-posts-ui-spec.md` (계산기 상세 하단 동일 블록)

## 0. 이 블록이 하는 일 / 현재 상태

블로그↔계산기 내부링크는 이미 양방향 완성. 이번 작업은 그 **카테고리 레벨** 확장이다. 각 카테고리 페이지(예: /salary) 하단에, **그 카테고리에 속한 블로그 글** 카드를 노출해 카테고리 → 블로그 동선을 연다.

현재 데이터 상태:
- 블로그 4편, 각 카테고리(salary/loan/date/life) **정확히 1편씩**. 발행일 전부 `2026-07-15` 동일.
- 즉 이 블록은 **각 카테고리에서 카드 1장**만 보여준다(당분간). 카테고리당 글이 늘면 자동으로 여러 장.

현재 `CategoryPage.tsx` 구조(라인 17~40, 실물 확인 완료):
```
<div className="mx-auto max-w-5xl px-4 py-8 sm:px-6 sm:py-12">
  <h1>{info.title}</h1>              // 예: "급여 계산기"
  <p>{info.description}</p>          // 예: "...급여와 관련된 계산기 모음입니다."
  <div className="mt-8 grid grid-cols-1 gap-4 sm:grid-cols-2 lg:grid-cols-3">
    {items.map(...)} // CalculatorCard, 카드 제목 <h3>
  </div>
</div>
```
**핵심: 계산기 그리드 위에 섹션 헤딩(h2)이 없다.** 그래서 현재 아웃라인은 `h1 → (h2 없음) → h3(카드들)` 로 이미 h2 스킵이 있다. 여기에 "관련 글"(h2)을 붙이면 문제가 생긴다(§2에서 해결).

## 1. 레이아웃 / 배치 (결정 1)

기존 계산기 그리드 **아래**에 `RelatedBlogPosts`를 그대로 재사용해 붙인다. 컨테이너(`max-w-5xl` = 1024px) 안에 형제 섹션으로 스택.

```
<div className="mx-auto max-w-5xl px-4 py-8 sm:px-6 sm:py-12">
  <h1 .../>
  <p .../>

  <section aria-labelledby="calc-list-heading">        // ← 계산기 목록 섹션(§2 참고)
    <h2 id="calc-list-heading" className="sr-only">계산기 목록</h2>
    <div className="mt-8 grid grid-cols-1 gap-4 sm:grid-cols-2 lg:grid-cols-3">
      {items.map(...)} // CalculatorCard (h3)
    </div>
  </section>

  <RelatedBlogPosts posts={categoryPosts} />           // ← 신규. 내부에서 mt-10 + <h2>관련 글</h2>
</div>
```

### 간격 검토 (mt-8 그리드 → mt-10 관련글)
- 계산기 그리드는 설명문에서 `mt-8`(2rem) 떨어져 있다.
- `RelatedBlogPosts`는 자체 `mt-10`(2.5rem)을 가진다. 즉 그리드 하단 → "관련 글" 섹션 간격이 `mt-10`.
- **적절하다.** 섹션 전환(계산기 도구 목록 → 읽을거리)은 그리드-헤더 내부 간격(`mt-8`)보다 살짝 더 벌어져야 "다른 성격의 블록"임이 드러난다. `mt-10 > mt-8` 리듬이 이 위계를 정확히 표현한다. 상세 페이지에서 `RelatedCalculators`/`RelatedBlogPosts`가 모두 `mt-10`으로 쓰이는 사이트 관례와도 일치. **간격 신규 조정 불필요.**

### 컨테이너 폭 차이 주의 (검수 필수)
- 카테고리 페이지 컨테이너는 `max-w-5xl`(1024px)로, 상세 페이지(`max-w-3xl`=768px)보다 넓다.
- 따라서 데스크톱 3열에서 블로그 카드 1장 폭 ≈ `(1024 - 좌우패딩 - gap×2) / 3` ≈ **약 320px**. 상세 페이지(약 230px)보다 여유롭다 → 오버플로 위험은 상세 페이지보다 오히려 낮다. BlogPostCard는 고정폭/min-w 없고 텍스트 `line-clamp-2`라 구조상 안전.
- **정합성:** 위 계산기 그리드와 아래 블로그 그리드가 **동일한 그리드 클래스**(`grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`, `gap-4`)를 쓰므로, 두 그리드의 열 경계·거터가 픽셀 단위로 정렬된다. 카드 폭도 동일 → 시각적으로 완전 정합. 이 동일성을 반드시 유지할 것(§5 체크리스트).

### 카드 1장일 때의 배치 (현 상태 — 검수 시 놀라지 말 것)
- 지금은 카테고리당 글이 1편이라, 데스크톱(lg)에서 블로그 카드 1장이 3열 그리드의 **왼쪽 1/3 칸**에 놓이고 오른쪽 2/3는 빈다.
- 이는 결함이 아니라 **의도된 반응형 동작**이며, 상세 페이지 "관련 글"(관련 글 1편일 때) 및 `RelatedCalculators`(관련 항목 적을 때)와 **동일한 이미 승인된 거동**이다(commit e4f9574). 카테고리별 글이 늘면 자연히 채워진다.
- 위 계산기 그리드는 여러 장이라 3열이 꽉 차는데 아래 블로그 그리드는 1장이라 왼쪽만 참 → 상하 비대칭이 보인다. **허용.** 별도 중앙정렬/폭 축소 커스텀을 넣지 말 것(그리드 일관성·향후 확장성 우선).

## 2. 헤딩 위계 문제 & 해결 (결정 2 — 핵심)

### 문제
현재 카테고리 페이지 아웃라인: `h1(카테고리명) → h3(CalculatorCard 카드들)`. 이미 h2가 통째로 빠져 있다.
여기에 `RelatedBlogPosts`의 `<h2>관련 글</h2>`를 붙이면:
```
h1 → h3 h3 h3 (계산기 카드) → h2 (관련 글) → h3 (블로그 카드)
```
h3 다음에 h2가 오는 **아웃라인 역전**이 발생한다. 스크린리더 문서 개요에서 "관련 글"이 앞의 계산기 카드들보다 상위로 잡히며 계층이 꼬인다.

### 검토한 3안
- **(A) 그리드 위에 시각적으로 보이는 h2 신설** (예: "계산기 목록"). 아웃라인은 `h1→h2→h3→h2→h3`로 완전 정상화되지만, **화면에 헤딩 텍스트가 추가되는 디자인 변화**가 생긴다.
- **(B) 그리드 위에 sr-only h2 신설.** 화면 변화 0, 아웃라인만 `h1→h2(sr-only)→h3→h2→h3`로 보정.
- **(C) 관련 글 블록 헤딩을 h2가 아닌 다른 방식으로.** → 재사용 컴포넌트를 카테고리 전용으로 분기해야 하고, 상세 페이지와 시맨틱이 어긋나 일관성이 깨진다. **기각.**

### 권고: (B) sr-only h2 "계산기 목록"

근거:
1. **h1이 이미 목록을 설명한다.** h1 "급여 계산기" + 설명문 "…급여와 관련된 계산기 모음입니다"가 그리드의 정체를 완전히 서술한다. 그 바로 아래에 **보이는** "계산기 목록" h2를 또 넣으면 **h1과 의미가 중복**되어 시각적 노이즈가 된다(글 본문에 "본문"이라는 소제목을 붙이는 격).
2. **상세 페이지 선례와 정합.** 상세 페이지는 1차 콘텐츠(계산기 UI)를 h1 바로 아래 **섹션 헤딩 없이** 두고, "관련 계산기"·"관련 글" 같은 2차 섹션만 보이는 h2를 붙인다. 카테고리 페이지도 "1차 콘텐츠=보이는 섹션 헤딩 없음 / 2차 섹션=보이는 h2"라는 동일 원칙을 따르는 게 사이트 일관성에 맞다. 카테고리 페이지만 다른 점은 1차 콘텐츠(그리드)가 h3 카드를 내보낸다는 것 → 그래서 **접근성용 앵커 h2**만 있으면 충분.
3. **역전·스킵 동시 해결 + 시각 회귀 0.** sr-only h2는 (a) h3 카드들에 정당한 부모를 주어 기존 h1→h3 스킵을 없애고, (b) 뒤의 "관련 글" h2를 형제 h2로 만들어 역전을 없앤다. 화면 픽셀 변화가 전혀 없으므로 톤앤매너/브랜딩 검토가 필요 없고, "신규 색/컴포넌트 없이 기존 재사용" 원칙과도 부합. 리스크 최저.

**(A)를 택하지 않는 이유이자, (A)로 뒤집을 조건:** 만약 마스터/기획이 "두 섹션이 스택되므로 스캔성을 위해 둘 다 보이는 라벨을 달자"고 판단하면 (A)가 대안이 된다. 그 경우 아래 스펙을 그대로 쓴다. 디자인팀 1순위는 (B), (A)는 명시 지시가 있을 때만.

#### (B) 구현 스펙
- 그리드를 `<section aria-labelledby="calc-list-heading">`로 감싼다.
- 그 안 첫 요소로 `<h2 id="calc-list-heading" className="sr-only">계산기 목록</h2>`.
- **주의:** `sr-only` 요소가 문서 흐름에서 높이를 차지하지 않아야 하므로, `mt-8`은 지금처럼 **그리드 `<div>`** 에 그대로 둔다(h2로 옮기지 말 것). h2를 section 최상단에 두고 그 아래 `mt-8 grid …`를 유지하면 시각적 간격이 100% 보존된다.

#### (A) 대안 스펙 (지시 있을 때만)
- 위와 동일 구조에서 `sr-only`만 제거하고 보이는 스타일 부여:
  `<h2 id="calc-list-heading" className="mb-4 text-xl font-bold text-brand-text">계산기 목록</h2>`
- 이때 그리드의 `mt-8`은 유지하되, h2가 생기며 설명문↔그리드 간격이 시각적으로 커 보일 수 있으니 검수에서 확인. 문구는 h1과 덜 겹치도록 "전체 계산기" 등 대안 검토 여지 있음 → 채택 시 기획팀 문구안과 합의 필요.

## 3. 관련 글 블록 헤딩 문구 (결정 3)

### 권고: "관련 글" 그대로 유지 (RelatedBlogPosts 무수정 재사용)

근거:
1. **사이트 일관성.** 계산기 상세 페이지의 동일 블록이 이미 "관련 글"이다. 같은 컴포넌트·같은 라벨을 카테고리에서도 쓰면 사용자 멘탈 모델이 사이트 전역에서 일관된다("관련 글 = 함께 볼 블로그 글" 학습이 페이지마다 재사용됨).
2. **문맥상 충분히 명확.** 카테고리 페이지 전체가 하나의 주제(예: 급여)이므로, 하단 "관련 글"은 자연스럽게 "이 주제와 관련된 글"로 읽힌다. 게다가 블로그 카드는 카테고리 배지·발행일·읽는 시간이 붙어 계산기 카드와 시각적으로 명확히 구분되므로 라벨이 짧아도 혼동이 없다.
3. **최소 변경·최저 리스크.** headingText prop 없이 컴포넌트를 그대로 넘기면 됨. 상세 페이지 사용처도 건드리지 않는다.

### 대안(전용 문구를 원할 경우) — 개발 이관 명시
마스터/기획이 카테고리 맥락 강조를 원하면, RelatedBlogPosts에 **`headingText?: string` prop 추가**(기본값 `"관련 글"`). 카테고리에서는 예: `headingText="이 주제의 블로그 글"` 로 주입.
- **필수 제약:** 기본값을 `"관련 글"`로 두어 **상세 페이지 기존 동작이 100% 불변**이어야 한다. 시각 스타일(`mb-4 text-xl font-bold text-brand-text`)은 문구만 바뀌고 클래스 불변.
- 디자인팀 1순위는 "관련 글" 유지(무 prop). 위 prop은 지시가 있을 때만.

## 4. 반응형 / 모바일 (결정 4)

두 그리드 모두 Tailwind 기본 브레이크포인트, 동일 클래스:

| 뷰포트 | 조건 | 계산기 그리드 | 블로그 그리드 | 근거 |
|---|---|---|---|---|
| 모바일 | ≤ 639px | 1열 | 1열 | `grid-cols-1` |
| 태블릿 | 640~1023px | 2열 | 2열 | `sm:grid-cols-2` |
| 데스크톱 | ≥ 1024px | 3열 | 3열 | `lg:grid-cols-3` |

### 모바일(≤640px) 점검
- 컨테이너 `px-4`, 두 그리드 모두 `grid-cols-1` + `gap-4` → 계산기 카드와 블로그 카드가 **동일 폭(컨테이너 폭 − px-4×2)**으로 세로 스택. 폭 정합 보장.
- BlogPostCard는 고정폭/min-w 없음, 제목·요약 `line-clamp-2`, 카테고리 pill `self-start`(콘텐츠 폭) → **가로 오버플로/가로 스크롤 0**. iPhone SE(360px)에서도 안전. 검수에서 실측 확인.
- 세로 순서(모바일): h1 → 설명 → 계산기 카드들(세로 스택) → "관련 글" → 블로그 카드(1장). 자연스러운 읽기 흐름.

## 5. 구현 체크리스트 (개발팀)

대상 파일: **`components/CategoryPage.tsx`** (다른 파일 최소 변경)

- [ ] 계산기 그리드를 `<section aria-labelledby="calc-list-heading">`로 감싼다.
- [ ] section 최상단에 `<h2 id="calc-list-heading" className="sr-only">계산기 목록</h2>` 추가. **(B)안. `sr-only` 필수 — 화면 변화 없어야 함.**
- [ ] 그리드 `<div>`의 기존 `mt-8`·그리드 클래스는 그대로 둔다(간격 보존).
- [ ] 그리드 섹션 **아래**(컨테이너 `</div>` 직전)에 `<RelatedBlogPosts posts={categoryPosts} />` 배치.
- [ ] `RelatedBlogPosts`는 **무수정 재사용**. 내부 `mt-10` + `<h2>관련 글</h2>` + 동일 그리드 그대로. (문구 "관련 글" 유지 — §3.)
- [ ] **데이터 이관 이슈(로직=개발팀 몫, 아래는 요구사항만):** 현재 `lib/blog.ts`에는 카테고리로 글을 거르는 헬퍼가 없다(`getBlogPostsForCalculator`·`getAllBlogPosts`만 존재). **카테고리별 글 목록을 반환하는 헬퍼가 필요**하다. 요구사항:
  - 입력: `CalculatorCategory` (blog의 `category`와 동일 4값).
  - 출력: 해당 카테고리 글, `publishedDate` 내림차순(동일 발행일이면 slug 오름차순 등 결정적 정렬 — SSG 안정성). 상한은 개발 재량(현재 카테고리당 1편이라 사실상 무관, 향후 대비 6편 등 권장).
  - `RelatedBlogPosts`는 이미 `posts.length === 0`이면 `return null` → 글 없는 카테고리에서 빈 제목 안 뜸(이중 안전).
- [ ] **신규 색 토큰·신규 컴포넌트 도입 없음** 확인(brand-* 및 기존 컴포넌트만).
- [ ] 두 그리드가 **동일 그리드 클래스**(`grid grid-cols-1 gap-4 sm:grid-cols-2 lg:grid-cols-3`)를 유지하는지 확인(열 경계 정합).

### 개발팀 회신 요망(핑퐁 가능 지점)
- sr-only h2 방식이 SSG/린트(빈 텍스트·중복 id 등)와 충돌 없는지. `id`는 페이지당 1회만 렌더되므로 문제없어야 함.
- 카테고리 헬퍼 신설 위치/시그니처가 기존 `lib/blog.ts` 관례와 맞는지(디자인은 화면만 규정, 구현은 개발 재량).

## 6. 구현 후 디자인 검수 항목 (역할 ②)

- [ ] dev 서버에서 4개 카테고리(/salary /loan /date /life) 모두 하단에 "관련 글" + 카드 1장 노출.
- [ ] (글 없는 상태를 만들 수 있다면) 글 0편 카테고리에서 "관련 글" 섹션이 아예 안 뜨는지.
- [ ] **화면 변화 0 확인:** (B) sr-only 적용 후 계산기 그리드 위에 **보이는 헤딩 텍스트가 생기지 않았는지**(설명문 바로 아래 그리드가 지금과 동일 위치·간격).
- [ ] 개발자도구 요소 검사: 계산기 카드=h3, 그리드 부모=sr-only h2, "관련 글"=h2, 블로그 카드=h3. 문서 개요가 `h1→h2→h3→h2→h3`로 역전·스킵 없이 정상.
- [ ] 데스크톱(≥1024, 컨테이너 1024px) 3열 / 태블릿(640~1023) 2열 / 모바일(≤639) 1열, 두 그리드 열 경계가 세로로 정렬.
- [ ] 데스크톱에서 블로그 카드 1장이 왼쪽 1/3에 놓이는 것 = 의도된 동작(결함 아님) 재확인.
- [ ] 모바일 360px에서 계산기 카드·블로그 카드 동일 폭, 가로 스크롤 0.
- [ ] "관련 글" 카드가 블로그 목록/상세 관련글 카드와 픽셀 동일(외형 회귀 없음).
- [ ] 그리드 하단 → "관련 글" 간격이 `mt-10` 리듬으로 자연스러운지.
- [ ] 키보드 탭 이동 시 블로그 카드 포커스가 시각적으로 드러나는지(안 보이면 BlogPostCard/CalculatorCard 공통 `focus-visible` 보강을 개발팀에 요청 — related-blog-posts-ui-spec §5-2와 동일 사안).
