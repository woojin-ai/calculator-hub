## 0. 이 페이지가 다른 계산기와 다른 점 (설계 전제)

- **입력 4종**(연봉 / 월 비과세액 / 부양가족 수 / 8~20세 자녀 수)으로 대출 계산기와 동일하게
  많다. design-system.md 4번 원칙 1("입력→결과 무스크롤")은 대출과 마찬가지로 문자 그대로
  달성이 어렵다 → §5에서 절충(입력 폼 압축 + 계산 후 결과로 스무스 스크롤).
- **결과의 정보량이 지금까지 계산기 중 가장 많다.** 기획 §1-1 파이프라인이 정의한 중간값은
  최소 8개 숫자(4대보험 4항목 + 근로소득세 + 지방소득세 + 월 실수령 + 연 환산 실수령)이며,
  중간값(월 과세대상 급여 등)까지 더하면 10개를 넘는다. 이걸 모바일 375px에서 가로 오버플로
  없이, 무엇을 크게/무엇을 접을지 위계를 잡는 것이 이 문서의 핵심이다(§3, 기획 §7-3에 대한
  디자인팀 확정 답).
- **YMYL 등급 = 중간**(기획 §0). 대출(높음)보다 한 단계 낮으나, 근로소득세를 **간이세액표
  근사 공식(Option B)** 로 산정하므로 "간이세액표 근사, 실제와 다를 수 있음" 고지가 필수다.
  대출의 상단 면책 배너 패턴을 재사용하되 톤을 조정한다(§6).
- **YMYL-중간 CTA 금지**: 급여 인상·이직을 부추기거나 오인시키는 2차 버튼/문구를 계산기
  카드 안팎에 절대 넣지 않는다(§6-3). "더 받는 법", "이직하면 얼마?" 류 카피·버튼 금지.

## 1. 페이지 전체 레이아웃

`app/calculator/[slug]/page.tsx` 기존 골격을 그대로 따른다. 신규 설계는 "2. 입력 설계"와
"3. 결과 카드"이며, 대출 계산기와 동일하게 계산기 카드 상단에 **근사 고지 배너**를 삽입한다.

```
[Breadcrumb: 홈 / 연봉 계산기]
[H1] 연봉 실수령액 계산기
[본문 요약 1줄]
──────────────────────────────
[SalaryNetCalculator 컴포넌트]  ← 이번 스펙의 범위 (2~6장)
  [ⓘ 근사 고지 배너]  ← 컴포넌트 최상단, 계산 여부와 무관하게 항상 노출 (6장). 광고 아님.
  [계산기 카드]
    - 연봉 입력 (전체 폭)
    - 월 비과세액 입력 (전체 폭)
    - 부양가족 수 / 8~20세 자녀 수 (2열)
    - 계산 버튼
    - 결과 카드 (계산 후 노출, 3장)
──────────────────────────────
[결과 해석]     ← ResultInterpretation 재사용 (카피=기획 §2, 공통 템플릿 — 범위 밖)
[본문 중간 광고]  ← design-system.md 5장 공통 규칙 (범위 밖, §6-4에서 원칙만 재확인)
```

(사이드바 도입은 페이지 공통 이슈로 이 문서 범위 밖).

## 2. 입력 설계

### 2-1. 필드 순서 및 배치 원칙

기획 §4 입력 항목표를 그대로 반영하되, **금액 입력(자릿수 김)은 전체 폭, 소수 정수 입력(짧음)은
2열**로 묶는 BMI/대출 패턴을 따른다.

| 순서 | 필드 | 배치 | 근거 |
|---|---|---|---|
| 1 | 연봉 (원) | 전체 폭 | 억 단위까지 가는 큰 금액이라 한 줄 확보(대출 원금과 동일 판단). 2열로 묶으면 자릿수가 잘려 보임 |
| 2 | 월 비과세액 (원) | 전체 폭 | 콤마 포함 6~7자리, 라벨이 길어("식대 등 비과세") 한 줄 확보 |
| 3 | 부양가족 수 / 8~20세 자녀 수 | 2열 그리드 (`grid grid-cols-2 gap-3 sm:gap-4`, BMI 패턴 재사용) | 둘 다 한 자릿수 셀렉트라 나란히 배치해도 여유. 부양가족↔자녀 의존관계(자녀 ≤ 부양−1)를 나란히 보여줘 관계를 직관적으로 인지시킴 |
| 4 | 계산 버튼 | 전체 폭 | 대출/BMI와 동일, `h-12`(48px) 터치타깃 |

### 2-2. 필드별 스펙

design-system.md 3-1(테두리 `#E5E7EB`/`border-brand-border`, radius 8px, 높이 `h-12`/48px,
포커스 시 `focus:border-brand-primary focus:ring-4 focus:ring-brand-primary/15`)을 모든 입력에
동일 적용. 대출 컴포넌트의 `inputBase` 상수를 그대로 재사용한다.

| 필드 | id | 라벨 | 컨트롤 | placeholder / 기본값 | 도움말(라벨 아래 또는 우측) |
|---|---|---|---|---|---|
| 연봉 | `annualSalary` | "연봉 (세전, 원)" | `text` + `inputMode="numeric"`, 입력 중 천단위 콤마 서식 | placeholder `40,000,000` / 기본 미입력(필수) | "성과급·상여가 연봉에 포함된 계약이면 연 총액을 입력하세요" (Caption, 선택 노출) |
| 월 비과세액 | `taxFreeMonthly` | "식대 등 비과세 (월, 원)" | `text` + `inputMode="numeric"`, 천단위 콤마 서식 | placeholder/기본값 **`200,000`** (기본 채움) | "식대는 월 20만원까지 비과세(기본값). 자가운전보조금 등 있으면 증액" (Caption) |
| 부양가족 수 | `dependents` | "부양가족 수 (본인 포함)" | `select` 권장(1~10, "10명 이상" 등은 v1 불필요) | 기본 선택 **`1`** | "본인 포함 최소 1명. 간이세액표 공제대상 가족 수" (Caption) |
| 8~20세 자녀 수 | `children` | "8~20세 자녀 수" | `select` 권장(0 ~ 부양가족−1, 동적) | 기본 선택 **`0`** | "자녀세액공제 대상. 부양가족 수보다 적어야 함" (Caption) |

**연봉·비과세액 천단위 콤마 = 필수**: 대출 컴포넌트의 `formatPrincipalInput(raw)`(숫자만 남기고
`toLocaleString("ko-KR")`)을 두 필드에 그대로 재사용한다. 결과 카드의 모든 금액 표시에도
천단위 콤마 필수(`formatWon` 재사용).

**셀렉트 vs 숫자입력 — 셀렉트 권장(디자인팀 결정)**: 부양가족·자녀는 **1~10, 0~9의 좁은
정수 범위**이고 모바일 네이티브 피커가 오타·범위이탈을 원천 차단하며, 아래 **의존 검증(자녀 ≤
부양−1)** 을 "선택 불가 옵션 비활성화"로 자연스럽게 표현할 수 있어 셀렉트가 낫다. 숫자입력으로
가면 사용자가 유효범위를 모른 채 큰 값을 넣고 제출 후 에러를 보게 되어 UX가 떨어진다. (개발
공수상 숫자입력이 유리하면 대체 가능하나, 그 경우 §2-4 의존 검증을 제출 시 처리해야 함.)

> **셀렉트 스타일 = 신규 컴포넌트 유형**(design-system.md 3장에 select 정의 없음). `inputBase`와
> 동일한 높이(`h-12`)·테두리·radius·포커스 링을 쓰고, 우측 `pr-9`에 chevron(▾) 아이콘을
> `pointer-events-none`으로 겹쳐 배치, `appearance-none`으로 브라우저 기본 화살표 제거를
> 권장한다. 라벨은 다른 입력과 동일하게 위쪽 Body Small + Text Secondary. → §8 마스터 확인
> 항목(design-system 편입 여부).

### 2-3. 의존 검증 (자녀 수 ≤ 부양가족 수 − 1)

기획 §1-0 / §4의 검증 규칙을 UI로 구현한다. **셀렉트 채택 시(권장)**: 부양가족 수 선택값이
바뀌면 자녀 수 셀렉트의 옵션을 `0 ~ (부양가족−1)`로 **동적 갱신**한다. 현재 자녀 선택값이 새
상한을 초과하면 자동으로 상한값(또는 0)으로 보정한다. 이러면 "유효하지 않은 조합 자체가
선택 불가"라 별도 에러 메시지가 거의 필요 없다.

- 예: 부양가족 1 → 자녀 옵션 `0`만(자녀 불가). 부양가족 3 → 자녀 옵션 `0,1,2`.
- 부양가족 = 1이고 자녀 셀렉트가 사실상 `0` 고정이 되는 상황은, 자녀 셀렉트를 `disabled`로
  두고 Caption에 "부양가족을 2명 이상으로 설정하면 자녀 수를 입력할 수 있습니다"를 노출한다
  (헷갈림 방지).

### 2-4. 에러 상태

대출/BMI 패턴 재사용. 연봉 미입력·0 이하·비숫자 시 해당 입력 테두리 `border-brand-warning` +
하단 `text-xs text-brand-warning` 메시지, `role="alert"`. 비과세액은 기본값이 채워져 있고
0 허용(전액 과세)이라 필수 아님. 정확한 검증 규칙·문구는 개발팀 재량(이 문서는 시각 스타일만
규정). 예: "연봉을 올바르게 입력해주세요."

## 3. 결과 카드 설계 — [기획 §7-3에 대한 디자인팀 확정안]

### 3-0. 정보구조 확정 (무엇을 크게 / 무엇을 요약 / 무엇을 접이식)

기획 §7-3의 트레이드오프(정보량 = 공제 8줄+ vs 가독성 = 모바일 무스크롤)에 대한 디자인팀
**최종 결정**은 아래 3-tier 구조다. 기획팀 제안("4대보험 4항목 + 소득세 + 지방세 + 실수령
월/연을 표로")을 버리지 않되, **한 번에 다 펼치지 않고 위계를 3단으로 나눈다.**

| Tier | 내용 | 강조/배치 | 노출 |
|---|---|---|---|
| **① 대표값** | **월 실수령액** | Result 스케일(36~40px, Bold, Accent `#10B981`, tabular-nums) — 카드 내 가장 큼 | 항상 |
| ① 보조 대표 | 연 환산 실수령액 | Heading2(20px, SemiBold, `text-brand-text` — Accent 아님, 위계 1단 낮춤) | 항상 |
| ① 요약 문구 | 기획 §2 1문단(동적) 압축 1~2줄 | Body Small, Text Secondary | 항상 |
| **② 공제 요약** | 3행: 월 급여총액(세전) / 4대 보험 합계(−) / 소득세·지방소득세 합계(−) | `flex justify-between py-1.5 text-sm`(대출 보조항목 패턴), 값 `font-semibold tabular-nums` | 항상 |
| **③ 공제 상세** | 8줄 개별: 국민연금·건강보험·장기요양·고용보험(+4대보험 소계) / 근로소득세·지방소득세(+세금 소계) / 월 과세대상 급여·비과세액(중간값) | 접이식(`<details>` "공제 내역 자세히 보기 ▾") 내부. 열면 라벨-값 행으로 세로 나열 | 접힘(기본) |

**결정 근거(기획 §7-3 답변)**:
- **월 실수령액을 단일 대표값으로 강조**(마스터 권장 방향 채택). 사용자가 이 페이지에 오는
  핵심 질문은 "그래서 내 통장에 매달 얼마?"이므로 이 숫자 하나만 히어로로 키운다. 연 환산은
  바로 아래 한 단계 작은 위계로 붙여 "월/연"을 한 시야에 보여준다.
- **8줄을 결과 카드에 기본으로 다 펼치면** 모바일에서 히어로 아래로 8~10행이 쌓여 카드 높이가
  과도해지고(§5 실측 ~500px+), 정작 대표값 대비 위계가 뭉개진다. 그래서 **항상 보이는 건 "세전
  → 4대보험 합계 → 세금 합계"의 3행 요약**(무엇이 빠졌는지 큰 그림은 즉시 파악)이고, 4대보험을
  4개로 쪼갠 개별 항목·중간값(과세대상 급여)은 **원하는 사용자만 접이식으로 펼친다.**
- 이는 정보 은닉이 아니다: 기획팀이 원한 "4대보험 4항목 + 소득세 + 지방세 개별 표시"는 ③에
  **전부 그대로 유지**되고, 접이식 라벨을 "공제 내역 자세히 보기"로 명확히 노출해 **찾기 쉽게**
  둔다(CS "안 보인다" 리스크 대비 — §8에서 기획팀에 접힘/펼침 기본값 확인 요청).

### 3-1. 결과 카드 공통 골격

design-system.md 3-2 준수: `mt-6` 간격, `rounded-xl border-l-4 border-brand-accent
bg-gradient-to-br from-emerald-50 to-white p-4 sm:p-6`. 대출 결과 카드와 동일한 Accent 고정
(BMI처럼 상태별 색상 분기 없음 — 실수령액에 위험/경고 색을 부여하지 않는다). 갱신 시
`transition-colors duration-200`. `resultRef`로 계산 후 스무스 스크롤(대출 패턴 재사용).

### 3-2. Tier ① — 대표값 영역 (항상)

```
예상 월 실수령액                         ← 라벨: text-sm font-medium text-brand-text-secondary
2,912,578 원                             ← 히어로: text-4xl sm:text-[2.5rem] font-bold
                                            tabular-nums text-brand-accent, "원"은 text-base
                                            font-medium text-brand-text-secondary
연 환산 약 34,950,936 원                  ← 보조 대표: mt-1 text-xl font-semibold text-brand-text
                                            tabular-nums (Accent 아님)
세전 연봉 4,000만원에서 4대 보험 약        ← 요약 문구(동적): mt-2 text-sm text-brand-text-secondary
30만원과 세금 약 11만원이 공제된 금액입니다.   (기획 §2 1문단을 입력값 기반으로 동적 축약, §3-5)
```

- 히어로/보조 대표 모두 `tabular-nums`, 천단위 콤마 필수.
- "약"·"원" 단위 텍스트는 히어로보다 작게(Body) Text Secondary — design-system.md 3-2 단위
  규칙.

### 3-3. Tier ② — 공제 요약 3행 (항상)

Tier ① 아래 구분선(`mt-4 pt-4 border-t border-brand-border/60`, 대출 패턴) 다음, 큰 그림
3행을 나열한다. 대출의 "총 이자/총 상환액" 2행 보조항목 패턴을 그대로 확장.

```
월 급여총액 (세전)        3,333,333 원
4대 보험 (근로자 부담)    − 304,230 원
소득세 · 지방소득세        − 116,525 원
```

- 각 행: `flex items-center justify-between py-1.5 text-sm`
- 라벨(좌): `text-brand-text-secondary`
- 값(우): `font-semibold tabular-nums text-brand-text`(Accent 아님 — 컬러 강조는 히어로 전용,
  시선 위계 유지). 공제 항목은 값 앞에 `−`(마이너스)를 붙여 "빠지는 돈"임을 시각적으로 명확화.
- 3행 고정. 이 3행 = "세전에서 무엇이 얼마 빠졌나"의 최소 요약. 합(세전 − 4대보험 − 세금)이
  Tier ① 월 실수령액과 일치함을 사용자가 검산할 수 있다.

### 3-4. Tier ③ — 공제 상세 (접이식, 기본 접힘)

Tier ② 아래에 접이식 토글. `<details>` 네이티브 요소 권장(JS 없이 접힘/펼침, 접근성 기본
확보). summary는 클릭 영역 전체(터치타깃 확보).

```
▸ 공제 내역 자세히 보기        ← summary: text-sm font-medium text-brand-primary, chevron 회전
  ─────────────────────────
  [4대 보험]                  ← 소그룹 헤더: text-xs font-semibold text-brand-text-secondary, mt-2
    국민연금        − 148,810 원
    건강보험        − 112,640 원
    장기요양보험    −  14,580 원
    고용보험        −  28,200 원
    소계            − 304,230 원   ← 소계 행만 font-semibold, 상단 얇은 구분선
  [세금]
    근로소득세      − 105,935 원
    지방소득세      −  10,590 원
    소계            − 116,525 원
  [과세 정보]                 ← 중간값(기획 §1-1 ②) 노출: 계산 근거 투명성
    월 과세대상 급여   3,133,333 원
    적용 비과세액      200,000 원
```

- 각 상세 행: Tier ②와 동일한 `flex justify-between py-1 text-sm`(padding만 `py-1`로 조금
  촘촘). 소그룹 헤더는 `[4대 보험]` 형태 대신 실제로는 작은 텍스트 라벨로.
- 소계 행: 상단 `border-t border-brand-border/40 mt-1 pt-1`, 값 `font-semibold`.
- **월 과세대상 급여 = 중간값 노출**(기획 §7-3이 UI 논의 대상으로 넘긴 값). 여기 접이식 안에
  둬서, 세금·4대보험이 "무슨 금액 기준으로" 계산됐는지 검증하고 싶은 사용자에게만 보인다.
  일반 사용자에겐 노출되지 않아 대표값 위계를 해치지 않는다.
- **근사 고지 인라인 노트**(§6-2): [세금] 그룹 소계 바로 아래 `text-xs text-brand-text-secondary`
  로 "근로소득세는 간이세액표 근사값으로 실제 원천징수액과 다를 수 있습니다." 1줄. → 근사
  고지 "3중"(상단 배너 + interpretation + FAQ) 중 결과 지점 보강.

### 3-5. Tier ① 요약 문구 = 기획 §2 1문단 동적 배치

기획 §2 1문단은 정적 예시(연봉 4,000만 기준)이며 **실제로는 사용자 입력값 기반 동적 생성**
(기획 §8-4, 기존 라이브 계산기와 동일). 이 문서는 **배치 위치만 확정**: Tier ① 히어로/연환산
바로 아래(§3-2 세 번째 줄). BMI의 `getResultSummary(...)`처럼 컴포넌트 내부 함수로 입력·결과를
받아 1~2문장을 조립한다. 정식 카피 톤은 기획 §2 담당.

- 길이: 모바일 2줄 이내 권장(길면 히어로 위계 방해). 4대보험·세금을 "약 XX만원"으로 반올림해
  간결하게(상세 숫자는 Tier ②·③에 있으므로 요약 문구는 개략치로).
- **YMYL-중간 카피 가드**: 이 문구에 "이직하면 더 받을 수 있다", "연봉 협상" 등 부추김/오인
  표현 금지(§6-3). 사실 서술만.

## 5. 모바일(375px) 반응형 규칙

### 5-1. 무스크롤 목표 재검토 — 대출과 동일하게 문자 그대로는 불가

iPhone SE급(375×667, 실사용 높이 약 550~600px) 기준 개략 높이 추정.

| 영역 | 예상 높이 |
|---|---|
| 근사 고지 배너(단일~2줄, §6-1) | 약 56~72px |
| 카드 전 여백 없음(배너-카드 `mb`만) + 카드 상단 padding | 16px |
| 연봉 필드(라벨+`h-12`+도움말) | 약 92px |
| gap-4 | 16px |
| 월 비과세액 필드(라벨+`h-12`+도움말) | 약 92px |
| gap-4 | 16px |
| 부양가족/자녀 2열 행(라벨+`h-12`+도움말) | 약 92px |
| gap-4 | 16px |
| 계산 버튼(`h-12`) | 48px |
| 카드 하단 padding | 16px |
| **입력 폼까지 소계** | **약 468~484px + 배너** ≈ **540~556px** |
| 결과 카드 Tier ①+②(접이식 접힘 상태, padding 포함) | 약 300~340px |
| **입력+결과(상세 접힘) 합계** | **약 840~896px** |

→ 입력 폼만으로 실사용 뷰포트를 거의 채우고, 결과까지는 한 화면에 안 들어온다. **대출 계산기와
동일한 판단**: design-system.md 4번 원칙 1을 문자 그대로 달성 불가하므로 아래 절충안을 택한다.

1. **상단 배너 광고는 이 페이지에 넣지 않는다**(모바일·데스크톱 공통). 최상단은 근사 고지 배너가
   우선(§6-1). 지면 예산 + YMYL 원칙.
2. **입력 폼은 최대한 압축**하되 라벨·도움말·48px 터치타깃은 유지(YMYL 정보 명확성 우선).
   도움말 Caption이 세로를 늘리면, 도움말을 입력 포커스 시에만 노출하거나 첫 필드에만 두는
   식으로 개발팀이 조정 가능(강제 아님).
3. **계산 버튼 클릭 시 결과 영역(`resultRef`)으로 스무스 스크롤** — 대출 컴포넌트에 이미 구현된
   패턴 그대로 재사용(강제 아님, 강력 권장). 사용자가 결과를 놓치지 않게 함.
4. **접이식(Tier ③)이 무스크롤을 지키는 핵심 장치**다. 상세를 기본 접힘으로 두어 결과 카드 초기
   높이를 300px대로 억제하고, 대표값(월 실수령)은 스크롤 직후 즉시 above-the-fold에 온다.

### 5-2. 결과 카드의 모바일 세로 스택 — 가로 오버플로 0 보장

이 페이지에서 가장 신경 쓸 반응형 리스크는 **공제 항목이 많은데 375px에서 가로로 터지는 것**
이다. 이를 **원천 차단**하기 위한 규칙:

- **HTML `<table>`(다열)로 만들지 않는다.** Tier ②·③의 모든 항목은 **라벨-값 1행**을
  `flex items-center justify-between`으로 구성한다(대출 보조항목과 동일). 라벨은 좌측 자연
  줄바꿈 허용, 값은 우측 고정 → **가로 스크롤이 구조적으로 발생할 수 없다.**
- 값은 `tabular-nums`로 자릿수 정렬. 최장 값(연 환산 실수령, 8자리 "34,950,936 원")도
  `justify-between` 우측 정렬이라 375px에서 라벨과 겹치지 않음(라벨이 짧아 여유). 만약 특정
  행에서 라벨+값이 한 줄을 초과하면 라벨이 2줄로 줄바꿈되고 값은 우측 유지 → 여전히 오버플로
  없음.
- Tier ① 히어로 숫자: 최장 8자리(연봉 상한 고소득자 "9,999,999 원") 기준 `text-4xl`
  (36px)에서 375px 카드 안쪽 폭(약 311px = 375 − 좌우 16 − border-l-4 등)에 들어감. 안전 위해
  히어로는 모바일 `text-4xl`(36px), `sm` 이상 `text-[2.5rem]`(40px). design-system.md Result
  스케일 "모바일 28px 이하 축소 금지" 준수(36px ≥ 28px OK).
- 연 환산 보조 대표(`text-xl`, 20px)는 8자리 여유 있게 수용.
- Tier ③ 접이식은 세로로만 늘어나므로 폭 이슈 없음.

### 5-3. 부양가족/자녀 2열의 모바일 처리

`grid grid-cols-2 gap-3`(BMI 패턴)을 375px에서도 유지한다(각 셀 약 150px). 셀렉트는 콘텐츠가
"3명" 정도로 짧아 2열에서도 잘림 없음. 라벨("8~20세 자녀 수")이 2줄이 될 수 있으므로 두
셀렉트의 라벨 영역 높이가 달라 셀렉트가 어긋나지 않도록, 라벨 컨테이너에 `min-h`를 주거나
`items-end`로 셀렉트 상단선을 맞추는 것을 권장(개발팀 재량). 좁으면 `sm` 미만에서 1열
(`grid-cols-1`) 폴백도 허용하나, 두 값의 의존관계(자녀 ≤ 부양−1)를 나란히 보여주는 이점이
있어 2열 유지를 우선한다.

## 6. YMYL(중간) 반영 — 근사 고지 배치 및 CTA 금지

### 6-1. 근사 고지 배너 (상단, 대출 배너 패턴 재사용 + 톤 조정)

마스터 확정: "간이세액표 근사, 실제와 다를 수 있음" 고지 필요 + 대출 상단 면책 배너 패턴 재사용
검토. **디자인팀 결정: 재사용한다.** 대출 컴포넌트의 배너 마크업을 그대로 가져온다.

- 위치: 컴포넌트 최상단, 계산기 카드 위(§1 와이어프레임). 계산 여부와 무관하게 항상 노출.
- 스타일(대출과 동일): `mb-6 flex items-start gap-2 rounded-lg border border-amber-200
  bg-amber-50 px-4 py-3`, `role="note"`. 좌측 `ⓘ`(`text-brand-warning`, `aria-hidden`),
  본문 `text-sm text-amber-900`.
- 카피 골자(정확한 문안은 기획팀, §8): "근로소득세는 국세청 간이세액표를 **근사한** 값으로
  계산되어 실제 급여명세서·원천징수액과 다를 수 있습니다. 참고용으로만 활용하세요."
- **광고 유닛 아님** — "광고" 캡션 금지, 항상 노출.
- **YMYL 등급 차이 반영**: 대출(높음)은 배너 카피가 "상담·신청이 아님 + 금융기관 확인"까지
  강한 어조였으나, 이 페이지(중간)는 **근사·참고용 고지에 초점**을 맞춰 1~2줄로 간결하게. 톤은
  대출보다 약간 가볍게(경고가 아니라 안내). → 배너를 YMYL-중간에도 적용할지(대출 전용 요소를
  확장하는지)는 사이트 톤앤매너 영향 사안이라 §8 마스터 확인.

### 6-2. 결과 지점 근사 고지 (인라인, §3-4)

Tier ③ [세금] 소계 아래 `text-xs text-brand-text-secondary` 1줄(§3-4). interpretation 하단
면책(기획 §2 ⑤단)·FAQ 2번(기획 §3)과 함께 근사 한계를 3중 고지한다.

### 6-3. CTA 디자인 금지 원칙 (YMYL-중간, 급여/이직 부추김 절대 금지)

- 계산기 카드 안에는 **"연봉 실수령액 계산하기" 버튼 1개만** 존재. "이직하면 얼마?", "연봉
  협상 팁", "더 받는 법", "채용 공고 보기" 류 2차 버튼/링크를 결과 카드나 카드 주변에 **절대**
  추가하지 않는다.
- 결과·요약 문구(§3-5)에도 급여 인상·이직·전직을 부추기거나 암시하는 표현 금지. 사실 서술만.
- 공식 기관(홈택스 등)을 참조·링크할 필요가 있으면(주로 interpretation 본문, 범위 밖)
  **일반 본문 링크 스타일**(밑줄 + Primary 컬러 인라인)만 사용하고 박스형 CTA 버튼 금지 —
  버튼 스타일은 계산 버튼 전용 예약.
- "관련 계산기" 카드(공통 템플릿)는 내부 링크로만. 외부 채용/이직/연봉정보 서비스 연결 카드
  금지.

### 6-4. 광고 배치 (design-system.md 5장 + 이 페이지 규칙)

- 상단 배너 광고: 넣지 않음(§5-1, 근사 고지 배너가 그 자리).
- 본문 중간/하단 광고: 공통 템플릿이 처리(범위 밖). design-system.md 5장 공통 규칙(위아래
  margin 24px+, "광고" Caption 라벨, 결과·경고 영역과 배경/테두리 구분) 준수.
- **이 페이지 규칙**: 광고 컨테이너를 결과 카드(에메랄드 틴트 `bg-emerald-50` + 좌측 강조선)나
  근사 고지 배너(amber 틴트)와 유사한 톤으로 만들지 않는다 — 급여/재테크/대출 광고 소재가
  "연봉 올리기" 류 문구를 포함하더라도 사이트 UI와 명확히 구분되는 중립 배경(White/Gray 50 +
  얇은 회색 테두리)으로 감싸 콘텐츠 오인·클릭 유도로 보이지 않게 한다.

## 8. 다른 팀 전달 사항 (마스터 팀장 경유)

**기획팀 확인 요청**
- **[중요] 접이식(Tier ③) 기본값 = 접힘으로 확정한 것에 대한 합의**: 기획 §7-3은 "4대보험
  4항목 + 소득세 + 지방세를 표로" 제안했다. 디자인팀은 이 8줄을 **버리지 않고 접이식 안에 전부
  유지**하되 **기본은 접고**, 항상 보이는 것은 "세전/4대보험 합계/세금 합계" 3행 요약으로 정했다
  (모바일 무스크롤·대표값 위계 근거, §3-0). 기획팀이 "공제 개별 항목은 펼치지 않아도 사용자에게
  충분히 전달되는가"를 콘텐츠 관점에서 확인해달라. 만약 "개별 공제 노출이 이 계산기의 핵심
  가치라 기본 펼침이어야 한다"고 판단하면 회신 요망(그 경우 Tier ③ 기본 열림으로 조정 가능).
- 상단 근사 고지 배너(§6-1)·결과 인라인 노트(§6-2) **정확한 카피** 확정 필요.
- 요약 문구(§3-5) 동적 조립 문안·톤(YMYL-중간 부추김 금지) 확정.

**개발팀 전달 사항**
- 신규 `components/SalaryNetCalculator.tsx`, `lib/salary.ts` 구현. `CALCULATOR_COMPONENTS`
  매핑 추가, `lib/calculators.ts` 콘텐츠 확보 후 `status: "live"` 전환.
- 기획 §8-2 파이프라인 순서(4대보험→소득세), §8-3 국민연금 clamp·천원절사, 장기요양 12.9457%,
  §8-5 요율 상수 모듈화 준수.
- 연봉·비과세액 천단위 콤마(`formatPrincipalInput` 재사용), 결과 전 금액 `formatWon`.
- §2-2 셀렉트 채택 시 §2-3 자녀 옵션 동적 캡(부양가족−1) 구현. 숫자입력 대체 시 제출 시
  의존 검증 처리.
- §3-4 `<details>` 접이식, §5-1 계산 후 결과 스무스 스크롤 재사용 권장.
- 375×667 / 390×844에서 §5-1 추정치(입력~540px, 결과 접힘 ~340px) 및 §5-2 가로 오버플로 0을
  실 구현 후 디자인팀이 재검수 예정.

**마스터 팀장 확인 필요**
1. **상단 근사 고지 배너를 YMYL-중간 계산기에 도입(§6-1)**: 상단 배너는 원래 대출(YMYL-높음)
   전용으로 신규 도입한 요소다. 이번에 YMYL-중간인 연봉 계산기에 (톤을 낮춰) 확장 적용하는 것이
   맞는지 확인 요망. 승인되면 향후 퇴직금·4대보험료 등 YMYL-중간 이상 계산기의 공통 규칙으로
   삼을 수 있다(사이트 톤앤매너 영향 사안이라 임의 확장하지 않고 문의).
2. **design-system.md 신규 컴포넌트 편입 여부**: (a) `select` 입력 스타일(§2-2), (b) 결과 카드
   내 접이식 `<details>` 상세 블록(§3-4), (c) "대표값(월) + 보조 대표(연 환산)"의 2단 히어로
   숫자 패턴(§3-2). 대출 문서의 세그먼트 토글·병렬 숫자 패턴처럼, 재사용 가능성이 높으므로
   `design-system.md`에 정식 편입할지 결정 요망(승인 전까지 본 문서 제안 단계로 취급, 진행은
   막지 않음).

## 9. 대안으로 검토했으나 채택하지 않은 안

- **8줄 공제표를 결과 카드에 전부 기본 노출(기획 원안)**: 정보 투명성은 최고이나, 모바일에서
  히어로 아래 10행이 쌓여 카드 높이 500px+ → 대표값 위계 훼손·무스크롤 완전 실패. 3-tier로
  "요약은 항상, 개별은 접이식"으로 절충(§3-0). 개별 항목을 없앤 게 아니라 접었을 뿐이라 투명성
  손실은 최소.
- **부양가족/자녀를 숫자입력(`type="number"`)으로**: 구현 간단하나 유효범위(자녀 ≤ 부양−1)를
  사용자가 모른 채 입력→제출→에러 흐름이 되어 UX 저하. 좁은 정수 범위라 셀렉트가 오입력을
  원천 차단(§2-2). 개발 공수 이슈 시 대체 허용하되 의존 검증 필수.
- **월/연 실수령을 나란히 2개 히어로로**(대출 원금균등 "1회차→마지막" 병렬처럼): 두 숫자 위계가
  동등해져 "그래서 매달 얼마?"의 초점이 흐려짐. 월을 단일 대표, 연은 한 단계 낮춰 종속시키는
  §3-2가 초점이 명확하다고 판단.
