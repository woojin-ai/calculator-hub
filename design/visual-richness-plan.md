# 비주얼 리치니스(이미지+색채) 보강 방향 (v1.0, 2026-07-13)

> 배경: 마스터 팀장 피드백 — "사이트가 전체적으로 밋밋하다, 이미지나 색채감이 부족하다."
> 마스터가 힉스필드(이미지 생성 도구)로 이미지를 제작해 넣을 예정이며, 이 문서는 (1) 어디에
> 어떤 스타일로 넣을지, (2) 기존 팔레트 안에서 추가로 줄 수 있는 포인트, (3) 힉스필드
> 프롬프트에 그대로 쓸 수 있는 스타일 가이드를 제공한다.

원칙: **기존 팔레트(블루600/에메랄드500 기반)를 뒤집지 않는다.** 새 색상 계열 도입, 다크모드
전환, 완전히 다른 톤(예: 파스텔/비비드 컬러 전면 도입) 같은 큰 톤앤매너 변경은 이 문서에서
제안하지 않으며, 그런 변경이 필요하다고 판단되면 별도로 마스터 팀장에게 먼저 확인한다. 아래
제안은 모두 "기존 팔레트 위에 포인트를 더하는" 범위로 한정했다.

또한 `design/design-system.md`가 이미 명시한 애드센스 승인 조건("모바일 최적화, 빠른 로딩")과
충돌하지 않도록, 이미지는 **가볍고 절제된 방식**으로만 추가한다(5장 기술 가이드 참고).

## 1. 홈페이지 히어로 섹션 이미지

### 1-1. 배치 위치 및 방식

현재 `app/page.tsx` 히어로는 텍스트만 있는 `text-center sm:text-left` 블록이다. 아래처럼
변경할 것을 제안한다.

- **데스크톱(`lg` 이상, 1024px~)**: 히어로 섹션을 `lg:flex lg:items-center lg:justify-between
  lg:gap-10` 2단 구성으로 전환. 좌측에 기존 H1+리드 문단(`lg:max-w-md` 정도로 폭 제한), 우측에
  일러스트 이미지 1개(정사각형에 가까운 비율, 폭 320~380px 권장).
- **모바일/태블릿(`lg` 미만)**: **히어로 이미지는 노출하지 않는다** (`hidden lg:block`).
  이유: design-system.md 4번 원칙 1 "모바일에서 입력 관련 핵심 콘텐츠가 스크롤 없이 보이는 것"
  및 애드센스 로딩 속도 조건과 상충하지 않도록, 화면이 좁아 이미지가 실질적 여백만 차지하는
  모바일에서는 과감히 뺀다. 이는 "이미지를 아예 안 넣는다"던 기존 about 페이지 스펙(이미지
  없음 원칙)과는 다른 결정이다 — 데스크톱에서는 넣되 모바일에서는 뺀다는 절충안으로, 마스터의
  "밋밋하다"는 피드백과 기존 로딩속도 원칙을 동시에 만족시키는 방향이다.
- 이미지 자체(일러스트 내용): 계산기·동전·달력·집 아이콘이 떠 있는 느낌의 플랫 벡터 구성,
  또는 "스마트폰/노트북으로 계산기를 사용하는 사람" 실루엣 + 주변에 결과 숫자·체크 아이콘이
  떠 있는 구성. 사람을 넣을 경우 특정 성별·연령이 드러나지 않는 심플한 실루엣으로.

### 1-2. 배경 처리 (이미지 없이도 가능한 즉시 반영 가능 항목)

이미지 자산이 준비되기 전에도 아래는 **CSS만으로 즉시 적용 가능한 저위험 개선**이라 우선
반영을 제안한다.

- 히어로 섹션 배경을 기존 `bg-brand-bg`(단색 그레이) 대신 아주 옅은 대각선 그라디언트로 변경:
  `bg-gradient-to-br from-blue-50 via-white to-emerald-50`
- 팔레트 안의 두 브랜드 컬러(Primary blue/Accent emerald)를 아주 옅은 톤(50 단계)으로만 섞어
  "생기"를 주되, 본문 텍스트 대비(WCAG AA)에는 영향이 없도록 컬러 값 자체를 바꾸지 않는다.
- 히어로 섹션에만 적용하고 나머지 페이지 배경은 기존 `bg-brand-bg` 유지(전체 톤 변경 아님).

## 2. 카테고리 카드 아이콘 (급여/대출/날짜/생활)

### 2-1. 배치

- `app/page.tsx`의 "카테고리" 섹션 카드, `app/about/page.tsx`의 카테고리 요약 카드, 각
  `/salary` `/loan` `/date` `/life` 카테고리 페이지 상단(H1 옆)까지 **동일한 아이콘 세트를
  재사용**해 사이트 전반의 시각적 일관성을 만든다.
- 카드 내부 배치: 기존 `<h3>` 제목 위에 아이콘을 추가, `mb-3` 간격.
  - 아이콘을 감싸는 원형 배지: 48px 원, 배경은 카테고리 공용이 아니라 **Primary 톤 하나로
    통일**(`bg-blue-50`), 아이콘 자체는 24px, 컬러는 `text-brand-primary`(또는 결과 관련
    뉘앙스가 있는 카테고리는 `text-brand-accent`로 구분 — 예: 급여/대출처럼 숫자·금액 계산
    카테고리는 blue, 생활(BMI 등 건강)처럼 "결과/성과" 느낌이 강한 카테고리는 emerald. 날짜는
    중립적으로 blue 유지). 이 구분은 새로운 색상 의미를 만드는 것이 아니라 기존 Primary/
    Accent 두 색을 아이콘 배지에 재사용하는 것이므로 팔레트 확장이 아니다.
- 카테고리별 아이콘 소재(제안):
  - 급여(`/salary`) → 지폐/동전 아이콘
  - 대출(`/loan`) → 집 + 퍼센트, 또는 계약서 아이콘
  - 날짜(`/date`) → 캘린더 아이콘
  - 생활(`/life`) → 저울 또는 하트 아이콘(BMI 등 건강 계산기 이미지와 연결)

### 2-2. 구현 방식 선택지 (마스터 판단 필요)

| 방식 | 장점 | 단점 |
|---|---|---|
| (A) 코드 인라인 SVG 아이콘 (개발팀이 직접 심플한 라인 아이콘 작성/현재 미설치된 아이콘 라이브러리 없이) | 파일 크기 거의 0, `currentColor`로 손쉽게 재색상, 어떤 해상도에서도 선명 | 힉스필드로 "생성"하는 것은 아님(마스터의 "이미지 생성" 취지와는 약간 다를 수 있음) |
| (B) 힉스필드로 카테고리별 아이콘 이미지 4장 생성 후 `public/images/icons/`에 저장 | 마스터가 원하는 "이미지 생성" 방식과 정확히 일치, 스타일 통일감을 이미지 자체로 확보 | 아이콘처럼 작은 요소도 래스터/SVG 파일로 관리해야 함, 4장 모두 톤이 어긋나지 않게 한 번에 생성해야 함(4장 프롬프트를 최대한 동일 스타일 가이드로 통일 — 4장 참고) |

**제안: 카테고리 아이콘 4개는 (B) 힉스필드 생성 방식을 권장한다.** 마스터가 이미지 생성 도구를
이미 쓸 계획이므로, 4장을 한 스타일 가이드로 일괄 생성하면 통일감을 얻기 쉽고 "이미지가
없다"는 원 피드백에도 더 직접적으로 대응한다. 다만 최종 채택은 개발팀 구현 편의성도 함께
고려해 결정할 것.

## 3. 기존 팔레트 안에서 포인트를 줄 수 있는 곳 (이미지 없이 가능)

이미지 자산 준비와 무관하게, 코드만으로 바로 반영 가능한 항목들이다. design-system.md의
"신뢰감·가독성" 원칙은 유지한다.

1. **라이브 계산기 배지**: `CalculatorCard.tsx`는 현재 "준비 중"일 때만 배지를 보여준다.
   `status: "live"`인 카드에도 작은 초록 배지를 추가 제안: `inline-flex items-center gap-1
   rounded-full bg-emerald-50 px-2 py-0.5 text-xs font-medium text-emerald-600` + 좌측에
   작은 점(●). 문구 예: "바로 사용 가능". 카드가 "일부만 활성화됨"을 시각적으로 더 명확히
   전달하면서 톤은 기존 Accent 컬러 재사용이라 팔레트 변경이 아니다.
2. **결과 카드 틴트 강화**: design-system.md 3-2의 결과 카드(연한 그린/블루 틴트 + 좌측 4px
   강조선)를 단색 틴트에서 **아주 옅은 2톤 그라디언트**로 바꾸는 것을 제안:
   `bg-gradient-to-br from-emerald-50 to-white`(또는 blue 계열은 `from-blue-50 to-white`).
   시각적으로 살짝 입체감을 주되 컬러 값 자체(Accent `#10B981`, Primary `#2563EB`)는 그대로
   유지하므로 결과 숫자의 대비·가독성에는 영향 없음.
3. **카테고리 페이지(`/salary` 등) 상단**: H1 옆에 2장의 아이콘 배지를 재사용해 배치하면
   카테고리 진입 시에도 홈페이지 카드와 동일한 아이콘을 다시 보게 되어 정보 구조 인지에도
   도움이 된다(브랜드 일관성).
4. **버튼/CTA는 현행 유지**: "계산하기" 등 solid Primary 버튼에 그라디언트나 과한 그림자를
   추가하는 것은 권장하지 않는다. 애드센스 사이트는 버튼이 광고처럼 화려해지는 순간 오히려
   신뢰도가 떨어지고, "클릭 유도 디자인 금지" 원칙과도 맞닿아 있어 액션 버튼은 지금의 절제된
   solid 스타일을 유지하는 편이 낫다.

## 4. 이미지 생성 스타일 가이드 (힉스필드 프롬프트용)

마스터가 힉스필드 프롬프트에 아래 내용을 그대로 붙여 넣어 쓸 수 있도록 공통 가이드와, 용도별
예시 프롬프트를 함께 제공한다.

### 4-1. 공통 스타일 가이드 (모든 이미지에 공통 적용)

```
플랫 벡터 일러스트 스타일(flat vector illustration), 미니멀 디자인, 과도한 디테일과 복잡한
텍스처 지양. 색상은 블루(#2563EB)를 메인으로, 에메랄드(#10B981)를 보조로 하는 2색 조합만
사용하고, 필요 시 흰색/아주 옅은 그레이(#F9FAFB) 정도만 배경 요소로 추가. 배경은 완전 투명
(transparent PNG) 또는 순수 흰색(#FFFFFF) 단색 중 하나로 통일. 그림자는 넣더라도 아주 옅게,
과장된 입체감·네온·글로우 효과 금지. 사람을 그릴 경우 특정 성별·연령·인종이 드러나지 않는
심플한 실루엣 또는 아이콘화된 형태로 표현. 텍스트·숫자·로고·워터마크는 이미지 안에 넣지 않음
(별도로 코드에서 얹을 예정). 전반적 톤은 금융/생활 서비스에 어울리는 차분하고 신뢰감 있는
느낌으로, 만화적 과장된 표정이나 원색 남발은 피한다.
```

### 4-2. 용도별 예시 프롬프트

**홈페이지 히어로 일러스트 (1-1 참고)**
```
[공통 스타일 가이드 적용] 스마트폰을 들고 계산 결과를 확인하는 사람의 심플한 실루엣, 주변에
계산기, 동전, 달력, 체크 표시 아이콘이 가볍게 떠 있는 구성. 정사각형에 가까운 비율(1:1~4:3),
좌우 여백을 넉넉히 두어 텍스트와 나란히 배치해도 답답하지 않게. 배경 투명.
```

**카테고리 아이콘 — 급여**
```
[공통 스타일 가이드 적용] 지폐와 동전을 단순한 형태로 표현한 아이콘, 정사각형 프레임(1:1),
배경 투명, 라인 두께 일정하게 유지.
```

**카테고리 아이콘 — 대출**
```
[공통 스타일 가이드 적용] 작은 집 모양과 퍼센트(%) 기호를 함께 배치한 아이콘, 정사각형
프레임(1:1), 배경 투명.
```

**카테고리 아이콘 — 날짜**
```
[공통 스타일 가이드 적용] 캘린더에 체크 표시가 된 아이콘, 정사각형 프레임(1:1), 배경 투명.
```

**카테고리 아이콘 — 생활**
```
[공통 스타일 가이드 적용] 저울 또는 하트 모양을 단순화한 아이콘(건강/생활 지표를 상징),
정사각형 프레임(1:1), 배경 투명.
```

4장의 아이콘은 반드시 같은 세션/같은 프롬프트 톤으로 연속 생성해 스타일이 서로 어긋나지 않게
할 것을 권장한다(라인 두께, 색 배합 비율 등이 장마다 다르면 오히려 통일감이 깨져 "밋밋하다"는
문제보다 "제각각이다"는 새 문제가 생길 수 있음).

## 5. 기술/성능 가이드 (개발팀 전달용, 이미지 반영 시 준수)

- **파일 형식**: 아이콘류는 가능하면 SVG로 받거나 변환(힉스필드가 SVG를 직접 못 주면 PNG로
  받은 뒤 개발팀이 벡터화 검토), 히어로처럼 복잡한 일러스트는 WebP 권장(PNG 대비 용량 절감).
- **용량 기준**: 히어로 이미지 1장 150KB 이하, 아이콘 1장 20KB 이하를 목표로 한다(로딩 속도
  우선 원칙, design-system.md와 동일 원칙 재확인).
- **저장 위치**: 힉스필드에서 생성한 이미지는 마스터가 다운로드해 `public/images/` 하위에
  저장하고 로컬 경로로 참조하는 것을 권장한다(외부 이미지 호스팅 도메인을 `next.config`에
  추가하고 원격 로딩하는 방식보다 로딩 속도·안정성 면에서 유리, CORS/도메인 만료 리스크도 없음).
- **`next/image` 사용**: 폭/높이(width/height)를 명시하고, 히어로 이미지처럼 above-the-fold에
  있는 이미지는 `priority` 속성을 고려(개발팀 판단), 아이콘류는 지연 로딩(lazy, 기본값)으로
  충분.
- **대체 텍스트(alt)**: 모든 이미지에 의미 있는 한국어 alt 텍스트 필수(예: "급여 계산기
  아이콘", "계산기 허브 소개 일러스트") — 접근성뿐 아니라 SEO에도 도움. 순수 장식 목적이라면
  `alt=""` 처리도 허용(개발팀 판단).

## 6. 다른 팀 전달 사항

**마스터 팀장 확인 필요**
- 2-2 "카테고리 아이콘 구현 방식"은 (A) 코드 SVG / (B) 힉스필드 생성 이미지 중 선택이 필요하다
  — 힉스필드 활용 취지에 맞춰 (B)를 권장하지만 최종 선택 요청.
- 1-2 히어로 배경 그라디언트는 이미지 없이 즉시 적용 가능한 저위험 항목이라, 이미지 생성을
  기다리지 않고 먼저 반영해도 되는지 확인 요청(색상 계열 자체는 바뀌지 않으므로 큰 톤앤매너
  변경은 아니라고 판단했으나, 배경색이 처음으로 그라디언트를 쓰는 시도라 확인차 보고).

**개발팀 전달 사항**
- 1장(히어로), 2장(카테고리 아이콘) 반영 시 `app/page.tsx`, `app/about/page.tsx`, 각 카테고리
  페이지, `components/CalculatorCard.tsx` 수정 범위에 해당. 이미지 자산 도착 전까지는 3장(코드
  만으로 가능한 항목)부터 먼저 반영해도 무방.
- 5장 기술 가이드(파일 형식/용량/저장 위치/next-image/alt) 준수 요청.

이미지 자산이 실제로 준비되면, 반영된 화면을 데스크톱/모바일 뷰포트에서 이 문서와 대조해
검수할 예정이다.
