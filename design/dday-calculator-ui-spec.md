# D-Day 계산기 — UI 구성안 (v1.0, 2026-07-09)

> 대상: `lib/calculators.ts`의 `dday-calculator` (category: `date`)
> 실제 라우팅은 기존 구조(`app/calculator/[slug]/page.tsx`)를 그대로 따른다 → 실 URL은
> `/calculator/dday-calculator` 이다. (요청에서 언급된 "date/dday-calculator"는 카테고리/슬러그를
> 개념적으로 표기한 것으로 보이며, 기존 만 나이 계산기와 동일하게 카테고리 접두어가 붙는 별도
> 라우트가 아님. 혼동 방지를 위해 기획팀·개발팀 확인 필요 — 하단 "다른 팀 전달 사항" 참고.)

본 문서는 확정된 `design/design-system.md` 스펙을 그대로 따르며, 기존 라이브 컴포넌트인
`components/AgeCalculator.tsx` 의 구조·톤을 최대한 재사용해 일관성을 유지한다. 코드는 작성하지
않으며 개발팀이 본 문서를 참고해 `components/DdayCalculator.tsx`(신규)를 구현한다.

## 1. 페이지 전체 레이아웃 (기존 계산기 상세 페이지 구조 재사용)

`app/calculator/[slug]/page.tsx`의 기존 골격을 그대로 따른다. 신규 레이아웃 설계가 필요한 부분은
"2. 계산기 입력/결과 영역"뿐이다.

```
[Breadcrumb: 홈 / 날짜 계산기]
[H1] D-Day 계산기
[본문 요약 1줄]
─────────────────────────────
[계산기 카드]  ← 이번 스펙의 핵심 (2번 항목)
  - 기준일 입력
  - (선택) 이벤트 이름 입력
  - 계산 버튼
  - 결과 표시 영역
─────────────────────────────
[결과 해석]         ← ResultInterpretation 재사용
[본문 중간 광고 여백]  ← design-system.md 5번 가이드 준수
[FAQ 아코디언]        ← FaqAccordion 재사용
[관련 계산기]         ← RelatedCalculators 재사용
[하단 광고]           ← design-system.md 5번 가이드 준수
```

데스크톱 `lg` 이상에서는 design-system.md 4번 원칙에 따라 좌측 콘텐츠(2/3) + 우측 사이드바(1/3,
sticky 광고)의 2단 레이아웃을 고려할 수 있으나, 이는 페이지 공통 레이아웃 이슈이므로 만 나이
계산기 페이지와 통일된 방식으로 처리한다 (현재 만 나이 페이지가 사이드바 없이 단일 컬럼
`max-w-3xl`이므로, D-Day 페이지도 동일하게 단일 컬럼 유지 — 별도 사이드바 도입은 전체 페이지
공통 변경 사항이라 이 문서 범위 밖. 필요 시 마스터 팀장 확인 후 전체 계산기 공통으로 적용).

## 2. 계산기 입력/결과 영역 (신규 구성)

### 2-1. 입력 UI

| 요소 | 스펙 |
|---|---|
| 기준일 입력 | `type="date"`, 라벨 "기준일", placeholder 불필요(date input 특성상). AgeCalculator와 동일하게 `h-12`(48px), radius `8px`, 포커스 시 Primary 컬러 링. **AgeCalculator와 달리 `max` 제한(오늘까지)을 걸지 않는다** — D-Day는 미래(예: 수능일, 결혼식)와 과거(예: 이미 지난 결혼기념일 경과일 확인) 날짜를 모두 입력할 수 있어야 하므로 min/max 제한 없음. |
| 이벤트 이름 입력 (선택) | `type="text"`, 라벨 "이벤트 이름 (선택)", placeholder `예: 수능, 결혼기념일, 여행 출발일`. 아래 2-2 참고 — 기본 접힘 상태로 제공. |
| 계산 버튼 | 필요함. **AgeCalculator와 동일하게 "즉시 자동계산"이 아니라 버튼 클릭(제출) 방식으로 통일한다.** 사이트 전체의 계산기 인터랙션 패턴을 하나로 유지하는 것이 사용자 학습비용을 줄이고, 이벤트 이름을 입력하는 동안(키 입력마다) 결과가 계속 재계산/리렌더되는 것을 방지하는 실익도 있다. 버튼 라벨: "D-Day 계산하기". 스타일은 AgeCalculator 버튼과 동일(`h-12`, Primary 배경, hover 시 Primary Hover). |

**판단: 이벤트 이름 입력은 넣는 것을 권장한다.** 이유:
- "D-100"이라는 숫자만 보여주는 것보다 "수능까지 D-100"처럼 자신이 지정한 이벤트명이 결과에
  그대로 반영되면 개인화 체감이 커지고, 결과 캡처/공유 시에도 의미가 분명해져 재방문·체류시간
  확보에 도움이 된다.
- 다만 D-Day 계산의 핵심 입력은 "날짜" 하나뿐이므로, 이벤트 이름을 필수 항목처럼 상단에 나란히
  노출하면 입력 단계가 늘어난 것처럼 느껴지고 모바일에서 세로 공간을 추가로 차지해 "입력→결과가
  스크롤 없이 한 화면에" 라는 최우선 원칙(design-system.md 4번)을 위협한다.
- 따라서 **기본 상태에서는 이벤트 이름 입력을 접어두고, 텍스트 토글로 펼치는 방식**을 아래
  2-2에서 규정한다.

### 2-2. 이벤트 이름 입력 — 접이식(토글) UI

- 기준일 입력 필드 아래, 버튼 위에 링크형 토글 텍스트를 배치: `+ 이벤트 이름 추가 (선택)`
  - 스타일: Body Small(14px), Text Secondary 컬러, 클릭 시 Primary 컬러로 전환
  - 터치 타겟: 텍스트만이 아니라 **한 줄 전체를 클릭 영역으로 확보**하고 세로 padding을 주어
    최소 44px 높이를 만족시킨다 (design-system.md 4번 원칙 3 "터치 타겟 최소 44x44px").
- 토글을 탭하면 그 자리에 "이벤트 이름" 텍스트 입력 필드가 펼쳐진다 (AgeCalculator의 날짜
  입력 필드와 동일한 `h-12`, radius `8px`, 라벨 스타일 재사용). 펼쳐진 뒤에는 토글 텍스트가
  사라지고 입력 필드만 남는다 (다시 접는 기능은 필수는 아님 — 개발팀 판단에 맡김, 접는 기능을
  넣더라도 UX상 문제 없음).
- 이 방식으로 기본 화면(모바일 최초 진입 시)은 "기준일 입력 1개 + 토글 링크 1줄 + 버튼"만
  노출되어, AgeCalculator와 거의 동일한 세로 높이를 유지하면서도 원하는 사용자는 한 번의 탭으로
  이벤트 이름을 추가할 수 있다.
- 이벤트 이름을 입력하지 않고 계산해도 정상 동작해야 한다 (완전 선택 항목).

### 2-3. 결과 표시 영역

design-system.md 3-2 "결과 표시 영역" 스펙을 그대로 따른다 — 카드 형태, 좌측 강조선(4px,
Accent), 연한 그린 틴트 배경(`#F0FDF4` = Tailwind `emerald-50`), 입력 영역 바로 아래
above-the-fold 배치.

**레이아웃 (위에서 아래 순서):**

1. **결과 라벨** (Body Small, Text Secondary)
   - 이벤트 이름 입력 시: `{이벤트 이름}까지` (미래) 또는 `{이벤트 이름}으로부터` (과거)
     - 예: "수능까지", "결혼기념일로부터"
   - 이벤트 이름 미입력 시: `선택하신 날짜까지` / `선택하신 날짜로부터`
   - 당일(D-Day)인 경우: `오늘은` 또는 이벤트 이름 그대로 사용 (문구는 개발팀 재량, 의미만 준수)

2. **D-Day 숫자 표시** (design-system.md Result 스케일 그대로 적용: 36~40px, Bold 700,
   모바일에서도 축소 금지)
   - 표기 형식 3종:
     - 미래 날짜: `D-123` (남은 일수)
     - 당일: `D-Day` (숫자 없이 텍스트 그대로, 동일한 Result 크기/굵기 유지)
     - 과거 날짜: `D+45` (지난 일수)
   - 컬러: design-system.md 그대로 Accent(`#10B981`) 사용. 미래/과거를 색으로 구분하는 별도
     컬러 규칙은 만들지 않는다 (디자인 시스템에 정의되지 않은 새 색상 의미 부여는 마스터 팀장
     확인 없이 임의로 추가하지 않음).
   - 숫자에는 `tabular-nums` 적용 (기존 결과 영역과 동일).
   - `D-Day`(당일)의 경우 살짝의 축하 톤을 원한다면 숫자 아래 보조 문구로 "바로 오늘입니다"
     정도의 텍스트만 추가하고, 이모지·색상 변경 등 톤앤매너에 영향을 주는 장식은 넣지 않는다
     (필요하다고 판단되면 별도로 마스터 팀장에게 제안 후 반영).

3. **보조 정보 라인** (Body Small, Text Secondary) — AgeCalculator의 "다음 생일까지 N일
   남았습니다" 라인과 동일한 위계
   - 대상 날짜의 실제 날짜 표기: 예 `2026년 11월 12일 (목)`
   - 과거 날짜인 경우 "이미 지난 날짜입니다." 같은 보조 안내 1줄 추가 가능

### 2-4. 에러 상태

AgeCalculator 패턴 그대로 재사용:
- 기준일 미입력 후 계산 버튼 클릭 시: 입력 필드 테두리 Warning 컬러, 하단 Caption 크기 에러
  메시지 "기준일을 입력해주세요." (`role="alert"`)
- 날짜 범위 제한이 없으므로 "미래 날짜 입력 불가" 류의 에러 문구는 사용하지 않는다 (AgeCalculator와
  다른 점 — 개발팀 구현 시 주의).

## 3. 모바일 레이아웃 검토 (스크롤 없이 한 화면 확보)

목표 기준: iPhone SE급 소형 뷰포트(375×667, 브라우저 UI 제외 실사용 높이 약 550~600px)에서도
"입력 → 결과"가 한 화면에 보이는 것을 1순위로 한다 (design-system.md 4번 원칙 1).

**기본 상태(이벤트 이름 토글 접힘) 예상 높이 구성:**

| 영역 | 예상 높이 |
|---|---|
| 상단 breadcrumb + H1 + 요약문 | 약 90px |
| 계산기 카드 padding + 기준일 라벨/입력 | 약 90px |
| 이벤트 이름 토글 링크 1줄 | 약 44px |
| 계산 버튼 | 약 48px (+상하 gap) |
| 계산기 카드 상하 padding | 약 32px |
| 결과 카드(라벨+숫자+보조정보, padding 포함) | 약 190~210px |
| **합계** | **약 500~530px** |

→ 기본 상태에서는 여유 있게 한 화면에 들어온다. **단, 사용자가 이벤트 이름 토글을 펼치면
입력 필드 1개(약 68px)가 추가되어 총 570~600px 수준까지 늘어난다.** 이 경우 소형 기기에서는
결과 영역 하단 일부가 스크롤 없이는 살짝 가려질 수 있다.

**결론 및 지침:**
- 기본 상태(토글 접힘)는 반드시 한 화면 안에 들어오도록 유지한다 — 이것이 최우선.
- 이벤트 이름을 펼친 뒤 화면 하단이 살짝 잘리는 것은 "선택적으로 사용자가 추가 정보를 입력하기로
  한 결과"이므로 완전히 금지할 필요는 없으나, 개발팀은 실제 375×667 뷰포트에서 토글 펼침 상태까지
  QA 확인해 결과 카드 상단(숫자 표기 부분)만큼은 반드시 화면에 보이도록 여백을 조정해야 한다.
  (예: 토글 펼침 시 카드 내부 gap을 `gap-4`(16px)에서 `gap-3`(12px)로 축소하는 등 미세 조정은
  개발팀 재량으로 허용)

**터치 타겟 체크:**
- 기준일 입력, 이벤트 이름 입력, 계산 버튼: 모두 `h-12`(48px) → 44px 기준 충족
- 이벤트 이름 토글 링크: 텍스트 자체는 작지만 클릭 영역을 padding으로 44px 이상 확보 (2-2 참고)

## 4. 기존 컴포넌트 재사용 여부

| 컴포넌트 | 재사용 가능 여부 | 비고 |
|---|---|---|
| `ResultInterpretation.tsx` | 그대로 재사용 가능 | 구조 변경 불필요. `lib/calculators.ts`의 `dday-calculator` 항목에 `interpretation` 텍스트만 채우면 됨 (콘텐츠는 기획팀/개발팀 작성 필요 — 5번 참고) |
| `FaqAccordion.tsx` | 그대로 재사용 가능 | 구조 변경 불필요. `faq` 배열만 채우면 됨 |
| `RelatedCalculators.tsx` / `CalculatorCard.tsx` | 그대로 재사용 가능 | `getRelatedCalculators`가 자동으로 같은 카테고리(`date`)를 우선 노출하므로 만 나이 계산기가 관련 카드로 자연 노출됨. 별도 작업 불필요 |

최대한 그대로 재사용/변형해 구현할 것을 권장한다 (스타일 일관성 + 개발 공수 절감).

## 6. 다른 팀 전달 사항

**기획팀 확인 요청**
- `lib/calculators.ts`의 `dday-calculator` 항목은 현재 `status: "coming-soon"`이며
  `interpretation`, `faq` 필드가 비어 있다. 라이브 전환 전에 만 나이 계산기 수준
  (500~1000자 해석 텍스트 + FAQ 3~4개)의 콘텐츠 작성이 필요하다. 디자인팀 영역이 아니므로
  기획팀에서 문구를 작성해 개발팀에 전달해야 한다.
- 라우팅 표기 확인: 요청서에는 slug가 "date/dday-calculator"로 언급되었으나, 기존 구현
  구조상 실제 URL은 카테고리 접두어 없이 `/calculator/dday-calculator`가 된다 (만 나이
  계산기와 동일 패턴). 만약 향후 `/date/dday-calculator` 같은 카테고리 접두어 URL 구조로
  변경할 계획이 있다면 이는 라우팅/정보구조 변경이므로 디자인 스펙이 아니라 기획팀+개발팀이
  먼저 결정하고 마스터 팀장 확인 후 진행해야 한다. 현재 스펙은 기존 라우팅 구조를 그대로
  따른다고 가정하고 작성했다.

**개발팀 전달 사항**
- 신규 컴포넌트 `components/DdayCalculator.tsx` 구현 필요 (2번 항목 스펙 참고)
- `app/calculator/[slug]/page.tsx`의 `CALCULATOR_COMPONENTS`에 매핑 추가 필요
- `lib/calculators.ts`의 `dday-calculator.status`를 콘텐츠 확보 후 `"live"`로 변경
- 이벤트 이름 토글 펼침 상태에서 375×667 뷰포트 스크롤 여부 QA 필요 (3번 항목 참고)
- 날짜 입력에 `min`/`max` 제한을 걸지 않는 것(과거/미래 모두 허용)이 AgeCalculator와의 의도적
  차이점이므로 복붙 구현 시 실수로 `max={today}` 를 남기지 않도록 주의

이상으로 D-Day 계산기 개발 착수가 가능하다. 구현 완료 후 데스크톱/모바일 실제 화면을 본
스펙과 대조해 검수할 예정이다.
