### 0-1. 정확한 것 ✅

- line 24 nav에 `overflow-x-auto whitespace-nowrap` — 맞음
- 헤더는 `sticky`/`fixed`가 아닌 일반 흐름 요소 (`design/cross-site-nav-widget-spec.md:39`에서 "사이트 전체에 뷰포트 고정 요소가 단 하나도 없다"고 확인된 사실과 일치)

**[v1.1 정정] 초과폭은 42px가 아니라 82px다.** v1.0 작성 시점에는 브라우저 재측정 없이 마스터 전달값 42px를 산술 추정(가용폭 ~247px 대비 항목 실폭 ~300px)으로 방증했으나, 개발팀 실측 결과 **375px 베이스라인(1행) 초과폭은 82px**, nav 콘텐츠 실폭은 **326px**(추정 ~311px보다 +15px)이었다.

이 정정은 §0~§4의 진단·비교·결론을 **바꾸지 않는다**(오히려 §1에서 보듯 문제를 더 심각한 쪽으로 확정한다). 다만 §5-3 임계값 계산의 입력값이었으므로 그 결과는 정정된다(→ §5-3).

### 0-2. 수정 ① — "기존 의도된 설계다"는 절반만 맞다 ⚠️

마스터는 이를 "버그가 아니라 의도된 설계"로 규정했다. **가로 스크롤이 의도된 것은 맞지만, "완성된 설계"로 승인된 적은 없다.** 근거는 이 결정을 내린 원본 문서 `design/support-page-ui-spec.md` §1 (line 46~58)이다:

- line 51: **"권장 안전장치:"** `overflow-x-auto whitespace-nowrap` — 즉 이것은 최적 UX가 아니라 **레이아웃 깨짐/헤더 높이 변화를 막기 위한 방어책(safety net)** 으로 도입됐다.
- line 46: **"단, 모바일 폭 이슈를 반드시 확인해야 한다 (개발팀 QA 요청 사항)"**
- line 57: "개발팀은 실제 iPhone SE(375px) 등 소형 뷰포트에서 6개 메뉴 + 로고가 줄바꿈 없이 보이는지, **또는 가로 스크롤로 자연스럽게 처리되는지 반드시 확인할 것.**"
- line 54: "**구조 변경(햄버거 메뉴 등)은 이번 범위에 포함하지 않는다.** 항목이 앞으로 더 늘어날 계획이라면 그때는 모바일 전용 축약 메뉴 도입을 별도로 마스터 팀장에게 제안해야 한다."

즉 07-13 시점 디자인팀은 375px 초과 가능성을 **미리 예상하고 명시적으로 QA 확인을 요청해 뒀으며**, "자연스럽게 처리되는지"를 합격 조건으로 걸어 뒀다. 07-20 실측 결과는 **그 조건을 충족하지 못한다**(스크롤 단서가 전무하므로 "자연스럽게 처리"되지 않음).

**결론: 이것은 "손대면 안 되는 확정 설계"가 아니라, 07-13에 예약해 둔 QA 항목이 이제 실측으로 회수된 미완료 설계 과제다.** 따라서 이번 개선은 기존 결정을 뒤집는 것이 아니라 예정된 후속 작업이다.

### 0-3. 수정 ② — 푸터에 "고객센터"는 **없다** ⚠️

마스터는 안 (B)의 근거로 "푸터에 이미 있는지 확인하라, 이미 있으면 중복 제거 근거가 강해진다"고 했다. `components/SiteFooter.tsx` 확인 결과:

```
FOOTER_LINKS = 사이트 소개(/about) · 문의하기(/support#ask) · 개인정보처리방침(/privacy) · 이용약관(/terms)
```

- **"고객센터"라는 라벨은 푸터에 없다.** 있는 것은 **"문의하기"**이고, 링크도 `/support`가 아니라 **`/support#ask`** (앵커)다.
- `/support` 페이지 구조 확인 결과 `#faq`(line 97)와 `#ask`(line 101) 두 섹션이 있고, 푸터 링크는 그중 **질문하기 폼으로 직행**한다. 즉 푸터 링크로 들어가면 FAQ 섹션을 건너뛴다.

**따라서 "nav 중복 제거"라는 안 (B)의 근거는 성립하지 않는다.** 헤더 "고객센터"(FAQ 포함 페이지 전체)와 푸터 "문의하기"(폼 직행)는 목적지도 라벨도 다르다. (B)를 택하면 중복 제거가 아니라 **FAQ로 가는 유일한 발견 경로가 사라지는** 기능 축소가 된다.

### 0-4. 추가 발견 — 안 (B)는 마스터 결정을 뒤집는 안이다 🚩

`design/support-page-ui-spec.md:3`: **"오늘 마스터 팀장 결정: 헤더 끝에 '고객센터' 신설"**

헤더에 고객센터를 넣은 것은 디자인팀 판단이 아니라 **마스터 팀장의 직접 결정**이다. 따라서 안 (B)(헤더에서 제거)는 디자인팀이 단독으로 추천·실행할 수 없고, 채택하려면 **마스터의 명시적 재승인이 선행되어야 한다.** 이 문서에서는 (B)를 검토 대상으로는 다루되 추천하지 않는다.

### 0-5. 추가 발견 — 다크모드는 이 프로젝트에 존재하지 않는다 ⚠️

마스터 지시에 "다크모드/기존 brand-* 토큰 체계를 따를 것"이 있었으나, 전체 `.tsx`/`.css`에 `dark:` 사용처가 **0건**이고 `app/globals.css`에도 다크 스킴 정의가 없다. **이번 작업에서 `dark:` 변형을 새로 도입하지 않는다** (전역 컴포넌트에 다크모드를 여기서 처음 도입하면 나머지 60개 페이지와 불일치가 생긴다). brand-* 토큰 체계만 따른다.

## 1. 문제 정의

**레이아웃 문제가 아니라 발견가능성(discoverability) 문제다.** — 이 규정은 마스터 판단이 정확하다.

- `body.scrollWidth == 375` → 페이지 가로 스크롤 없음. 레이아웃은 깨지지 않았다.
- 그러나 잘린 "고객센터"는 **잘렸다는 사실 자체가 보이지 않는다.** 스크롤바는 모바일에서 기본 숨김(overlay scrollbar)이고, 페이드/화살표/그림자 등 어떤 단서도 없다.
- 사용자 입장: 헤더 메뉴는 "급여/대출/날짜/생활/블로그" 5개로 **완결되어 보인다.** 6번째가 있다는 사실을 알 방법이 없으므로, 스크롤을 시도할 동기 자체가 생기지 않는다.
- **[v1.1 정정]** v1.0은 "잘린 폭 42px가 작아서 여백으로 오인된다"고 썼으나, 실측 초과폭은 **82px**이고 "고객센터" 항목 폭은 **약 72px**이다. 즉 **초과폭 > 항목 폭** → 375px 베이스라인에서 "고객센터"는 일부가 잘린 게 아니라 **완전히 비가시**였고, 직전 "블로그"까지 일부 잘렸을 가능성이 높다. (개발팀이 320px 베이스라인에서 "고객센터 100% 비가시"를 측정한 것과 정합한다.)
- 이 정정은 **위 세 번째 항목을 문자 그대로 확정한다** — 사용자에게 헤더 메뉴는 실제로 5개로 보였다. 6번째 항목의 존재를 암시하는 픽셀이 **화면에 단 하나도 없었으므로**, 이는 "단서가 약한" 문제가 아니라 **단서가 0인** 문제였다. 안 (A)(페이드)만으로는 부족하고 안 (C)(원인 제거)가 필요하다는 §4의 판단이 실측으로 더 강해진다.

**부가 영향:** `/support`는 AdSense 심사에서 중요한 신뢰 신호(문의 경로)이며, FAQ는 체류시간 자산이다. 모바일 사용자의 상당수에게 이 경로가 사실상 비가시 상태다.

## 3. 검토한 안

### 안 (A) — 현행 가로 스크롤 유지 + 우측 페이드 마스크

우측 20~24px에 `mask-image` 그라데이션을 걸어 "더 있음"을 암시.

- 👍 변경 최소, C1~C6 전부 자동 충족, diff 3줄
- 👎 **문제를 완화할 뿐 해소하지 못한다.** 사용자는 여전히 (a) 페이드를 스크롤 단서로 해석하고 (b) 헤더를 가로로 밀어야 목적지를 안다. 2단계 인지 부담이 남는다
- 👎 **보상이 초라하다.** 82px 밀면 항목 하나 나오는 게 전부라, 스크롤 rail 패턴(뉴스·커머스의 10개+ 카테고리)이 주는 "긴 목록" 기대와 어긋난다. *(v1.1: 수치를 42→82px로 정정. 초과폭이 커진 만큼 페이드가 가려야 할 콘텐츠도 늘어나므로 이 단점은 오히려 강화된다)*
- 👎 페이드는 **항상 켜져 있다.** CSS만으로는 "실제로 넘칠 때만" 판정이 불가능(`animation-timeline: scroll()`은 Chrome 전용)하므로, 375px에서 마지막 가시 항목이 상시 흐려 보인다

### 안 (B) — "고객센터"를 nav에서 제거하고 푸터로 이관

- 👍 6개 → 5개로 줄어 375px에서 초과가 사라짐 (추정 가용폭 247px vs 5개 ~243px)
- 👎 **§0-3: 푸터 중복이 아니다.** 푸터는 `/support#ask` 폼 직행 링크이므로, 제거 시 FAQ 발견 경로가 소실된다
- 👎 **§0-4: 마스터 팀장 직접 결정을 뒤집는 안**이라 디자인팀 단독 추천 불가
- 👎 여유가 **4px**뿐이라 폰트 렌더링 편차·320px 기기에서 곧바로 재발한다. 문제를 해결하는 게 아니라 임계점 바로 아래로 옮기는 것
- 👎 AdSense 신뢰 신호(문의 경로)를 헤더에서 내리는 것은 심사 대기 중인 현 시점에 역방향

### 안 (C) — 모바일에서만 헤더를 2행으로 (로고 행 / 메뉴 행) ★ 추천

`sm:` 미만에서만 컨테이너를 `flex-col`로 바꿔 nav가 **컨테이너 전체 폭(343px)** 을 쓰게 한다.

- 👍 **문제를 없앤다.** 가용폭 247px → 343px. 6개 항목 실제 폭 **326px**(v1.1 실측, 추정 ~311px에서 정정)이 들어가 **375px에서 잘림·스크롤 자체가 발생하지 않는다.** 단서를 학습시킬 필요 없이 6개가 그냥 다 보인다. *(실측 여유는 17px로 v1.0 추정 32px보다 빠듯하다 — 항목 추가 시 즉시 재실측 필요)*
- 👍 **선행 결정을 하나도 뒤집지 않는다.** 항목 6개·순서·라벨·스타일·구분선 없음(`support-page-ui-spec.md:35`) 전부 유지. 마스터 재승인 불필요
- 👍 순수 CSS, 반응형 클래스만 (C3)
- 👍 데스크톱은 `sm:` 클래스로 전량 원복 → C1 충족
- 👎 **모바일 헤더 높이 64px → 약 92px (+28px).** 유일한 실질 비용
- 👍 단 헤더가 **sticky가 아니므로** 이 비용은 최초 1화면에만 발생하고 스크롤하면 사라진다. 뷰포트를 상시 잠식하지 않는다
- 👎 **≤357px에서는 여전히 초과**(v1.1 실측 확정 경계. 예: 320px는 가용 288px < 326px) → **안 (A)의 페이드를 `max-[358px]:`로 한정 병용**하여 보완. 이 잔여 구간에서 키보드 포커스 잘림 이슈가 남는다(§6-1)

**기각한 안:** 햄버거 메뉴 — 6개짜리 nav를 클릭 뒤로 숨기는 것은 카테고리 발견성 손해가 크고, `support-page-ui-spec.md:54`가 "구조 변경은 마스터 확인 대상"으로 못박아 뒀다. **nav에 `flex-wrap`만 추가** — 2번째 줄이 로고 폭만큼 들여쓰기된 채 항목 1개만 놓여 시각적으로 불안정.

## 4. 추천: 안 (C) + (A)의 320px 한정 병용

**근거 요약:** (A)는 증상 완화, (B)는 임계점 이동 + 마스터 결정 번복, **(C)는 원인 제거**. 비용은 모바일 헤더 +28px 한 가지이고, 헤더가 sticky가 아니라 그 비용의 체감이 작다. 무엇보다 (C)는 07-13 마스터 결정과 디자인팀 결론(6개 유지·구분선 없음)을 **전부 보존한 채** 문제만 없앤다.

## 5. 구현 스펙 (개발팀 — 이대로 옮기면 됨)

대상 파일 **`components/SiteHeader.tsx` 단 1개.** `NAV_ITEMS` 배열, import, 컴포넌트 구조는 **손대지 않는다.** 바뀌는 것은 `className` 문자열 2개뿐이다.

### 5-1. 컨테이너 (현 line 16)

```diff
- <div className="mx-auto flex h-16 max-w-5xl items-center gap-2 px-4 sm:px-6">
+ <div className="mx-auto flex max-w-5xl flex-col items-start gap-1 px-4 py-3 sm:h-16 sm:flex-row sm:items-center sm:gap-2 sm:px-6 sm:py-0">
```

| 클래스 | 역할 |
|---|---|
| `flex-col items-start` | 모바일: 로고 행 / 메뉴 행 2행. 로고 좌측 정렬 |
| `gap-1` | 두 행 사이 4px |
| `py-3` | 모바일 상하 12px (기존 `h-16` 대체) |
| `sm:h-16 sm:flex-row sm:items-center sm:gap-2 sm:py-0` | **데스크톱 완전 원복** — 높이 64px, 1행, 수직 중앙, gap 8px, 세로 padding 0 |

`mx-auto max-w-5xl px-4 sm:px-6`은 그대로 유지 (사이트 최대폭 관례 — `cross-site-nav-widget-spec.md:32`).

### 5-2. 로고 (현 line 17~22)

**변경 없음.** `shrink-0`은 `flex-col`에서 무해하므로 그대로 둔다. `FOCUS_RING_LINK_ROUNDED` 유지.

### 5-3. nav (현 line 24)

```diff
- {/* 항목이 6개로 늘어나 소형 모바일(375px)에서는 줄바꿈 대신 가로 스크롤로 처리 */}
- <nav className="flex min-w-0 flex-1 items-center gap-1 overflow-x-auto whitespace-nowrap sm:justify-end sm:gap-2">
+ {/*
+   모바일은 2행 헤더(로고 행 / 메뉴 행)로 nav가 컨테이너 전체 폭을 쓴다 → 375px에서 6개 전부 표시.
+   overflow-x-auto는 320px급 초소형 기기용 안전장치로 유지하며,
+   그 구간에서만 우측 페이드로 스크롤 가능 단서를 준다.
+   design/site-header-mobile-nav-spec.md §5-3
+ */}
+ <nav
+   aria-label="주요 메뉴"
+   className="flex w-full min-w-0 items-center gap-1 overflow-x-auto whitespace-nowrap max-[358px]:[mask-image:linear-gradient(to_right,black_calc(100%-20px),transparent)] sm:w-auto sm:flex-1 sm:justify-end sm:gap-2"
+ >
```

| 클래스 | 역할 |
|---|---|
| `w-full` | 모바일: nav가 컨테이너 전체 폭 사용 (**이 안의 핵심**) |
| `min-w-0` | 유지 — flex 자식 축소 허용 |
| `overflow-x-auto whitespace-nowrap` | **유지** — 320px급 안전장치 (C2: 줄바꿈으로 인한 높이 변화 방지) |
| `max-[358px]:[mask-image:...]` | **실제로 넘치는 구간(≤357px)에서만** 우측 20px 페이드. 358px 이상에서는 미적용 → 375px에서 마지막 항목이 흐려지는 안 (A)의 부작용이 없다 |
| `sm:w-auto sm:flex-1 sm:justify-end sm:gap-2` | **데스크톱 완전 원복** — `flex-1`로 남은 폭 차지 후 우측 정렬 |

**[v1.1 정정] 임계값은 339px가 아니라 `358px`다.** 개발팀 실측(Chromium 149 / playwright) 근거:

1. **nav 콘텐츠 실폭 = 326px** (v1.0 산술 추정 ~311px보다 +15px). 좌우 padding 32px를 더하면 초과가 사라지는 경계는 **358px**이며, **≤357px에서 초과 발생**한다.
2. **Tailwind v4의 `max-[Npx]:`는 `width < N`이지 `width <= N`이 아니다.** 생성되는 CSS가 `@media not (min-width: 358px)`이므로, `max-[358px]:`는 **357px 이하에만** 적용된다. 초과 구간(≤357px)을 정확히 덮는 값이 곧 **358**이다.

**v1.0의 339px를 그대로 구현했다면 340~357px 구간(초과 1~18px)에 페이드 단서가 없는 사각지대가 생겼을 것이다.** v1.0이 "실측하여 상향 조정 후 회신하라"고 요구한 항목이 실제로 회수된 사례이므로, 이 문서는 **358px를 확정값으로 고정**한다. 이후 `NAV_ITEMS` 항목 추가·라벨 변경·폰트 변경이 있으면 nav 콘텐츠 실폭을 **재실측**하고 이 값을 갱신해야 한다(추정 금지).

**Tailwind v4 확인:** `app/globals.css`가 `@import "tailwindcss"` + `@theme inline`, `focusRing.ts`가 `outline-hidden`을 쓰므로 v4 확정. `max-[358px]:` 임의 변형과 `[mask-image:...]` 임의 속성 모두 v4 지원. **주의: `max-[Npx]:`의 의미는 `width < N`(생성 CSS `@media not (min-width: N)`)이며 `<= N`이 아니다** — 경계값을 다룰 때 off-by-one의 원인이 되므로 위 §5-3 정정 근거를 반드시 확인할 것.

### 5-4. nav 링크 (현 line 26~33)

**변경 없음.** `px-2 py-2 text-sm ... sm:px-3` + `FOCUS_RING_LINK` 그대로.

> 모바일에서 `px-2 py-2` + `text-sm`는 터치 타깃이 약 36px 높이로 권장치(44px)에 못 미친다. 이는 **이번 변경으로 새로 생긴 문제가 아니라 기존 상태**이며, 헤더 높이에 다시 영향을 주므로 이번 범위에서 건드리지 않는다. 별건으로 마스터에게 제기한다(§8).

### 5-5. (선택) SiteFooter 라벨 정합성

**이번 범위 아님, 마스터 판단 사항.** §0-3에서 드러난 대로 푸터 "문의하기"(`/support#ask`)와 헤더 "고객센터"(`/support`)는 목적지가 다르다. 현행 유지가 기능적으로는 타당하나(폼 직행은 푸터 맥락에 적합), 두 라벨이 같은 페이지를 가리킨다는 점을 사용자가 알기 어렵다. 조정한다면 푸터 nav에도 `aria-label="푸터 메뉴"`를 붙이는 정도만 권장한다(헤더/푸터 두 `<nav>`가 모두 무명이라 스크린리더에서 구분되지 않음).

## 6-1. [v1.1 신설] 키보드 포커스 자동 스크롤 — 실측 반증 및 대응 방침

### 무엇이 틀렸나

v1.0 §6과 §7-2 S4는 "좁은 뷰포트에서 Tab으로 포커스하면 브라우저 네이티브 동작으로 항목이 스크롤되어 나타난다"고 단언했다. **이 주장은 300~357px 구간에서 거짓이다.**

원인은 Chromium의 포커스 스크롤이 `ScrollAlignment::…IfNeeded` 의미라는 데 있다 — **요소가 부분적으로라도 보이면 스크롤하지 않는다.** 개발팀 실측(Chromium 149 / playwright):

| 뷰포트 | "고객센터" 가시폭 | focus 시 `scrollLeft` | 포커스 후 잘린 폭 |
|---|---|---|---|
| 240 / 260 / 280px | **0px** (완전 비가시) | 118 / 98 / 78 | 0 — 정상 스크롤 ✅ |
| **320px** | 34px / 72px (부분 가시) | **0** | **38px** ❌ |
| **340px** | 54px / 72px (부분 가시) | **0** | **18px** ❌ |

`mask` · `w-full` · `flex-1`을 각각 제거한 4가지 변형에서도 동일하게 재현되어, **이번 변경의 부작용이 아니라 Chromium 고유 동작**으로 확인됐다.

### 역설적 트레이드오프 (정확히 기록해 둘 것)

베이스라인(1행 레이아웃)에서는 320px에서 "고객센터"가 **100% 비가시**였기 때문에 자동 스크롤이 **작동했다**(`scrollLeft` 127). 안 (C)가 항목을 **부분 가시(0px → 34px)로 개선한 결과**, 역설적으로 자동 스크롤 트리거 조건에서 벗어났다.

즉 **가시성은 개선됐고(0 → 34px) 자동 스크롤은 사라졌다.** 순 효과는 여전히 개선(마우스·터치 사용자에게 항목의 존재가 드러남 + 페이드 단서)이지만, **키보드 사용자에게는 "포커스했는데 일부가 잘려 보이는" 상태가 새로 생겼다.** 이 문서는 이것을 "개선"으로 뭉뚱그리지 않고 트레이드오프로 명시한다.

### 영향 범위 — 과소평가하지 말 것

"300~357px는 희귀 기기"로 치부하기 쉬우나, **이 폭은 WCAG 1.4.10 Reflow가 요구하는 320px CSS 폭과 정확히 겹친다.** 데스크톱 1280px에서 **400% 확대**하면 CSS 뷰포트가 320px가 된다. 즉 실제 영향 대상은 "구형 소형 폰 사용자"가 아니라 **고배율 확대를 쓰는 저시력 + 키보드 사용자**이며, 이는 접근성 관점에서 우선순위가 낮은 집단이 아니다. 다만 **AA 기준(2.4.11)은 충족**하고 미충족은 **AAA(2.4.12)** 뿐이다.

### 선택지

| | 방안 | 평가 |
|---|---|---|
| **ⓐ** | 현 구현 수용 + 한계 문서화 | AA 충족, 코드 변경 0, C3 유지. AAA 격차가 ≤357px에 한정 |
| **ⓑ** | JS 도입 (`focusin` → `scrollIntoView`) | **C3(순수 CSS) 위반.** 전역 컴포넌트가 클라이언트 컴포넌트화되어 61개 정적 페이지 전부에 JS 번들·hydration 추가 |
| **ⓒ** | ≤357px에서만 nav를 **줄바꿈**시켜 초과 자체를 제거 (`max-[358px]:flex-wrap`) | 순수 CSS로 근본 해결 가능성. **미검증 가설** |

### 권고: **ⓐ를 채택하고, 향후 개선이 필요해지면 ⓑ가 아니라 ⓒ로 간다**

**이번 릴리스는 ⓐ.** 근거:

1. **AA는 충족하고 미충족은 AAA뿐이다.** 이 사이트는 AAA 준수를 선언한 바 없다. 검증 단계에 들어간 전역 컴포넌트를 AAA 격차 하나로 되돌리는 것은 위험 대비 이득이 맞지 않는다.
2. **순 효과는 명백히 플러스다.** 주력 뷰포트(≥358px)에서 문제가 완전히 사라졌고, ≤357px에서도 가시폭이 0 → 34px로 늘었다. 되돌리면 **더 나쁜 상태**로 간다.
3. **C3는 장식적 제약이 아니다.** 빠른 로딩은 AdSense 승인 조건이며(§2), 61개 정적 페이지 전역에 JS를 얹는 비용은 ≤357px 한정 AAA 격차보다 크다.

**ⓑ를 1순위 해법으로 삼지 않는 이유:** "접근성 문제 = JS로 보정"은 이 경우 성급하다. 문제의 근인은 *초과가 남아 있다는 것*이지 *브라우저가 스크롤을 안 해준다는 것*이 아니다. ⓑ는 근인을 남긴 채 증상을 JS로 덮는 방식이라, §4에서 안 (A)를 기각한 논리와 같은 이유로 차선이다.

**ⓒ가 지정 후속안인 이유:** ≤357px에서 `flex-wrap`을 허용하면 6개 항목이 2줄로 모두 표시되어 **초과·스크롤·페이드·포커스 잘림·2.4.12 미충족이 한 번에 소멸**하고 C3도 지켜진다. 비용은 해당 구간 헤더 높이 증가뿐인데, 이는 400% 확대 사용자가 이미 수용한 트레이드오프다.

> ⚠️ **ⓒ는 디자인팀 가설이며 실측되지 않았다.** v1.0의 339px가 추정이어서 틀렸던 전례가 있으므로, 채택 시 개발팀이 **①`max-[358px]:flex-wrap` 적용 후 240~357px 전 구간에서 실제로 초과가 0이 되는지 ②C2(스크롤 중 헤더 높이 불변)가 유지되는지 ③≥358px에 영향이 없는지**를 실측한 뒤 확정할 것. 검증 전에는 이 문서도 이를 확정 스펙으로 취급하지 않는다.
>
> 참고로 `scroll-padding-inline-end`로 스크롤포트 유효 영역을 좁혀 보는 방법도 이론상 후보이나, Chromium이 *부분 가시*만으로 스크롤을 생략하는 것이라면 무효일 가능성이 높다. 저비용이므로 ⓒ 검증 시 함께 확인해 보는 정도로만 남긴다.

## 8. 마스터 팀장 확인 요청 사항

1. ~~**[결정 요청] 안 (C) 채택 여부**~~ → **[해결됨 v1.1] 마스터 팀장이 안 (C) 채택 결정, 개발팀 구현 완료.**
1-A. **[결정 요청 — v1.1 신설] §6-1 대응 방침 승인** — 300~357px에서 키보드 포커스 시 "고객센터"가 부분적으로 잘린 채 남는다(WCAG **AA 충족 / AAA 미충족**). 디자인팀 권고는 **ⓐ 현 구현 수용 + 한계 문서화**이며, 향후 개선이 필요해지면 **ⓑ(JS 도입, C3 위반)가 아니라 ⓒ(`max-[358px]:flex-wrap`, 순수 CSS)** 로 갈 것을 제안한다. 이 방침을 승인하는지, 아니면 ⓒ를 즉시 별건 과제로 착수할지 판단 요청.
1-B. **[보고 — v1.1 신설] 디자인팀 추정 오류 2건이 개발팀 실측으로 반증됐다** — 임계값 339px(→358px), "포커스 자동 스크롤" 접근성 주장(→거짓). 두 건 모두 **브라우저 실측 없이 산술·일반 지식으로 단정한 것이 원인**이다. 이후 구성안에서 경계값과 브라우저 네이티브 동작은 **실측 전까지 "가설"로 표기**하고 확정 스펙으로 쓰지 않겠다.
2. **[보고] 안 (B)는 추천하지 않음** — 헤더 "고객센터"는 07-13 **마스터 팀장 직접 결정** 사항이고(`support-page-ui-spec.md:3`), 푸터에 동일 링크가 있다는 전제도 사실과 다르다(푸터는 `/support#ask` "문의하기"). 그래도 (B)를 원하시면 재지시 바란다.
3. **[보고] 전제 정정 2건** — §0-2(가로 스크롤은 "완성 설계"가 아니라 07-13에 예약해 둔 QA 항목), §0-3(푸터에 "고객센터" 없음).
4. **[별건 제기] 헤더 nav 터치 타깃** — 현행 약 36px로 권장치 44px 미달(기존 문제, 이번 변경과 무관). 헤더 높이에 다시 영향을 주므로 이번 범위에서 제외했다. 별도 과제로 다룰지 판단 요청.
